| https://javarush.com/quests/lectures/questhibernate.level14.lecture05 | **При использовании двухуровневого кэша (например, кэша второго уровня в Hibernate) в Java могут возникнуть проблемы, связанные с параллельным доступом к объектам и кэшированием зависимостей**. Эти проблемы связаны с особенностями работы кэша второго уровня, который не хранит сами объекты, а хранит информацию в виде массивов строк, чисел и т. д.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| https://habr.com/ru/articles/833478/                                  | **Проблемы**<br><br>- **Устаревшие данные** из-за стратегии параллельного доступа к объектам в кэше. Например, если используется стратегия **NONSTRICT_READ_WRITE**, кэш обновляется после совершения транзакции, которая изменила данные в базе данных, но существует небольшое временное окно между обновлением данных в БД и обновлением тех же данных в кэше. Во время этого окна параллельная транзакция может получить из кэша устаревшие данные.<br>- **Не кэширование зависимостей**. Например, если класс сущности не аннотирован аннотацией @Cache, при выборке коллекция зависимостей достаётся из БД, а не из кэша второго уровня.<br>- **Ошибки кэширования** из-за неправильной настройки стратегии параллельного доступа. Например, если не указана стратегия **READ_WRITE**, которая гарантирует строгую согласованность, параллельные транзакции могут получить из кэша устаревшие данные. |
| https://telegra.ph/34-Kak-rabotat-s-kehshem-2-urovnya-01-29           | **Решения**<br><br>- **Использовать стратегию параллельного доступа, подходящую для данных**. Например, для данных, которые редко меняются и часто читаются, подходит стратегия **READ_WRITE**. Для данных, предназначенных главным образом для чтения, — **NONSTRICT_READ_WRITE**.<br>- **Настроить режим кэширования** для каждой отдельной сессии или даже для каждого запроса к базе данных. Например, в Hibernate можно настроить режим **CacheMode**, который управляет взаимодействием сессии с кэшем второго уровня.<br>- **Указывать аннотацию @Cache** для сущностей, для которых нужно задействовать кэш второго уровня. В аннотации можно указать стратегию параллельного доступа и другие параметры.<br>- **Создавать регионы кэша** для хранения сущности, если нужно отправлять часто изменяющиеся объекты в один регион, а все остальные — во второй.                                       |
