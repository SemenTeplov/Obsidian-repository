| https://www.hostragons.com/ru/блог/стратегии-бэкэнд-кэширования/                                                | **Стратегию кэширования на уровне приложения можно реализовать, используя разные подходы, учитывая потребности приложения и данные**. Кэширование — это временное хранение результатов операций или запросов для быстрого доступа к ним в будущем без выполнения дорогостоящих вычислений или запросов к источнику данных.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| https://www.hostragons.com/ru/блог/стратегии-кэширования-кеша-и-произво/                                        | **Типы**<br><br>Некоторые стратегии кэширования, которые можно реализовать на уровне приложения:<br><br>- **Cache-Aside (ленивая загрузка)**. Приложение сначала проверяет кэш: если данные есть (cache hit), они возвращаются, если нет (cache miss) — данные извлекаются из основного хранилища, сохраняются в кэш и затем возвращаются. Подходит для ситуаций, когда чтение данных происходит чаще, чем их обновление.<br>- **Write-Through (сквозная запись)**. Данные записываются одновременно в кэш и в основное хранилище. Подходит для приложений с частыми записями, чтобы кэш всегда был актуальным.<br>- **Write-Around (обход при записи)**. Данные записываются только в основное хранилище, минуя кэш. Кеш заполняется только при чтении. Подходит для ситуаций, когда данные записываются часто, но читаются редко, например, логи или аналитические данные. |
| https://dtf.ru/id678675/2174546-kak-pravilno-ispolzovat-keshirovanie-dannyh-dlya-uluchsheniya-proizvoditelnosti | **Настройка**<br><br>Некоторые аспекты, которые нужно настроить:<br><br>- **Время жизни кэша**. Определить, как долго данные могут быть считаны из кэша, прежде чем они будут признаны устаревшими. Слишком короткое время жизни может привести к частым списаниям данных, а слишком длинное — к отображению устаревших данных.<br>- **Аннулирование по событиям**. При изменении данных в источнике (например, в базе данных) необходимо иметь механизм, уведомляющий кэш об этих изменениях, чтобы он мог обновить свои данные.<br>- **Ручное аннулирование**. В некоторых случаях может потребоваться ручное аннулирование кэша, например, когда в данные вносятся изменения, которые невозможно обнаружить автоматически.                                                                                                                                                |
| https://www.cyberforum.ru/blogs/2408867/10449.html                                                              | **Ошибки**<br><br>Некоторые ошибки, которых следует избегать:<br><br>- **Кеширование всего подряд**. Нужно определить, какие данные действительно выиграют от кэширования — обычно это часто читаемые и редко изменяемые данные.<br>- **Кэширование изменяемых объектов без создания их копий**.<br>- **Кэширование null-значений без явной необходимости** — это может маскировать ошибки.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| https://www.geeksforgeeks.org/system-design/caching-strategies-for-api/                                         | **Мониторинг**<br><br>Для мониторинга эффективности кэширования на уровне приложения можно использовать метрики, например:<br><br>- **Процент попадания (hitrate или hit ratio)** — показывает, сколько запросов в процентном выражении получают ответы из кэша и какой — из базы данных.<br>- **Задержка получения данных из кэша (latency)** — время, нужное на получение данных из кэша. Задержка должна быть меньше задержки получения данных из оригинального источника.                                                                                                                                                                                                                                                                                                                                                                                                |
