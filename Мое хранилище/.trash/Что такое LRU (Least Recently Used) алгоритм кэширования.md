| https://blog.skillfactory.ru/glossary/lru/                            | **LRU (Least Recently Used)** — алгоритм кэширования, который **вытесняет элемент, который дольше всего не использовался**. Предполагается, что если к элементу давно не обращались, то вероятность его использования в ближайшее время мала.<br><br>Алгоритм применяют при организации кэша — данных приложений, которые те «держат под рукой», чтобы не загружать каждый раз при открытии. LRU подходит, если данные, к которым недавно обращались, скорее всего, вскоре понадобятся повторно.                                                                                                                                                                                                                                                                                                          |
| https://dzen.ru/a/Z9vlQ2VrjVU7BOQp                                    | **Принцип работы**<br><br>Алгоритм запоминает, когда каждый элемент кэша был использован последний раз. Для этого часто используют: <br><br>- **Двусвязный список** — для управления порядком элементов.<br>- **Хеш-таблицу** — для быстрого поиска элементов.<br><br><br>**Алгоритм работает так**: <br><br>- Когда программа обращается к данным, она проверяет, есть ли они в кэше. Если данные есть, их временная метка обновляется. Если данных нет, они добавляются в кэш.<br>- Пока в кэше есть место, новые данные добавляются туда без вытеснения имеющихся.<br>- Когда место заканчивается, перед добавлением новых данных алгоритм проверяет временные метки. Он находит элемент с самой старой временной меткой. Этот элемент удаляется из кэша, а на его место добавляется новая информация. |
| https://www.geeksforgeeks.org/system-design/lru-cache-implementation/ | **Реализация**<br><br>Обычно реализация алгоритма включает две структуры: хеш-таблицу с данными и список, где хранятся временные метки. Каждому элементу в таблице соответствует метка в списке — она определяет приоритет. Когда программа обращается к кэшу, чтобы добавить туда данные, алгоритм сверяется со списком приоритетов. А если к данным, которые есть в кэше, нужно обратиться, алгоритм не просто сверяется, а перезаписывает временную метку.                                                                                                                                                                                                                                                                                                                                             |
|                                                                       | **Эффективность**<br><br>**Преимущества LRU**:<br><br>- прост в реализации;<br>- эффективен для частых запросов — часто используемые данные гарантированно остаются в кэше.<br><br>**Недостатки**:<br><br>- нужно хранить метаданные о порядке использования;<br>- при большом кэше управление порядком может быть ресурсозатратным;<br>- алгоритм не учитывает ситуации, когда к данным обращаются раз в определённый промежуток времени (например, при циклическом сканировании).                                                                                                                                                                                                                                                                                                                       |
