| https://garden.struchkov.dev/ru/dev/architecture/highload/Инвалидация-кэша                                                              | **В системах с высокой доступностью для инвалидации кэша (удаления данных из кэша при изменении состояния источника) используются разные подходы**, которые зависят от характера данных и задач системы. Некоторые из них: стратегия по времени жизни (TTL), подход по событиям, использование алгоритмов вытеснения и, в некоторых случаях, применение репликации.                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| https://sky.pro/wiki/javascript/keshirovanie-v-php/                                                                                     | **TTL (Time-to-Live)**<br><br>**Содержимое кэша автоматически очищается через определённый период**. Это подходит, когда релевантность закэшированных данных предсказуемо снижается со временем. Например:<br><br>- **Приложение-агрегатор новостей** — закэшированные записи истекают через день, чтобы ответы на похожие запросы о свежих событиях генерировались быстро, а старые новости не занимали память. <br><br>**Важно подобрать оптимальное время жизни кэша**: слишком маленькое время жизни будет давать плохую производительность, слишком большое ухудшит опыт пользователей.                                                                                                                                                                                                                         |
| https://alesha.pro/posts/kv-cache-explain/                                                                                              | **События**<br><br>**Данные кэша удаляются по какому-то событию**. Например, при вызове метода обновления данных можно удалить кэш, который связан с этими данными. В некоторых случаях можно не удалять, а сразу класть новые данные в кэш, если они скоро потребуются. <br><br>**Инвалидацию по событию можно комбинировать с TTL**: при запросе данных из кэша можно проверять оставшееся время жизни, и если оно меньше, чем некоторое заданное значение, то установить TTL заново. Таким образом данные остаются в кэше, если ими активно пользуются.                                                                                                                                                                                                                                                           |
| https://kirya522.tech/posts/caching-base/                                                                                               | **Алгоритмы вытеснения**<br><br>**Для инвалидации в высоконагруженных системах используются, например:**<br><br>- **LRU (Least Recently Used)** — удаляются записи, у которых время последнего использования старше остальных.<br>- **LFU (Least Frequently Used)** — удаляются записи, которые использовались реже всего.<br><br>**Также применяется гранулярность кэша**: чем меньше объём связанных данных в одном кэше, тем реже нужно его полностью сбрасывать.                                                                                                                                                                                                                                                                                                                                                 |
| https://www.alachisoft.com/ru/blogs/ncache-essentials-consistency-and-data-replication/                                                 | **Репликация**<br><br>**В распределённых системах** **кэшированные записи дублируются на нескольких узлах** для обеспечения высокой доступности и отказоустойчивости в случае сбоя какого-либо сервера. **Механизмы синхронизируют копии всех узлов**, так что любое обновление отражается в масштабах всего кластера, избегая конфликтующих данных.<br><br>**Репликация может быть синхронной или асинхронной**:<br><br>- **Синхронная репликация** — данные записываются во все реплики одновременно, обеспечивая строгую согласованность, но потенциально увеличивая задержку.<br>- **Асинхронная репликация** — реплики обновляются после записи данных на основной сервер кэша, что повышает производительность, но может привести к временным несоответствиям данных между репликами и основным сервером кэша. |
