### Опишите, что такое хранимая процедура в контексте SQL.

| Под темы    | Заметка                                                                                                                                                      |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение | **Хранимая процедура в SQL** — это заранее скомпилированный набор инструкций, который хранится в базе данных и может выполняться по запросу.                 |
| Особенности | **Могут включать в себя** SQL-запросы (например, SELECT, INSERT, UPDATE, DELETE), условные операторы, циклы, обработку ошибок и даже вызовы других процедур. |
|             | **Могут принимать параметры** и возвращать результат (например, набор данных или код завершения).                                                            |
|             | **Поддерживаются** большинством реляционных СУБД, таких как MySQL, PostgreSQL, Oracle, SQL Server и другие.                                                  |

### Как можно создать простейшую функцию в SQL?

| Под темы   | Заметка                                                                                                                 |
| ---------- | ----------------------------------------------------------------------------------------------------------------------- |
| SQL Server | `CREATE FUNCTION dbo.Square(@Number INT) RETURNS INT AS BEGIN RETURN @Number * @Number; END`                            |
| Oracle     | `CREATE OR REPLACE FUNCTION sum_numbers(a IN NUMBER, b IN NUMBER) RETURN NUMBER IS BEGIN RETURN a + b; END sum_numbers` |
| PostgreSQL | `CREATE FUNCTION get_total_products() RETURNS INTEGER AS $$ BEGIN RETURN 100; END; LANGUAGE plpgsql`                    |

### Что такое триггер в базах данных и для чего он используется?

| Под темы     | Заметка                                                                                                                                                                                                                                                                                                   |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение  | **Триггер в базах данных** — это специальный тип хранимой процедуры, которая автоматически выполняется при наступлении определённого события. Эти события обычно связаны с операциями вставки, обновления или удаления данных в определённой таблице.                                                     |
| Используется | **Поддержание целостности данных** — проверка данных перед их вставкой, обновлением или удалением, чтобы предотвратить попадание некорректных данных в базу данных.                                                                                                                                       |
|              | **Автоматическое обновление связанных таблиц** — при изменении данных в одной таблице триггеры выполняют обновления в связанных таблицах, обеспечивая согласованность данных. Например, изменение цены товара в одной таблице может автоматически пересчитывать общую стоимость заказов в другой таблице. |
|              | **Логирование изменений** — триггеры автоматически создают записи в журнале при вставке, обновлении или удалении данных, фиксируя информацию о том, кто и когда внёс изменения.                                                                                                                           |
|              | **Реализация сложной бизнес-логики** — некоторые бизнес-правила требуют выполнения сложных вычислений или проверок, которые проще и эффективнее реализовать на уровне базы данных.                                                                                                                        |

### Какие преимущества использования хранимых процедур перед кодом на стороне клиента?

| Под темы           | Заметка                                                                                                                                                                                                                                                                                                                                                                  |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Производительность | **Предварительная компиляция**. Хранимые процедуры компилируются один раз и затем выполняются многократно с предсохранённым планом выполнения, что сокращает затраты на компиляцию запросов при каждом вызове. Это особенно важно для часто используемых запросов или сложных операций с базой данных.                                                                   |
|                    | **Уменьшение сетевого трафика**. Вместо отправки нескольких SQL-запросов по сети клиентское приложение может вызвать одну хранимую процедуру, которая выполнит все необходимые операции на стороне сервера.                                                                                                                                                              |
|                    | **Оптимизация планов выполнения**. Поскольку хранимые процедуры компилируются и хранятся, сервер базы данных может генерировать оптимизированные планы выполнения на основе статистики хранимых процедур и распределения данных.                                                                                                                                         |
| Безопасность       | **Ограничение доступа к таблицам**. Пользователи получают доступ только к процедуре, а не к самой SQL-таблице, что снижает риск несанкционированных изменений. Например, можно предоставить права на выполнение хранимой процедуры, но не на чтение или изменение базовых таблиц.                                                                                        |
|                    | **Защита от SQL-инъекций**. Параметризованные хранимые процедуры используют параметры, что снижает риск вредоносного кода, в отличие от динамических SQL-запросов, где пользовательские данные могут быть вставлены в текст запроса.                                                                                                                                     |
| Модульность        | **Инкапсуляция сложной логики** в отдельные модули, которые можно повторно использовать в различных частях приложения. Например, если есть стандартная процедура, которая возвращает список товаров, её можно вызывать из разных приложений, не дублируя код в каждом из них.                                                                                            |
|                    | **Устранение дублирования кода**. SQL-код, включающий многошаговые операции с базой данных, который многократно используется в приложении в разных местах, можно вынести в хранимую процедуру, чтобы не копировать один и тот же код.                                                                                                                                    |
| Сопровождение      | **Централизованное хранение** — в отличие от SQL-запросов, разбросанных по различным частям клиентского приложения, хранимые процедуры централизованы на сервере базы данных. Это упрощает управление и модификацию кода: если необходимо изменить логику работы с данными, это можно сделать только в хранимой процедуре, без необходимости изменения клиентского кода. |
|                    | **Согласованное поведение** — изменения, внесённые в хранимую процедуру, отражаются во всех приложениях, которые её используют, что упрощает процессы обновления и отладки.                                                                                                                                                                                              |

### Как передать параметры в SQL функцию и использовать их внутри неё?

| Под темы    | Заметка                                                                                                                                                                                                                  |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение | **Параметры в SQL-функции передаются по значению**, то есть любые изменения входных параметров внутри функции не влияют на значения этих параметров в вызывающей программе.                                              |
|             | Параметры указываются в круглых скобках после имени функции, разделены запятыми. У каждого параметра указывается тип данных. Для параметра можно указать ограничение NOT NULL, запретив передавать в него значение NULL. |
|             | Параметры могут иметь значение по умолчанию. Параметры, для которых заданы значения, должны располагаться в конце списка параметров.                                                                                     |

### Какие типы триггеров существуют и в чем их основные отличия?

| Под темы                               | Заметка                                                                                                                                                                                                              |
| -------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Row-level (на уровне строки)           | **Выполняются отдельно для каждой строки**, затронутой операцией (INSERT, UPDATE, DELETE).                                                                                                                           |
|                                        | **BEFORE ROW** — срабатывают перед вставкой, обновлением или удалением строки, но до фиксации изменений в базе данных. Могут использоваться для проверки и модификации данных перед фактическим внесением изменений. |
|                                        | **AFTER ROW** — срабатывают после того, как изменения в строке были зафиксированы в базе данных. Их можно использовать, например, для обновления связанных данных или ведения логов.                                 |
| Statement-level (на уровне инструкции) | **Срабатывают один раз на всю операцию**, независимо от количества затронутых строк.                                                                                                                                 |
|                                        | **BEFORE STATEMENT** — срабатывают перед выполнением операции (INSERT, UPDATE, DELETE). Применяются к операции в целом, а не к каждой строке отдельно.                                                               |
|                                        | **AFTER STATEMENT** — срабатывают после выполнения операции. Их можно использовать для выполнения дополнительных действий после вставки, обновления или удаления набора строк.                                       |

### Как можно использовать транзакции внутри хранимых процедур для управления изменениями в базе данных?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                              |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Транзакции можно использовать внутри хранимых процедур для управления изменениями в базе данных**, чтобы обеспечить целостность данных при выполнении операций, которые логически связаны.                                                                                                                                                                                                                         |
|             | **Изолировать ряд операций** в виде единой транзакции.                                                                                                                                                                                                                                                                                                                                                               |
|             | **Фиксировать изменения** в базе данных, если все операции успешно выполнены, или **откатывать изменения** при возникновении ошибки.                                                                                                                                                                                                                                                                                 |
| SQL Server  | **Явные транзакции** — пользователь явно указывает начало и конец транзакции с помощью специальных команд. Например, в хранимой процедуре, которая переводит средства с одного счёта на другой, все операции (снятие средств и зачисление) пишут внутри транзакции. Если в любой из операций возникает ошибка, все изменения отменяются.                                                                             |
|             | **Сочетание с обработкой ошибок** — транзакции можно сочетать с конструкцией TRY…CATCH, которая отслеживает ошибки в блоке инструкций и в блоке CATCH откатывает транзакцию.                                                                                                                                                                                                                                         |
| MySQL       | **Явные транзакции** — по умолчанию ядро базы данных считает каждый оператор SQL атомарной единицей, которая выполняется как своя собственная транзакция. В хранимой процедуре можно определить явные транзакции, которые выполняют несколько операторов SQL. Например, в процедуре, которая выполняет несколько связанных операций обновления, вставки или удаления, все эти операции объединяют в одну транзакцию. |
|             | **Фиксация изменений** — в случае успешного выполнения всех инструкций изменения фиксируются с помощью команды COMMIT.                                                                                                                                                                                                                                                                                               |
|             | **Откат изменений** — при сбое выполнения какой-либо операции изменения инициируются с помощью команды ROLLBACK.                                                                                                                                                                                                                                                                                                     |
| PostgreSQL  | **Транзакция в хранимой процедуре** начинается не с ключевого слова begin, а с первой команды DML. Все DML-команды в процедуре, а также в функциях и процедурах, которые вызваны командами select и call, выполняются в контексте этой транзакции.                                                                                                                                                                   |
|             | **Команда commit** заканчивает транзакцию, фиксируя её. Если commit не используется, транзакция будет зафиксирована после конца работы команды call, которая вызвала процедуру.                                                                                                                                                                                                                                      |
|             | **Откат (rollback)** — если параметр, который вызывает транзакцию, равен null, то происходит откат всей транзакции на её начало (перед первой командой DML). В противном случае выполняется вторая команда DML, и транзакция фиксируется.                                                                                                                                                                            |

### Как создать функцию, которая возвращает таблицу в SQL?

| Под темы   | Заметка                                                                                                                                                                                                                                                                |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ      | **В SQL можно создать функцию, которая возвращает таблицу**, с помощью конструкции **RETURNS TABLE**. Это позволяет инкапсулировать логику при выводе набора записей, что полезно при работе со сложными запросами или передаче параметров для фильтрации результатов. |
| SQL Server | В T-SQL существуют табличные функции с одним выражением (Inline Table-Valued Functions, ITVF) и табличные функции с множественными операторами (Multi-Statement Table-Valued Functions, MSTVF).                                                                        |
| PostgreSQL | **В PostgreSQL** для объявления функции, возвращающей таблицу, используется **конструкция RETURNS TABLE** с указанием списка столбцов.                                                                                                                                 |

### Каким образом можно обработать исключения внутри триггера и каковы лучшие практики для их использования?

| Под темы  | Заметка                                                                                                                                                  |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ     | Исключения (ошибки) внутри триггера (на языке PL/SQL) можно обработать с помощью встроенного механизма обработки исключений — ключевого слова EXCEPTION. |
| Позволяет | перехватывать ошибки                                                                                                                                     |
|           | выводить сообщения                                                                                                                                       |
|           | выполнять альтернативные действия                                                                                                                        |
| Код       | `BEGIN -- основной код EXCEPTION WHEN тип_исключения THEN -- действия при ошибке END;`                                                                   |
