### Что такое JPA и для чего она используется?

| Под темы    | Заметка                                                                                                                                                                                                                                  |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Java Persistence API (JPA)** — это **стандарт языка Java для управления реляционными данными в Java-приложениях**. Он предоставляет удобный механизм взаимодействия с базой данных через объектно-реляционное отображение (ORM).       |
| Цели        | **Сокращение количества шаблонного кода**. Это становится возможным благодаря автоматической генерации SQL-запросов и встроенному механизму маппинга.                                                                                    |
|             | **Работа со сложными схемами данных**. Система наследования, различные типы отношений между сущностями (One-to-One, One-to-Many, Many-to-Many) легко реализуются через аннотации и XML-конфигурацию.                                     |
|             | **Встроенная поддержка транзакций и механизмов контроля версий данных**. Это важно для приложений, работающих с финансовыми операциями или другими чувствительными данными.                                                              |
|             | **Работа с большими данными**. Технология предоставляет пагинацию, динамические запросы через Criteria API и нативные запросы, когда это необходимо.                                                                                     |
|             | **Защита от уязвимостей**. JPA предлагает встроенную защиту от SQL-инъекций и других распространённых уязвимостей, связанных с работой с базами данных.                                                                                  |
|             | **Лёгкая миграция между различными СУБД**. Достаточно изменить конфигурацию persistence.xml. Это особенно ценно для проектов, которые могут начинаться с одной базы данных, но впоследствии требовать масштабирования на другие решения. |

### Какие основные преимущества использования JPA по сравнению с JDBC?

| Под темы                          | Заметка                                                                                                                                                                     |
| --------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| JPA (Java Persistence API)        | **Чистый и лаконичный код** — не нужно вручную управлять соединениями и ResultSet.                                                                                          |
|                                   | **Работа на уровне объектов** — легче оперировать бизнес-логикой.                                                                                                           |
|                                   | **Поддержка кэширования, ленивой загрузки** и других оптимизаций «из коробки».                                                                                              |
|                                   | **Возможность лёгкой миграции между различными СУБД** без необходимости переписывания кода — достаточно изменить конфигурацию persistence.xml.                              |
|                                   | **Встроенная поддержка транзакций** и механизмов контроля версий данных, что важно для приложений, работающих с финансовыми операциями или другими чувствительными данными. |
|                                   | **Инструменты для работы с большими данными**: пагинация, динамические запросы через Criteria API и нативные запросы, когда это необходимо.                                 |
| Недостатки                        | Потенциально ниже производительность на сложных запросах, ограниченный контроль над SQL (иногда ORM генерирует неоптимальные запросы).                                      |
| JDBC (Java Database Connectivity) | **Полный контроль над SQL** и структурой запросов.                                                                                                                          |
|                                   | **Максимальная производительность** при оптимизации вручную.                                                                                                                |
|                                   | **Независимость от ORM и фреймворков**.                                                                                                                                     |
|                                   | **Простота использования** — интерфейс предоставляет готовый набор методов для подключения к базе, отправки запросов и обработки результатов.                               |
|                                   | **Надёжность работы с транзакциями** — JDBC позволяет контролировать выполнение SQL-команд, управлять фиксацией и откатом изменений, а также отслеживать ошибки соединений. |
| Недостатки                        | Код громоздкий: ручное управление соединениями и ResultSet, высокая вероятность ошибок при сложной логике, трудоёмкость поддержки больших моделей данных.                   |

### Как определить сущность в JPA?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                   |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Сущность (entity) в JPA (Java Persistence API) определяется с помощью аннотаций в классе**. Сущность — это POJO-класс, который указывает, что может быть сохранён в базе данных. Как правило, сущность представляет таблицу в реляционной базе данных, и каждый экземпляр сущности соответствует строке в этой таблице. |

### Какие стратегии наследования поддерживает JPA?

| Под темы        | Заметка                                                                                                                                                                                      |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Single Table    | **Все классы в иерархии наследования сопоставляются с одной таблицей**. В таблице есть столбец-дискриминатор, который определяет тип каждой строки.                                          |
| Особенности     | Подходит, когда сущности в иерархии имеют большинство общих атрибутов и небольшое количество разных полей.                                                                                   |
|                 | Может привести к разреженной таблице, если иерархия большая.                                                                                                                                 |
|                 | По умолчанию — стратегия, если не указана явно.                                                                                                                                              |
| Joined Table    | **Каждый класс в иерархии наследования сопоставляется с отдельной таблицей**. Таблица суперкласса содержит общие атрибуты, а таблицы подклассов — атрибуты, специфичные для этого подкласса. |
| Особенности     | В таблице подкласса создаётся внешний ключ, который ссылается на первичный ключ таблицы суперкласса.                                                                                         |
|                 | При получении данных выполняется операция JOIN между таблицами суперкласса и подклассов.                                                                                                     |
| Table Per Class | **Для каждого конкретного класса в иерархии наследования генерируется отдельная таблица**. В таблице хранятся все свойства класса и свойства, унаследованные от суперклассов.                |
| Особенности     | Обеспечивает изоляцию между таблицами, но может привести к дублированию данных и плохо поддерживает полиморфные запросы.                                                                     |
|                 | Согласно стандарту JPA, стратегия TABLE_PER_CLASS **не является обязательной**, поэтому другими реализациями может не поддерживаться.                                                        |

### Как реализовать отношения один-ко-многим в JPA?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                             |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Отношения «один-ко-многим» (OneToMany) в JPA реализуются с помощью аннотаций и классов сущностей**. Это взаимосвязь, при которой один экземпляр объекта связан с несколькими экземплярами другого объекта. Например, в одном отделе может быть несколько сотрудников: отдел — одна сторона, а сотрудник — многие. |

### Как работает кэширование в JPA и какие типы кэшей поддерживаются?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Принцип работы | Результат запроса сохраняется в кэше, и последующие запросы обслуживаются из кэша, а не из базы данных, пока данные в кэше остаются актуальными. Например, если приложение часто запрашивает список всех активных пользователей из базы данных, без кэширования каждый запрос будет вызывать выполнение SQL-запроса, что может быть дорогостоящим, особенно если данные редко меняются. С кэшированием результат запроса сохраняется в кэше, и последующие запросы обслуживаются из кэша. |
| Типы           | **Кэш первого уровня (First-Level Cache)** — встроенный кэш, предоставляемый JPA-провайдером. Автоматически включён и работает в рамках одной сессии EntityManager (или одной транзакции). Хранит сущности, загруженные в рамках текущей сессии, чтобы избежать повторных запросов к базе данных для тех же данных.                                                                                                                                                                       |
|                | **Кэш второго уровня (Second-Level Cache)** — опциональный кэш, также предоставляемый JPA-провайдером, но работающий на уровне всей EntityManagerFactory. Общий для всех сессий и транзакций в приложении, что позволяет кэшировать данные между различными запросами и пользователями.                                                                                                                                                                                                   |

### Как можно интегрировать пользовательские SQL запросы в JPA?

| Под темы | Заметка                                                                                                                                                                                                                                                                                         |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **В Spring Data JPA пользовательские SQL-запросы (JPQL или native SQL) можно интегрировать с помощью аннотации @Query**. Она позволяет определять запросы непосредственно в методах репозитория, что даёт гибкость для выполнения любых запросов без соблюдения conventions именования методов. |

### Какие подходы к оптимистичной и пессимистичной блокировке поддерживаются в JPA?

| Под темы                  | Заметка                                                                                                                                                                                                                                                                                                       |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Оптимистичная блокировка  | **Предполагает, что конфликты редки**, и позволяет параллельным транзакциям выполняться без блокировки данных. Для обнаружения конфликтов используется **поле версии** (обычно помечается аннотацией @Version).                                                                                               |
| Как работает              | Когда транзакция обновляет запись, она включает текущую версию записи. База данных проверяет, совпадает ли версия в транзакции с версией в базе данных. Если они могут совпадать, обновление продолжается.                                                                                                    |
|                           | Если другая транзакция обновила запись и изменила версию, текущая транзакция завершается с ошибкой **OptimisticLockException**. Это указывает на конфликт, который необходимо разрешить.                                                                                                                      |
|                           | Приложение должно обработать исключение OptimisticLockException путём повторной попытки транзакции или информирования пользователя о конфликте.                                                                                                                                                               |
| Пессимистичная блокировка | **Предполагает возникновение конфликтов** и блокирует данные, чтобы другие транзакции не могли получить к ним одновременный доступ.                                                                                                                                                                           |
| Как работает              | Когда транзакция считывает запись с пессимистической блокировкой, база данных получает блокировку для этой записи. Блокировка может быть разных типов, например: **PESSIMISTIC_READ** (предотвращает запись других транзакций) или **PESSIMISTIC_WRITE** (предотвращает чтение или запись других транзакций). |
|                           | Другие транзакции, пытающиеся получить доступ к той же записи, будут заблокированы до тех пор, пока блокировка не будет снята. Это предотвращает несогласованность данных, вызванную одновременными изменениями.                                                                                              |
|                           | Блокировка удерживается на время транзакции и может быть автоматически снята, когда транзакция фиксируется или откатывается.                                                                                                                                                                                  |

### Как реализовать мульти-тенантность в приложении с использованием JPA?

| Под темы     | Заметка                                                                                                                                                                                                                                                                                      |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение  | **Мульти-тенантность в приложении с использованием JPA реализуется через настройку архитектуры, конфигурации и кода**. Цель — обеспечить каждому арендатору отдельную долю экземпляра приложения, но изолировать информацию, принадлежащую каждому арендатору.                               |
| Архитектура  | **Отдельная база данных** — данные каждого арендатора хранятся в отдельном экземпляре базы данных, изолированном от других арендаторов.                                                                                                                                                      |
|              | **Общая база данных и отдельная схема** — данные каждого арендатора хранятся в отдельной схеме в общей базе данных.                                                                                                                                                                          |
|              | **Общая база данных и общая схема** — все арендаторы используют общую базу данных, и в каждой таблице есть столбец с идентификатором арендатора.                                                                                                                                             |
| Конфигурация | **Стратегия мульти-тенантности** — например, для подхода на основе схемы используется значение MultiTenancyStrategy.SCHEMA.                                                                                                                                                                  |
|              | **Реализация интерфейса MultiTenantConnectionProvider** — обеспечивает динамическое определение источника данных на основе идентификатора текущего арендатора.                                                                                                                               |
|              | **Реализация интерфейса CurrentTenantIdentifierResolver** — определяет, какую схему использовать на основе текущей сессии.                                                                                                                                                                   |
|              | **Создание контекста TenantContext** — помогает управлять информацией текущего арендатора на протяжении сессии.                                                                                                                                                                              |
| Код          | **Реализация MultiTenantConnectionProvider** — например, класс, который предоставляет разные соединения на основе идентификатора арендатора.                                                                                                                                                 |
|              | **Реализация CurrentTenantIdentifierResolver** — в реализации можно получить данные аутентификации из контекста безопасности и использовать имя пользователя как идентификатор арендатора. Если аутентификация анонимная или отсутствует, возвращается стандартный идентификатор арендатора. |
|              | **Создание TenantContext** — в классе можно хранить информацию текущего арендатора в каждом запросе.                                                                                                                                                                                         |
