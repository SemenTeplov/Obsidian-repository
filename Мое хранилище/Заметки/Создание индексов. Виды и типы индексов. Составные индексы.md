### Для чего используются индексы в реляционных базах данных?

.  

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                    |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | Индексы в реляционных базах данных используются для ускорения операций поиска, сортировки и фильтрации данных.                                                                                                                                                                                                                                             |
| Цели        | **Ускорение выполнения запросов на выборку данных (SELECT)**, особенно на больших таблицах с миллионами или миллиардами записей. Если запрос включает условия фильтрации (в предложении WHERE) или требует соединения нескольких таблиц (через JOIN) по проиндексированным столбцам, СУБД может использовать индексы для быстрого нахождения нужных строк. |
|             | **Ускорение сортировки данных** — если данные запрашиваются в отсортированном виде (ORDER BY) по столбцу, для которого существует индекс, СУБД может избежать дополнительной операции сортировки, так как данные в индексе уже упорядочены.                                                                                                                |
|             | **Обеспечение уникальности значений** — уникальные индексы гарантируют, что в индексируемом столбце или наборе столбцов не будет дублирующихся значений.                                                                                                                                                                                                   |

### Что такое кластеризованный и некластеризованный индекс?

| Под темы                  | Заметка                                                                                                                                                                                                                                                 |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Кластеризованный индекс   | **Определяет физический порядок хранения записей в таблице**. Данные упорядочиваются и хранятся на основе значений ключевых столбцов этого индекса.                                                                                                     |
| Особенности               | Каждая таблица может иметь только один кластеризованный индекс, поскольку данные физически могут быть упорядочены только одним способом.                                                                                                                |
|                           | По умолчанию создаётся на основе первичного ключа таблицы.                                                                                                                                                                                              |
|                           | Листовые узлы индекса содержат фактические данные таблицы.                                                                                                                                                                                              |
|                           | Обычно реализуется как B-дерево (B-дерево).                                                                                                                                                                                                             |
| Некластеризованный индекс | **Создаёт отдельную структуру данных, независимую от фактической таблицы**. Содержит подмножество столбцов и связанных с ними указателей на соответствующие записи данных.                                                                              |
| Особенности               | Листья некластеризованного индекса содержат только те столбцы таблицы, по которым определён индекс.                                                                                                                                                     |
|                           | Указатель на данные зависит от способа хранения данных: кластеризованная таблица или куча. Если указатель ссылается на кластеризованную таблицу, то он ведёт к кластеризованному индексу, если на кучу — к конкретному идентификатору строки с данными. |
|                           | Некластеризованные индексы нельзя отсортировать, в отличие от кластеризованных, но можно создать несколько некластеризованных индексов для одной таблицы.                                                                                               |

### Какие плюсы и минусы вы можете выделить при использовании индексов?

| Под темы | Заметка                                                                                                                                                                                                                                        |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Плюсы    | **Ускорить поиск данных** — быстро находить строки, соответствующие условиям запроса, без полного сканирования таблицы.                                                                                                                        |
|          | **Улучшить производительность соединений** — при выполнении соединений между таблицами индексы помогают быстро находить соответствующие строки.                                                                                                |
|          | **Оптимизировать сортировку и группировку** — индексы уже хранят значения в отсортированном порядке.                                                                                                                                           |
|          | **Обеспечить уникальность значений** — уникальные индексы гарантируют, что в индексируемом столбце или наборе столбцов не будет дублирующихся значений.                                                                                        |
| Минусы   | **Увеличение занимаемого места на диске** — индексы занимают дополнительное место, что может привести к увеличению объёма хранимых данных.                                                                                                     |
|          | **Увеличение времени на обновление данных** — при вставке, обновлении или удалении данных база данных должна также обновлять соответствующие индексы, что может увеличить время выполнения этих операций.                                      |
|          | **Потребление ресурсов при создании и поддержке индексов** — создание и поддержка индексов требуют дополнительных ресурсов процессора и памяти, особенно на больших объёмах данных.                                                            |
|          | **Риск ухудшения производительности при неправильном использовании** — неправильное применение индексов может привести к ухудшению производительности запросов из-за ненужных индексов или неэффективного использования существующих индексов. |

### Какие принципы нужно соблюдать при проектировании составного индекса?

| Под темы         | Заметка                                                                                                                                                                                                                                                                                                        |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Порядок столбцов | **Первыми должны идти столбцы с наибольшей селективностью** (наибольшим количеством уникальных значений). Это связано с тем, как устроено B-дерево: сначала идёт сортировка по первому столбцу, затем по второму.                                                                                              |
|                  | **Первой должна быть колонка, которая чаще всего используется в WHERE-условиях**. Даже если её селективность ниже.                                                                                                                                                                                             |
|                  | **Порядок столбцов в индексе не обязательно должен совпадать с порядком использования столбцов в запросах** в операторах WHERE и ORDER BY.                                                                                                                                                                     |
| Количество       | **Составной индекс из 2–3 полей** обычно считается нормальным и эффективно поддерживаемым. Если количество колонок в индексе превышает три, следует тщательно анализировать его использование, чтобы избежать излишней нагрузки на производительность.                                                         |
|                  | **По возможности создавать частичные индексы** — в индексе должны быть только те данные, которые будут реально выбираться. Это поможет уменьшить объём индекса.                                                                                                                                                |
| Метод            | **Учитывать тип индекса** — составные индексы могут быть созданы на разных структурах, например, B-Tree (MySQL) или GIN, BRIN и GiST (PostgreSQL).                                                                                                                                                             |
|                  | **Минимизировать использование неравенств в начале составных индексов** — это может ограничить их применение для последующих колонок.                                                                                                                                                                          |
| Обновление       | **Каждый раз при вставке, обновлении или удалении данных, которые попадают в составной индекс, СУБД должна обновлять весь индекс**. Это может увеличивать время выполнения таких операций.                                                                                                                     |
|                  | **Стараться минимизировать использование неравенств в начале составных индексов** — это может ограничить их применение для последующих колонок.                                                                                                                                                                |
|                  | **Обновлять статистику и обслуживать индексы** — со временем из-за многочисленных вставок, обновлений и удалений индексы могут фрагментироваться и терять свою эффективность. СУБД обычно имеют механизмы для сбора статистики по данным и для перестроения (rebuild) или реорганизации (reorganize) индексов. |

### Какие есть структуры для хранения индекса?

| Под темы    | Заметка                                                                                                                                                                                                         |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| B-дерево    | **Основная структура** для хранения индексов в большинстве современных СУБД. Это сбалансированное дерево, где каждый узел содержит несколько ключей и указателей на дочерние узлы.                              |
| Особенности | Все листовые узлы находятся на одном уровне, что гарантирует предсказуемое время доступа.                                                                                                                       |
|             | Ключи в узлах упорядочены, что позволяет использовать бинарный поиск внутри каждого узла.                                                                                                                       |
|             | При поиске значения система начинает с корня, сравнивает значение с ключами и переходит к нужному дочернему узлу, пока не достигнет листа, где хранится указатель на строку в таблице.                          |
| Хэш-таблица | **Обеспечивает быстрый доступ к данным** благодаря хеш-функциям, но **подходит только для точного поиска**.                                                                                                     |
| Особенности | Элементы в хеш-таблице не хранятся в отсортированном виде, поэтому доступ к ним возможен только по ключам.                                                                                                      |
|             | Если два разных ключа приводят к одному и тому же индексу, применяются методы разрешения коллизий, например, цепочки (связные списки внутри бакетов) или открытая адресация (поиск свободного места в таблице). |

### Опишите разницу между индексами B-дерева и хеш-индексами. В каких случаях предпочтительно использовать каждый из них?

| Под темы       | Заметка                                                                                                                              |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------ |
| Определение    | **B-дерево** представляет собой сбалансированное дерево поиска, где каждый узел содержит множество ключей и ссылки на дочерние узлы. |
| Характеристики | Поддерживает сортированный порядок данных                                                                                            |
|                | Эффективно для диапазонных запросов                                                                                                  |
|                | Хорошо работает с данными, которые хранятся на диске                                                                                 |
|                | Обеспечивает быстрый доступ при последовательном чтении                                                                              |
|                | Подходит для операций сравнения (<, >, BETWEEN)                                                                                      |
| Используется   | При необходимости сортировки данных                                                                                                  |
|                | Для запросов с условиями диапазона                                                                                                   |
|                | Когда важна устойчивость к фрагментации                                                                                              |
|                | При работе с большими наборами данных                                                                                                |
|                | В системах с частыми обновлениями                                                                                                    |
| Определение    | **Хеш-индекс** использует хеш-функцию для преобразования ключа в адрес памяти.                                                       |
| Характеристики | Обеспечивает быстрый доступ по точному совпадению                                                                                    |
|                | Не поддерживает сортировку и диапазонные запросы                                                                                     |
|                | Эффективен для операций равенства (=, IN)                                                                                            |
|                | Требует меньше памяти для хранения индекса                                                                                           |
|                | Производительность близка к константной                                                                                              |
| Используется   | Для быстрого поиска по точным значениям                                                                                              |
|                | Когда преобладают операции равенства                                                                                                 |
|                | При работе с небольшими наборами данных                                                                                              |
|                | Когда важна максимальная скорость поиска                                                                                             |
|                | В случаях, где не требуется сортировка                                                                                               |

### Как работает индекс с использованием внешних функций (например, lower()) и как это влияет на производительность запросов?

| Под темы                      | Заметка                                                                                                                                                                                                                                                                                                                                                           |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                   | **Индекс с использованием внешней функции (например, lower()) в базах данных работает как функциональный индекс** — структура данных, которая индексирует результаты вычислений, а не просто значения столбцов. Это позволяет быстро находить данные в таблице по результатам вычислений, что ускоряет запросы.                                                   |
| Принцип работы                | Индекс создаётся не для самого столбца, а для выражения, вычисленного на основе одного или нескольких столбцов таблицы. Например, для сравнения без учёта регистра символов часто используется функция lower. Запрос, который ищет по значениям поля, приведённым к нижнему регистру, может использовать индекс, определённый для результата функции lower(col1). |
| Влияние на производительность | **Использование индекса с внешней функцией (например, lower()) ускоряет запросы** по сравнению с полным сканированием таблицы. Это особенно важно, если запрос требует поиска по значениям поля, приведённым к нижнему регистру.                                                                                                                                  |

### В чем особенности использования индексов в распределенных базах данных?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                              |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Особенности использования индексов в распределённых базах данных** связаны с необходимостью **поддержания актуальности и согласованности индексов** в системах, где данные могут быть реплицированы на нескольких узлах или разбиты на шардированные части. Это требует реализации сложных алгоритмов синхронизации индексов между узлами, чтобы обеспечить корректность и своевременность данных. |
|             | Также индексы в распределённых системах могут быть **покрывающими** — содержать не только ключевые столбцы для поиска, но и все остальные столбцы, запрашиваемые в запросе. Это позволяет избежать обращения к исходной таблице для получения данных — вся необходимая информация уже содержится в индексе.                                                                                          |
