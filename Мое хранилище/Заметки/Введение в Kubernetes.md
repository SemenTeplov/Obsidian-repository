### Что такое Kubernetes и для чего он используется?

[[Kubernetes]]

### Какие основные компоненты архитектуры Kubernetes вы знаете?

| Под темы      | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение   | **Архитектура Kubernetes** следует классической клиент-серверной схеме. Кластер разделён на **управляющий слой ([[Заметки/Control plane.md\|Control Plane]])**, который принимает решения и управляет состоянием системы, и **рабочие узлы (Worker Nodes)**, на которых фактически развёртываются контейнеры. Control Plane выступает в роли «мозга» кластера, а Nodes — это «мышцы», обеспечивающие выполнение задач.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| Control Plane | Некоторые компоненты Control Plane:<br>- **API-сервер** — центральная точка взаимодействия с кластером, принимает запросы от пользователей и компонентов.<br>- **etcd** — распределённая база данных типа «ключ-значение», которая хранит состояние кластера.<br>- **Scheduler** — определяет, на какой рабочей ноде запускать поды. Учитывает доступные ресурсы (CPU, RAM), требования приложения и другие параметры, чтобы выбрать оптимальный вариант.<br>- **Controller Manager** — следит за состоянием кластера, синхронизируя желаемое состояние с текущим. Например, если указано, что должно быть три экземпляра контейнера, Controller Manager будет следить, чтобы их всегда было ровно три. Если один контейнер упадёт, Controller Manager автоматически запустит новый.<br>В крупных коммерческих контейнерных средах важно развёртывать сразу несколько мастер-узлов на кластер для обеспечения высокой доступности и отказоустойчивости.                                                                       |
| Worker Nodes  | Некоторые компоненты Worker Nodes:<br>- **Kubelet** — агент, который работает на каждом рабочем узле. Его задача — убедиться, что все контейнеры на этом узле запущены и работают корректно. Kubelet общается с API-сервером и получает от него инструкции.<br>- **Kube-proxy** — отвечает за сетевую связь между контейнерами. Обеспечивает балансировку нагрузки и маршрутизацию трафика, чтобы запросы доходили до нужных контейнеров.<br>- **Container Runtime** — программное обеспечение, которое фактически запускает контейнеры. Kubernetes поддерживает различные runtime, такие как Docker, containerd и CRI-O.<br>Помимо компонентов Control Plane и Worker Nodes, Kubernetes имеет и другие системы и инструменты, которые отвечают за безопасность, сетевые операции, хранение данных и другие процессы работы в контейнерных средах. Например, RBAC (Role-Based Access Control) обеспечивает управление правами доступа к ресурсам API, а Secrets Management — безопасное хранение конфиденциальной информации. |

### Перечислите основные преимущества использования Kubernetes.

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                                              |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Основные  преимущества | **Автоматизация рутинных операций**<br>    - развёртывание и обновление  приложений без ручного вмешательства;<br>    - автоматическое размещение  подов на узлах с учётом ресурсов и политик;<br>    - самовосстановление: перезапуск  упавших контейнеров, перенос подов при сбоях узлов.                                          |
|                        | **Масштабируемость**<br>    - **горизонтальное масштабирование**:  автоматическое добавление/удаление экземпляров контейнеров при изменении нагрузки;<br>    - **вертикальное масштабирование**:  динамическое увеличение CPU/RAM для контейнеров;<br>    - экономия ресурсов: отключение  лишних контейнеров при снижении нагрузки. |
|                        | **Высокая доступность и отказоустойчивость**<br>    - непрерывная работа приложений даже при  выходе из строя отдельных узлов;<br>    - автоматическое перераспределение  нагрузки на рабочие узлы;<br>    - минимизация времени простоя (минуты вместо часов).                                                                      |
|                        | **Обновления без простоя**<br>    - постепенная замена старых подов  новыми (_Rolling Update_);<br>    - стратегия _Blue‑Green Deployment_: параллельный  запуск новой версии с последующим переключением трафика;<br>    - автоматический откат при ошибках в новой версии.                                                         |
|                        | **Балансировка нагрузки**<br>    - равномерное распределение трафика  между экземплярами приложения; <br>    - использование объекта `Service`  как встроенного балансировщика;<br>    - поддержка HTTP/HTTPS через `Ingress`.                                                                                                       |
|                        | **Оптимизация ресурсов**<br>    - эффективное использование CPU,  памяти, дискового пространства;<br>    - сокращение затрат на инфраструктуру  на **20–40 %** за счёт плотного размещения контейнеров;<br>    - гибкое выделение ресурсов по  требованию (pay‑as‑you‑go).                                                           |
|                        | **Унификация сред разработки и эксплуатации**<br>    - идентичные окружения для разработки,  тестирования и продакшена;<br>    - снижение ошибок при деплое из‑за  различий в настройках;<br>    - инфраструктура как код: конфигурация в  YAML‑файлах, версионирование.                                                             |
|                        | **Портативность и мультиоблачность**<br>    - работа на локальных серверах,  публичных облаках (AWS, GCP, Azure) и гибридных средах;<br>    - лёгкий перенос приложений между провайдерами;<br>    - независимость от конкретного облачного вендора.                                                                                 |
|                        | **Управление конфигурацией и секретами**<br>    - отделение настроек от кода через `ConfigMap`;<br>    - безопасное хранение паролей, токенов,  ключей в `Secrets`;<br>    - централизованное распространение конфигураций.                                                                                                          |
|                        | **Экосистема и расширяемость**<br>    - интеграция с инструментами мониторинга  (Prometheus, Grafana);<br>    - поддержка плагинов для сети  (Calico, Flannel), хранилища (Ceph, NFS);<br>    - активное сообщество и регулярные обновления.                                                                                         |
|                        | **Упрощение микросервисной архитектуры**<br>    - изоляция сервисов в отдельных подах;<br>    - автоматическое обнаружение сервисов;<br>    - управление сложными взаимодействиями  между десятками/сотнями микросервисов.                                                                                                           |
|                        | **Снижение операционных затрат**<br>    - уменьшение числа администраторов за  счёт автоматизации;<br>    - сокращение времени на рутинные задачи  (деплой, мониторинг, восстановление); <br>    - предсказуемые расходы на инфраструктуру.                                                                                          |
|                        | **Безопасность**<br>    - сетевые политики для изоляции подов;<br>    - контроль доступа через RBAC (Role‑Based Access Control);<br>    - шифрование трафика между компонентами.                                                                                                                                                     |
|                        | **Гибкость в управлении жизненным циклом приложений**<br>    - поддержка разных стратегий  развёртывания (Canary, Rolling Update);<br>    - управление состояниями через `StatefulSet` для  БД и других stateful‑приложений;<br>    - оркестрация фоновых задач через `Jobs` и  `CronJobs`.                                          |
|                        | **Ускорение вывода продукта на рынок**<br>    - сокращение времени релиза с дней/недель  до часов/минут;<br>    - быстрая итерация за счёт автоматизированных  CI/CD‑пайплайнов;<br>    - возможность экспериментировать с новыми  функциями без риска для продакшена.                                                               |

### Какие объекты kubernetes вы знаете и для чего они нужны?(прим. pod, service, deployment, etc)

| Под темы                              | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Базовые  объекты                      | **Pod**<br>    - **Что это**: минимальная единица развёртывания,  один или несколько контейнеров на одном узле.<br>    - **Для чего**: запуск приложения; контейнеры  в поде делят сеть, хранилище и жизненный цикл.  <br>    - **Особенности**: эфемерны (при уничтожении теряются  данные), имеют общий IP‑адрес.                                                                                                                               |
|                                       | **Service**<br>    - **Что это**: абстракция для стабильного доступа к группе подов.<br>    - **Для чего**:<br>        - обеспечивает постоянный IP и DNS‑имя для набора подов;<br>        - балансирует нагрузку между подами;  <br>        - скрывает изменения IP подов при перезапусках. <br>    - **Типы**: ClusterIP (внутри кластера), NodePort  (доступ через порт узла), LoadBalancer (внешний балансировщик), ExternalName (DNS‑алиас). |
|                                       | **Namespace**<br>    - **Что это**: логическое разделение кластера на виртуальные подкластеры.<br>    - **Для чего**: изоляция ресурсов для разных команд, проектов или сред (dev/test/prod).                                                                                                                                                                                                                                                     |
| Объекты  управления  приложениями     | **Deployment**<br>    - **Что это**: декларативный способ управления подами через ReplicaSet.<br>    - **Для чего**:<br>        - развёртывание и обновление приложений;<br>        - стратегии обновления (RollingUpdate, Recreate);<br>        - автоматический откат при ошибках.                                                                                                                                                              |
|                                       | **ReplicaSet**<br>    - **Что это**: гарантирует заданное число идентичных подов.<br>    - **Для чего**: поддержание масштабируемости  и отказоустойчивости (перезапуск подов при сбоях).                                                                                                                                                                                                                                                         |
|                                       | **StatefulSet**<br>    - **Что это**: аналог Deployment для stateful‑приложений (БД, кеши).<br>    - **Для чего**:<br>        - упорядоченное развёртывание/удаление подов;<br>        - стабильные сетевые идентификаторы (DNS‑имена);<br>        - постоянное хранилище для каждого пода.                                                                                                                                                       |
|                                       | **DaemonSet**<br>    - **Что это**: запускает один под на каждом (или выбранных) узле кластера.<br>    - **Для чего**: развёртывание системных служб  (логирование, мониторинг, сетевые плагины).<br>                                                                                                                                                                                                                                             |
|                                       | **Job**<br>    - **Что это**: запускает поды до успешного завершения задачи.<br>    - **Для чего**: одноразовые операции (миграция данных, бэкапы).                                                                                                                                                                                                                                                                                               |
|                                       | **CronJob**<br>    - **Что это**: Job по расписанию (аналог cron).<br>    - **Для чего**: автоматизация периодических задач (ежедневные отчёты, очистка).                                                                                                                                                                                                                                                                                         |
| Объекты  конфигурации и  безопасности | **ConfigMap**<br>    - **Что это**: хранилище неконфиденциальных данных конфигурации (ключ‑значение).<br>    - **Для чего**: отделение настроек от кода (например, параметры приложения).                                                                                                                                                                                                                                                         |
|                                       | **Secret**<br>    - **Что это**: безопасное хранение чувствительных данных (пароли, токены).<br>    - **Для чего**: передача секретов в поды без жёсткой привязки к манифестам.                                                                                                                                                                                                                                                                   |
|                                       | **PersistentVolume (PV)** и **PersistentVolumeClaim (PVC)**<br>    - **Что это**:<br>        - PV — ресурс хранилища, управляемый администратором;<br>        - PVC — запрос пользователя на объём хранилища.<br>    - **Для чего**: предоставление постоянного хранилища для stateful‑приложений.                                                                                                                                                |
| Сетевые объекты                       | **Ingress**<br>    - **Что это**: ресурс для управления внешним HTTP/HTTPS‑доступом к сервисам.<br>    - **Для чего**:<br>        - маршрутизация трафика по URL/хостам;<br>        - SSL‑терминация;<br>        - балансировка нагрузки.                                                                                                                                                                                                         |
|                                       | **NetworkPolicy**<br>    - **Что это**: правила сетевой изоляции между подами.<br>    - **Для чего**: ограничение трафика на уровне сети (например, запрет доступа из dev в prod).                                                                                                                                                                                                                                                                |
| Объекты  контроля  доступа            | **Role** и **ClusterRole**<br>    - **Что это**: определение набора прав (разрешений) для ресурсов.<br>    - **Для чего**: описание, что можно делать с объектами (читать, создавать, удалять).                                                                                                                                                                                                                                                   |
|                                       | **RoleBinding** и **ClusterRoleBinding**<br>    - **Что это**: привязка ролей к пользователям/группам/сервисным аккаунтам.<br>    - **Для чего**: назначение прав в конкретном  Namespace (RoleBinding) или во всём кластере (ClusterRoleBinding).                                                                                                                                                                                                |
|                                       | **ServiceAccount**<br>    - **Что это**: учётная запись для подов для взаимодействия с API Kubernetes.<br>    - **Для чего**: аутентификация подов при обращении к API (например, для мониторинга).                                                                                                                                                                                                                                               |
| Вспомогательные  объекты              | **HorizontalPodAutoscaler (HPA)**<br>    - **Что это**: автоматически масштабирует число подов на основе нагрузки.<br>    - **Для чего**: адаптация к трафику (CPU, память, кастомные метрики).                                                                                                                                                                                                                                                   |
|                                       | **LimitRange** и **ResourceQuota**<br>    - **Что это**: ограничения на использование ресурсов в Namespace.<br>    - **Для чего**: предотвращение перегрузки кластера (лимиты CPU/памяти, числа объектов).                                                                                                                                                                                                                                        |
|                                       | **Endpoint**<br>    - **Что это**: список IP‑адресов подов, обслуживающих Service.<br>    - **Для чего**: связь между Service и реальными подами (обычно управляется автоматически).                                                                                                                                                                                                                                                              |

### Как работает механизм автоматического масштабирования в Kubernetes?

| Под темы                                                                  | Заметка                                                                                                                                                                                                                                                                                                                                                    |
| ------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Cluster  Autoscaler  (автомасштабирование  кластера)                      | **Что делает**: меняет количество узлов (worker‑нод) в кластере.  <br>**Как работает**:<br>- отслеживает нагрузку на существующие узлы;<br>- если поды не могут разместиться из‑за  нехватки ресурсов — добавляет узлы (до заданного максимума);<br>- если узлы долго простаивают — удаляет  их (до заданного минимума).                                   |
| Horizontal Pod  Autoscaler (HPA,  горизонтальное  масштабирование  подов) | **Что делает**: изменяет количество реплик  подов в Deployment/StatefulSet/ReplicaSet.  <br>**Как работает**:<br>- собирает метрики (CPU, память,  кастомные показатели) через `metrics-server`;<br>- сравнивает текущую загрузку с  целевым уровнем (например, 50 % CPU);<br>- увеличивает или уменьшает число  подов, чтобы удержать заданный уровень.   |
| Vertical Pod  Autoscaler (VPA,  вертикальное  масштабирование  подов)     | **Что делает**: меняет выделенные ресурсы (CPU/память) для каждого пода.  <br>**Как работает**:<br>- анализирует фактическое потребление ресурсов;<br>- автоматически корректирует `requests` и `limits` для подов;<br>- может работать в режиме:<br>    - `Auto` — самостоятельное изменение ресурсов;<br>    - `Off` — только рекомендации без действий. |
| Взаимодействие                                                            | **При росте нагрузки**:<br>    - сначала срабатывает **HPA** — увеличивает число подов;<br>    - если узлов не хватает для новых подов — срабатывает  **Cluster Autoscaler** и добавляет ноды.                                                                                                                                                             |
|                                                                           | **При снижении нагрузки**:<br>    - **HPA** уменьшает число подов;<br>    - **Cluster Autoscaler** удаляет простаивающие узлы.                                                                                                                                                                                                                             |
|                                                                           | **VPA** работает параллельно, оптимизируя  ресурсы внутри подов.                                                                                                                                                                                                                                                                                           |

### Какие типы сервисов поддерживает Kubernetes и в чем их различия?

| Под темы                  | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ClusterIP  (по умолчанию) | **Суть**: внутренний IP‑адрес в кластере.  <br>**Доступность**: только внутри кластера.  <br>**Назначение**:<br>- связь между микросервисами внутри Kubernetes;<br>- бэкенд‑сервисы, не требующие внешнего доступа.<br>**Особенности**:<br>- IP назначается из внутреннего диапазона кластера;<br>- балансировка нагрузки между подами через `kube-proxy`;<br>- самый простой и безопасный вариант для  внутренних коммуникаций.                                                                                                             |
| NodePort                  | **Суть**: открывает порт на **всех рабочих узлах** кластера.  <br>**Доступность**: извне кластера через IP любого узла + фиксированный порт.  <br>**Назначение**:<br>- быстрый тестовый доступ к сервису;<br>- сценарии, где не нужен балансировщик нагрузки.<br>**Особенности**:<br>- порт выбирается из диапазона `30000–32767` (по умолчанию);<br>- трафик перенаправляется на поды через `kube-proxy`;<br>- нет встроенной балансировки между  узлами — клиент сам выбирает узел.                                                        |
| LoadBalancer              | **Суть**: создаёт **внешний балансировщик нагрузки**  через облачного провайдера (AWS, GCP, Azure).  <br>**Доступность**: публичный IP/DNS для  доступа из интернета.  <br>**Назначение**:<br>- продакшен‑приложения с высокой нагрузкой;<br>- распределение трафика между узлами кластера.<br>**Особенности**:<br>- автоматически интегрируется с облачными  сервисами (ELB, GCE Load Balancer);<br>- предоставляет единый публичный IP;<br>- поддерживает здоровье‑чеки и сессию‑стиккинг;<br>- платный (зависит от облачного провайдера). |
| ExternalName              | **Суть**: CNAME‑запись, перенаправляющая трафик **вне кластера**.  <br>**Доступность**: внутри кластера через DNS.  <br>**Назначение**:<br>- интеграция с внешними сервисами  (SaaS, legacy‑системы);<br>- миграция сервисов без изменения кода приложений.<br>**Особенности**:<br>- не имеет селектора (`selector`);<br>- работает только через DNS (нет  проксирования трафика);<br>- используется для «псевдо‑сервисов».                                                                                                                  |

### Как Kubernetes обеспечивает высокую доступность и устойчивость приложений?

| Под темы                              | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Самовосстановление  (Self‑healing)    | **Как работает**:<br>- **kubelet** на каждом узле мониторит состояние подов и перезапускает их при сбоях;<br>- **контроллеры** (ReplicaSet, Deployment)  следят за количеством реплик и воссоздают поды при их падении;<br>- **liveness‑пробы** (`livenessProbe`)  автоматически перезапускают контейнеры, если они «зависли» (например, при утечке памяти).                                                                              |
| Репликация и  распределение подов     | **Механизмы**:<br>- **ReplicaSet/Deployment** гарантируют заданное число реплик приложения;<br>- **PodDisruptionBudget** ограничивает число одновременно недоступных  подов при обновлениях или сбоях узлов;<br>- **topologySpreadConstraints** распределяют  поды по зонам доступности, стойкам или узлам для защиты от единичных отказов.<br>**Эффект**: приложение остаётся доступным, даже если часть подов или узлов вышла из строя. |
| Балансировка  нагрузки                | **Компоненты**:<br>- **Service** (тип `ClusterIP`, `LoadBalancer`)  равномерно распределяет трафик между работающими подами;<br>- **kube‑proxy** на узлах управляет правилами маршрутизации (iptables/IPVS);<br>- **Ingress** добавляет HTTP(S)‑балансировку и маршрутизацию.<br>**Результат**: отказ одного пода не приводит к  потере доступности — трафик перенаправляется на здоровые реплики.                                        |
| Управление  обновлениями  без простоя | **Стратегии**:<br>- **Rolling Update** (по умолчанию для Deployment):  постепенная замена старых подов новыми (по одному или группами);<br>- **Blue‑Green Deployment**: параллельный запуск новой версии и переключение трафика;<br>- **Canary Releases**: частичное перенаправление трафика на новую версию для тестирования.<br>**Плюс**: обновления не прерывают работу сервиса.                                                       |
| Защита от  сбоев узлов                | **Механизмы**:<br>- **Node Affinity/Anti‑Affinity**: размещение подов на  надёжных узлах или разнесение по разным зонам;<br>- **Taints и Tolerations**: предотвращение запуска подов на проблемных узлах;<br>- **DaemonSet**: автоматический перезапуск  системных подов (логирование, мониторинг) на новых узлах.<br>**Эффект**: приложения продолжают работать при выходе из строя отдельных узлов.                                     |
| Постоянное  хранилище                 | **Компоненты**:<br>- **PersistentVolume (PV)** и  **PersistentVolumeClaim (PVC)** обеспечивают сохранность данных при перезапуске подов;<br>- **StorageClasses** позволяют выбирать типы хранилищ (локальные, сетевые, облачные);<br>- **StatefulSet** гарантирует стабильные имена подов и привязку к PV при перезапусках.<br>**Итог**: данные не теряются при сбоях подов или узлов.                                                    |
| Мониторинг и  здоровье‑пробы          | **Типы проб**:<br>- **livenessProbe**: проверяет, жив ли контейнер (перезапускает при сбоях);<br>- **readinessProbe**: определяет, готов ли под  принимать трафик (исключает из балансировки при инициализации);<br>- **startupProbe**: ждёт полной загрузки приложения перед проверкой liveness/readiness.                                                                                                                               |
| Ограничение  воздействия сбоев        | **Инструменты**:<br>- **Resource Requests/Limits**: предотвращают «вытеснение» подов из‑за перегрузки CPU/памяти;<br>- **NetworkPolicies**: изолируют трафик  между сервисами (например, запрет доступа из dev в prod);<br>- **PodSecurityPolicies**: ограничивают привилегии контейнеров для снижения рисков.                                                                                                                            |
| Автоматическое  масштабирование       | **Компоненты**:<br>- **Horizontal Pod Autoscaler (HPA)**: увеличивает число подов при росте нагрузки;<br>- **Cluster Autoscaler**: добавляет узлы при нехватке ресурсов;<br>- **Vertical Pod Autoscaler (VPA)**: корректирует ресурсы (CPU/память) для подов.<br>**Результат**: система адаптируется к нагрузке, избегая перегрузок и отказов.                                                                                            |
| Резервирование и  мультизональность   | **Практики**:<br>- развёртывание кластера в нескольких зонах доступности (AZ) облака;<br>- использование **StatefulSet** с распределёнными PV для баз данных;<br>- настройка **PodAntiAffinity** для размещения реплик в разных зонах.<br>**Эффект**: устойчивость к сбоям целой зоны или дата‑центра.                                                                                                                                    |

### Как настроить горизонтальное автоматическое масштабирование подов на основе пользовательских метрик в Kubernetes?

| Под темы            | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ               | Горизонтальное автоматическое масштабирование подов в Kubernetes на основе пользовательских метрик реализуется с помощью **Horizontal Pod Autoscaler (HPA)** и интеграции с системами мониторинга, такими как Prometheus. HPA позволяет динамически изменять количество реплик подов в зависимости от заданных пользовательских метрик, например, количества запросов в секунду, глубины очереди или активных соединений.                                                                                                                                                                                       |
| Основные компоненты | 1. **Horizontal Pod Autoscaler (HPA)** — контроллер Kubernetes, который отслеживает метрики и корректирует количество подов в Deployment, StatefulSet или ReplicaSet.<br>2. **Пользовательские метрики** — специфические для приложения показатели, которые могут собираться через API `custom.metrics.k8s.io` или `external.metrics.k8s.io`. <br>3. **Адаптер метрик** (например, `prometheus-adapter`) — компонент, который позволяет HPA получать доступ к пользовательским метрикам из систем мониторинга. <br>4. **Система мониторинга** (например, Prometheus) — собирает и хранит метрики от приложения. |

### В чем заключается принцип работы операторов в Kubernetes и как они могут быть использованы для управления состоянием приложения?

| Под темы                                 | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                       |
| ---------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                              | **Оператор** — это специализированный контроллер,  расширяющий функционал Kubernetes для управления сложными приложениями через пользовательские ресурсы (Custom Resources). Его ключевая задача — автоматизировать операции, специфичные для конкретного приложения, и поддерживать желаемое состояние системы.                                                                                                              |
| Цикл согласования  (reconciliation loop) | Оператор работает по принципу **декларативного согласования состояний**:<br>1. Пользователь задаёт _желаемое состояние_ (desired state) через манифест пользовательского ресурса.<br>2. Оператор непрерывно мониторит _фактическое состояние_ (actual state) системы.<br>3. При расхождении состояний оператор выполняет действия для приведения системы к желаемому виду.<br>4. Цикл повторяется до достижения соответствия. |
| Ключевые  компоненты  оператора          | **Custom Resource Definition (CRD)**<br>    - Описывает схему пользовательского ресурса (поля, типы данных, валидацию).<br>    - Регистрирует новый API‑ресурс в Kubernetes.                                                                                                                                                                                                                                                  |
|                                          | **Контроллер (Controller)**<br>    - Реализует логику согласования.<br>    - Состоит из: <br>        - **Informer** — следит за изменениями ресурсов через watch API Kubernetes;  <br>        - **Indexer** — обеспечивает быстрый доступ к кэшу состояний;<br>        - **Рабочая очередь** — обрабатывает события (create/update/delete).                                                                                   |
|                                          | **Пользовательские ресурсы (Custom Resources, CR)**<br>    - YAML‑манифесты, описывающие желаемое состояние приложения.<br>    - Пример: `PostgreSQLCluster`, `KafkaCluster`.                                                                                                                                                                                                                                                 |
| Управление  состоянием                   | **Автоматизация жизненного цикла**<br>    - Установка/обновление/удаление приложения.<br>    - Миграция данных при обновлении версии.<br>    - Настройка резервного копирования.                                                                                                                                                                                                                                              |
|                                          | **Самовосстановление**<br>    - Обнаружение сбоев компонентов.<br>    - Автоматический перезапуск/восстановление.<br>    - Перераспределение нагрузки при отказах.                                                                                                                                                                                                                                                            |
|                                          | **Конфигурация и настройка**<br>    - Генерация конфигов на основе CR.<br>    - Обновление настроек без простоя.<br>    - Управление сертификатами.                                                                                                                                                                                                                                                                           |
|                                          | **Масштабирование**<br>    - Горизонтальное/вертикальное масштабирование.<br>    - Балансировка нагрузки.                                                                                                                                                                                                                                                                                                                     |
|                                          | **Мониторинг и отчётность**<br>    - Сбор метрик состояния.<br>    - Формирование статуса в CR (`status`‑поле).                                                                                                                                                                                                                                                                                                               |
| Преимущества                             | - **Автоматизация рутинных операций** — снижение ручного труда.<br>- **Стандартизация** — единый подход к развёртыванию и управлению.<br>- **Устойчивость** — самовосстановление при сбоях.<br>- **Расширяемость** — поддержка сложных приложений (БД, Kafka, Elasticsearch).<br>- **Декларативность** — описание желаемого состояния вместо пошаговых инструкций.                                                            |
| Приминение                               | - Управление базами данных (PostgreSQL, MySQL, MongoDB).<br>- Развёртывание и масштабирование Kafka‑кластеров.<br>- Автоматизация CI/CD‑пайплайнов.<br>- Управление сетевыми политиками.<br>- Мониторинг и логирование (Prometheus, ELK).                                                                                                                                                                                     |
