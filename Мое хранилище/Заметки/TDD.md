### Что такое TDD и для чего оно используется?

| Под темы    | Заметка                                                                                                                                                                                                                       |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **TDD (Test-Driven Development) — методология разработки программного обеспечения, при которой тесты пишутся до написания кода**. Цель — не просто проверить функциональность, а спроектировать её «с нуля» через требования. |
| Принципы    | **Красный** — написание теста, который проверяет новую функциональность (пока тест не проходит).                                                                                                                              |
|             | **Зелёный** — написание минимального кода, чтобы тест прошёл.                                                                                                                                                                 |
|             | **Рефакторинг** — улучшение кода без изменения его поведения.                                                                                                                                                                 |

### Какие основные этапы процесса TDD?

| Под темы    | Заметка                                                                                                                                                                                                 |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Красный     | **Цель** — создать тест, который проверяет ещё нереализованную функцию. Тест должен определять, что нужно, чтобы код делал.                                                                             |
| Особенности | Тест заведомо проваливается, так как соответствующий код ещё не написан.                                                                                                                                |
|             | Разработчик должен чётко понимать требования к новой функциональности, рассматривать возможные сценарии использования и пользовательские истории.                                                       |
|             | Этот этап доказывает, что тест работает правильно, и устанавливает ясную, конкретную цель для следующего шага.                                                                                          |
| Зелёный     | **Задача** — написать самый простой код, который заставит красный тест стать «зелёным» (пройти успешно). Не нужно думать об оптимизации или красоте кода — главное — заставить тест пройти.             |
| Особенности | Важно реализовать только те компоненты, которые непосредственно необходимы для прохождения теста.                                                                                                       |
|             | Код может быть неэлегантным или неэффективным — это нормально, фокус исключительно на выполнении требования, определённого тестом.                                                                      |
|             | После написания кода тест проходит, что свидетельствует о соответствии кода заданным требованиям.                                                                                                       |
| Рефакторинг | **После успешного прохождения теста** написанный код оптимизируется и улучшается, при этом сохраняется его исходная функциональность. **Цель** — сделать код более понятным, читабельным и эффективным. |
| Особенности | Убираются дубли, улучшается читаемость, применяются паттерны проектирования.                                                                                                                            |
|             | Тесты должны по-прежнему проходить, что даёт уверенность в безопасности изменений.                                                                                                                      |
|             | Рефакторинг должен проводиться аккуратно, чтобы не нарушить работоспособность созданных тестов.                                                                                                         |

### В чем преимущества использования TDD при разработке ПО?

| Под темы                  | Заметка                                                                                                                                                                                                                                                                                                |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Раннее обнаружение ошибок | **Написание тестов до кода** позволяет разработчикам выявлять потенциальные проблемы в самом начале цикла разработки. Это снижает вероятность, что ошибки будут пропущены и всплывут на более поздних этапах.                                                                                          |
|                           | **Автоматические тесты запускаются при каждом изменении**, и команда мгновенно узнаёт, если что-то сломалось. Это снижает риск регрессии — когда исправление одной ошибки вызывает другую.                                                                                                             |
|                           | **Выявленная ошибка сначала покрывается тестами**, а затем уже исправляется, что исключает регресс с повторным появлением ранее исправленных ошибок.                                                                                                                                                   |
| Качество кода             | **Создание модульного, хорошо организованного кода**. Поскольку разработчики обязаны писать тесты для каждой функциональной части, код структурирован так, что это способствует удобству сопровождения. В результате получается более чистый и читабельный код, который легче понять и модифицировать. |
|                           | **Уверенность в рефакторинге** — набор тестов гарантирует сохранение существующей функциональности после каждого рефакторинга, что позволяет разработчикам постоянно улучшать и оптимизировать код без ущерба для его надёжности.                                                                      |
|                           | **Снижение избыточности** — если компонент проходит тест, он считается готовым. Если существующие тесты проходят, но работает компонент не так, как ожидается, это значит, что тесты пока не отражают всех требований, а это повод добавить новые тесты.                                               |
| Документация              | **Тесты становятся «живой документацией»**, описывающей требуемое поведение системы. Любой новый разработчик может посмотреть, как должна работать функция, просто прочитав тест.                                                                                                                      |
|                           | **Тесты дают представление об ожидаемом поведении кода** — это облегчает понимание функциональности новыми разработчиками и помогает в процессе внедрения.                                                                                                                                             |
|                           | **Регрессионное тестирование** — любые изменения в коде могут быть проверены с помощью существующего набора тестов, что гарантирует, что новые изменения не приведут к непреднамеренному нарушению ранее функционировавших функций.                                                                    |
| Скорость разработки       | **Сокращение времени на исправление дефектов** — поскольку ошибки выявляются сразу, время на исправление дефектов значительно сокращается.                                                                                                                                                             |
|                           | **Меньше времени на отладку** — обычно отладка начинается, когда основной функционал уже сделан, но при TDD концепцию функционала отлаживают заранее, пока пишут тесты.                                                                                                                                |
|                           | **Быстрый цикл обратной связи** — разработчики получают мгновенную обратную связь об успешности или неуспешности своих изменений, что позволяет им быстро выполнять итерации и решать проблемы по мере их возникновения.                                                                               |

### Как TDD влияет на дизайн и архитектуру ПО?

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Влияние на дизайн      | **Опираясь на тесты, разработчики могут быстрее представить, какая функциональность необходима пользователю**. Таким образом, детали интерфейса появляются задолго до окончательной реализации решения.                                                                                                                                                                                                                               |
|                        | **Тесты заставляют делать код более приспособленным для тестирования**. Например, отказываться от глобальных переменных, делать классы менее связанными и лёгкими для использования. Сильно связанный код или код, который требует сложной инициализации, будет труднее протестировать.                                                                                                                                               |
|                        | **Модульное тестирование способствует формированию чётких и небольших интерфейсов**. Каждый класс выполняет определённую роль, как правило, небольшую. Как следствие, зацепление между классами снижается, а связность повышается.                                                                                                                                                                                                    |
|                        | **Тесты помогают лучше понять требования** к программному обеспечению. Это упрощает создание и управление тестовыми сценариями.                                                                                                                                                                                                                                                                                                       |
| Влияние на архитектуру | **TDD способствует более модульному, гибкому и расширяемому коду**. Это связано с тем, что при этой методологии разработчику необходимо думать о программе как о множестве небольших модулей, которые написаны и протестированы независимо и лишь потом соединены вместе. Это приводит к меньшим, более специализированным классам, уменьшению связанности и более чистым интерфейсам.                                                |
|                        | **TDD помогает улучшить архитектуру системы**. Например, при создании нового компонента (например, нового класса) по TDD разработчики должны нарушить зависимости между этим компонентом и всеми остальными, чтобы протестировать его в изоляции. Это требует проектирования (или рефакторинга) для внедрения зависимостей и улучшает использование механизмов гибкости: интерфейсов, полиморфизма, шаблонов проектирования и прочих. |
|                        | **TDD позволяет обеспечить безопасность рефакторинга** исходного кода приложения. Лучше всего это проявляется при обращении к написанному коду через длительное время либо другим разработчиком — при наличии тестов всегда есть гарантия того, появилась ли неполадка в результате изменений.                                                                                                                                        |

### Можете ли вы описать цикл "Красный-Зеленый-Рефактор" в контексте TDD?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                     |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Красный** — написание неудачного теста. Изначально тест будет неудачным, поскольку соответствующий код для выполнения его требований ещё не написан.                                                                                                                                                                                                      |
|          | **Зелёный** — написание минимального кода, прохождение теста. После того как неудачный тест создан, разработчик пишет минимальное количество кода, необходимое для прохождения теста. Эта фаза называется «зелёной», поскольку автоматизированный тест теперь проходит успешно, что свидетельствует о соответствии кода заданным требованиям.               |
|          | **Рефакторинг** — повышение качества кода. После успешного прохождения теста разработчик приступает к рефакторингу кода с целью повышения его качества. Рефакторинг заключается в реструктуризации кода без изменения его внешнего поведения. На этом этапе обеспечивается чистота кода, возможность его сопровождения и соблюдение стандартов кодирования. |

### Какие существуют распространенные ошибки при использовании TDD?

| Под темы                        | Заметка                                                                                                                                                                                                                                                                                |
| ------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Неправильное написание тестов   | **Попытка охватить тестами всё подряд**. Не каждый метод требует отдельного теста. Например, геттеры и сеттеры в простых DTO (Data Transfer Objects) можно не тестировать — они не содержат логики. Перегрузка тестами замедляет сборку и усложняет поддержку.                         |
|                                 | **Тесты копируют реализацию, а не специфицируют поведение**. Например, если в тесте написано expect(result).toBe(calculate(x, y)), это не проверка, а дублирование. Нужно проверять ожидаемый результат: expect(add(2, 3)).toBe(5).                                                    |
|                                 | **Тест плохо сформулирован**. Он проверяет не то, что нужно. Например, тест может проверять, что функция вернула объект, но не проверяет его поля. Решение — переписать тест, добавив больше деталей.                                                                                  |
| Игнорирование рефакторинга      | **Многие разработчики останавливаются на «зелёном» состоянии** и не улучшают код. Это нарушает принцип TDD и приводит к техническому долгу. Рефакторинг — не опция, а обязательный этап.                                                                                               |
|                                 | **Неправильное понимание цикла «тест → код → рефакторинг»**. Многие нарушают ритм TDD, пытаясь сразу написать всё: тест и полноценную реализацию. Настоящий цикл — это серия маленьких шагов: пишем минимальный тест, реализуем минимум для его прохождения, затем улучшаем структуру. |
| Медленное выполнение тестов     | **Тестирование внешних зависимостей**. Если тест зависит от базы данных, API или файловой системы, он становится медленным и хрупким. Решение — использование моков и заглушек (mocks/stubs). Например, вместо реального вызова HTTP-запроса в тесте можно имитировать ответ сервера.  |
| Недостаточное тестовое покрытие | **Пропуск написания тестов**. Когда новый код пишется без тестов в голове, написание тестов впоследствии, скорее всего, будет более сложным. Это приводит к снижению тестового покрытия.                                                                                               |
|                                 | **Неправильное тестирование деталей реализации**. Тесты должны быть сосредоточены на публичном API («что»), а не на внутренней реализации («как»). Тестирование приватных методов делает тесты хрупкими и затрудняет рефакторинг.                                                      |

### Как TDD влияет на поддержку и расширяемость ПО?

| Под темы | Заметка                                                                                                                                                                                                                                                      |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ    | **Повышается надёжность изменений**. При внесении изменений в хорошо протестированный код риск появления новых ошибок значительно ниже. Если новая функциональность приводит к сбоям, тесты сразу же это покажут.                                            |
|          | **Упрощается добавление новых функций**. Легко вносить изменения, не боясь навредить существующим функциям. Цикл TDD позволяет наращивать количество функций постепенно и контролируемо.                                                                     |
|          | **Позволяется легко локализовать причину багов**. Тесты рассчитаны на проверку небольших фрагментов кода, что позволяет быстро выяснять то место проекта, в котором произошёл сбой. Ошибки реже обнаруживаются в уже развёрнутой обновлённой версии системы. |
|          | **Улучшаются возможности по масштабированию и расширению**. Разработчики планируют будущее устройство систем в плане их структуры и архитектуры с самого начала работы над системой.                                                                         |

### Как можно интегрировать TDD в существующий процесс разработки без значительных потерь в производительности?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                     |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Интегрировать TDD (разработку через тестирование) в существующий процесс разработки без значительных потерь в производительности можно, сочетая принципы TDD с инструментами и обучением**. Важно найти баланс между написанием тестов и написанием кода, чтобы не снижать качество кода, торопя процесс. |
| Принципы | **Декомпозировать задачи** — разбивать большие функции на мелкие тестируемые части.                                                                                                                                                                                                                         |
|          | **Писать независимые тесты** — каждый тест должен проверять одну конкретную возможность.                                                                                                                                                                                                                    |
|          | **Проводить рефакторинг тестов** — удалять устаревшие проверки и оптимизировать актуальные.                                                                                                                                                                                                                 |
|          | **Автоматизировать рутину** — использовать скрипты для массового запуска тестов.                                                                                                                                                                                                                            |
|          | **Интегрировать TDD с процессами непрерывной интеграции (CI) и непрерывного развёртывания (CD)** — это гарантирует непрерывное выполнение автоматизированных тестов.                                                                                                                                        |
|          | **Объединять TDD с инструментами отслеживания требований** — это позволяет связать каждый модульный тест с его требованием, бизнес-правилом или пользовательской историей, гарантируя сквозное покрытие требований.                                                                                         |

### Какие стратегии можно применить для обработки унаследованного кода, который не был разработан с использованием TDD?

| Под темы     | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ        | **Для обработки унаследованного кода, который не был разработан с использованием TDD (Test-Driven Development), можно применить стратегии, связанные с рефакторингом, тестированием и документацией**, а также использовать подходы, адаптированные под специфику проекта. Универсальной стратегии, подходящей абсолютно для всех проектов, не существует — каждая стратегия должна быть настроена и адаптирована к конкретным условиям и требованиям. |
| Рефакторинг  | **Разбиение крупных функций** на более мелкие логические блоки. Например, функция, обрабатывающая множественные этапы операции, делится на отдельные методы с чётко обозначенной задачей.                                                                                                                                                                                                                                                              |
|              | **Переименование переменных и методов** — применение понятных и однозначных имён, что улучшает структуру и снижает количество ошибок.                                                                                                                                                                                                                                                                                                                  |
|              | **Устранение дублирования** — вынос повторяющегося кода в общие методы или классы.                                                                                                                                                                                                                                                                                                                                                                     |
|              | **Пошаговое изменение** — рефакторинг не проводится монолитно, а разбивается на небольшие части. Каждый шаг вносится постепенно, сохраняя работоспособность и тестируемость.                                                                                                                                                                                                                                                                           |
| Тестирование | **Автоматизированные тесты** — юнит-тесты проверяют отдельные модули, а интеграционные тесты контролируют взаимодействие компонентов.                                                                                                                                                                                                                                                                                                                  |
|              | **Регресс-тестирование** — проверяет отсутствие новых багов после изменений.                                                                                                                                                                                                                                                                                                                                                                           |
|              | **Тестирование модулей** — фокусируется на отдельных функциях.                                                                                                                                                                                                                                                                                                                                                                                         |
|              | **Непрерывная интеграция** — автоматический запуск тестов при каждом обновлении кода.                                                                                                                                                                                                                                                                                                                                                                  |
|              | **Использование тестовых покрытий** — помогает оценить полноту проверок.                                                                                                                                                                                                                                                                                                                                                                               |
| Документация | **Использовать модульные тесты** — они документируют, что должен делать код по мнению разработчиков.                                                                                                                                                                                                                                                                                                                                                   |
|              | **Документировать поведение библиотек**, на которые полагается код, — это позволяет поддерживать согласованное поведение приложения во время обновления.                                                                                                                                                                                                                                                                                               |
