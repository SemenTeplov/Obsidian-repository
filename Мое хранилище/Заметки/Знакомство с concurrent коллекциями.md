### Что такое concurrent коллекции и в чем их основное предназначение?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Concurrent коллекции в Java** — это потокобезопасные (thread-safe) реализации стандартных коллекций, разработанные для многопоточных приложений. Их основное предназначение — **обеспечить безопасное изменение и доступ к данным из нескольких потоков** без необходимости внешней синхронизации.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| Виды        | Некоторые примеры concurrent коллекций в Java:<br>- **ConcurrentHashMap** — потокобезопасная реализация Map, позволяет одновременные операции чтения и записи. Вместо блокировки всей карты блокирует только конкретные сегменты (бакеты) для лучшей производительности.<br>- **CopyOnWriteArrayList** — потокобезопасная реализация List, при каждой модификации (добавление, удаление элементов) создаёт новую копию внутреннего массива. Это обеспечивает безопасное итерирование по списку, не приводя к ConcurrentModificationException. <br>- **ConcurrentSkipListMap** — потокобезопасная реализация NavigableMap на основе списка пропуска, поддерживает одновременный доступ и поддерживает порядок, что подходит для приложений, требующих отсортированных пар ключ-значение.<br>- **ConcurrentLinkedQueue** — потокобезопасная неблокирующая очередь, основана на неблокирующем алгоритме, позволяет одновременные вставки и удаления. |

### Какие проблемы решают concurrent коллекции по сравнению с обычными коллекциями Java?

| Под темы                                    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                                 | **Concurrent-коллекции в Java решают проблемы, связанные с одновременным доступом к общим данным в многопоточной среде, по сравнению с обычными коллекциями из пакета java.util**. Они обеспечивают потокобезопасность (thread-safe) и оптимизированы для параллельного доступа, что снижает риск проблем синхронизации данных и улучшает производительность.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| Обычные коллекции (java.util)               | **Стандартные коллекции Java**, такие как ArrayList или HashMap, **не являются потокобезопасными**. Использование их в многопоточной среде без синхронизации может привести к неопределённому поведению или даже к ошибкам выполнения. Некоторые проблемы, которые решают concurrent-коллекции: <br>- **Проблемы с согласованностью данных** — например, если несколько потоков пытаются одновременно обновить карту без синхронизации, целостность карты может быть нарушена.<br>- **Снижение производительности** — из-за постоянной синхронизации использование обычных коллекций может приводить к снижению производительности, особенно при высоких нагрузках.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| Concurrent коллекции (java.util.concurrent) | **Concurrent-коллекции** предоставляют потокобезопасные реализации стандартных коллекций, таких как списки, множества и отображения. Некоторые механизмы, которые решают проблемы:<br>- **Встроенные механизмы синхронизации** — снижают необходимость внешней синхронизации, например, синхронизированных блоков или блокировок.<br>- **Атомарные операции** — позволяют безопасно выполнять их из нескольких потоков без необходимости явной синхронизации.<br>- **Механизмы блокировки сегментов (lock striping)** — минимизируют блокировки и обеспечивают высокую производительность при параллельных операциях чтения и записи.<br>- **Стратегия копирования при записи** — например, в CopyOnWriteArrayList операция записи создаёт копию всего массива, что обеспечивает безопасное итерирование по списку, не приводя к ConcurrentModificationException.<br>**Преимущества concurrent-коллекций**:<br>- **Масштабируемость** — позволяют нескольким потокам обращаться к коллекции одновременно.<br>- **Производительность** — могут достигать высокой производительности по сравнению с синхронизированными коллекциями для высококонкурентоспособных нагрузок.<br>- **Неблокирующее поведение** — не блокируют потоки, обеспечивая отзывчивость даже при тяжёлом одновременном доступе. |

### Перечислите несколько примеров concurrent коллекций в Java

| Под темы        | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Queue (Очередь) | - **ConcurrentLinkedQueue** — неблокирующая очередь, основанная на связанных узлах. Реализует принцип FIFO — «первым прибыл, первым убыл». Новые элементы размещаются в хвосте очереди, а операции извлечения получают элементы из головы. Класс не разрешает использование null-элементов.<br>- **ConcurrentLinkedDeque** — неограниченная по ёмкости двухсторонняя очередь, основанная на связанных узлах. Обеспечивает безопасные операции размещения, удаления и доступа параллельно выполняющимся потокам. Не разрешает размещение null-элементов в очереди.                                                                                                                                                                |
| Map (Карта)     | - **ConcurrentHashMap** — потокобезопасная карта, поддерживает быстрые операции чтения и записи. Данные представлены в виде сегментов, разбитых по хэш-ключам. Для доступа к данным блокируется сегмент, а не объект. Итераторы не вызывают ConcurrentModificationException и представляют данные на определённый отрезок времени.<br>- **ConcurrentSkipListMap** — аналог TreeMap для многопоточности. Данные сортируются по ключу, гарантируется усреднённая производительность log(N) для операций containsKey, get, put, remove и других похожих операций.<br>- **ConcurrentNavigableMap** — расширяет интерфейс NavigableMap, все итераторы безопасны для использования и не вызывают ConcurrentModificationException.      |
| Set (Множество) | - **ConcurrentHashMap.newKeySet** — создаёт потокобезопасное множество на основе ConcurrentHashMap. Подходит для сценариев с высокой степенью параллелизма и не требует ручной синхронизации.<br>- **ConcurrentSkipListSet** — обеспечивает упорядоченный доступ, подходит для сценариев, где активно происходит чтение, запись и итерация. Внутренняя структура «прыгучего списка» предоставляет логарифмическую эффективность для основных операций, соблюдая при этом порядок элементов.<br>- **CopyOnWriteArraySet** — гарантирует надёжность итераций, так как итератор отражает состояние множества на момент его создания. Наиболее эффективен в ситуациях, где чтения происходят часто, а обновления не так значительны. |

### В чем разница между Collections.synchronizedMap и ConcurrentHashMap?

| Под темы                    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| --------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                 | **Collections.synchronizedMap и ConcurrentHashMap в Java отличаются подходом к обеспечению потокобезопасности**. Несмотря на внешнюю схожесть, эти реализации используют разные механизмы синхронизации, что влияет на производительность и функциональность.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| Collections.synchronizedMap | **Классический подход к обеспечению потокобезопасности**, появившийся ещё в Java 1.2. Оборачивает обычную Map и синхронизирует каждый метод, используя единственный объект-монитор.<br>**Особенности**:<br>- Блокирует всю структуру данных для чтения и записи, что означает, что только один поток может обращаться к карте одновременно.<br>- Предоставляет итератор, который не является потокобезопасным — при итерации необходимо внешнее синхронизирование на объекте карты.<br>- Полностью поддерживает null в качестве ключей и значений (если базовая Map это позволяет).<br>**Применение**: Collections.synchronizedMap подходит для сценариев, где к карте обращаются несколько потоков, но она не сильно используется в многопоточной среде.                                                              |
| ConcurrentHashMap           | **Потокобезопасная реализация Map**, появившаяся в Java 1.5 как часть пакета java.util.concurrent. Позволяет одновременно читать и записывать данные из разных потоков без блокировки всей структуры данных.<br>**Особенности**:<br>- Внутренняя структура разделена на сегменты, каждый из которых может блокироваться независимо.<br>- Чтение происходит без блокировок в большинстве случаев, запись блокирует только конкретный сегмент.<br>- Обеспечивает безопасную итерацию без необходимости внешней синхронизации — итераторы работают по принципу fail-safe, отражая состояние карты на момент создания итератора и не выбрасывая ConcurrentModificationException.<br>- Не допускает использование null ни в качестве ключа, ни в качестве значения — попытка вставить null приведёт к NullPointerException. |

### Как работает CopyOnWriteArrayList и в каких случаях его использование оправдано?

| Под темы           | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение        | **CopyOnWriteArrayList** — потокобезопасный вариант ArrayList в Java, который использует подход «копирование при записи» (copy-on-write).                                                                                                                                                                                                                                                                                                                                                               |
| Принцип работы     | **CopyOnWriteArrayList оправдан в следующих случаях**:<br>- список будет использоваться в параллельной среде;<br>- число итераций превышает число операций мутации;<br>- итераторы должны иметь моментальную версию списка на момент их создания;<br>- не хочется синхронизировать доступ к потокам программно.<br>**Важно**: в CopyOnWriteArrayList только один поток может выполнять операцию записи за раз, но несколько потоков могут выполнять операции чтения одновременно.                       |
| Использование      | **Из-за дополнительного шага создания нового резервного массива** каждый раз при обновлении списка производительность CopyOnWriteArrayList хуже, чем у ArrayList. **Операции чтения** не приводят к снижению производительности, и оба класса выполняют их одинаково. <br>**Рекомендуется**: CopyOnWriteArrayList полезен только в случаях, когда количество операций записи мало по сравнению с количеством операций чтения, иначе это может вызвать проблемы с памятью при большом объёме данных.     |
| Производительность | **Некоторые особенности использования CopyOnWriteArrayList**:<br>- **Операции мутации на итераторах** (удаление, установка и добавление) не поддерживаются — эти методы выдают исключение UnsupportedOperationException.<br>- **Итератор не отражает** добавления, удаления или изменения в списке с момента создания.<br>- **Действия в потоке** перед помещением объекта в CopyOnWriteArrayList происходят до действий, следующих за доступом или удалением этого элемента из списка в другом потоке. |

### Почему использование блокировок в concurrent коллекциях может быть предпочтительнее, чем внешнее синхронизирование?

| Под темы                     | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Блокировки (Lock)            | **Блокировки (Lock) позволяют более гибко управлять доступом к общему ресурсу** по сравнению с синхронизированными блоками. Некоторые преимущества:<br>- **Возможность ожидания блокировки с таймаутом** — блокировка может быть получена за заданный промежуток времени.<br>- **Прерываемое ожидание блокировки** — поток может быть прерван во время ожидания блокировки, что позволяет адекватно реагировать на запросы о завершении работы.<br>- **Получение и освобождение блокировки в разных местах кода** — Lock даёт возможность более гибкого управления блокировками.<br>- **Масштабируемость в условиях состязательности** — когда много потоков соперничают за право получения блокировки, общая пропускная способность обычно лучше у Lock, чем у synchronized.<br>Однако использование блокировок требует определённых обязанностей: например, блокировка должна обязательно сниматься в блоке finally, чтобы свести к минимуму риск взаимных блокировок из-за неожиданных исключений. |
| Синхронизация (synchronized) | **Синхронизация (synchronized) обеспечивает атомарность операций и видимость изменений между потоками**. Когда поток входит в synchronized-блок или метод, он получает монопольную блокировку на объект-монитор, не позволяя другим потокам одновременно выполнять код, защищённый той же блокировкой.<br>**Некоторые особенности синхронизации**:<br>- **Синхронизированные методы** — блокировка автоматически устанавливается на объект this (для нестатических методов) или на объект класса (для статических методов). Это простой способ синхронизации, но блокировка распространяется на весь метод, что может снизить производительность.<br>- **Синхронизированные блоки** — позволяют минимизировать область действия блокировки, повышая производительность.                                                                                                                                                                                                                               |

### Каковы основные принципы работы блокировок в ConcurrentHashMap?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Основные принципы работы блокировок в ConcurrentHashMap** заключаются в **минимизации блокировок** и **обеспечении параллельного доступа к частям карты**. В отличие от обычной HashMap, которая не является потокобезопасной, ConcurrentHashMap гарантирует потокобезопасность операций, позволяя нескольким потокам считывать и записывать данные одновременно.                                                                                                                                                                                                                                                         |
| Механизм    | В ConcurrentHashMap используются **мелкие блокировки** (Fine-Grained Locks) для управления доступом к отдельным бакетам или небольшим группам бакетов. Также для некоторых операций, таких как вставка и обновление значений, используется **механизм CAS (Compare-And-Swap)**, который позволяет атомарно обновлять значения без использования явных блокировок.<br>**Внутренне** для блокировок ConcurrentHashMap использует **ReentrantLock**, что позволяет потокам безопасно входить в блокировку несколько раз.                                                                                                       |
| Виды        | На работу блокировок в ConcurrentHashMap влияет **параметр concurrencyLevel**. По умолчанию он равен 16, но можно указать другое значение. **Занижение concurrencyLevel** ведёт к тому, что более вероятны блокировки потоками сегментов карты при записи, **завышение** — к неэффективному использованию памяти. Если лишь один поток будет изменять карту, а остальные будут производить чтение, рекомендуется использовать значение 1.<br>**Увеличение количества сегментов** способствует тому, что операции модификации будут затрагивать различные сегменты, что уменьшит вероятность блокировок во время выполнения. |

### В каких случаях использование ConcurrentLinkedQueue предпочтительно перед LinkedBlockingQueue?

| Под темы              | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение           | **Использование ConcurrentLinkedQueue предпочтительно в случаях, когда требуется высокая производительность при работе с множеством потоков-производителей и потоков-потребителей**. Также ConcurrentLinkedQueue подходит для **неограниченных нагрузок**, где производители генерируют данные в всплесках, и блокировка их может нарушить операции.<br>**LinkedBlockingQueue** стоит использовать, если **требуется контроль над ресурсами** и необходимость поддерживать фиксированный размер очереди. Также LinkedBlockingQueue подходит, если **приемлема блокировка** — производители и потребители могут ждать данных или пространства (например, рабочие процессы с постоянной скоростью).<br>Перед выбором очереди рекомендуется провести тестирование и бенчмаркинг.                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ConcurrentLinkedQueue | **Некоторые особенности ConcurrentLinkedQueue:**<br>- **Неблокирующий дизайн** — минимизирует накладные расходы при высокой параллельности (например, более 1000 потоков).<br>- **Отсутствие конкуренции за блокировки** — избегает рисков взаимоблокировки или голода потоков, связанных с блокировками.<br>- **Неограниченная гибкость** — производители не блокируются (так как метод offer() всегда успешно выполняется), что подходит для всплесков нагрузок, где блокировка <br>**Недостатки**:<br>- **Утечки памяти** — неограниченный рост очереди может привести к сбоям в работе JVM, если потребители отстают. Нужно контролировать размер очереди и при необходимости ограничивать производителей.<br>- **Занятое ожидание** — использование метода poll() в плотном цикле (без сна) тратит CPU.                                                                                                                                                                                                                                                                                                                                                                                                                    |
| LinkedBlockingQueue   | **Некоторые особенности LinkedBlockingQueue:**<br>- **Структура** — дважды связанный список (узлы содержат указатели prev и next).<br>- **Контроль параллельности** — использует два отдельных экземпляра ReentrantLock (один для производителей, один для потребителей) для минимизации конкуренции за блокировки.<br>- **Ёмкость** — может быть ограниченной (фиксированный размер) или неограниченной (по умолчанию — Integer.MAX_VALUE). Ограниченная ёмкость предотвращает переполнение памяти, блокируя производителей при заполнении.<br>- **Блокирующие методы** — наследует методы BlockingQueue, например, take() (блокирует, пока не станет доступен элемент) и put() (блокирует, пока не станет доступно пространство).<br>**Недостатки**:<br>- **Стоимость блокировки** — каждая операция put или take конкурирует за блокировку между производителями или потребителями (если их много), поэтому в сценариях с многими производителями и потребителями операции могут быть медленнее. <br>**Важно**: перед использованием методов LinkedBlockingQueue и ConcurrentLinkedQueue рекомендуется проверить реализацию, так как это может привести к неожиданной блокировке, влияющей на производительность приложения. |

### Каким образом реализована атомарность операций в коллекциях из пакета java.util.concurrent?

| Определение    | **Атомарность операций в многопоточной среде в Java реализована с помощью классов из пакета java.util.concurrent.atomic**. Эти классы обеспечивают безопасные операции над переменными в многопоточной среде без явной синхронизации.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| Принцип работы | Атомарность операций обеспечивается **механизмом compare-and-swap (CAS)**. Он обеспечивает безопасную операцию записи, которая выполняется только если значение переменной соответствует ожидаемому. Основные компоненты CAS:<br>- **Сравнение (Compare)** — сравнение текущего значения переменной с ожидаемым значением.<br>- **Замена (And-Swap)** — если сравнение успешно, происходит замена значения переменной на новое.<br>Механизм CAS позволяет избежать использования явных блокировок и улучшить эффективность в средах с высоким параллелизмом.                                                                                                                                                                                                                         |
| Классы         | Некоторые классы из пакета java.util.concurrent.atomic, реализующие атомарность операций:<br>- **AtomicInteger** — обеспечивает атомарные операции для переменных типа int.<br>- **AtomicLong** — предоставляет атомарные операции для переменных типа long.<br>- **AtomicBoolean** — позволяет безопасно выполнять атомарные операции над переменными типа boolean.<br>- **AtomicReference** — позволяет проводить атомарные операции с переменными типа ссылки.<br>- **AtomicIntegerArray, AtomicLongArray, AtomicReferenceArray** — предоставляют атомарные операции для соответствующих массивов.<br>Классы в этом пакете также содержат методы для получения и безусловной установки значений, а также более слабую условную атомарную операцию обновления (weakCompareAndSet). |
