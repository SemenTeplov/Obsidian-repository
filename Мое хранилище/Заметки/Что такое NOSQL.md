### Что такое NOSQL

| Под темы                | Заметка                                                                                                                                                                                                                                                   |
| ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение             | **NoSQL** (от англ. _Not Only SQL_) — это подход к базам данных, который не ограничивается реляционной моделью. Основная цель — обеспечить высокую масштабируемость, гибкость и производительность при работе с большими объёмами разнородной информации. |
| NoSQL                   | **Структура данных**: вместо таблиц с фиксированными структурами и отношениями между ними используются гибкие форматы: документы, пары «ключ-значение», графы или колонки.                                                                                |
| Реляционные базы данных | **Структура данных**: информация хранится в таблицах с чёткой структурой: строки — это записи, а столбцы — свойства этих записей. Таблицы связаны между собой через первичные и внешние ключи, чтобы объединять данные из нескольких таблиц.              |

### Перечислите несколько типов NoSQL баз данных.

| Под темы                 | Заметка                                                                                                                                                                                                                                 |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ключ-значение            | **Основаны на простой структуре**: каждая запись — пара «ключ-значение». Ключ используется для идентификации и извлечения записи, значение может быть любого формата (строки, числа, JSON-объекты и др.).                               |
| Документоориентированные | **Хранят данные в виде документов** (обычно в формате JSON, BSON или XML). Документы — самостоятельные единицы, которые содержат всю необходимую информацию. Они могут быть вложенными, что позволяет хранить сложные структуры данных. |
| Колоночные               | **Ориентированы на хранение данных в виде колонок, а не строк**. Колонки логически группируются в колоночные семейства, что позволяет обращаться к целому семейству столбцов одновременно, а не к одному отдельно взятому столбцу.      |
| Графовые                 | **Оптимизированы для работы с графами и связанными данными**. Элементы базы данных хранятся в узлах (вершинах), между узлами существуют рёбра, которые определяют отношения элементов друг к другу.                                     |

### Какие преимущества имеет использование NoSQL баз данных перед реляционными?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                            |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| NoSQL       | **Гибкость модели данных и отсутствие жёсткой схемы**. В отличие от реляционных баз, где данные должны соответствовать строго определённой схеме, NoSQL позволяет хранить их в различных форматах без необходимости заранее определять структуру. Это полезно, когда структура данных часто меняется или данные имеют сложную вложенную структуру. |
|             | **Масштабируемость и распределённость**. NoSQL-базы поддерживают горизонтальное масштабирование, что позволяет легко добавлять новые серверы для обработки растущих объёмов данных. Также данные реплицируются на несколько серверов, что обеспечивает отказоустойчивость и высокую доступность.                                                   |
|             | **Высокая производительность**. Распределение данных между множеством серверов позволяет обрабатывать миллионы запросов в секунду, а репликация данных обеспечивает их доступность даже при выходе из строя части серверов.                                                                                                                        |
| Реляционные | **Подходят для сложных SQL-запросов и аналитики**, где требуется соединение данных из разных таблиц. Если запросы требуют сложных операций JOIN и агрегации, реляционные базы данных могут быть предпочтительнее.                                                                                                                                  |
|             | **Обеспечивают целостность данных** с помощью ACID-свойств (Atomicity, Consistency, Isolation, Durability). Если приложение требует строгой гарантии целостности данных, то реляционная база данных может быть более подходящим выбором.                                                                                                           |
|             | **Обеспечивают предсказуемую схему данных**, если данные имеют строго определённую структуру и не подвержены частым изменениям.                                                                                                                                                                                                                    |

### Какие сценарии использования являются наиболее подходящими для NoSQL баз данных?

| Под темы    | Заметка                                                                                                                                                                                                                              |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ       | Базы данных NoSQL (Not Only SQL) подходят для сценариев, где необходимо обрабатывать большие объёмы данных, обрабатывать данные в реальном времени и использовать гибкие модели данных.                                              |
| Особенности | **Горизонтальное масштабирование** — возможность добавлять новые серверы для обработки увеличивающихся объёмов данных.                                                                                                               |
|             | **Гибкость данных** — возможность хранить данные в различных форматах и схемах, что упрощает работу с неструктурированными и полуструктурированными данными.                                                                         |
|             | **Высокая производительность** — благодаря простой архитектуре и возможности хранить данные в памяти многие NoSQL базы данных предлагают высокую скорость обработки запросов.                                                        |
| Примеры     | **Хранение данных, поступающих с миллионов сенсоров**. NoSQL базы данных подходят для работы с данными в разных форматах: от JSON-документов до данных временных рядов.                                                              |
|             | **Аналитика в реальном времени** — NoSQL базы данных, особенно базы в памяти, поддерживают высокую скорость обработки данных.                                                                                                        |
|             | **Хранение и анализ огромных объёмов логов и метаданных**, генерируемых веб-сайтами, мобильными приложениями и сенсорами IoT. Системы, такие как MongoDB или Cassandra, используются для этого.                                      |
|             | **Хранение и предобработка данных для машинного обучения** — NoSQL базы данных могут использоваться для хранения и предобработки данных, необходимых для обучения моделей машинного обучения.                                        |
|             | **Управление профилями пользователей**, публикациями и взаимодействием. NoSQL обеспечивает быстрый и эффективный поиск персонализированного контента, системы рекомендаций и анализ взаимодействия пользователей в реальном времени. |
|             | **Использование графовых баз данных** — они хранят данные в виде узлов и связей между ними, что подходит для задач, ориентированных на связи, например, для алгоритмов рекомендаций контента.                                        |
|             | **Обработка обширных каталогов продуктов**, отзывов пользователей и транзакционных данных. NoSQL базы данных обеспечивают надёжное управление данными и обработку данных в реальном времени для беспрепятственного процесса покупок. |
|             | **Управление корзинами пользователей**, которые динамически меняются в процессе взаимодействия с сайтом.                                                                                                                             |
|             | **Анализ покупательской способности** в режиме реального времени.                                                                                                                                                                    |

### Объясните, что такое документо-ориентированная база данных и приведите примеры таких баз.

| Под темы    | Заметка                                                                                                                                                                                                                                                         |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Документоориентированная база данных** — это система управления базами данных, предназначенная для хранения иерархических структур данных (документов).                                                                                                       |
|             | Такие базы данных хранят данные в слабоструктурированных документах, таких как JSON или BSON. Это обеспечивает большую гибкость моделирования данных и позволяет создавать динамичные схемы, которые могут развиваться по мере изменения требований приложения. |
| Пример      | CouchDB, Couchbase, MarkLogic, MongoDB, eXist.                                                                                                                                                                                                                  |

### Как работает индексация в NoSQL базах данных и чем она отличается от индексации в реляционных базах данных?

| Под темы                | Заметка                                                                                                                                                                                                            |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Реляционные базы данных | **Индексация в RDBMS** представлена в виде **B-деревьев**. Они отражают иерархическую структуру реляционных данных.                                                                                                |
| Особенности             | **Кластеризованные индексы** — используют первичный ключ для структуризации данных в таблице, создаются по умолчанию.                                                                                              |
|                         | **Некластеризованные индексы** — не перестраивают физическую структуру таблицы, а лишь организуют ссылки на соответствующие строки.                                                                                |
|                         | **Индексирование по нескольким столбцам** — индекс можно создать более чем для одного столбца, важен порядок следования полей.                                                                                     |
| NoSQL базы данных       | **Индексация в NoSQL-базах данных** указывает на **документы** или на части документов, между которыми, в основном, нет отношений.                                                                                 |
| Особенности             | **Первичные индексы** — дают уникальную идентификацию каждой записи и организуют данные в соответствии с определённым ключом. Определяются автоматически при создании записи.                                      |
|                         | **Вторичные индексы** — дают возможность быстрого поиска по полям, отличным от первичного ключа. Помогают выполнять сложные запросы, которые не могут быть реализованы с использованием только первичных индексов. |
|                         | **Индексы на основе диапазонов** — позволяют обрабатывать запросы, где необходимо работать с интервалами значений: временными метками, числовыми диапазонами или алфавитными строками.                             |

### Может ли Elasticsearch выступать как обычная база данных, а не только для логов?

| Под темы | Заметка                                                                                                                                                                                                                                        |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Elasticsearch может выступать как обычная база данных**, но **не только для логов**. Система сочетает функции базы данных, поисковой и аналитической системы, и её можно использовать для хранения, поиска и анализа больших объёмов данных. |
|          | Это связано с особенностями функций Elasticsearch, архитектуры системы и возможностей настройки.                                                                                                                                               |

### Какие есть преимущества и особенности Cassandra?

| Под темы     | Заметка                                                                                                                                                                                                                                                                                                                                     |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение  | **Apache Cassandra** — распределённая NoSQL-база данных с открытым исходным кодом, предназначенная для работы с большими объёмами данных в высоконагруженных системах. Изначально разработана в Facebook, позже проект был передан в Apache Software Foundation.                                                                            |
| Преимущества | **Масштабируемость и высокая доступность**. Отсутствие центрального сервера (master node) устраняет единую точку отказа. Новые узлы добавляются в кластер без остановки сервиса и ручного вмешательства. Система может масштабироваться линейно до тысяч узлов, при этом производительность растёт пропорционально количеству оборудования. |
|              | **Гибкая схема данных**. Система поддерживает до двух миллионов столбцов в одной таблице и позволяет определять различные наборы столбцов для строк в одной таблице. Такой подход решает проблему разреженных таблиц — типичную ситуацию в Big Data-проектах.                                                                               |
|              | **SQL-подобный язык запросов (CQL)**. Cassandra Query Language обеспечивает привычный синтаксис для разработчиков, знакомых с SQL, поддерживая основные операции SELECT, UPDATE и работу с пространствами имён.                                                                                                                             |
|              | **Расширяемость функциональности**. Начиная с версии 2.1 поддерживаются пользовательские типы данных (UDT), а версия 2.2 добавила возможность создания хранимых процедур и агрегатных функций.                                                                                                                                              |
| Особенности  | **Настраиваемая согласованность**. Пользователи могут сами определить необходимое количество реплик и задать уровень согласованности данных по каждой операции хранения.                                                                                                                                                                    |
|              | **Репликация данных**. Cassandra позволяет развернуть кластер в нескольких центрах обработки данных в разных регионах, что позволяет реплицировать данные для аварийного восстановления.                                                                                                                                                    |
|              | **Комплексная безопасность**. Поддержка аутентификации, ролевой модели доступа и шифрованного соединения по SSL обеспечивает корпоративный уровень безопасности данных.                                                                                                                                                                     |
|              | **ACID на уровне записи**. Хотя Cassandra не поддерживает транзакции между записями, она гарантирует ACID-свойства для операций с одним ключом — все столбцы записи будут обновлены атомарно с обеспечением изолированности и долговечности.                                                                                                |

### Объясните концепцию CAP теоремы и как она применяется к NoSQL базам данных.

| Под темы    | Заметка                                                                                                                                                                                                                       |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Теорема CAP (известна также как теорема Брюера) объясняет ограничения распределённых систем, включая NoSQL-базы данных**. Она помогает определить приоритеты и балансировать между основными характеристиками таких систем. |
| Концепция   | Теорема утверждает, что в распределённой системе невозможно одновременно обеспечить три свойства                                                                                                                              |
|             | **Согласованность (Consistency)** — все узлы системы видят одни и те же данные в один и тот же момент времени.                                                                                                                |
|             | **Доступность (Availability)** — каждый запрос к системе получает ответ, даже если некоторые узлы недоступны или вышли из строя.                                                                                              |
|             | **Устойчивость к разделению (Partition tolerance)** — система продолжает функционировать даже в условиях сетевых разделений, когда узлы не могут общаться друг с другом.                                                      |
| Приминение  | Теорема CAP помогает разработчикам выбирать, какие свойства важны для конкретной системы.                                                                                                                                     |
|             | Если требуется построить систему для финансовых транзакций, то согласованность может быть более важной, чем доступность или устойчивость к разделению.                                                                        |
|             | Если нужно построить систему для обработки данных в режиме реального времени, то доступность может быть более важной, чем согласованность или устойчивость к разделению.                                                      |

### Как NoSQL базы данных обеспечивают горизонтальное масштабирование и как это отличается от вертикального масштабирования в реляционных базах?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                              |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Базы данных NoSQL обеспечивают горизонтальное масштабирование**, в то время как реляционные базы данных обычно масштабируются вертикально. Это отличие связано с архитектурой и подходами к хранению данных                                                                                        |
| NoSQL       | **Горизонтальное масштабирование** в базах данных NoSQL обеспечивается за счёт **распределения данных по нескольким серверам** (узлам) в кластере.                                                                                                                                                   |
| Реляционные | **Вертикальное масштабирование** в реляционных базах данных предполагает **увеличение мощности одного сервера** для обработки более высокой нагрузки. Это можно достичь путём обновления аппаратного обеспечения сервера, например добавления дополнительных ядер ЦП, увеличения оперативной памяти. |

### В чем состоит принцип BASE в контексте NoSQL и как он отличается от принципа ACID в реляционных базах данных?

| Под темы | Заметка                                                                                                                                                                                                                                                                           |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | BASE ориентирован на **доступность и гибкость**, а не строгую согласованность, в то время как ACID обеспечивает **строгую согласованность** после каждой транзакции.                                                                                                              |
| Различия | **ACID** обеспечивает строгую согласованность после каждой транзакции, строго соблюдая правила целостности. **BASE** жертвует строгой согласованностью ради производительности и доступности — данные могут быть временно несогласованными, но придут в порядок позже.            |
|          | **ACID-системы** ставят во главу угла согласованность, из-за чего могут становиться недоступными при сбоях. **BASE-системы** ориентированы на высокую доступность и остаются отзывчивыми даже при сетевых проблемах или отказах отдельных узлов.                                  |
|          | **ACID-системам** сложно масштабироваться горизонтально, поскольку строгая согласованность на распределённых системах требует значительных ресурсов. **BASE-системы** легко масштабируются и рассчитаны на распределённую архитектуру, где допустимы временные несогласованности. |
