### Что такое проблема N+1 в контексте JPA / Hibernate?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Проблема N+1** — это проблема производительности в фреймворках ORM, таких как JPA и Hibernate. Она возникает, когда приложение выполняет **один запрос** для получения списка записей, а затем **N дополнительных запросов** для загрузки связанных данных для каждой записи. Это приводит к выполнению **1 + N запросов** вместо одного эффективного запроса, что может замедлить производительность приложения при больших объёмах данных. |

### Какие существуют базовые подходы к решению проблемы N+1 в JPA / Hibernate?

| Под темы                             | Заметка                                                                                                                                                                                                                                                                                                                               |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Изменение стратегии загрузки (fetch) | **Использовать FetchType.EAGER** для ассоциаций, которые по умолчанию используют FetchType.LAZY. Это заставляет Hibernate загружать связанные сущности заранее, что снижает количество запросов.                                                                                                                                      |
|                                      | **Указывать FetchType.LAZY** для ассоциаций, которые не должны загружаться по умолчанию. Например, в ассоциациях @ManyToOne и @OneToOne по умолчанию используется FetchType.EAGER, но это может вызывать проблему N+1.                                                                                                                |
| Использование join fetch             | **Явно заставлять Hibernate подгрузить связанную коллекцию в рамках основного запроса** с помощью ключевого слова JOIN FETCH в JPQL или HQL. Например, в запросе: SELECT a FROM Author a JOIN FETCH a.books — Hibernate выполняет один SQL-запрос с объединением таблиц автора и книг, вместо отдельного запроса на каждую коллекцию. |
|                                      | **Использовать JOIN FETCH глобально** для коллекций, которые часто лениво загружаются, — это страховка. Например, @BatchSize(size = 20) на всех @OneToMany в важных сущностях.                                                                                                                                                        |
| Аннотирование запросов               | **Использовать аннотацию @BatchSize** для пакетной выборки. Например, в запросе: @OneToMany(mappedBy = "user") @BatchSize(size = 10) private List<Order> orders; — Hibernate группирует несколько запросов вместе, что снижает количество обращений к базе данных.                                                                    |
|                                      | **Указывать @Fetch(FetchMode.SUBSELECT)** — Hibernate использует один запрос для всех сущностей и другой — для всех дочерних коллекций, избегая N+1.                                                                                                                                                                                  |
|                                      | **Использовать EntityGraph** (как через аннотации, так и программно) — позволяет селективно загружать отношения во время выполнения.                                                                                                                                                                                                  |
| Профилирование                       | **Обнаружить проблему N+1 помогает логирование запросов** (Hibernate show_sql + format_sql + BasicBinder TRACE для параметров) или профилирование на уровне БД. Если виден шаблон повторяющихся запросов, это сигнал применить один из подходов.                                                                                      |
|                                      | **Использовать инструменты для профилирования** — например, встроенную статистику Hibernate или инструменты APM (Application Performance Monitoring) (например, NewRelic, AppDynamics) — чтобы выявить чрезмерное поведение запросов.                                                                                                 |

### Может ли использование ленивой загрузки (lazy loading) привести к проблеме N+1? Если да, то как?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                           |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ          | **Использование ленивой загрузки (lazy loading) в ORM (Object-Relational Mapping) может привести к проблеме N+1**. Это происходит, когда приложение делает один запрос для получения основных сущностей, а затем дополнительный запрос для каждой сущности для загрузки связанных данных. Если есть N сущностей, это приводит к **N+1 запросам**. |
| Принцип работы | Ленивая загрузка откладывает получение связанных данных до момента, когда к ним действительно обращаются. Например, если получен объект Hippo, а связанные Meal загрузятся только тогда, когда вызовут hippo.getMeals(), это порождает множество запросов — по одному на каждый элемент коллекции.                                                |

### Как использование JPQL/HQL запросов может помочь в решении проблемы N+1?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| JPQL     | **Использование JOIN FETCH**. С помощью ключевого слова JOIN FETCH Hibernate явно указывает подгрузить связанную коллекцию в рамках основного запроса. Например, если есть сущности Hippo (бегемот) и Meal (обед), связанные через @OneToMany, вместо ленивого выполнения запросов.                    |
|          | **Использование аннотации @BatchSize**. Если невозможно или неудобно использовать JOIN FETCH, аннотация @BatchSize позволяет Hibernate загружать данные пакетами, а не по одному. Например, если запросят 50 бегемотов, Hibernate выполнит не 50 отдельных запросов, а 5 запросов по 10 обедов каждый. |
|          | **Использование графов сущностей (Entity Graphs)**. Графы сущностей позволяют гибко указывать, какие связи нужно загрузить, прямо на уровне запросов или аннотаций.                                                                                                                                    |
| HQL      | **Использование JOIN FETCH**. Например, если нужно получить список всех пользователей с их заказами.                                                                                                                                                                                                   |

### Какие аннотации в JPA могут быть использованы для оптимизации и предотвращения проблемы N+1?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| EntityGraph | **@EntityGraph**. Позволяет явно указывать стратегию загрузки данных во время выполнения запроса. Например, можно добавить @EntityGraph к методу репозитория, чтобы указать, что связанные сущности должны загружаться с помощью JOIN при выполнении этого метода. Это помогает избежать проблемы N+1. |
|             | **Есть два типа EntityGraph**: Fetch и Load, которые определяют, следует ли лениво или с нетерпением извлекать сущности, не указанные в атрибутивных узлах.                                                                                                                                            |
| BatchSize   | **@BatchSize**. Позволяет контролировать количество связанных сущностей, загружаемых в одном запросе, при ленивой загрузке коллекций или отношений. Это помогает снизить количество запросов, отправленных в базу данных, и улучшить эффективность процесса извлечения данных.                         |
| Query       | **@Query**. Позволяет явно указывать JPQL-запрос, например, использовать JOIN FETCH в запросе. Это instructs JPA загружать связанные сущности в одном запросе, избегая ленивой загрузки.                                                                                                               |

### Как роль графа сущностей (entity graph) связана с решением проблемы N+1?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                          |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | Entity Graph помогает решить проблему N+1, позволяя заранее определить, какие ассоциации и поля сущности должны быть загружены вместе с основной записью.                                                                                                                                                                                                                                                                        |
|          | **Проблема N+1** возникает, когда при загрузке сущностей из базы данных автоматически генерируется большое количество дополнительных SQL-запросов, что снижает производительность системы. Например, если выбрать список заказов, и каждый заказ ссылается на клиента, то без правильной настройки будет выполнен один запрос на заказы и по одному запросу на каждого клиента — итого N+1 запросов, где N — количество заказов. |
|          | **Entity Graph позволяет объединить эти операции в один эффективный JOIN-запрос**. По данным исследования Baeldung в 2024 году, использование Entity Graph позволило снизить количество SQL-запросов на 40–70% в типовых CRUD-операциях, а время отклика API — в среднем на 55%.                                                                                                                                                 |

### Как можно решить проблему N+1 при использовании Criteria API?

| Под темы               | Заметка                                                                                                                                                                                                                                                                                |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Настройка конфигурации | **Использовать жадную загрузку (Eager Loading)**. Позволяет извлекать все необходимые данные в одном запросе, включая связанные дочерние сущности. Это можно сделать с помощью аннотации @ManyToOne(fetch = FetchType.EAGER) или @OneToMany(fetch = FetchType.EAGER).                  |
|                        | **Использовать явную загрузку (Explicit Loading)**. Позволяет загружать связанные дочерние сущности по требованию, когда они действительно нужны. Это можно сделать с помощью метода Hibernate.initialize(entity.getChildEntities()) или с использованием метода fetch в HQL-запросах. |
|                        | **Использовать пакетную загрузку (Batch Loading)**. Позволяет выполнить несколько запросов одновременно для извлечения связанных дочерних сущностей. Это можно сделать с помощью настройки параметров пакетной загрузки в файле конфигурации Hibernate.                                |
|                        | **Использовать кэширование (Caching)**. Позволяет сохранять уже извлечённые данные в памяти, чтобы избежать повторных запросов к базе данных. Hibernate предоставляет различные уровни кэширования, такие как уровень сессии, уровень второго уровня и уровень запроса.                |
| Код запроса            | Использовать предложение JOIN FETCH в запросах Criteria API. Это позволяет избежать проблемы N+1, так как Hibernate выполняет один SQL-запрос для извлечения всех данных. Например, в запросе Criteria API можно указать: criteria.setFetchMode("roles", FetchMode.EAGER).             |
|                        | Использовать режим FetchMode.SUBSELECT для связанных коллекций. При доступе к неинициализированной коллекции этот режим запускает загрузку всех элементов всех коллекций одной роли для всех владельцев, связанных с контекстом persistence, с помощью одного вторичного запроса.      |
| Тесты                  | **Использовать инструменты для автоматического обнаружения проблемы N+1**. Например, проект Hypersistence Utils с открытым исходным кодом позволяет автоматически обнаруживать проблему N+1 на уровне доступа к данным.                                                                |
|                        | **Проводить тесты, проверяющие, насколько эффективно приложение использует Criteria API**. Например, можно использовать утилиту AssertHibernateSQLCount для утверждения базовых операторов SQL, которые генерируются.                                                                  |

### Какие есть подходы к программному управлению инициализацией связанных сущностей для избежания проблемы N+1?

| Под темы | Заметка                                                                                                                                                                            |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Подходы  | **Стратегия JOIN**. Объединяет родительскую и дочернюю таблицы с помощью оператора join, что позволяет избежать выполнения дополнительных запросов для получения связанных данных. |
|          | **Режим SELECT с пакетной выборкой зависимых коллекций**. Зависимые коллекции загружаются не одиночными запросами, а пакетно. Размер пакета можно настроить вручную.               |
|          | **Жадная загрузка (eager loading)**. Позволяет загрузить связанные данные одним запросом, что сокращает число SQL-запросов и повышает производительность.                          |
|          | **Стратегия, при которой для каждого отдельного метода устанавливается целевое количество вызовов репозитория**. Любое превышение этого количества будет считаться проблемой N+1.  |
|          | Также для оптимизации работы с базами данных могут использоваться **индексы**, которые позволяют ускорить поиск данных, сортировку и фильтрацию.                                   |

### Как использование проекций (DTO) может способствовать решению проблемы N+1?

| Под темы       | Заметка                                                                                                                                                                                                                               |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ          | **Использование проекций (DTO) может способствовать решению проблемы N+1** в ORM (например, Hibernate, JPA) **за счёт выборочной загрузки данных**. Это позволяет:                                                                    |
|                | - **Избегать большого количества запросов** для загрузки связанных данных: вместо того, чтобы загружать 100 пользователей и потом лениво тянуть их 1000 заказов, написать один запрос, который сразу вернёт нужную комбинацию данных. |
|                | - **Использовать проекции для специализированных выборок** — например, отчётов, отображения на дашборде агрегированных данных, экспорта списков, где не требуется дальнейшее взаимодействие с БД через сессионные объекты.            |
| Принцип работы | **Выбирать только необходимые данные** вместо загрузки всей сущности.                                                                                                                                                                 |
|                | **Обрабатывать результаты запроса** — вместо того, чтобы сопоставлять каждую строку с Object, провайдер persistence (например, Hibernate) вызывает конструктор DTO, чтобы создать новый объект.                                       |
|                | **Учитывать, что DTO должен иметь конструктор**, который соответствует столбцам, выбранным запросом.                                                                                                                                  |
