### Что такое план запроса в контексте SQL?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                              |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение | **План запроса (план выполнения запроса) в контексте SQL** — это детализированное описание последовательности операций, которые система управления базами данных (СУБД) выполняет для получения результата SQL-запроса. План показывает, как данные извлекаются из таблиц, какие индексы используются, как выполняются соединения и сортировки, и другие аспекты выполнения запроса. |
|             | Анализ плана запроса — инструмент для оптимизации производительности SQL-запросов. Оптимально спроектированный план может сократить время выполнения запроса и использование ресурсов базы данных.                                                                                                                                                                                   |

### Как можно просмотреть план запроса в SQL?

| Под темы   | Заметка                                                                                                                                                                                                                                                                              |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| SQL Server | **Предполагаемый план** (Display Estimated Execution Plan) — открывается в SSMS на панели инструментов. Запрос фактически не выполняется, план строится без выполнения.                                                                                                              |
|            | **Фактический план** (Include Actual Execution Plan) — отображается на вкладке «План выполнения» на панели результатов после выполнения запроса. План содержит фактические метрики использования ресурсов.                                                                           |
|            | **Динамический план** (Live Query Statistics) — создаётся в режиме реального времени, доступен во время выполнения запроса. Отображает общий ход выполнения и текущую статистику на уровне оператора, например, число полученных строк, затраченное время.                           |
| MySQL      | **Команда EXPLAIN** — показывает план запроса, показывает, какие индексы используются, как таблицы соединяются и какие операции выполняются. Чтобы получить план, нужно добавить ключевое слово EXPLAIN перед SQL-запросом. Например: `EXPLAIN SELECT * FROM users WHERE age > 30;`. |
|            | **Формат вывода EXPLAIN** — план запроса в виде вложенного дерева, каждая вложенная строка означает новый цикл. В скобках указана информация об оценочной стоимости и расчётное количество прочитанных строк.                                                                        |
|            | **С версии 8.0.18** — можно использовать выражение EXPLAIN ANALYZE, которое, помимо плана запроса и оценочных затрат, выводит фактические затраты отдельных итераторов в плане выполнения.                                                                                           |

### Почему важно понимать план запроса при оптимизации запросов?

| Под темы | Заметка                                                                                                                                                                                                                                       |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | Понимать план запроса (план выполнения) важно при оптимизации запросов в базах данных, чтобы выявить потенциальные узкие места и реализовать соответствующие оптимизации. План показывает, как база данных интерпретирует и выполняет запрос. |
| Полезно  | **Выявить дорогостоящие операции** — полные сканирования таблиц, операции сортировки, соединения с большим количеством записей.                                                                                                               |
|          | **Проверить использование индексов** — если индексы не используются, рассмотреть возможность их добавления или изменения структуры запроса.                                                                                                   |
|          | **Сравнить планы** — если есть несколько версий одного и того же запроса, можно сравнить их планы выполнения, чтобы увидеть, какая версия более эффективна.                                                                                   |

### Какие типы операций можно найти в плане запроса?

| Под темы   | Заметка                                                                                                                                                |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| SQL Server | **Операции соединения (JOIN)**: Nested Loop (вложенные циклы), Hash Join (хэш-соединение), Merge Join (соединение слиянием).                           |
|            | **Операции доступа к данным**: Index Seek (поиск конкретных строк), Index Scan (сканирование всего индекса), Table Scan (полное сканирование таблицы). |
|            | **Операции сортировки**: Sort (сортировка входящих строк).                                                                                             |
|            | **Операции параллельной обработки**: Parallelism (деление данных на несколько частей для параллельной обработки).                                      |
| Greenplum  | **Broadcast motion** — каждый сегмент отправляет свои собственные отдельные строки всем остальным сегментам.                                           |
|            | **Redistribute motion** — каждый сегмент перехеширует данные и отправляет строки в соответствующие сегменты в соответствии с хэш-ключом.               |
|            | **Gather motion** — данные результатов со всех сегментов собираются в один поток.                                                                      |
| PostgreSQL | **Seq Scan** — последовательное сканирование таблицы.                                                                                                  |
|            | **Index Scan** — индексное сканирование.                                                                                                               |
|            | **Bitmap Index Scan** — сканирование с построением битовых карт.                                                                                       |
|            | **Merge Join** — соединение слиянием.                                                                                                                  |
|            | **Nested Loop** — соединение с вложенным циклом.                                                                                                       |
|            | **Hash Join** — соединение по хэшу.                                                                                                                    |
|            | **Sort** — операция сортировки по столбцам.                                                                                                            |
|            | **Aggregate** — вычисление агрегатных функций (COUNT, SUM, AVG, MAX, MIN и прочих).                                                                    |

### Как индексы влияют на план запроса?

| Под темы | Заметка                                                                                                                                                                                               |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Первичные индексы** — создаются автоматически для каждой таблицы, в которой объявлен PRIMARY KEY. Гарантируют уникальность значений и позволяют быстро находить записи по основному идентификатору. |
|          | **Уникальные индексы** — предотвращают дублирование данных, например, для столбцов, в которых должны храниться уникальные значения.                                                                   |
|          | **Составные индексы** — создаются для нескольких столбцов одновременно, полезны для ускорения запросов, которые используют фильтры по двум и более полям.                                             |

### Какие факторы могут влиять на выбор оптимизатора при составлении плана запроса?

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Характеристики запроса | **Сложность запроса**. Сложные запросы с большим числом подзапросов, операций группировки и фильтрации требуют значительных вычислительных ресурсов. Оптимизатор может выбрать неоптимальный план.                                                                                                                                                                                                                                                                  |
|                        | **Порядок соединения таблиц**. Оптимизатор принимает решение на основе количества строк в выборке каждой таблицы, степени сложности выборки строк и плотности данных в таблицах. В некоторых СУБД можно использовать подсказки для оптимизатора (hints), в которых можно приказать соединять таблицы в определённом порядке.                                                                                                                                        |
|                        | **Использование индексов**. Оптимизатор решает, использовать индекс или нет, учитывая селективность выражения (отношение количества выбираемых строк к общему количеству строк в таблице). Высокоселективные индексы позволяют быстро находить нужные строки, так как сокращают количество данных для сканирования.                                                                                                                                                 |
| Структура базы данных  | **Наличие индексов**. Если в структуре базы данных отсутствует индекс, удовлетворяющий условиям запроса, оптимизатор будет вынужден сканировать таблицу или один из её индексов. Это увеличивает время выполнения запроса.                                                                                                                                                                                                                                          |
|                        | **Расположение данных**. Оптимизатор предпочитает выполнять как можно большую часть работы на сегментах кластера и минимизировать объём данных, которые необходимо передать между сегментами для выполнения запроса.                                                                                                                                                                                                                                                |
| Параметры конфигурации | **Настройки памяти, параметров кэширования, параллелизма, размеров буферов**. Неправильные настройки снижают эффективность использования ресурсов. Например, в PostgreSQL оптимизатор учитывает размер пулов буферов, заданный при их создании или изменении, — это влияет на оценку стоимости использования индекса.                                                                                                                                               |
|                        | **Скорость процессора**. Учитывается оптимизатором при оценке стоимости выполнения некоторых операций. Оптимальный план доступа для запроса выбирается исходя из затрат процессорного времени и затрат на ввод-вывод.                                                                                                                                                                                                                                               |
| Статистика             | **Статистика таблиц и индексов**. Оптимизатор анализирует статистику, чтобы определить, какой индекс использовать, в каком порядке соединять таблицы, использовать ли параллельное выполнение. Например, селективность — мера уникальности значений в столбце или комбинации столбцов — влияет на производительность индексов. Если индекс имеет низкую селективность, оптимизатор может решить, что сканирование таблицы будет быстрее, чем использование индекса. |
|                        | **Устаревшие статистические данные**. Если статистика устарела или неполная, выбор плана может быть ошибочным.                                                                                                                                                                                                                                                                                                                                                      |

### Какие существуют методы оценки стоимости выполнения запроса?

| Под темы                         | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| -------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| В информационных системах (СУБД) | **Стоимостный подход**. Оптимизатор генерирует множество планов выполнения, основываясь на возможных порядках объединения, операциях объединения и доступных методах доступа. Затем оптимизатор оценивает стоимость каждого плана и выбирает план с наименьшей стоимостью. Стоимость рассчитывается на основе **стоимостной модели** — набора аналитических формул для расчёта стоимости физических операций. Для каждой операции вычисляется как стоимость операций ввода-вывода с внешними устройствами, так и стоимость выполнения процессорных инструкций. Сумма стоимостей всех физических операций оптимального плана — общая стоимость выполнения запроса. |
| В бизнес-процессах               | **Анализ затрат на выполнение операций**. Время выполнения операции умножают на стоимость единицы рабочего времени для данной должности или группы должностей. Однако эта оценка не даёт полной картины стоимости бизнес-процесса — необходимо проанализировать другие категории затрат: аренду, расходные материалы, затраты на автоматизацию и т. д..                                                                                                                                                                                                                                                                                                           |
|                                  | **Расчёт стоимости бизнес-процесса**. Для каждой операции в бизнес-процессе известен норматив времени, коэффициент использования операции в процессе, исполнитель и стоимость единицы рабочего времени исполнителя. Стоимость операции рассчитывают, умножая стоимость единицы рабочего времени на длительность выполнения операции. Суммирование стоимости всех операций показывает, во сколько обходится компании весь бизнес-процесс.                                                                                                                                                                                                                          |

### Как можно влиять на выбор плана запроса без изменения самого запроса?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                        |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Использовать указания хранилища запросов**. Этот метод позволяет влиять на планы выполнения запросов без изменения кода приложения или объектов базы данных. Подсказки Query Store можно добавлять и удалять для пакетов запросов, чтобы настроить производительность.                                       |
|          | **Изменить веса низкоуровневых операций**. Можно попробовать убрать из поля зрения текущий план, который планировщик считает оптимальным, чтобы оптимальным стал какой-нибудь из ближайших его конкурентов. Однако нужно помнить, что, улучшив производительность в одном запросе, можно ухудшить её в другом. |
|          | **Использовать расширения базы данных**. Например, SR_PLAN позволяет сохранить рассчитанный план и принудительно использовать его в дальнейшем. Это помогает сделать поведение БД более предсказуемым в смысле выбора плана.                                                                                   |
|          | **Использовать Adaptive Query Optimizer**. Это расширение реализует обратную связь планировщику от реального времени выполнения запроса. Планировщик измеряет фактические результаты выполненного запроса и корректирует свои планы в будущем с учётом этого.                                                  |

### Как производится оптимизация запросов на основе плана запроса?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Оптимизация запросов на основе плана запроса** — это процесс поиска наиболее эффективного способа выполнения запроса в системе управления базами данных (СУБД). План запроса — это последовательность операций, которые выполняет БД для получения требуемого результата. Один и тот же результат может быть получен различными способами, которые отличаются по затратам ресурсов и времени выполнения. Задача оптимизации — найти оптимальный план. |
| Алгоритм    | **Генерация логических планов** — оптимизатор устраняет избыточные условия, подставляет представления, преобразует предикаты, исключает бесполезные соединения.                                                                                                                                                                                                                                                                                         |
|             | **Генерация физических планов** — для каждого узла дерева (например, операции соединения) рассматриваются все доступные алгоритмы его реализации (Nested Loop Join, Hash Join, Merge Join).                                                                                                                                                                                                                                                             |
|             | **Оценка стоимости** — на основе статистики о данных вычисляется предполагаемая стоимость каждого варианта.                                                                                                                                                                                                                                                                                                                                             |
|             | **Выбор оптимального плана** — план с наименьшей совокупной расчётной стоимостью выбирается для последующего выполнения.                                                                                                                                                                                                                                                                                                                                |
