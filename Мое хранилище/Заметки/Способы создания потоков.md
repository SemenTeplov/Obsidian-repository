### Какой класс в Java используется для создания нового потока?

| Под темы                       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Наследование от класса Thread  | **Пошаговый процесс**:<br>1. Создать класс, расширяющий Thread.<br>2. Переопределить метод run() — здесь размещается код, который будет выполняться в отдельном потоке.<br>3. Создать экземпляр этого класса.<br>4. Вызвать метод start() для запуска потока.<br>**Особенности**:<br>- Java не поддерживает множественное наследование — если класс наследует Thread, он не может наследовать другие классы.<br>- Если вызвать run() напрямую, код выполнится в текущем потоке, а не в новом. Метод start() создаёт новый поток и затем вызывает run()  уже в контексте этого нового потока. |
| Реализация интерфейса Runnable | **Интерфейс Runnable** содержит метод run(), который нужно реализовать, а затем передать экземпляр этого класса в конструктор Thread. **Особенности**:<br>- Этот подход отделяет поведение потока от класса Thread, что способствует лучшей организации кода и его повторному использованию.<br>- Интерфейс Runnable позволяет разделить задачу (логику потока) от средств выполнения (самого потока).                                                                                                                                                                                       |

### Каким образом можно создать поток, используя интерфейс Runnable?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Алгоритм | Для создания потока с помощью Runnable нужно выполнить следующие шаги:<br>1. **Создать класс**, реализующий интерфейс Runnable.<br>2. **Реализовать метод run** — в нём определяется код, который выполняется при запуске потока.<br>3. **Создать экземпляр** этого класса.<br>4. **Передать экземпляр** в конструктор Thread.<br>5. **Вызвать метод start** у экземпляра Thread.<br>**Важно**:<br>- **start()** — метод класса Thread, который создаёт новый поток и вызывает метод run() в этом новом потоке. Вызов start() инициирует параллельное выполнение потока.<br>- **run()** — метод, содержащий код, который будет выполнен в потоке. Если вызвать run() напрямую, без вызова start(), код будет выполнен в текущем потоке, как обычный метод, и новый поток не будет создан. |

### Чем отличается создание потока с использованием класса Thread от создания с использованием интерфейса Runnable?

| Под темы           | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Класс Thread       | **Класс java.lang.Thread** — это основной инструмент для создания и управления потоками. Сами экземпляры класса не являются фактическими потоками, а скорее предоставляют интерфейс для управления потоками.<br>**Создание потока с использованием класса Thread** предполагает наследование от класса Thread и переопределение метода run(). Этот способ позволяет определить поведение потока непосредственно в подклассе.<br>**Особенности**:<br>- Простота и наглядность кода — всё в одном классе.<br>- Прямой доступ к методам Thread, таким как getName(), getPriority(), isAlive().<br>- Подходит для простых задач, где не требуется наследование от других классов.<br>**Ограничение**: Java не поддерживает множественное наследование — если класс наследует Thread, он не может наследовать другие классы.                                                                                                                                                                           |
| Интерфейс Runnable | **Интерфейс Runnable** определяет единственный метод run(). Объекты, реализующие этот интерфейс, могут выполняться потоком.<br>**Создание потока с использованием интерфейса Runnable** предполагает реализацию интерфейса Runnable, создание экземпляра этого класса и передачу экземпляра в конструктор Thread. Этот подход отделяет поведение потока от класса Thread, что способствует лучшей организации кода и его повторному использованию.<br>**Особенности**:<br>- Позволяет разделить задачу (логику потока) от средств выполнения (самого потока).<br>- Даёт больше гибкости: класс, реализующий Runnable, не обязательно должен быть просто потоком — его можно запустить в потоке, передать какой-либо службе-исполнителю или передать куда-то как задачу в однопоточном приложении.<br>- Классы, реализующие интерфейс Runnable, могут быть повторно использованы в различных контекстах, таких как исполнители (Executor), без привязки к специфическим деталям реализации потока. |

### В чем преимущество использования интерфейса Callable по сравнению с Runnable при создании потоков?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Интерфейс Callable позволяет задачам возвращать результат и явно обрабатывать исключения**, что делает его более гибким по сравнению с интерфейсом Runnable.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Runnable    | **Runnable** — простейший способ описания задачи для выполнения в отдельном потоке. Он содержит единственный метод run(), который не принимает аргументов и ничего не возвращает. Некоторые особенности Runnable:<br>- **Разделение задачи и механизма её выполнения** (соответствие принципу единственной ответственности).<br>- **Возможность наследовать другие классы**, кроме Thread.<br>- **Повторное использование логики** в разных контекстах.<br>- **Возможность передавать одну и ту же задачу нескольким потокам**.<br>- **Лучшая совместимость** с современными API (ExecutorService, ThreadPool).<br>**Преимущества использования Runnable**:<br>- простота использования и понимание;<br>- прямая совместимость с классом Thread;<br>- меньший накладочный расход при выполнении.                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Callable    | **Callable** был добавлен в Java 5 вместе с пакетом java.util.concurrent. Он предоставляет более гибкий механизм, позволяя задаче возвращать результат и явно обрабатывать исключения. Некоторые преимущества использования Callable:<br>- **Возвращаемый результат** — задачи могут возвращать результат своего выполнения, что полезно для передачи данных между потоками.<br>- **Обработка исключений** — Callable позволяет задачам бросать проверяемые исключения, которые могут возникнуть во время их выполнения, что облегчает обработку ошибок в фоновых задачах.<br>- **Нативная интеграция с Future API** для контроля выполнения.<br>- **Поддержка таймаутов и отмены выполнения**.<br>**Выбор между Runnable и Callable** зависит от требований конкретной задачи:<br>- **Для простых задач без возврата результата** — Runnable остаётся более компактным и понятным выбором.<br>- **Если необходимо получить результат выполнения** или важна гибкая обработка исключений — Callable будет оптимальным решением.<br>**Важно**: если задача, реализующая Callable, бросает проверяемое исключение, оно будет обернуто в ExecutionException, когда попытается получить результат выполнения задачи с помощью Future. |

### Как использовать пул потоков для управления множеством потоков в Java?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **Пул потоков в Java используется для управления множеством потоков, переиспользуя существующие потоки** вместо создания новых для каждой задачи. Это устраняет накладные расходы на создание и удаление потоков, что может быть дорогостоящей процедурой.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| Принцип работы | Пул потоков содержит **фиксированное или переменное количество рабочих потоков**. Эти потоки создаются и запускаются при инициализации пула.<br>**Задачи отправляются в очередь задач**. Рабочие потоки постоянно отслеживают эту очередь на наличие ожидающих задач.<br>**Пул назначает задачи доступным рабочим потокам**. Если доступных потоков нет, задачи ждут в очереди, пока поток не освободится.<br>**После завершения задачи** поток не завершается, а возвращается обратно в пул для обслуживания следующего запроса.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Реализация     | Для работы с пулами потоков в Java используется **платформа Executor**. Она предоставляет набор интерфейсов и классов для одновременного выполнения задач с использованием пулов потоков.<br>**Некоторые методы класса Executors для создания пулов**:<br>- **Executors.newCachedThreadPool** — создаёт новые потоки по мере необходимости и повторно использует ранее созданные потоки, когда они становятся доступны. Бездействующие потоки сохраняются в течение 60 секунд.<br>- **Executors.newFixedThreadPool** — создаёт пул из фиксированного количества потоков; бездействующие потоки сохраняются неограниченное время. Если число отправленных задач превышает число бездействующих потоков, необработанные задачи помещаются в очередь, они запускаются после завершения других задач.<br>- **Executors.newSingleThreadExecutor** — создаёт объект Executor (фактически пул из одного потока) с одним рабочим потоком, который последовательно выполняет отправленные задачи.<br>- **Executors.newVirtualThreadPerTaskExecutor** — создаёт объект Executor, который запускает каждую задачу в новом виртуальном потоке. |

### Как реализовать паттерн Producer-Consumer с использованием многопоточности в Java?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Паттерн Producer-Consumer (Производитель-Потребитель) можно реализовать с использованием многопоточности в Java**. Для этого используют **блокирующие очереди (Blocking Queues)**, которые безопасны для потоков. Несколько потоков могут добавлять и удалять сообщения из такой очереди без проблем с параллельностью.<br>Популярные реализации:<br>- **ArrayBlockingQueue** — очередь фиксированного размера на массиве, честная (FIFO).<br>- **LinkedBlockingQueue** — очередь на связном списке, может быть безлимитной, имеет два лока (отдельно на take, отдельно на put).                                                                                                                                                                                                                                                                                                                                                                                              |
| Алгоритм    | Некоторые особенности реализации:<br>- **Создание потоков**. Производитель и потребитель запускаются в отдельных потоках. Для создания потоков можно использовать интерфейс Runnable.<br>- **Использование методов очереди**. Производитель может безопасно размещать элементы в очереди с помощью метода put. Потребитель может безопасно читать элементы из очереди с помощью метода take или poll. Метод take — блокирующий, поток ждёт, пока в очереди не будет элемента. Метод poll позволяет указать время ожидания: если в очереди нет элементов, метод возвращает null.<br>- **Управление процессом**. Можно создать несколько потоков-производителей и потребителей, запустить их и gracefully остановить процесс, вызвав метод stop() у объектов производителя и потребителя.<br>**Важно**: поведение многопоточной программы не определено, нет гарантии, что последовательность выполнения потоков будет следующей: producer, consumer, producer, consumer и т. д.. |

### Какие механизмы синхронизации потоков можно использовать при реализации многопоточных приложений?

| Под темы            | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Мьютексы            | **Мьютекс** (от англ. mutual exclusion — «взаимное исключение») — **примитив синхронизации, который обеспечивает исключительный доступ к общему ресурсу для одного потока**.<br>**Особенности**:<br>- Мьютекс — переменная, которая может принимать два значения: 1 (владелец) и 0 (свободен).<br>- Когда поток получает доступ к ресурсу, он должен сначала захватить мьютекс (установит значение в 1). Пока мьютекс захвачен, другие потоки не могут получить доступ к ресурсу.<br>- Когда поток заканчивает работу с ресурсом, он должен освободить мьютекс (установит значение в 0).                                                                                                                                                                                                                                                                                                       |
|                     | **Особенности**:<br>- Мьютексы блокирующие — поток, который пытается захватить захваченный мьютекс, блокируется до тех пор, пока мьютекс не будет освобождён.<br>- Чрезмерное использование мьютексов может привести к снижению производительности, поэтому рекомендуется минимизировать критические секции и избегать взаимных блокировок.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Условные переменные | **Условная переменная** — **механизм синхронизации, который позволяет потокам ожидать, пока некоторое условие не станет истинным**.<br>**Особенности**:<br>- Когда условие не выполняется, поток, ожидающий его, переходит в состояние ожидания, освобождая процессорное время.<br>- Другой поток, который изменяет состояние, может «разбудить» ожидающий поток, когда условие станет истинным.<br>**Особенности**:<br>- Условные переменные всегда используются с мьютексом — мьютекс защищает доступ к данным, которые проверяются в условии.<br>- Для ожидания (wait) поток, который должен ждать условия, вызывает функцию ожидания, которая атомарно освобождает мьютекс и переводит поток в состояние ожидания.<br>- Для сигнализации (signal/broadcast) другой поток, который изменяет условие, вызывает функцию сигнализации, чтобы «разбудить» один или несколько ожидающих потоков. |
| Атомарные операции  | **Атомарная операция** — **неделимая единица выполнения**, которая либо выполняется целиком, либо не выполняется вообще, без промежуточных состояний. В контексте многопоточности это означает, что ни один поток не может наблюдать частично выполненную атомарную операцию.<br>**Механизмы реализации**:<br>- В разных языках программирования есть специализированные классы для атомарных операций, например, AtomicInteger, AtomicLong, AtomicBoolean в Java.<br>- Частным случаем атомарных операций являются read-modify-write-операции: compare-and-swap, test-and-set, fetch-and-add.                                                                                                                                                                                                                                                                                                 |
| Барьеры             | **Барьер** — **участок кода, в котором синхронизируется состояние потоков**. Например: <br>- Если для функции в главном потоке требуется, чтобы все дочерние потоки закончили свою работу, можно поставить барьер перед ней. Тогда она будет ждать завершения работы дочерних потоков, после чего все потоки продолжат свою работу.<br>- Примером реализации барьера может быть критическая секция, код которой разрешается выполняться только последнему потоку, запросившему выполнение. Остальные потоки должны ожидать его.<br>**Важно**: избыточная синхронизация или неэффективное распределение потоков могут снизить производительность, вызывая дополнительные накладные расходы, связанные с переключением контекста и ожиданием блокировок. Поэтому при проектировании многопоточных приложений рекомендуется следовать принципу минимальной достаточной синхронизации.             |

### Какие существуют способы управления состоянием потоков в Java и как они влияют на производительность приложения?

| Под темы            | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Синхронизация       | **Ключевое слово synchronized** гарантирует, что только один поток может выполнять определённый блок кода в один момент времени. Это предотвращает ошибки, связанные с параллельным доступом к общим ресурсам (переменным, объектам).<br>**Особенности синхронизации**:<br>- Получение и освобождение блокировок требует дополнительных ресурсов CPU.<br>- Потоки, ожидающие блокировку, приостанавливаются, что вызывает переключение контекста.<br>- При высокой конкуренции за блокировку возрастают накладные расходы.<br>**Важно**: synchronized не является серебряной пулей — он создаёт накладные расходы и может стать источником проблем, таких как взаимная блокировка (deadlock), если используется неправильно. Для оптимизации многопоточных приложений Java предлагает несколько альтернатив синхронизации, предоставляющих различный баланс между безопасностью и производительностью.                                                                                                                                                                 |
| Блокировки          | **Интерфейс Lock** предоставляет больше функциональности для управления блокировками, чем synchronized. Некоторые особенности Lock:<br>- Ожидание блокировки с таймаутом — блокировка может быть получена за заданный промежуток времени.<br>- Прерываемое ожидание блокировки — поток может быть прерван во время ожидания блокировки, что позволяет адекватно реагировать на запросы о завершении работы.<br>- Получение и освобождение блокировки в разных местах кода — Lock даёт возможность более гибкого управления блокировками по сравнению с синхронизированными блоками.<br>**Важно**: блокировку, полученную через Lock, нужно освободить в блоке finally, чтобы свести к минимуму риск взаимных блокировок из-за неожиданных исключений.                                                                                                                                                                                                                                                                                                                  |
| Распределения задач | **Многопоточность** позволяет выполнять несколько задач одновременно, что повышает производительность. Однако **чрезмерное увеличение числа потоков** может снизить производительность, так как затраты на управление потоками начинают превышать выгоды от их параллельного выполнения.<br>**Некоторые стратегии распределения задач**:<br>- **Создание пула потоков** — задачи передаются в пул, и пул назначает их доступным рабочим потокам. Если доступных потоков нет, задачи ждут в очереди, пока поток не освободится.<br>- **Разбиение задачи на подзадачи** — каждая подзадача отправляется в отдельный поток для выполнения, главный поток ожидает, пока все подзадачи завершатся.<br>- **Work dealing** — у каждого потока своя очередь, когда поток понимает, что ему тяжело, он просит других забрать у него часть задач.<br>**Важно**: не все задачи хорошо распараллеливаются, и накладные расходы на создание и управление потоками могут превысить выигрыш от параллельности.                                                                        |
| Профилирование      | **Профилирование** помогает отслеживать и анализировать поведение приложения во время выполнения, выявляя потенциальные узкие места и неэффективность. **Профилирование потоков** позволяет выявить, какие участки кода занимают больше всего времени на выполнение в каждом потоке. <br>**Некоторые инструменты профилирования**:<br>- **JProfiler** — позволяет анализировать производительность Java-приложений в режиме реального времени.<br>- **VisualVM** — инструмент профилирования, который поставляется вместе с JDK, позволяет анализировать производительность локальных и удалённых приложений.<br>- **YourKit** — инструмент профилирования, обеспечивает множество функций для оптимизации производительности, включая анализ потребления памяти и профилирование SQL-запросов.<br>**Анализ результатов профилирования** помогает выявить узкие места в производительности и оптимизировать программу. Например, если в профиле есть широкие блоки или высокие проценты, это указывает на горячие точки, которые потребляют большую часть времени CPU. |
