### Что такое протокол HTTP?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                              |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **HTTP (HyperText Transfer Protocol)** — **протокол прикладного уровня**, разработанный для обмена данными между клиентом (например, веб-браузером) и сервером в сети Интернет. Используется для передачи HTML-страниц, изображений, видео, текстов и других типов данных, составляющих веб-ресурсы. |
|             | HTTP имеет клиент-серверную архитектуру: клиент отправляет запросы серверу, а сервер отвечает на них.                                                                                                                                                                                                |

### Каковы основные методы запросов HTTP?

| Под темы    | Заметка                                                                                                                                                                                                                                       |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| GET         | **Используется для запроса данных с сервера**. Параметры передаются через строку запроса в URL.                                                                                                                                               |
| Особенности | Ответы на GET-запросы могут быть кешированы, что улучшает производительность и уменьшает нагрузку на сервер.                                                                                                                                  |
|             | GET-запросы нельзя использовать для передачи конфиденциальных данных, так как параметры передаются в URL и могут быть видны в истории браузера или логах сервера.                                                                             |
|             | GET имеет ограничение по длине URL, обычно не более 2048 символов.                                                                                                                                                                            |
| POST        | **Используется для отправки данных на сервер**. В отличие от GET, данные передаются в теле запроса.                                                                                                                                           |
| Особенности | POST-запросы не имеют ограничений на объём передаваемых данных, что позволяет отправлять большие формы и файлы.                                                                                                                               |
|             | Они более безопасны для передачи конфиденциальной информации, так как данные передаются в теле запроса и не отображаются в URL.                                                                                                               |
| PUT         | **Предназначен для полного замещения ресурса по указанному URI**. Клиент передаёт серверу новое представление ресурса, а сервер либо создаёт новый ресурс (если URI не существовал), либо полностью заменяет старый (если URI уже был занят). |
| Особенности | Повторные идентичные PUT-запросы к одному URI дают тот же результат, что и первый успешный запрос.                                                                                                                                            |
| DELETE      | **Применяется для удаления ресурсов с сервера**. Это тоже идемпотентный метод: повторные запросы дадут один и тот же результат — ресурс или будет удалён, или будет получено подтверждение, что ресурс отсутствует.                           |
| Особенности | Обычно в запросе DELETE есть URI, который идентифицирует ресурс, который нужно удалить.                                                                                                                                                       |
|             | Сервер обрабатывает запрос и удаляет указанный ресурс, если он существует, возвращая код состояния, который указывает на успех или ошибку операции.                                                                                           |

### Что такое статус-код в ответе HTTP?

| Под темы    | Заметка                                                                                                                                              |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Статус-код (код состояния) в ответе HTTP** — это стандартизированное трёхзначное число, которое веб-сервер передаёт клиенту в ответ на его запрос. |
|             | **Статус-коды служат индикатором результата обработки запроса**, позволяют быстро определить успешность операции или характер возникшей проблемы.    |
| Категории   | **1xx** — информационный, запрос получен, сервер готов к продолжению.                                                                                |
|             | **2xx** — успешный, запрос получен, понятен и обработан.                                                                                             |
|             | **3xx** — перенаправление, следующие действия нужно выполнить для обработки запроса.                                                                 |
|             | **4xx** — ошибка клиента, запрос содержит ошибки или не отвечает протоколу.                                                                          |
|             | **5xx** — ошибка сервера, сервер не смог обработать запрос, хотя был составлен верно.                                                                |
| Примеры     | 200 OK — запрос получен и успешно обработан;                                                                                                         |
|             | 201 Created — запрос получен и успешно обработан, в результате чего создан новый ресурс или его экземпляр;                                           |
|             | 301 Moved Permanently — запрашиваемый ресурс был перемещён навсегда, и последующие запросы к нему должны происходить по новому адресу;               |
|             | 403 Forbidden — запрос понятен, но нужна авторизация;                                                                                                |
|             | 404 Not Found — сервер не нашёл ресурс по этому адресу;                                                                                              |
|             | 501 Not Implemented — сервер не поддерживает функциональность для ответа на этот запрос;                                                             |
|             | 505 HTTP Version Not Supported — сервер не поддерживает указанную версию HTTP-протокола.                                                             |

### Как работает механизм перенаправления (редиректа) в HTTP?

| Под темы        | Заметка                                                                                                                                                                                                         |
| --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Перенаправление | Пользователь или поисковый робот запрашивает определённый URL.                                                                                                                                                  |
|                 | Сервер получает запрос и определяет, что для данного URL настроен редирект.                                                                                                                                     |
|                 | Сервер отправляет ответ с соответствующим HTTP-кодом и указанием нового URL.                                                                                                                                    |
|                 | Браузер или поисковый робот автоматически выполняет новый запрос по полученному URL.                                                                                                                            |
|                 | В результате пользователь оказывается уже на другой странице, не замечая перехода. Поисковые системы также реагируют на такие сигналы, обновляют свою базу и учитывают перенаправление при ранжировании сайтов. |

### В чем разница между HTTP и HTTPS?

| Под темы    | Заметка                                                                                                                                                                                                                                          |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ       | **Главное отличие между протоколами HTTP (HyperText Transfer Protocol) и HTTPS (HyperText Transfer Protocol Secure) — уровень безопасности**. HTTP передаёт данные в открытом виде, а HTTPS обеспечивает защиту информации с помощью шифрования. |
| HTTP        | **HTTP** — базовый протокол передачи данных в интернете, отвечает за связь между браузером пользователя и сервером, на котором размещён сайт.                                                                                                    |
| Особенности | Работает по принципу «запрос-ответ»: браузер отправляет запрос на сервер, который обрабатывает его и возвращает нужные данные (текст, изображения, видео).                                                                                       |
|             | Не предусматривает механизмов защиты, передаваемый трафик остаётся открытым и может быть перехвачен злоумышленниками.                                                                                                                            |
|             | Подходит для нечувствительных данных — например, статических веб-страниц.                                                                                                                                                                        |
|             | Адрес сайта начинается с «http://».                                                                                                                                                                                                              |
| HTTPS       | **HTTPS** — расширенная версия HTTP, дополненная механизмом шифрования.                                                                                                                                                                          |
| Особенности | Использует SSL (Secure Sockets Layer) или его более современный вариант — TLS (Transport Layer Security) для защиты передаваемых данных между браузером и сервером.                                                                              |
|             | Данные передаются в зашифрованном виде, что предотвращает их перехват.                                                                                                                                                                           |
|             | Поддерживает сертификаты безопасности, выдаваемые сертификационными центрами, — они подтверждают подлинность сайта и гарантируют пользователям, что их данные защищены.                                                                          |
|             | Используется для сайтов, работающих с конфиденциальной информацией: интернет-банков, онлайн-магазинов, социальных сетей и других сервисов, где требуется защита пользователей.                                                                   |
|             | Адрес сайта начинается с «https://».                                                                                                                                                                                                             |

### Какие существуют типы содержимого (Content-Type) в HTTP?

| Под темы      | Заметка                                                                                                                                                                                                          |
| ------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ         | **В HTTP существуют различные типы содержимого (MIME-типы)**, которые указываются в заголовке **Content-Type**. Этот заголовок показывает, в каком формате передаются данные в теле запроса или ответа.          |
| Классификация | **Текстовые типы** — используются для представления простого текста, HTML-контента, CSS-стилей и кода JavaScript. Примеры: text/plain, text/html, text/css, text/javascript и др..                               |
|               | **Типы изображений** — применяются для различных форматов изображений. Примеры: image/jpeg, image/png, image/gif и др..                                                                                          |
|               | **Типы аудио и видео** — используются для обработки аудио- и видеофайлов. Примеры: audio/mpeg, video/mp4 и др..                                                                                                  |
|               | **Типы приложений** — применяются для различных видов бинарных данных и документов, например, JSON-данных в API, XML-данных и документов PDF. Примеры: application/json, application/xml, application/pdf и др.. |
|               | **Типы мультичасти** — используются для поддержки данных, разделённых на части, например, загрузки файлов. Примеры: multipart/form-data, multipart/byteranges и др..                                             |
|               | **Типы сообщений** — применяются для encapsulating сообщений, например, HTTP-запросов и ответов или сообщений электронной почты. Примеры: message/http, message/rfc822 и др..                                    |

### Как реализован механизм сжатия данных в HTTP и какие алгоритмы сжатия обычно используются?

| Под темы  | Заметка                                                                                                                                                                                                                                         |
| --------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ     | **Механизм сжатия данных в HTTP реализован на уровне протокола** и позволяет уменьшать объём передаваемой информации между клиентом и сервером, что помогает ускорить загрузку веб-страниц и снизить использование пропускной способности сети. |
| Механизм  | **Запрос клиента**. Клиент указывает, какие алгоритмы сжатия он поддерживает, с помощью заголовка **Accept-Encoding**.                                                                                                                          |
|           | **Обработка на сервере**. Получив запрос, сервер проверяет заголовок Accept-Encoding и, если поддерживает один из указанных алгоритмов, сжимает ответ перед отправкой обратно клиенту.                                                          |
|           | **Декомпрессия на клиенте**. После получения сжатого ответа клиент (браузер) автоматически распознаёт заголовок Content-Encoding и использует соответствующий алгоритм для декомпрессии данных.                                                 |
| Алгоритмы | **Gzip** — используется для сжатия текстовых файлов (HTML, CSS, JS, JSON).                                                                                                                                                                      |
|           | **Brotli** — применяется для статических файлов, даёт лучшее сжатие, но требует больше CPU.                                                                                                                                                     |
|           | **Deflate** — используется для динамического контента, но не всегда широко распространён из-за проблем с совместимостью с некоторыми серверами и браузерами.                                                                                    |

### Какие механизмы безопасности используются в HTTPS?

| Под темы           | Заметка                                                                                                                                                                                                                                                                                                                 |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ              | **В протоколе HTTPS (HyperText Transfer Protocol Secure) используются механизмы безопасности, связанные с шифрованием, аутентификацией, обеспечением целостности данных и управлением ключами**. Эти механизмы обеспечивают конфиденциальность данных, их целостность и аутентификацию.                                 |
| Шифрование         | **Симметричное шифрование** — использует один и тот же ключ как для шифрования, так и для дешифрования данных. Обеспечивает высокую скорость обработки, но создаёт проблему безопасной передачи самого ключа — если злоумышленник перехватит ключ, он сможет расшифровать все сообщения.                                |
|                    | **Асимметричное шифрование** — использует пару ключей: открытый (публичный) и закрытый (приватный). Информация, зашифрованная открытым ключом, может быть расшифрована только соответствующим закрытым ключом, и наоборот. Это позволяет безопасно обмениваться данными без предварительного обмена секретными ключами. |
| Аутентификация     | HTTPS обеспечивает **аутентификацию сервера** с использованием **цифровых сертификатов**. Это гарантирует, что клиент связывается с настоящим сервером и не подвергается атакам посредника (Man-in-the-Middle Attacks).                                                                                                 |
|                    | Сертификаты выдаются доверенными центрами сертификации. Это предотвращает возможность того, что злоумышленник предоставит свой публичный ключ, выдавая себя за сервер, к которому обращается клиент.                                                                                                                    |
| Целостность        | **Криптографические хеш-функции** — создают уникальный «отпечаток» для каждого блока данных. Если кто-то попытается изменить данные, хеш изменится, что сразу же будет замечено.                                                                                                                                        |
|                    | **Цифровые подписи** — каждый фрагмент данных подписывается цифровой подписью или кодом аутентификации, чтобы соблюсти целостность и подлинность данных.                                                                                                                                                                |
| Управление ключами | В основе работы HTTPS лежит **процесс обмена ключами**, который начинается с «рукопожатия» (handshake). Этот процесс гарантирует, что соединение между клиентом и сервером будет безопасным с самого начала.                                                                                                            |

### Какие нововведения были введены в HTTP/2 по сравнению с HTTP/1.1?

| Под темы  | Заметка                                                                                                                                                                                                                                                                                                                     |
| --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| HTTP/1.1  | **Текстовый формат** — запросы и ответы состоят из строк ASCII с определёнными разделителями (CRLF), заголовки передаются в виде текста без сжатия.                                                                                                                                                                         |
|           | **Режим постоянного соединения** — вместо закрытия TCP после каждого ответа соединение остаётся открытым для новых запросов.                                                                                                                                                                                                |
|           | **Обязательное указание заголовка Host** в каждом запросе, что позволило реализовать виртуальный хостинг — размещение нескольких доменов на одном IP-адресе.                                                                                                                                                                |
|           | **Новые методы** (PUT, DELETE, OPTIONS, TRACE, а позже CONNECT и PATCH) для расширения семантики протокола.                                                                                                                                                                                                                 |
|           | **Кодирование ответа частями** (chunked transfer encoding) для потоковой передачи больших или неизвестных заранее по размеру данных.                                                                                                                                                                                        |
| HTTP/2    | **Бинарное кадрирование** — передача данных происходит в двоичном виде вместо текста. Сообщения разбиваются на фреймы (кадры) — небольшие блоки данных. Каждый фрейм состоит из заголовка, который содержит информацию о типе фрейма, его размере и идентификаторе потока, и полезной нагрузки — данных запроса или ответа. |
|           | **Мультиплексирование** — позволяет отправлять несколько запросов и получать несколько ответов одновременно по одному соединению.                                                                                                                                                                                           |
|           | **Приоритизация потоков** — позволяет разработчикам назначать уровень важности для различных запросов. Это помогает сначала загружать самые критичные ресурсы.                                                                                                                                                              |
|           | **Сжатие заголовков** — каждый отправленный заголовок содержит информацию об отправителе и получателе, а это — избыточные объёмы. Благодаря сжатию полная информация отправляется только в первом заголовке, в последующих отправленных заголовках такой информации уже нет.                                                |
|           | **Server Push** — если в версии HTTP/1 браузер должен был сначала получить домашнюю страницу и лишь из неё понять, какие ресурсы ему необходимы для рендеринга, то HTTP/2 позволяет отправить все необходимые ресурсы сразу, при первичном обращении к серверу.                                                             |
| Сравнение | **Формат данных** — HTTP/1.1 использует текстовый формат, HTTP/2 — бинарный.                                                                                                                                                                                                                                                |
|           | **Мультиплексирование** — в HTTP/1.1 для каждого запроса требуется устанавливать отдельное TCP-соединение, в HTTP/2 все запросы идут по одному каналу, но обрабатываются параллельно.                                                                                                                                       |
|           | **Компрессия заголовков** — HTTP/2 сжимает заголовки алгоритмом HPACK, в HTTP/1.1 заголовки передаются в виде текста без сжатия.                                                                                                                                                                                            |
