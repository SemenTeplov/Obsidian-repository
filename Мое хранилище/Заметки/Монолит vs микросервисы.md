| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Монолит     | Слои приложения с монолитной архитектурой составляют единую структуру, в которой все части связаны. Приложения можно разделить по модулям и слоям, но части приложения все равно связаны: их пишут внутри одного проекта и собирают одновременно.                                                                                                                                                                                                                                                                                                                                                                   |
|             | Преимущества: скорость разработки - проще продумывать взаимосвязи между сущностями, следовательно, можно быстро разработать несложную функциональность; линейный процесс деплоя - приложение как одна сущность, которую нужно собрать и развернуть; у всех частей приложения есть доступ друг к другу - нет нужды разрабатывать механизмы взаимодействия между частями; легко поддерживать транзакционность процессов - код храниться в одном месте и после сбоя легко откатить изменения; одна база данных - таблицы проще поддерживать в согласованном виде.                                                      |
|             | Недостатки: сложно изменить и добавить новую функциональность - все компоненты связаны, поэтому, все изменения повлекут за собой полную сборку и повторный деплой; сложно внедрить новые технологии - придётся переписывать код всего приложения; много кода в одном проекте - при усложнении логики, кода становится больше; нельзя масштабировать отдельную часть приложения - изменения затронут все компоненты; сложно разобраться в проекте - чем сложнее логика, тем сложнее понимать систему в целом.                                                                                                        |
| Микросервис | В микросервисной архитектуре приложения делятся на сервисы. У каждого микросервиса конкретная, небольшая и независимая задача. При этом они не полностью автономны: они могут передавать друг другу информацию. Чаще из микросервисов состоит только бизнес-логика, а вот UI остается монолитом. Каждый микросервис связан со своей базой данных. В ней хранятся только те данные, который относятся к его функциональности. К другим базам у микросервиса нет доступа.                                                                                                                                             |
|             | Преимущества: легко изменить приложение и добавить новую функциональность - микросервисы можно тестировать и развертывать по отдельности; можно использовать разные технологии - они автономны, поэтому внутри одного приложения можно использовать разные фреймворки и языки программирования; можно переиспользовать код - микросервисы реализуют конкретную функциональность, которую можно использовать в другом месте; можно масштабировать отдельную часть приложения; удобнее распределять задачи по разработке - микросервис проще изолировать от остального приложения и поручить отдельному разработчику. |
|             | Недостатки: скорость разработки - тщательней нужно анализировать бизнес-логику и связи между сервисами; задержки и ошибки - сервисы должны общаться между собой, а это, порой, долго и чревата появлением ошибок; производительность - может быть ниже, чем у монолита; сложный процесс CI/CD - сервисом много и может быть много способов деплоя; сложно поддерживать транзакционность процессов - может потребоваться откат изменений у нескольких сервисов; несколько баз данных - у каждого сервиса своя база данных, следовательно сложнее поддерживать их согласованность.                                    |
### Вывод
---
В современной разработке чаще всего используют микросервисы.
