### Что такое Apache Kafka?

| Под темы    | Заметка                                                                                                                                                                                                                  |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение | **Apache Kafka** — **распределённая платформа для обмена сообщениями**, предназначенная для обработки потоков данных в реальном времени. Её называют брокером сообщений, потому что она выступает в качестве посредника. |
| Особенности | Высокая пропускная способность и низкая задержка;                                                                                                                                                                        |
|             | Возможность горизонтального масштабирования — добавление новых брокеров без прерывания работы системы;                                                                                                                   |
|             | Отказоустойчивость — данные реплицируются на нескольких брокерах, что обеспечивает сохранность данных при отказе одного из них.                                                                                          |

### Какие основные компоненты имеет Kafka?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                           |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Producer    | **Продюсер** — приложение или сервис, которое публикует данные в Kafka.                                                                                                                                                                                                                                                                           |
| Особенности | Сообщения отправляются в топик.                                                                                                                                                                                                                                                                                                                   |
|             | Если у продюсера задан ключ сообщения, Kafka использует его для выбора партиции. Если ключа нет, сообщения распределяются равномерно.                                                                                                                                                                                                             |
|             | Продюсер может быть настроен для обеспечения гарантий доставки: «at least once», «at most once» или «exactly once».                                                                                                                                                                                                                               |
| Broker      | **Брокер** — сервер, который хранит и управляет данными в Kafka.                                                                                                                                                                                                                                                                                  |
| Особенности | Принимает данные от продюсеров и предоставляет их потребителям                                                                                                                                                                                                                                                                                    |
|             | Хранит сообщения в партициях — упорядоченных, неизменяемых журналах сообщений.                                                                                                                                                                                                                                                                    |
|             | В кластере Kafka может быть один или несколько брокеров, что обеспечивает надёжность и масштабируемость системы. Если один брокер выйдет из строя, остальные продолжат работу, не теряя данных.                                                                                                                                                   |
| Topic       | **Топик** — логический канал или категория, в которую отправляются данные.                                                                                                                                                                                                                                                                        |
| Особенности | Каждый топик имеет уникальное имя, которое используется производителями для публикации сообщений и потребителями для их чтения.                                                                                                                                                                                                                   |
|             | Топики делятся на разделы (partitions) — упорядоченные, неизменяемые журналы сообщений. Каждое сообщение, добавленное в раздел, получает последовательный идентификатор — смещение (offset).                                                                                                                                                      |
|             | Каждый раздел может быть реплицирован на несколько брокеров, что обеспечивает отказоустойчивость: если один брокер выходит из строя, его реплики на других брокерах могут продолжить обслуживать запросы.                                                                                                                                         |
| Consumer    | **Consumer** — приложение или сервис, который подписывается на топики и обрабатывает данные.                                                                                                                                                                                                                                                      |
| Особенности | Потребители подписываются на выбранные топики и извлекают сообщения из их разделов.                                                                                                                                                                                                                                                               |
|             | Обычно работают в составе **группы потребителей** — все потребители в одной группе совместно читают сообщения из одного или нескольких топиков. Каждый раздел топика назначается только одному Consumer внутри группы. Если потребитель выходит из строя, его разделы автоматически переназначаются другим активным потребителям в той же группе. |
|             | - Consumer непрерывно опрашивает брокеры на наличие новых сообщений (используя метод poll()). После успешной обработки он фиксирует (commits) смещения, указывая, до какого момента сообщения были прочитаны и обработаны.                                                                                                                        |

### Для каких задач обычно используется Kafka?

| Под темы   | Заметка                                                                                                                                                                                           |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Применение | **Журналирование событий** — многие компании используют Kafka для сбора журналов событий своих приложений и систем, что позволяет отслеживать действия пользователей и производить анализ данных. |
|            | **Аналитика в реальном времени** — системы, обрабатывающие данные в реальном времени, используют Kafka для передачи данных от источников к аналитическим инструментам.                            |
|            | **Интеграция данных** — Kafka часто используется для соединения различных микросервисов и систем, обеспечивая обмен данными между ними.                                                           |
|            | **Мониторинг и алертинг** — системы мониторинга используют Kafka для сбора метрик и агрегирования информации о состоянии систем.                                                                  |
|            | **Интернет вещей (IoT)** — Kafka может использоваться для обработки потоковых данных от множественных датчиков и устройств.                                                                       |

### Как работает механизм потребителей и продюсеров в Kafka?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ       | **Механизм работы продюсеров и потребителей в системе Apache Kafka** заключается в обмене сообщениями между приложениями-источниками данных (продюсерами) и приложениями-потребителями. Система работает по принципу публикации-подписки (pub/sub): множество источников публикуют информацию для множества потребителей одновременно. |
| Продюсеры   | **Продюсер** — приложение, которое генерирует и записывает события в систему Kafka.                                                                                                                                                                                                                                                    |
| Потребители | **Потребитель** — приложение, которое принимает и обрабатывает данные от продюсеров.                                                                                                                                                                                                                                                   |

### Что такое топик в Kafka и как он используется?

| Под темы    | Заметка                                                                                                                                                                       |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Топик (topic) в брокере Apache Kafka** — это логическая категория или канал для публикации и подписки на сообщения. Это фундаментальная единица организации данных в Kafka. |
|             | **Принцип работы**: производители публикуют сообщения определённой категории в топик, а потребители подписываются на этот топик и читают из него сообщения.                   |
| Применение  | **Сбор логов** — различные сервисы публикуют свои логи в топик «application_logs».                                                                                            |
|             | **Потоковая обработка данных** — данные о продажах отправляются в топик «sales_events» для анализа в реальном времени.                                                        |
|             | **Обмен сообщениями** — микросервисы обмениваются сообщениями через топики для асинхронной коммуникации.                                                                      |
|             | **Отслеживание событий** — события пользовательской активности (например, клики, просмотры страниц) публикуются в топик «user_activity».                                      |

### Как Kafka обеспечивает высокую доступность и надежность хранения данных?

| Под темы          | Заметка                                                                                                                                                                                                                                                            |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Репликация        | **Данные, которые записываются в один узел кластера (брокер), дублируются на другие**. Это помогает минимизировать риск потери данных в случае сбоя отдельных компонентов системы.                                                                                 |
| Партиционирование | **Каждый топик (topic) делится на партиции** — логические подмножества, в которых хранятся сообщения.                                                                                                                                                              |
| Кворум            | **Kafka использует концепцию кворума** для записи и чтения данных. Данные считаются успешно записанными, только если они попадают на определённое количество реплик из ISR. Такой подход помогает избежать ситуации, когда данные теряются из-за сбоя одного узла. |
| Мониторинг        | **Регулярный мониторинг статуса репликации** кластера Kafka важен для проактивного выявления потенциальных проблем до того, как они повлияют на доступность или надёжность.                                                                                        |

### Объясните, как работает механизм репликации в Kafka и зачем он нужен?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                   |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ          | **Механизм репликации в Kafka работает на основе принципа лидера и последователя**. Он создаёт копии партиций топика на нескольких брокерах, чтобы обеспечить **отказоустойчивость** и **надёжность данных**.                                                                                                             |
| Принцип работы | **Реплика-лидер** — одна реплика для каждой партиции, обрабатывает все запросы на чтение и запись для этой партиции. Производители всегда записывают данные лидеру, а потребители обычно читают данные у лидера.                                                                                                          |
|                | **Реплики-последователи** — все остальные реплики для партиции, пассивно реплицируют данные от своих соответствующих лидеров партиций. Их основная роль — выступать в качестве резервных копий, готовых принять на себя функции лидера в случае его сбоя.                                                                 |
|                | **Фактор репликации** — определяет количество копий партиции, существующих в кластере Kafka. Например, фактор репликации, равный 3, означает, что каждая партиция будет иметь одну реплику-лидера и две реплики-последователя.                                                                                            |
|                | **Синхронизированные реплики (ISRs)** — это реплика (как лидер, так и последователь), которая полностью синхронизирована с лидером и считается «в синхронизации». Kafka поддерживает список ISR для каждой партиции.                                                                                                      |
|                | **Выборы лидера** — когда текущая реплика-лидер для партиции становится недоступной (например, из-за сбоя брокера или проблемы с сетью), Kafka автоматически инициирует процесс выборов лидера. Цель — выбрать нового лидера из оставшихся ISR, чтобы гарантировать, что партиция остаётся доступной для чтения и записи. |

### Как настроить и масштабировать кластер Kafka для обработки больших объемов данных?

| Под темы        | Заметка                                                                                                                                                                                                                                                                                                                                               |
| --------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ           | **Кластер Kafka для обработки больших объёмов данных нужно настроить и масштабировать, учитывая распределённую архитектуру системы, разделение топиков на разделы и репликацию**. Важно оптимизировать параметры брокеров и продюсеров, а также регулярно мониторить работу кластера, чтобы выявлять узкие места и оптимизировать производительность. |
| Настройка       | **Выбрать правильное разделение темы**. Разделение должно быть сбалансированным и основано на ожидаемом объёме данных и нагрузке. Рекомендуется выбирать количество разделов так, чтобы оно соответствовало количеству потребителей.                                                                                                                  |
|                 | **Настроить параметры производителя**. Некоторые важные параметры: максимальный размер пакета (batch.size), задержка перед отправкой (linger.ms) и максимальное количество неотправленных сообщений (max.in.flight.requests.per.connection).                                                                                                          |
|                 | **Настроить параметры потребителя**. Некоторые параметры: количество потоков чтения (num.streams), задержка перед повторной попыткой чтения (retry.backoff.ms) и время ожидания сообщения (fetch.wait.max.ms).                                                                                                                                        |
|                 | **Настроить репликацию**. Kafka поддерживает репликацию данных для обеспечения отказоустойчивости. Можно настроить параметры репликации: количество реплик (replication.factor), фактор подтверждения записи (min.insync.replicas) и максимальное количество допустимых задержек (max.replication.lag).                                               |
| Масштабирование | **Добавить больше брокеров** в кластер, чтобы распределить нагрузку и обрабатывать больше данных.                                                                                                                                                                                                                                                     |
|                 | **Перераспределить разделы** существующих тем, чтобы равномерно распределить нагрузку между брокерами. Для этого можно использовать инструмент Kafka для перераспределения разделов.                                                                                                                                                                  |
|                 | **Оптимизировать настройки продюсера**. Например, увеличить размер пакета (batch.size) — это может улучшить пропускную способность, но за счёт задержки. Также можно включить сжатие сообщений (например, Snappy или LZ4) для снижения нагрузки на сеть.                                                                                              |

### Каким образом Kafka обеспечивает упорядоченность сообщений внутри одной партиции?

| Под темы       | Заметка                                                                                                                                                                                                                                             |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ          | **Kafka обеспечивает упорядоченность сообщений внутри одной партиции** за счёт механизма партиционирования — разделения топика на логические подмножества, в которых хранятся сообщения. Каждое сообщение принадлежит только одной партиции топика. |
| Принцип работы | **Присвоение порядкового номера (смещения) каждому сообщению** в партиции. Это гарантирует, что сообщения будут доставлены консьюмеру в том же порядке, в котором они были сохранены в партиции.                                                    |
|                | **Упорядоченный лог-файл** для каждой партиции. Сообщения добавляются в конец, и обычно их не изменяют и не удаляют вручную.                                                                                                                        |
|                | **Создание индексов** для каждой партиции, которые хранятся рядом с .log-файлами. Благодаря этим индексам консьюмер может мгновенно перейти к нужному сообщению, не перечитывая весь лог подряд.                                                    |
