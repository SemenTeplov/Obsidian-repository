### Расскажите что такое FetchType?

| Под темы | Заметка                                                                                                                                                                                                                                                                                             |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **FetchType** — это перечисляемый тип в Java Persistence API (JPA), который указывает, как следует загружать поле или свойство.                                                                                                                                                                     |
| LAZY     | По умолчанию в Hibernate. Означает, что связанные данные будут извлечены только при первом обращении к ним. Это может улучшить производительность в случаях, когда связанные данные не требуются чаще всего.                                                                                        |
| EAGER    | Означает, что связанные данные будут извлечены вместе с основной сущностью при её загрузке из базы данных. Это может быть полезно в случаях, когда связанные данные всегда необходимы, но может привести к снижению производительности, если связанные данные большие и/или имеют много ассоциаций. |

### Опишите, какие состояния Entity существуют в Hibernate?

| Под темы                 | Заметка                                                                                                                                                                          |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Transient (преходящее)   | **Объект находится в преходящем состоянии**, когда он создан, но ещё не связан с сессией Hibernate.                                                                              |
|                          | Объект не отслеживается Hibernate и не имеет соответствующей записи в базе данных.                                                                                               |
|                          | Любые изменения в объекте не влияют на базу данных, так как Hibernate ещё не отслеживает его.                                                                                    |
| Persistent (постоянное)  | **Объект находится в постоянном состоянии**, когда он связан с сессией Hibernate.                                                                                                |
|                          | Hibernate отслеживает изменения объекта и автоматически синхронизирует его с базой данных при необходимости.                                                                     |
|                          | Объект в постоянном состоянии имеет соответствующую запись в базе данных.                                                                                                        |
| Detached (отсоединенное) | **Объект находится в отсоединённом состоянии**, когда он был отсоединён от сессии Hibernate. Это может произойти, например, когда сессия закрыта или объект был явно отсоединён. |
|                          | Объект не отслеживается Hibernate и не синхронизируется с базой данных.                                                                                                          |
|                          | Любое изменение в объекте не отражается в базе данных, если объект не будет снова связан с сессией Hibernate.                                                                    |
| Removed (удаленное)      | **Состояние removed** представляет **удаленный объект**. Когда постоянный объект удалён с помощью метода session.delete(), он удаляется из обеих сессии и базы данных.           |
|                          | Объект по-прежнему существует в памяти, но уже удалён из базы данных.                                                                                                            |
|                          | Любые дальнейшие изменения в объекте не влияют на базу данных.                                                                                                                   |

### Что такое Hibernate и какие основные преимущества он предлагает?

| Под темы     | Заметка                                                                                                                                                                                                                                                                                                                                                 |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение  | **Hibernate** — фреймворк для работы с базами данных в языке программирования Java, который реализует концепцию **объектно-реляционного отображения (ORM)**. Его основное предназначение — упростить разработчикам процесс взаимодействия с базами данных, абстрагируя работу с данными на уровне объектов Java, вместо ручного написания SQL-запросов. |
| Преимущества | **Автоматизация работы с базами данных** — фреймворк автоматически генерирует SQL-код для основных операций с данными (создание, чтение, обновление, удаление) (CRUD).                                                                                                                                                                                  |
|              | **Независимость от базы данных** — Hibernate может работать с практически любым типом базы данных (MySQL, Oracle, SQL Server). Это позволяет разработчикам изменять используемую систему БД, не переписывая код.                                                                                                                                        |
|              | **Оптимизация производительности** — фреймворк включает функции, которые помогают ускорить работу с данными, снижая нагрузку на базу данных. Например, кэш первого уровня — хранит данные, к которым часто обращаются, ленивая загрузка — откладывает загрузку определённых данных до момента их необходимости.                                         |
|              | **Управление сложными связями между объектами** — в системах, где существуют сложные связи между данными (например, «один к одному», «один ко многим», «многие ко многим»), Hibernate позволяет легко управлять такими отношениями через конфигурацию и код.                                                                                            |

### Опишите, какой FetchType по умолчанию при каждой связи между двумя сущностями?

| Под темы | Заметка                                                                                      |
| -------- | -------------------------------------------------------------------------------------------- |
| Значения | **Для связей «один к одному»** (например, @OneToOne) — **FetchType.EAGER**.                  |
|          | **Для связей «многие ко многим»** (например, @OneToMany и @ManyToMany) — **FetchType.LAZY**. |
|          | **Для связи «многие к одному»** (например, @ManyToOne) — **FetchType.EAGER**.                |

### Расскажите что такое каскадирование и как каскадные операции влияют на управление данными в БД?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                            |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Каскадирование** в контексте баз данных — это **автоматическое распространение изменений от родительского элемента к связанным с ним дочерним элементам** в реляционной базе данных. Цель — обеспечить целостность и согласованность данных, предотвращая появление «осиротевших» записей, не имеющих надлежащих связей с родительской таблицей. |

### Что такое встраиваемый (Embeddable) класс? Какие стратегии наследования имеются в Hibernate?

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение            | **Встраиваемый (Embeddable) класс** в Hibernate — это класс, который не используется сам по себе, а **встроен в другой класс** как составная часть его структуры данных. Для работы с встраиваемыми классами в Hibernate используются аннотации @Embeddable и @Embedded.                                                                                                                                                                                                                                                                                                                                               |
| Стратегии наследования | **В Hibernate поддерживается наследование встраиваемых классов на основе столбца дискриминатора**. **Как это работает**: класс, отмеченный аннотацией @Embeddable, может быть расширен другими классами, отмеченными той же аннотацией, — они становятся подтипами. В этом случае свойства @Embedded, использующие этот тип, создают дополнительный столбец в отображениях сущностей, который хранит информацию о конкретном подтипе композитного значения. При получении унаследованного свойства Hibernate читает значение дискриминатора и создаёт правильный подтип @Embeddable с его соответствующими свойствами. |

### Для чего нужны callback методы в Hibernate? К каким сущностям применяются аннотации callback методов?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                   |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ          | **Callback-методы в Hibernate нужны для вызова при определённых событиях жизненного цикла сущности**. Это позволяет приложению реагировать на события, которые происходят внутри механизма persistence, и реализовывать общую функциональность или расширять встроенную функциональность. |
|                | Аннотации callback-методов применяются **к методам сущности** или **к классу прослушивателя сущности**. Можно использовать оба подхода одновременно.                                                                                                                                      |
| Принцип работы | Callback-методы вызываются при определённых событиях жизненного цикла сущности.                                                                                                                                                                                                           |
| Пример         | **@PrePersist** — выполняется перед операцией persist менеджера сущности или каскадированной;                                                                                                                                                                                             |
|                | **@PreRemove** — выполняется перед операцией remove менеджера сущности или каскадированной;                                                                                                                                                                                               |
|                | **@PostPersist** — выполняется после операции persist менеджера сущности или каскадированной;                                                                                                                                                                                             |
|                | **@PostRemove** — выполняется после операции remove менеджера сущности или каскадированной;                                                                                                                                                                                               |
|                | **@PreUpdate** — выполняется перед операцией UPDATE базы данных;                                                                                                                                                                                                                          |
|                | **@PostUpdate** — выполняется после операции UPDATE базы данных;                                                                                                                                                                                                                          |
|                | **@PostLoad** — выполняется после загрузки сущности в текущий контекст persistence или её обновления.                                                                                                                                                                                     |

### Расскажите, что такое EntityGraph?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Entity Graph** — это механизм декларативного управления тем, какие поля и связанные сущности должны быть загружены при выполнении запроса к базе данных.                                                                                                                                                                                                                                  |
| Особенности | **Позволяет точно указать, какие ассоциации нужны в конкретном контексте**. В отличие от стандартного подхода, при котором все связи могут загружаться лениво (lazy loading) или сразу (eager loading), Entity Graph даёт возможность указать, какие именно ассоциации необходимы.                                                                                                          |
|             | **Помогает сбалансировать нагрузку между базой данных и приложением**. Например, если выбрать список заказов, и каждый заказ ссылается на клиента, то без правильной настройки будет выполнен один запрос на заказы и по одному запросу на каждого клиента — итого N+1 запросов, где N — количество заказов. Entity Graph позволяет объединить эти операции в один эффективный JOIN-запрос. |
|             | **Поддерживает два типа графов**: fetch graph и load graph. Fetch graph загружает только те атрибуты, которые явно указаны в графе, игнорируя другие, даже если они помечены как EAGER. Load graph, напротив, загружает указанные атрибуты в дополнение к тем, что уже помечены как EAGER.                                                                                                  |

### Какие виды кэша есть в Hibernate и какими способами.

| Под темы       | Заметка                                                                                                                                                                                                                                                                                     |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Первый уровень | **Функционирует на уровне сессии (Session) или контекста постоянства (Persistence Context)**. Каждая сессия Hibernate имеет свой собственный L1-кэш, изолированный от других сессий.                                                                                                        |
|                | **По умолчанию** кэш первого уровня активен и не требует специальной настройки.                                                                                                                                                                                                             |
| Второй уровень | **Это глобальный кэш на уровне SessionFactory** (или EntityManagerFactory в терминах JPA). В отличие от L1, он разделяется между несколькими сессиями: все сессии, созданные одной фабрикой, могут читать и записывать в единое кэш-хранилище.                                              |
|                | **Важно**: кэш второго уровня в Hibernate не активируется автоматически, его нужно включить явным образом и настроить подходящий провайдер. По умолчанию Hibernate не будет использовать второй уровень (значение hibernate.cache.use_second_level_cache по умолчанию = false).             |
| Query Cache    | **Хранит результаты выполнения запросов к базе данных**. Когда приложение выполняет запрос, Hibernate сначала проверяет наличие результата в кэше запросов.                                                                                                                                 |
|                | **Если результат уже находится в кэше**, Hibernate возвращает его из кэша, что позволяет избежать выполнения запроса к базе данных. **Если результат отсутствует в кэше**, Hibernate выполняет запрос и помещает результат в кэш для последующего использования.                            |
|                | **Важно**: кэш запросов не кэширует состояние актуальных сущностей в кэше — он кэширует только значения идентификаторов и результаты типов. Поэтому кэш запросов следует использовать в паре с кэшем второго уровня для тех сущностей, которые должны кэшироваться как часть кэша запросов. |

### Какие варианты настройки есть для второго уровня кэша?

| Под темы  | Заметка                                                                                                                                                                                                                                                                                                               |
| --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Hibernate | **Использование провайдера кэша второго уровня**. Например, EHCache — open-source библиотека кэширования на Java, которая интегрируется с Hibernate. В настройках Hibernate (например, в application.properties или persistence.xml) указывают необходимые свойства.                                                  |
|           | **Настройка области кэша второго уровня** для каждой сущности или коллекции, которые нужно кэшировать. Это включает добавление аннотаций или конфигурации XML к классам сущностей или файлам сопоставления.                                                                                                           |
|           | **Настройка режима кеширования** для каждой отдельной сессии или даже для каждого запроса к базе данных. Например, можно указать режим GET (данные читаются из кэша, но не добавляются в него) или IGNORE (сессия не взаимодействует с кэшем).                                                                        |
| Spring    | **Реализация второго уровня кэша с помощью распределённого кэша**. Например, Redis — сервер, который позволяет нескольким экземплярам приложения делиться кэшированными данными. В настройках Spring нужно добавить конфигурацию, связанную с кэшем Redis, и включить второй кэш с аннотациями @Caching и @Cacheable. |
|           | **Синхронизация данных** между несколькими кэшами, даже если они объявлены для одного и того же метода. Например, при обновлении сущности через Hibernate нужно руками очистить соответствующий кэш Spring, аннотируя метод save с @CacheEvict.                                                                       |
| Redis     | **Установка лимитов памяти** с помощью директивы maxmemory. Если этот лимит достигнут, Redis начнёт вытеснять ключи на основе настроенной политики.                                                                                                                                                                   |
|           | **Выбор политики вытеснения** (maxmemory-policy). Определяет алгоритм, который Redis использует для выбора ключей, подлежащих удалению при превышении лимита памяти. Например, можно выбрать политику LRU (наименее недавно использованный) или LFU (наименее часто используемый).                                    |
|           | **Настройка работы в памяти** — если Redis используется исключительно как кеш, можно отключить запись на диск и настроить работу только в оперативной памяти.                                                                                                                                                         |
