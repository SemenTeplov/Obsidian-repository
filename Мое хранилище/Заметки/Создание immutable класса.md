### Что такое immutable класс в Java?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Immutable (неизменяемый) класс в Java** — это класс, **экземпляры которого не могут изменить своё состояние после инициализации**. То есть если в коде есть ссылка на экземпляр immutable класса, то любые изменения в нём приводят к созданию нового экземпляра.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Требования  | Чтобы класс считался immutable, он должен соответствовать следующим условиям: <br>- **Объявлен как final**, чтобы от него нельзя было наследоваться. Иначе дочерние классы могут нарушить иммутабельность.<br>- **Все поля класса должны быть приватными** в соответствии с принципами инкапсуляции.<br>- **Для корректного создания экземпляра** в классе должны быть параметризованные конструкторы, через которые осуществляется первоначальная инициализация полей класса.<br>- **В классе не должно быть сеттеров** — методов, которые изменяют состояние объекта.<br>- **Для полей-коллекций** необходимо делать глубокие копии, чтобы гарантировать их неизменность.<br>Также важно, чтобы все объекты, на которые ссылается класс, также были неизменяемыми, иначе можно было бы изменить состояние объекта. |
| Реализация  | Некоторые этапы создания собственного неизменяемого класса:<br>- **Объявить класс как final**.<br>- **Сделать все поля private**, чтобы прямой доступ был запрещён.<br>- **Не предоставлять методы setter** для переменных.<br>- **Сделать все изменяемые поля final**, чтобы значение поля можно было назначить только один раз.<br>- **Инициализировать все поля** с помощью метода конструктора, выполняющего глубокое копирование.<br>- **Выполнить клонирование объектов** в методах getter, чтобы возвращать копию, а не реальную ссылку на объект.<br>В Java 14 для создания неизменяемых объектов «только для чтения» можно использовать **записи (Records)** — они представляют собой неизменяемую структуру данных.                                                                                        |

### Какие преимущества предоставляют неизменяемые классы в Java?

| Под темы     | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Преимущества | Использование неизменяемых классов в Java имеет несколько преимуществ:<br>- **Потокобезопасность** — неизменяемые объекты не требуют синхронизации при доступе из нескольких потоков.<br>- **Простота и ясность** — неизменяемые объекты проще проектировать и использовать, так как они не меняют состояние.<br>- **Безопасность** — неизменяемые объекты предотвращают несанкционированное изменение их состояния, что полезно при передаче объектов в ненадёжный код.<br>- **Кеширование** — неизменяемые объекты можно безопасно кэшировать и использовать повторно, так как их состояние не изменяется. |

### Какие основные правила необходимо соблюдать при создании immutable класса в Java?

| Под темы         | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение      | **При создании неизменяемого (immutable) класса в Java необходимо соблюдать определённые правила, связанные с структурой класса, реализацией методов и копированием данных**. Эти правила направлены на обеспечение неизменяемости — состояния, в котором значение объектов нельзя изменить после их инициализации.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| Структура класса | - **Объявить класс как final**, чтобы от него нельзя было наследоваться. Иначе дочерние классы могут нарушить иммутабельность.<br>- **Сделать все поля приватными**, чтобы прямой доступ был запрещён.<br>- **Сделать все изменяемые поля final**, чтобы значение поля можно было назначить только один раз.<br>- **Инициализировать все поля** с помощью параметризованного конструктора, который выполняет глубокое копирование. Это предотвращает изменение полей по ссылке на объект. <br>- **Не допускать, чтобы дочерние классы переопределяли методы**. Проще всего это сделать, объявив класс как final.                                                                                                                                                                                                                                                                                                 |
| Методы           | - **Не предоставлять методы-установщики (setter)**, которые изменяют состояние объекта.<br>- **Если в классе есть изменяемые поля**, не предоставлять методы, которые их изменяют.<br>- **В методах получения** (getter) возвращать **копию** изменяемого объекта, а не реальную ссылку на него. Это предотвращает изменение исходного объекта.<br>- **Если класс содержит изменяемый объект**, в конструкторе использовать **клонированную копию** передаваемого аргумента, а не реальный экземпляр. Это предотвращает изменение объекта клиентами.                                                                                                                                                                                                                                                                                                                                                             |
| Копирование      | - **Для полей-коллекций** необходимо делать **глубокие копии** в конструкторе и методах получения, чтобы гарантировать их неизменность. Поверхностное копирование (shallow copy) не подходит — при нём копируется только ссылка на объект, что может привести к изменениям в оригинале.<br>- **В методах получения** для изменяемых объектов (например, коллекций) возвращать **новую копию** — это скрывает исходное значение от клиентов. Например, если поле — карта, метод getMap() должен возвращать new HashMap<>(map) вместо ссылки на оригинал.<br>- **Для внутренних изменяемых объектов** в конструкторе и методах получения создавать **клонированные экземпляры** (например, ImmutablePerson(String name, int age, Date birthDate) { this.name = name; this.age = age; this.birthDate = new Date(birthDate.getTime()) }) — это предотвращает изменение состояния при передаче изменяемых аргументов. |

### Как можно создать immutable класс, если в нем есть ссылочные поля, которые сами по себе могут быть изменяемыми?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение | Чтобы создать immutable (неизменяемый) класс в Java, если в нём есть ссылочные поля, которые сами по себе могут быть изменяемыми, необходимо обеспечить, чтобы эти объекты не менялись.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Правила     | - **Не предоставлять методы, которые изменяют изменяемые объекты**. Сеттеры обычно используются для изменения состояния объекта, и в immutable-классе их не должно быть.<br>- **Не предоставлять общий доступ к ссылкам на изменяемые объекты**. Например, не хранить ссылки на внешние изменяемые объекты, передаваемые конструктору, — при необходимости создавать копии и сохранять ссылки на копии. <br>- **Создавать копии внутренних изменяемых объектов** при необходимости, чтобы избежать возврата оригиналов в методах. Если нужно возвращать ссылочные типы вызывающему объекту, нужно возвращать защитные копии вместо ссылок на исходные экземпляры.<br>- **Сделать все поля класса приватными** и финальными, чтобы прямой доступ к ним был запрещён.<br>- **Инициализировать все поля** с помощью параметризованного конструктора, выполняющего глубокое копирование, чтобы данные не могли быть изменены по ссылке на объект.<br>- **Выполнять глубокое копирование объектов** в методах getter, чтобы возвращать копию, а не реальную ссылку на объект. |

### Какова роль ключевого слова final в создании immutable классов в Java?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **Ключевое слово final в Java играет важную роль в создании неизменяемых (immutable) классов**. Оно позволяет **запретить изменять состояние объекта** после его создания. Это обычно рекомендуется для создания объектов, которые должны оставаться постоянными во время жизни программы, например, уникальных идентификаторов или настроек приложения.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Принцип работы | - **Для классов** final означает, что **от этого класса нельзя наследоваться**. Иначе дочерние классы могут нарушить иммутабельность.<br>- **Для полей** final гарантирует, что **значение поля не может быть изменено** после создания объекта. Это важно, так как сложные объекты (не примитивы) будут полностью неизменяемыми, если каждое поле пометить как final. Иначе только ссылка на объект будет неизменяемой, но содержимое объекта может быть изменено.<br>- **Для методов** final означает, что **метод нельзя переопределить** в подклассах. Это полезно, когда нужно защитить реализацию от изменений.<br>**Некоторые рекомендации по созданию неизменяемого класса**:<br>- Все поля класса должны быть приватными, чтобы прямой доступ не был разрешён.<br>- Для корректного создания экземпляра в классе должны быть параметризованные конструкторы, через которые осуществляется первоначальная инициализация полей.<br>- Для полей-коллекций необходимо делать глубокие копии, чтобы гарантировать их неизменность.<br>- В классе не должно быть сеттеров — методов, которые изменяют состояние объекта. |

### Почему важно предоставлять только методы для чтения (getter methods), не предоставляя методов для изменения состояния (setter methods), в immutable классах?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **В immutable-классах важно предоставлять только методы для чтения (getter methods), не предоставляя методы для изменения состояния (setter methods), чтобы гарантировать неизменность объекта**. Это следует из концепции immutability — свойства объекта, которое означает, что его внутреннее состояние не может быть изменено после создания.                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| Принцип     | Требование предоставления только getter-методов связано с **инкапсуляцией** — принципом, согласно которому внутренние детали объекта должны быть скрыты от внешнего мира. Методы setter позволяют изменять состояние объекта, что противоречит идее immutability.<br>Кроме того, предоставление только getter-методов обеспечивает **доступ к атрибутам класса только для чтения**. Это важно, например: <br>- **Для безопасности** — неизменяемые объекты менее подвержены несанкционированным изменениям со стороны вредоносного кода.<br>- **Для потокобезопасности** — неизменяемые объекты можно безопасно использовать в многопоточных приложениях без риска повреждения данных.<br>- **Для кэширования** — неизменяемые объекты можно безопасно кэшировать, не беспокоясь о изменениях. |

### Как реализовать immutable класс с использованием паттерна Builder в Java?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Паттерн Builder в Java позволяет реализовать неизменяемый класс (Immutable Class) для создания сложных объектов**. Этот паттерн разделяет построение сложного объекта от его представления, что позволяет создавать объект шаг за шагом, устанавливая только нужные параметры.                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Требования  | - **Класс должен быть объявлен как final** — это предотвращает создание дочерних классов.<br>- **Поля в классе должны быть объявлены как final и private** — это делает их значения неизменяемыми после создания объекта.<br>- **Конструктор класса** должен быть приватным, чтобы объекты создавались только через Builder.<br>- **Методы установки параметров в Builder** должны возвращать this для поддержки цепочки вызовов.<br>- **Обязательные параметры** лучше передавать через конструктор Builder.<br>- **В методе build()** можно реализовать валидацию параметров перед созданием объекта.<br>Чаще всего используется упрощённая форма: Builder — это внутренний статический класс в классе Product. |

### Можно ли сделать класс immutable, если он содержит массивы, и как это можно сделать?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **В Java можно сделать класс неизменяемым (immutable), даже если он содержит массивы**, но **обычный массив не является неизменяемым**. Чтобы сделать массив неизменяемым, нужно создать класс-оболочку, который инкапсулирует обычный массив и предоставляет методы для доступа к его элементам без разрешения изменений.                                                                                                                                                                                                                                                                                                                                                                                                                           |
| Правила     | Некоторые рекомендации по созданию неизменяемого класса:<br>- **Объявить класс как final**, чтобы от него нельзя было наследоваться. Иначе дочерние классы могут нарушить иммутабельность.<br>- **Сделать все поля final и private**. Поля, объявленные private, не будут доступны за пределами класса, а сделав их final, можно гарантировать, что их нельзя изменить даже случайно.<br>- **Не предоставлять методы-сеттеры** — они предназначены для изменения состояния объекта, чего нужно избежать. <br>- **Для полей-коллекций** необходимо делать глубокие копии, чтобы гарантировать их неизменность.<br>- **В методах получения** нужно выполнять глубокое копирование объектов, чтобы возвращать копию, а не ссылку на фактический объект. |

### Как обеспечить идемпотентность методов в immutable классах и почему это важно?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Идемпотентность методов в immutable классах** — это свойство, при котором повторный вызов метода с теми же входными данными не меняет состояние объекта и результат. Это важно, так как: <br>-  **Обеспечивает предсказуемое поведение** — повторные вызовы не создают новые сущности, даже если ответ может чуть отличаться. <br>- **Помогает создавать «длинные транзакции»** — можно повторять методы, не заканчивая в недопустимом состоянии.<br>- **Обеспечивает безопасность повторных запросов** — например, в REST идемпотентные методы гарантируют, что даже при дублировании запроса система останется в том же состоянии и не вызовет нежелательных побочных эффектов. |
| Принцип     | Идемпотентность означает, что **при одном и том же вводном значении функция всегда должна возвращать один и тот же результат**. В строгом смысле это и про отсутствие повторных побочных эффектов, и про одинаковый ответ.<br>**Важно**: метод по своей природе либо идемпотентен, либо нет — изменить это свойство нельзя. Однако в системах, которые изначально не предполагают идемпотентность, можно искусственно симулировать это свойство с помощью определённых ограничений. Например, использовать уникальные идентификаторы запросов (ключи идемпотентности), которые позволяют системе распознать повторение.                                                             |
