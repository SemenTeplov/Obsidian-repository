### Что такое JPA?

| Под темы    | Заметка                                                                                                                                                                                                         |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **JPA (Java Persistence API)** — **спецификация для управления реляционными данными в приложениях Java**. Она обеспечивает механизм взаимодействия с базой данных через объектно-реляционное отображение (ORM). |
|             | JPA позволяет разработчикам работать с объектами вместо прямого взаимодействия с SQL-запросами. Технология действует как мост между объектно-ориентированным миром Java и реляционной моделью баз данных.       |

### Назовите хотя бы одну альтернативу Hibernate.

| Под темы     | Заметка                                                                                                                                                                                                                                                                                                                              |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Альтернативы | **EclipseLink**. Эталонная реализация JPA, во многом схожая с Hibernate. Поддерживает ряд других стандартов и имеет особенности, которых нет в Hibernate (например, аннотации @Struct и @ReadOnly).                                                                                                                                  |
|              | **Reladomo**. ORM-фреймворк, ключевая особенность которого состоит в генерации Java-классов из xml-описания структуры БД. Взаимодействие с БД происходит посредством вызова методов сгенерированных классов.                                                                                                                         |
|              | **Apache Cayenne**. Среда, которая позволяет генерировать ORM-объекты с помощью программы-конструктора. Создание, удаление, сохранение объектов в БД и более сложные запросы происходят с помощью вызова методов класса DataContext.                                                                                                 |
|              | **MyBatis**. Фреймворк для взаимодействия с базой данных в Java-приложении. С ним работают в тех случаях, когда в приложении используются сложные запросы и связи, имеется достаточно много различных сложных выборок из нескольких таблиц, происходит работа с хранимыми процедурами или возникает необходимость написать свой ORM. |

### В чем состоит основное предназначение JPA?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                           |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Основное предназначение JPA (Java Persistence API) — управление реляционными данными в приложениях Java**. Технология действует как мост между объектно-ориентированным миром Java и реляционной моделью баз данных, позволяя разработчикам работать с объектами вместо прямого взаимодействия с SQL-запросами. |
| Возможности | **Сосредоточиться на бизнес-логике приложения**, абстрагируясь от деталей работы с базой данных.                                                                                                                                                                                                                  |
|             | **Работать со сложными схемами данных**: система наследования, различные типы отношений между сущностями (One-to-One, One-to-Many, Many-to-Many) легко реализуются через аннотации и XML-конфигурацию.                                                                                                            |
|             | **Встроенную поддержку транзакций и механизмов контроля версий данных** — это важно для приложений, работающих с финансовыми операциями или другими чувствительными данными.                                                                                                                                      |
|             | **Инструменты для работы с большими данными**: пагинация, динамические запросы через Criteria API и нативные запросы, когда это необходимо.                                                                                                                                                                       |
|             | **Возможность лёгкой миграции между различными СУБД** без необходимости переписывания кода — достаточно изменить конфигурацию persistence.xml.                                                                                                                                                                    |

### Какие преимущества предоставляет использование JPA по сравнению с JDBC?

| Под темы | Заметка                                                                                                                                                                     |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| JPA      | **Чистый и лаконичный код** — не нужно вручную управлять соединениями и ResultSet.                                                                                          |
|          | **Работа на уровне объектов** — легче оперировать бизнес-логикой.                                                                                                           |
|          | **Поддержка кэширования, ленивой загрузки** и других оптимизаций «из коробки».                                                                                              |
|          | **Возможность лёгкой миграции между различными СУБД** без необходимости переписывания кода — достаточно изменить конфигурацию persistence.xml.                              |
|          | **Встроенная поддержка транзакций** и механизмов контроля версий данных, что важно для приложений, работающих с финансовыми операциями или другими чувствительными данными. |
|          | **Инструменты для работы с большими данными**: пагинация, динамические запросы через Criteria API и нативные запросы, когда это необходимо.                                 |
| JDBC     | **Полный контроль над SQL** и структурой запросов.                                                                                                                          |
|          | **Максимальная производительность** при оптимизации вручную.                                                                                                                |
|          | **Независимость от ORM и фреймворков**.                                                                                                                                     |
|          | **Надёжность работы с транзакциями**: JDBC позволяет контролировать выполнение SQL-команд, управлять фиксацией и откатом изменений, а также отслеживать ошибки соединений.  |
|          | **Лёгкость сопровождения кода**: структура соединений и SQL-запросов в JDBC прозрачна для разработчика.                                                                     |

| Spring Framework | **Поддержка паттерна инверсии контроля (Inversion of Control, IoC) и паттерна внедрения зависимостей (Dependency Injection, DI)**. Приложение не само управляет жизненным циклом и зависимостями объектов, а делегирует эту ответственность фреймворку.                                                            |
|                  | **Модульность** — разработчик подключает только те компоненты Spring, которые нужны под конкретные задачи. Это делает проекты на Spring более «лёгкими» и адаптивными: можно использовать фреймворк как для небольшого REST-сервиса, так и для большого монолитного приложения.                                    |
|                  | **Готовые решения** — Spring предоставляет множество готовых решений (подключение к базе, безопасность, обработка веб-запросов и т. д.), что избавляет программиста от необходимости писать эти механизмы с нуля.                                                                                                  |
|                  | **Интеграция с другими технологиями** — Spring поддерживает стандарты Java EE (Servlet API, JPA и пр.), работает с популярными СУБД, ORM-фреймворками (Hibernate).                                                                                                                                                 |
| Grails           | **Конвенционное программирование** — вместо того чтобы тратить время на настройку и конфигурирование проекта, разработчики могут сосредоточиться на написании бизнес-логики приложения.                                                                                                                            |
|                  | **Система GORM (Grails Object Relational Mapping)** — обеспечивает простой и интуитивный доступ к базам данных. GORM позволяет разработчикам использовать объектно-ориентированный подход к работе с данными, а также обрабатывать связи между объектами и таблицами базы данных с помощью генерации SQL-запросов. |
|                  | **Обширная экосистема плагинов** — позволяет легко добавлять новые функции и инструменты в приложение.                                                                                                                                                                                                             |
|                  | **Гибкость** — Grails позволяет использовать различные шаблоны разработки (например, Model-View-Controller), а также выбирать язык шаблонизации (GSP, Jade, Handlebars и другие) и фронтенд-фреймворки.                                                                                                            |
|                  | **Поддержка тестирования** — Grails предоставляет полную поддержку тестирования, включая модульное тестирование, интеграционное тестирование и функциональное тестирование.                                                                                                                                        |

### Какие стратегии наследования поддерживает JPA?

| Под темы              | Заметка                                                                                                                                                                                             |
| --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Single Table (STI)    | **Все классы в иерархии наследования сопоставляются в единую таблицу базы данных**. Для различения экземпляров подклассов используется столбец-дискриминатор.                                       |
| Особенности           | Упрощает управление схемой и выполнение запросов.                                                                                                                                                   |
|                       | Может быть эффективной с точки зрения производительности запросов, но может привести к пустой таблице, если иерархия большая.                                                                       |
|                       | В таблице есть столбец-дискриминатор, который идентифицирует тип каждой строки.                                                                                                                     |
| Joined Table (JTI)    | **Каждый класс в иерархии наследования сопоставляется в свою таблицу**. Таблица базового класса содержит общие атрибуты, а таблица каждого подкласса — атрибуты, специфические для этого подкласса. |
| Особенности           | Может нормализовать схему базы данных, но может привести к сложным соединениям SQL.                                                                                                                 |
|                       | Подходит, когда подклассы имеют значительные различия в атрибутах, сводя к минимуму избыточность в основной таблице.                                                                                |
|                       | В Java задаётся аннотацией @Inheritance(strategy = InheritanceType.JOINED).                                                                                                                         |
| Table Per Class (TPC) | **Каждому классу в иерархии наследования соответствует выделенная таблица базы данных**. Все свойства каждого класса хранятся в соответствующей таблице, включая унаследованные свойства.           |
| Особенности           | Может обеспечить лучшую изоляцию между таблицами, но может привести к избыточности данных и не хорошо поддерживать полиморфные запросы.                                                             |
|                       | В Java задаётся аннотацией @Entity @Inheritance(strategy = InheritanceType.TABLE_PER_CLASS).                                                                                                        |

### Опишите процесс интеграции JPA с Spring Framework.

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                  |
| ---------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Настройка конфигурации | **В приложении Spring Boot** JPA настраивается автоматически с помощью зависимостей spring-boot-starter и spring-boot-starter-data-jpa. Spring Boot настраивает Hibernate в качестве поставщика JPA по умолчанию, поэтому не нужно определять bean-компонент entityManagerFactory.                       |
|                        | **В стандартном проекте Spring** конфигурация может быть более явной с использованием Java или XML. Например, нужно настроить EntityManager с помощью фабричного компонента Spring — это может быть простой LocalEntityManagerFactoryBean или более гибкий LocalContainerEntityManagerFactoryBean.       |
|                        | **В файле application.properties** (или application.yml) указываются параметры подключения к базе данных, например, spring.datasource.url, spring.datasource.username, spring.datasource.password. Также можно настроить генерацию таблиц на основе JPA, например, spring.jpa.hibernate.ddl-auto=update. |
| Создание сущностей     | **Сущности сопоставляют таблицы в базе данных**. Каждое поле класса становится колонкой в таблице. Для этого используются аннотации, например: @Entity, @Id, @GeneratedValue, @Column.                                                                                                                   |
|                        | **Репозиторий** отвечает за операции с базой данных. В Spring Data JPA для этого используются интерфейсы, например, JpaRepository. Расширяя интерфейс JpaRepository, автоматически получаются все стандартные методы, такие как save(), findById(), deleteById() и другие.                               |
| Выполнение запросов    | **В репозитории** можно добавлять кастомные методы, которые выполняют запросы к базе данных. Например, можно использовать аннотацию @Query для выполнения JPQL или нативного SQL-запроса.                                                                                                                |
|                        | **Можно передавать параметры** в запрос с помощью аннотации @Param. Каждый параметр, помеченный аннотацией @Param, должен иметь строку значения, соответствующую соответствующему параметру запроса.                                                                                                     |
| Транзакции             | **В Spring Data JPA** есть поддержка транзакций. Например, можно аннотировать метод с аннотацией <br>@Transactional — она гарантирует, что все изменения либо зафиксируются, либо откатятся, если произойдёт ошибка.                                                                                     |
|                        | **Spring создаёт прокси** для всех классов, аннотированных с @Transactional, — это позволяет фреймворку вводить транзакционную логику перед и после запуска метода.                                                                                                                                      |
|                        | **Можно настраивать поведение** с помощью атрибутов аннотации @Transactional, например, типа распространения транзакции, уровня изоляции, времени ожидания для операции, флага readOnly.                                                                                                                 |

### Какие есть подходы к оптимизации производительности при использовании JPA?

| Под темы                            | Заметка                                                                                                                                                                                                                                                                                                                        |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Оптимизация запросов                | **Использовать индексы**. Анализировать шаблоны запросов, выявлять часто используемые столбцы и добавлять соответствующие индексы для улучшения скорости операций. Однако не стоит добавлять слишком много индексов, так как это может повлиять на производительность записи.                                                  |
|                                     | **Использовать проекции**. Указывать, какие поля нужно получить из базы данных, — это позволяет получить только необходимые данные.                                                                                                                                                                                            |
|                                     | **Избегать проблемы N+1 Select**. Например, получать связанные сущности в одном запросе вместо нескольких, чтобы избежать ненужных обращений к базе данных.                                                                                                                                                                    |
|                                     | **Использовать пагинацию** для больших наборов результатов. Позволяет получать только часть результатов на основе определённого размера страницы и номера страницы, что снижает использование памяти.                                                                                                                          |
| Кэширование                         | **Использовать встроенный кэш JPA** через аннотацию @Cacheable. Это позволяет включать кэширование второго уровня.                                                                                                                                                                                                             |
|                                     | **Настроить провайдера кэширования** (например, Ehcache, Hazelcast, Infinispan).                                                                                                                                                                                                                                               |
|                                     | **Использовать кэширование запросов** — сохранять результаты конкретных запросов, что полезно для приложений, где много операций чтения.                                                                                                                                                                                       |
|                                     | **Использовать стратегии вытеснения кэша** — определять, когда кэшированные данные должны быть удалены. Например, использовать время вытеснения (данные удаляются после определённого периода) или размер вытеснения (кэш удаляет наименее используемые элементы, когда достигает определённого размера).                      |
| Управление транзакциями             | **Определять границы транзакций** на соответствующих уровнях, чтобы транзакции были короткими. Долгосрочные транзакции могут привести к конкуренции ресурсов и повлиять на производительность.                                                                                                                                 |
|                                     | **Маркировать транзакции как только для чтения** — если транзакция включает только операции чтения, это позволяет базе данных оптимизировать транзакцию и потенциально улучшить производительность.                                                                                                                            |
|                                     | **Использовать пакетные операции** для обработки больших данных или обновлений — JPA поддерживает пакетную обработку, что позволяет выполнять несколько операций в одной партии, снижая количество обращений к базе данных.                                                                                                    |
| Настройка соединения с базой данных | **Использовать нативные запросы** для сложных запросов — они дают больше контроля над SQL, генерируемым JPA, и позволяют оптимизировать его для лучшей производительности. Однако нативные запросы нужно использовать осторожно — они теряют некоторые преимущества JPA, например, портативность и автоматическое отображение. |
|                                     | **Мониторить производительность** — использовать инструменты для профилирования, например, Spring Boot Actuator, чтобы выявлять узкие места в производительности. Также можно проводить тестирование нагрузки, чтобы simulate реальные сценарии использования и оценить производительность приложения при разных нагрузках.    |

### Как реализовать многотенантность в приложении, используя JPA?

| Под темы    | Заметка                                                                                                                                                                              |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение | Многотенантность — это архитектура, в которой один экземпляр приложения обслуживает несколько арендаторов, при этом данные и ресурсы, используемые арендаторами, отделены от других. |
| Подходы     | **Отдельная база данных** — данные каждого арендатора хранятся в отдельном экземпляре базы данных, изолированы от других.                                                            |
|             | **Общая база данных и отдельная схема** — данные каждого арендатора хранятся в отдельной схеме в общей базе данных.                                                                  |
|             | **Общая база данных и общая схема** — все арендаторы используют общую базу данных, и в каждой таблице есть столбец с идентификатором арендатора.                                     |
