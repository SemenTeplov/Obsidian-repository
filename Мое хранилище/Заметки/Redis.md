### Что такое Redis и какие основные сценарии его применения?

| Под темы            | Заметка                                                                                                                                                                                                                      |
| ------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение         | **Redis** (англ. Remote Dictionary Server) — **нереляционная NoSQL-система управления базами данных (СУБД) с открытым исходным кодом**. Данные хранятся в формате пар «ключ — значение» в оперативной памяти, а не на диске. |
| Сценарии применения | **Высоконагруженные веб-приложения** — ускорение отклика за счёт кеширования страниц и запросов к базам данных.                                                                                                              |
|                     | **API и микросервисная архитектура** — хранение сессий и токенов, передача данных между сервисами.                                                                                                                           |
|                     | **Чаты, трекинговые приложения, онлайн-игры, торговые платформы** — обеспечение передачи сообщений.                                                                                                                          |
|                     | **Системы аналитики и мониторинга** — подсчёт метрик, логирование, агрегация данных и построение статистики.                                                                                                                 |
|                     | **Рекомендательные и рейтинговые сервисы** — хранение таблиц с результатами рейтингов и персонализированных рекомендаций.                                                                                                    |
|                     | **Игровые платформы и социальные сети** — кеширование профилей пользователей, подсчёт очков и лайков, работа с онлайн-статусами.                                                                                             |

### Чем Redis отличается от других баз данных (например, от реляционных или MongoDB)?

| Под темы    | Заметка                                                                                                                         |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------- |
| Redis       | **Redis — нереляционная СУБД**. Данные хранятся в виде пар «ключ-значение» в оперативной памяти сервера, а не на жёстком диске. |
| Особенности | **Высокая производительность** — операции чтения и записи занимают меньше миллисекунды даже при большом потоке запросов.        |
|             | **Гибкость** — данные находятся не в жёстких структурах (таблицах), а в более удобных (строки, списки, множества).              |
|             | **Отсутствие языка SQL** — вместо него используются скрипты на языке Lua.                                                       |

### Какие типы данных поддерживает Redis (строки, списки, множества и т.д.)?

| Под темы | Заметка                                                                                                                                                                                                                                    |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ    | **Redis поддерживает несколько типов данных**, включая строки, списки, множества и хеш-таблицы. Эти структуры позволяют работать с динамическими данными, хранить временные данные, которые нужно быстро получать, и представлять объекты. |

### Что означает, что Redis — это in-memory хранилище?

| Под темы | Заметка                                                                                                                                                                                                                                               |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Redis — это in-memory хранилище** (хранилище структур данных в памяти). Это означает, что **данные хранятся в оперативной памяти (RAM), а не на жёстком диске**. Это обеспечивает быстрые операции чтения и записи (задержка — менее миллисекунды). |

### Как работает механизм TTL (time-to-live) в Redis?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                 |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Механизм TTL (time-to-live) в Redis работает как встроенная система самоудаления данных**. Он позволяет задать срок истечения для ключа, после истечения которого ключ автоматически удаляется. Это важно для управления памятью и обеспечения актуальности данных, особенно в сценариях кэширования. |

### Как подключиться к Redis из Java-приложения с помощью Lettuce или Jedis?

| Под темы    | Заметка                                                                                           |
| ----------- | ------------------------------------------------------------------------------------------------- |
| Lettuce     | **Поддерживает асинхронные операции** и позволяет использовать одно соединение из разных потоков. |
| Особенности | Lettuce поддерживает SSL-соединения.                                                              |
|             | Можно использовать несколько подключений.                                                         |
| Jedis       | **Простой и лёгкий клиент** для Redis.                                                            |
| Особенности | Jedis предлагает меньше функций, но по-прежнему может обрабатывать большие объёмы памяти.         |

| RedisTemplate | **RedisTemplate** — центральный интерфейс в Spring Data Redis, обеспечивает абстракцию для работы с Redis.                                                                                                                                                       |
| @Cacheable    | **Аннотация @Cacheable** указывает на то, что результат метода должен быть кэширован. Атрибут value определяет имя кэша, а атрибут key — ключ кэша.                                                                                                              |

### Как управлять TTL и инвалидацией кэша в Redis с помощью аннотаций Spring Cache?

| Под темы           | Заметка                                                                                                                                                                                                                                                                                |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| TTL (Time-to-Live) | **Аннотация @Cacheable** помечает метод, результаты которого будут сохранены в указанный кэш (с определённым ключом). **TTL** настраивается **через CacheManager** провайдера кэша (для Redis — RedisCacheManager).                                                                    |
| Инвалидация кэша   | **Аннотация @CacheEvict** помечает метод, после выполнения которого нужно удалить определённые записи из кэша (или весь кэш — атрибут allEntries = true). **Обычно ставится на методы изменения данных** (создание, обновление, удаление), чтобы инвалидировать устаревшие кэш-записи. |

### Какие типы сериализации можно использовать при работе с Redis в Java-приложении и как они влияют на производительность?

| Под темы                    | Заметка                                                                                                                                                                                          |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Стандартная сериализация    | Объекты преобразуются в последовательность байтов, которую можно сохранить в Redis.                                                                                                              |
| Влияние                     | **Размер сериализованных данных** может быть значительным, особенно для больших сложных объектов. Большие объёмы данных означают более медленную передачу по сети и больше использования памяти. |
|                             | **Скорость сериализации** в целом медленнее по сравнению с другими методами из-за сложности нативной сериализации Java.                                                                          |
| JSON (Jackson)              | Объекты сериализуются в JSON с помощью библиотеки Jackson                                                                                                                                        |
| Влияние                     | **Компактность формата** — JSON занимает меньше места по сравнению с текстовыми форматами (например, JSON).                                                                                      |
|                             | **Возможность настраивать процесс сериализации** — например, использовать аннотации для сериализации пользовательских типов и коллекций.                                                         |
| Protocol Buffers (Protobuf) | Формат сериализации компактен благодаря бинарному представлению и эффективному кодированию полей.                                                                                                |
| Влияние                     | **Производительность сериализации и десериализации** высока благодаря сгенерированному коду, который избегает рефлексии и интерпретации схемы во время выполнения.                               |
|                             | **Для повторяющихся полей** (например, списков или массивов) можно использовать упакованную кодировку, чтобы уменьшить размер сериализованных данных.                                            |
| Avro                        | Данные сериализуются на основе схемы, которая определяет структуру и типы данных записей.                                                                                                        |
| Влияние                     | **Компактный бинарный формат** — это уменьшает объём данных и ускоряет передачу данных.                                                                                                          |
|                             | **Поддержка эволюции схем** — Avro позволяет схемам развиваться со временем без нарушения совместимости между производителями и потребителями данных.                                            |

### Как работает Redis Pub/Sub и в каких случаях он применяется?

| Под темы              | Заметка                                                                                                                                                                                                                                                                                                                                              |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение           | **Redis Pub/Sub** (Publish/Subscribe) — механизм обмена сообщениями, который позволяет приложениям общаться друг с другом в реальном времени. В отличие от традиционных методов обмена сообщениями, таких как очереди, где сообщения хранятся до тех пор, пока они не будут обработаны, Pub/Sub работает по принципу **«опубликовать-подписаться»**. |
| Принцип работы        | **Издатель отправляет сообщения в канал (или тему)**, а **подписчики, которые подписаны на этот канал, получают опубликованные сообщения**. Канал выступает посредником между издателем и подписчиками, что позволяет компонентам приложения обмениваться информацией асинхронно, не блокируя выполнение своих операций.                             |
| Примеры использования | **Веб-приложения** — для реализации уведомлений в реальном времени. Когда на сервере происходит какое-либо событие, сервер может опубликовать сообщение на соответствующий канал, и все клиенты, подписанные на этот канал, немедленно получат уведомление.                                                                                          |
|                       | **Игровые серверы** — в многопользовательских играх необходимо мгновенно передавать информацию о действиях игроков, таких как движение, атака или использование предметов. С помощью Pub/Sub сервер может быстро отправлять обновления всем игрокам в одной комнате или на одной карте, обеспечивая синхронизацию действий и минимизируя задержки.   |
|                       | **Системы мониторинга** — сообщения о сбоях отправляются в специальный канал, а подписчики (системы визуализации, алерты и дашборды) получают эти данные для обработки.                                                                                                                                                                              |

### Как Redis обеспечивает отказоустойчивость через Sentinel и кластеризацию?

| Под темы      | Заметка                                                                                                                                                                                                                                                              |
| ------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение   | **Redis обеспечивает отказоустойчивость через Sentinel и кластеризацию**. Эти механизмы позволяют минимизировать простои и гарантировать, что данные всегда будут доступны, даже в случае отказа мастера.                                                            |
| Sentinel      | **Sentinel** — система мониторинга и автоматизации отказоустойчивости для базы данных Redis. Она отслеживает состояние основного (master) и реплик (slaves) узлов Redis, обнаруживает сбои и инициирует **failover** — автоматическое переключение на здоровый узел. |
| Кластеризация | **Кластеризация** (встроенная кластеризация в терминологии Redis) обеспечивает отказоустойчивость, так как в кластере есть **реплики** — полные копии мастер-ноды, доступные только для чтения данных.                                                               |

### Что такое Redis Cluster, и как устроена партиционизация и репликация данных?

| Под темы        | Заметка                                                                                                                                                                                                                                  |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение     | **Redis Cluster** — это распределённая реализация Redis, которая автоматически разделяет данные между несколькими узлами.                                                                                                                |
| Партиционизация | **Redis Cluster использует хэш-слоты** для автоматического распределения данных по узлам. Каждый ключ назначается на один из 16 384 хэш-слотов на основе его хэша, и каждый слот назначается на конкретный узел.                         |
| Репликация      | **Redis Cluster использует асинхронную репликацию**. Если клиент пишет что-то на мастер-узел, который затем выходит из строя перед реплицированием записи на реплики, то новый мастер-узел (повышенный реплика) не знает об этой записи. |

### Как бороться с проблемой cache stampede, cache avalanche и cache penetration в Redis?

| Под темы          | Заметка                                                                                                                                                                                                                                                                              |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Cache Stampede    | Большое количество параллельных запросов пытаются получить одни и те же данные из кэша одновременно, но данные отсутствуют или истекают. Это вызывает все запросы, чтобы обойти кэш и обратиться к источнику данных (базе данных).                                                   |
| Решение           | **Блокировка на уровне приложения**. Первый запрос, который наткнулся на cache miss, ставит «замочек» на этот ключ. Остальные запросы по этому ключу видят замочек и ждут (или получают старые данные, если это приемлемо).                                                          |
|                   | **Проактивное обновление**. Кэш обновляется с часто запрашиваемыми данными перед их истечением, чтобы снизить вероятность cache stampede.                                                                                                                                            |
| Cache Avalanche   | Большое количество записей в кэше истекает одновременно, что вызывает резкий скачок запросов к источнику данных. Это может привести к высокой нагрузке на базу данных.                                                                                                               |
| Решение           | **Рандомизация TTL**. Вместо фиксированного TTL (скажем, 300 секунд) использовать небольшой случайный разброс (например, от 280 до 320 секунд). Это «размажет» время истечения ключей и сгладит пик.                                                                                 |
|                   | **Многоуровневое кеширование**. Если один слой кеша падает, другой может смягчить удар.                                                                                                                                                                                              |
| Cache Penetration | Запросы для несуществующих данных обходят кэш и напрямую обращаются к базе данных. Это может произойти, когда приложение получает большое количество запросов для ключей, которые не существуют в кэше или базе данных.                                                              |
| Решение           | **Кэшировать пустые ответы** для несуществующих ключей. Хранить значение-заполнитель (например, null или конкретный код ошибки) в Redis для ключей, которые не существуют. Последующие запросы по этому ключу можно обслуживать непосредственно из кэша без обращения к базе данных. |
|                   | **Ограничение числа запросов** к базе данных на уровне приложения или проксирующего слоя. Это поможет контролировать количество запросов от одного пользователя или IP-адреса.                                                                                                       |

### Как реализовать распределённую блокировку с помощью Redis (например, через Redlock)?

| Под темы         | Заметка                                                                                                                                                                          |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Алгоритм Redlock | **Клиент пытается получить блокировку на нескольких экземплярах Redis**, предоставляя уникальный идентификатор и время истечения блокировки.                                     |
|                  | **Клиент проверяет, была ли блокировка получена на большинстве экземпляров Redis**. Если так, блокировка считается действительной, и клиент может продолжить критическую секцию. |
|                  | **После завершения критической секции** клиент снимает блокировку на всех экземплярах Redis, используя уникальный идентификатор, чтобы освободить только правильную блокировку.  |
