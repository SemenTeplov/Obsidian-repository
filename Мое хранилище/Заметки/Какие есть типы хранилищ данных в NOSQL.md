### Какие основные типы хранилищ данных используются в NoSQL?

| Под темы      | Заметка                                                                                                                                                                                                                                                                             |
| ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ         | В NoSQL (нереляционная база данных) используются **четыре основных типа хранилищ данных**: «ключ-значение», документы, столбцы и графы. Каждый тип имеет свои особенности и применяется для разных задач.                                                                           |
| Ключ-значение | **Хранилища «ключ-значение»** — простейший тип, использует ключ для доступа к значению. Данные хранятся в виде хеш-таблицы, где каждый ключ уникален, а значение может быть представлено строкой, JSON, блобом (BLOB) и т. д..                                                      |
| Документы     | **Документоориентированные базы данных** — данные хранятся в виде документов, обычно в формате JSON, BSON или XML. Документы — это самостоятельные единицы, которые содержат всю необходимую информацию. Они могут быть вложенными, что позволяет хранить сложные структуры данных. |
| Столбцы       | **Хранилища, ориентированные на столбцы** — данные хранятся в столбцах, а не в строках. Колонки логически группируются в колоночные семейства, чтение и запись происходят с использованием колонок, а не строк.                                                                     |
| Графы         | **Хранилища на основе графов** — данные хранятся в виде графовой структуры с узлами и рёбрами, соединяющими данные. Узлы представляют сущности, а рёбра определяют связи между ними.                                                                                                |

### Какие преимущества предоставляет документо-ориентированное хранилище данных по сравнению с реляционной базой данных?

| Под темы                              | Заметка                                                                                                                                                                                                         |
| ------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Документо-ориентированные базы данных | **Гибкая схема**. Сущности не нужно подгонять под табличный формат, можно добавлять новые поля, удалять или изменять структуру данных для каждой сущности.                                                      |
|                                       | **Горизонтальное масштабирование**. Если информации или запросов становится больше, достаточно добавить больше узлов. Реляционные базы придётся масштабировать вертикально — переносить на более мощный сервер. |
|                                       | **Возможность хранить сложные данные**. В одном документе могут содержаться вложенные объекты и списки, что удобно для работы с неструктурированными или полуструктурированными данными.                        |
|                                       | **Упрощённое поддержание базы данных**. На значимые изменения в данных (например, изменения модели) не требуется обновление схемы.                                                                              |
| Реляционные базы данных               | **Структурированность**. Данные представлены в виде таблиц, где строки и столбцы связаны заранее установленными правилами.                                                                                      |
|                                       | **Целостность данных**. Поддерживают ACID-свойства (атомарность, согласованность, изолированность, долговечность).                                                                                              |
|                                       | **Гибкий запрос данных**. SQL позволяет выполнить сложные запросы и анализ данных.                                                                                                                              |

### Можете ли вы объяснить, как концепция CAP-теоремы применяется к выбору типа хранилища данных в NoSQL, и какие компромиссы необходимо учитывать?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Концепция CAP-теоремы (теоремы Брюера) применяется к выбору типа хранилища данных в NoSQL** через анализ компромиссов между согласованностью, доступностью и устойчивостью к разделению сети. Теорема утверждает, что **распределённое хранилище данных может гарантировать только два из трёх свойств одновременно**. Это вынуждает разработчиков идти на компромиссы в зависимости от конкретных требований и ограничений приложения. |
