### В чем основное различие между Kafka и RabbitMQ с точки зрения архитектуры?

| Под темы    | Заметка                                                                                                                                                                                                                                                        |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Kafka       | **Реализует концепцию распределённого лога**. Сообщения сохраняются в брокере, подписчики подписываются на них, при этом гарантируется, что сообщения находятся в той же последовательности, в которой поступили.                                              |
| Особенности | **Подход к обмену сообщениями** — потребитель сам запрашивает (pull) данные.                                                                                                                                                                                   |
|             | **Удаление сообщений из очереди** — в Kafka сообщения после прочтения не удаляются и могут храниться неограниченное время.                                                                                                                                     |
|             | **Маршрутизация сообщений** — сообщения распределяются по разделам (partitions) темы, внутри группы потребителей Kafka гарантирует, что каждый раздел назначен только одному потребителю.                                                                      |
|             | **Масштабируемость** — Kafka легко масштабируется горизонтально, что позволяет добавлять новые брокеры для обработки большего объёма данных.                                                                                                                   |
| RabbitMQ    | **Использует push-модель доставки** — брокер сам отправляет сообщения потребителям, как только они появляются в очереди. В отличие от Kafka, где потребитель сам запрашивает данные, в RabbitMQ они приходят «сами».                                           |
| Особенности | **Очереди** — специализированные хранилища для временного удержания сообщений.                                                                                                                                                                                 |
|             | **Обмены** — играют роль маршрутизаторов, решают, в какие именно очереди будут направлены сообщения, исходя из установленных правил маршрутизации и конкретного типа обмена.                                                                                   |
|             | **Гибкость маршрутизации** — есть возможность конфигурации обменов и очередей с учётом разнообразных схем маршрутизации. Например, использование топик-обмена позволяет направлять сообщения одновременно в несколько очередей в зависимости от их содержания. |
|             | **Приоритезация сообщений** — RabbitMQ позволяет назначать приоритет сообщениям, в Kafka приоритет для всех сообщений одинаков и его нельзя изменить.                                                                                                          |

### Какие типы обмена сообщениями поддерживает RabbitMQ, которых нет в Kafka?

| Под темы | Заметка                                                                                                                                                                                                                                                                     |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **RabbitMQ поддерживает некоторые типы обмена сообщениями, которых нет в Kafka**, благодаря гибкой маршрутизации сообщений. В отличие от Kafka, которая хранит сообщения для многократного чтения, RabbitMQ доставляет сообщения один раз и удаляет их после подтверждения. |

### Что такое топик в Kafka и как он соотносится с RabbitMQ?

| Под темы    | Заметка                                                                                                                                                                                                       |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Kafka       | **Топик в Kafka** — **логическая категория или канал для публикации и подписки на сообщения**. Это фундаментальная единица организации данных в Kafka.                                                        |
| Особенности | Каждый топик имеет уникальное имя, которое используется производителями (producers) для публикации сообщений и потребителями (consumers) для их чтения.                                                       |
|             | Топики делятся на **разделы (partitions)** — упорядоченные, неизменяемые журналы сообщений. Каждое сообщение, добавленное в раздел, получает последовательный идентификатор — смещение (offset).              |
|             | Разделы позволяют Kafka масштабироваться горизонтально, так как данные топика могут быть распределены по нескольким брокерам (серверам Kafka). Чем больше разделов, тем выше параллелизм обработки сообщений. |
|             | Каждый раздел может быть реплицирован на несколько брокеров, что обеспечивает отказоустойчивость: если один брокер выходит из строя, его реплики на других брокерах могут продолжить обслуживать запросы.     |
| RabbitMQ    | **Топик в RabbitMQ** — **тип обменника (exchange)**, который распределяет входящие сообщения по очередям на основе шаблона темы.                                                                              |
| Особенности | Сообщения, отправленные в топик, не могут иметь произвольный ключ маршрутизации — он должен быть списком слов, разделённых точками.                                                                           |
|             | Логика работы: сообщение, отправленное с конкретным ключом маршрутизации, доставляется во все очереди, которые связаны с соответствующим ключом привязки.                                                     |
|             | Можно использовать подстановочные знаки: # (hash) — соответствует нулю или более слов, * (star) — одному слову.                                                                                               |
| Сравнение   | **В Kafka** топик — это логическая единица хранения сообщений, а в RabbitMQ — механизм маршрутизации, который позволяет настраивать сложные сценарии доставки.                                                |
|             | **Таким образом, топик в Kafka** — это концепция, связанная с организацией данных, а **топик в RabbitMQ** — с маршрутизацией сообщений на основе шаблона темы.                                                |

### Как реализуется устойчивость к сбоям в Kafka и RabbitMQ?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                   |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Kafka       | **Устойчивость к сбоям в Kafka реализуется с помощью механизма репликации**. Данные, которые записываются в один узел кластера (брокер), дублируются на другие. Это помогает минимизировать риск потери данных в случае сбоя отдельных компонентов системы.                               |
| Особенности | **Лидер и реплики**. Каждая партия имеет один «лидирующий» брокер (лидер), который принимает записи данных, и остальные брокеры, которые хранят копии этой партиции (реплики).                                                                                                            |
|             | **Синхронизация данных**. Реплики постоянно синхронизируются с лидером, копируя все изменения (записи), происходящие на уровне партиции.                                                                                                                                                  |
|             | **Кворумный подход**. Данные считаются успешно записанными, только если они попадают на определённое количество реплик из списка синхронизированных реплик (ISR).                                                                                                                         |
|             | **Выбор нового лидера**. Если брокер, выполняющий роль лидера, выходит из строя, одна из реплик из списка ISR автоматически становится новым лидером.                                                                                                                                     |
| RabbitMQ    | Устойчивость к сбоям в RabbitMQ реализуется с помощью устойчивых очередей и сообщений.                                                                                                                                                                                                    |
| Особенности | **Устойчивые очереди**. Устойчивые очереди повторно объявляются при запуске узла и переживают перезапуск, сбой системы или сбой сервера (до тех пор, пока сохраняются данные). Неустойчивые очереди и маршрутизация удаляются при перезапуске узла.                                       |
|             | **Устойчивые сообщения**. По умолчанию сообщения в RabbitMQ являются временными (transient) — если брокер перезапускается, все временные сообщения будут потеряны. Чтобы предотвратить это, очереди и сообщения необходимо объявлять как долговечные (durable).                           |
|             | **Кластеризация с зеркалированием очереди**. Можно объединить несколько узлов RabbitMQ в кластер, а затем добавить дополнительную избыточность путём репликации очередей между несколькими узлами. Таким образом, если падает один узел, данные не теряются и система остаётся доступной. |

### Какие сценарии использования лучше подходят для Kafka, а какие для RabbitMQ?

| Под темы    | Заметка                                                                                                                                                                                                         |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Kafka       | **Большие данные и аналитика** — если система обрабатывает большие объёмы событий, логов или метрик.                                                                                                            |
|             | **Стриминговые приложения** — когда требуется обработка потоков данных в реальном времени, например, для рекомендаций, мониторинга или детекции аномалий.                                                       |
|             | **Микросервисная архитектура** — для обеспечения событийно-ориентированного взаимодействия между большим количеством сервисов с высокой нагрузкой.                                                              |
|             | **Журналирование и отслеживание** — когда необходимо хранить и повторно читать историю событий за определённый период.                                                                                          |
| Особенности | Сообщения хранятся на протяжении заданного времени (retention period) или до достижения определённого объёма, позволяя повторно читать данные.                                                                  |
|             | Темы (topics) разделяются на разделы (partitions), которые могут распределяться по разным брокерам, обеспечивая параллельную обработку.                                                                         |
| RabbitMQ    | **Фоновая обработка данных** — сообщение помещается в очередь без его немедленной обработки.                                                                                                                    |
|             | **Интеграция внутри приложений и между ними** — RabbitMQ выступает средством связи между приложениями и помогает избегать узких мест при передаче сообщений.                                                    |
|             | **Распределённые нагрузки** — несколько копий одного приложения могут быть запущены на разных компьютерах, RabbitMQ распределяет нагрузку относительно производительности и выбирает, кому отправить сообщение. |
| Особенности | Сообщения маршрутизируются через обменник (exchange) перед попаданием в очереди.                                                                                                                                |
|             | RabbitMQ хранит сообщение до тех пор, пока принимающее приложение не подключится и не получит его из очереди.                                                                                                   |
|             | Поддерживает приоритезацию в очередях и позволяет настроить диапазон приоритетов.                                                                                                                               |

### Каким образом можно интегрировать Kafka и RabbitMQ в микросервисную архитектуру?

| Под темы | Заметка                                                                                                                                                                                                                                                                                  |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Kafka    | В Kafka используется подход pull: консьюмеры сами отправляют запросы в брокер раз в n миллисекунд для получения новой порции сообщений. Такой подход позволяет группировать сообщения в пакеты, достигая лучшей пропускной способности.                                                  |
|          | После прочтения консьюмерами сообщения в Kafka не удаляются и могут храниться неограниченное время.                                                                                                                                                                                      |
|          | Для обеспечения отказоустойчивости и надёжности можно внедрить механизмы обработки ошибок и повторных попыток в потребителях Kafka для обработки временных сбоев, а для производителей — настроить соответствующие подтверждения, чтобы обеспечить долговечность и надёжность сообщений. |
| RabbitMQ | В RabbitMQ все сообщения маршрутизируются через обменник (exchange) перед попаданием в очереди.                                                                                                                                                                                          |
|          | RabbitMQ поддерживает несколько стандартизированных протоколов: AMQP, MQTT, STOMP и др..                                                                                                                                                                                                 |
|          | RabbitMQ позволяет назначать приоритет сообщениям, в Kafka приоритет для всех сообщений одинаков, и его нельзя изменить.                                                                                                                                                                 |

### Как реализовать распределенные транзакции с использованием Kafka и RabbitMQ?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                         |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Kafka    | **Транзакционная публикация сообщений**. Все операции, связанные с публикацией сообщения (запись в лидере раздела, репликация по подписчикам, увеличение смещения), должны выполниться вместе. Если произойдёт сбой хотя бы одной операции в рамках транзакции, она целиком считается невыполненной, причём происходит откат всех выполненных операций, если какие-то завершились успешно.                      |
|          | **Атомарные циклы чтения-обработки-записи**. Например, ошибка во время обработки может привести к прерыванию транзакции, и в этом случае ни одно из сообщений транзакции не будет доступно потребителям.                                                                                                                                                                                                        |
|          | **За транзакции отвечают координатор транзакций** — модуль, работающий внутри каждого брокера Kafka, и **журнал транзакций** — внутренний топик Kafka под названием __transaction_state.                                                                                                                                                                                                                        |
| RabbitMQ | **Использование шаблона Saga**. Он позволяет микросервисам выполнять только локальные транзакции, а транзакции, охватывающие все микросервисы, выполняются масштабируемым образом с использованием стратегии, при которой неудачные транзакции могут быть логически компенсированы локально. RabbitMQ может использоваться как брокер обмена сообщениями, в который можно отправлять события и прослушивать их. |
|          | **Реализация распределённых транзакций между RabbitMQ и другой системой**. Например, можно отправлять сообщения в RabbitMQ до завершения транзакции, но при обработке сообщения в другой системе сохранять уникальный идентификатор задачи и номер потребителя в списке. Если в БД уже есть сообщение с таким идентификатором задачи и номером потребителя, то повторно его не отправляют.                      |

### Какие механизмы контроля потребления сообщений доступны в Kafka и RabbitMQ, и как они работают?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Kafka       | **В Kafka используется модель pull**. Потребители запрашивают пакеты сообщений с определённого смещения.                                                                                                                                                                                                                                                                                                                                     |
| Особенности | **Обработка и хранение оффсетов**. Каждый потребитель поддерживает оффсет, который указывает на позицию последнего прочитанного сообщения в теме. Оффсеты позволяют потребителям продолжать чтение сообщений с места, где они остановились после перезапуска или сбоев.                                                                                                                                                                      |
|             | **Автоматическое управление оффсетами**. Kafka самостоятельно отслеживает и хранит оффсеты с помощью функциональности группировки потребителей и координатора группы. В этом случае Kafka гарантирует, что каждый потребитель читает сообщения только один раз.                                                                                                                                                                              |
|             | **Транзакционное потребление**. Потребители, читающие из разделов, опубликованных транзакционно, могут быть настроены на чтение только зафиксированных данных. За это отвечает параметр isolation.level в конфигурации потребителя. Если для него установлено значение read_committed, метод потребителя poll() будет возвращать только зафиксированные транзакционные сообщения.                                                            |
| RabbitMQ    | **В RabbitMQ используется push-модель доставки**. Брокер сам отправляет сообщения потребителям, как только они появляются в очереди.                                                                                                                                                                                                                                                                                                         |
| Особенности | **Отслеживание доставки сообщений с гарантией at-least-once**. Когда сообщение попадает в очередь, оно остаётся там до тех пор, пока потребитель не подтвердит его обработку, отправив положительное подтверждение ack. Если подтверждение не получено из-за сбоя потребителя или разрыва соединения, сообщение возвращается в очередь и может быть доставлено другому потребителю — RabbitMQ повторяет доставку неподтверждённых сообщений. |
|             | **Контроль потребления упреждающей выборки**. Потребитель может запросить заранее установленное количество сообщений, которые следует получать прежде чем он выдаст квитанцию подтверждения приёма. Это позволяет RabbitMQ более эффективно отправлять сообщения, указывая, сколько сообщений предназначено для данного потребителя.                                                                                                         |
|             | **Настройка альтернативных очередей и обменников**. Например, можно направлять сообщения, которые не смог обработать потребитель, в специализированный обменник-сборщик недоставленных сообщений. К этому обменнику будет привязана очередь недоставленных сообщений, на которую подписывается обработчик.                                                                                                                                   |

### В чем заключается подход к шардированию в Kafka и как он отличается от механизмов масштабирования в RabbitMQ?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                              |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Kafka    | **В Kafka партиционирование заложено в основу архитектуры**. Каждый топик разделён на **партиции**, распределённые между узлами кластера.                                                                                                                                                                                                                            |
|          | Производитель может явно указать целевую партицию или предоставить ключ, по которому Kafka определит партицию. Сообщения с одинаковым ключом всегда попадают в одну и ту же партицию, что гарантирует их строгую упорядоченность.                                                                                                                                    |
| RabbitMQ | **В RabbitMQ горизонтальное масштабирование обеспечивается благодаря шардированию очереди по сегментам (шардам)**. Одна очередь как логическая сущность разбивается на несколько независимых шардов, каждый из которых хранит часть сообщений. Каждому шарду назначается определённый диапазон сообщений. Шарды могут размещаться на разных узлах кластера RabbitMQ. |
|          | **При публикации новых сообщений в очередь** они распределяются между разными шардами согласно алгоритму шардирования, например, по хэшу ключа сообщения или простым перебором (Robin-Round). Потребители могут читать сообщения параллельно из разных шардов, что увеличивает производительность и позволяет масштабировать обработку нагрузки.                     |
| Отличия  | **В Kafka** масштабирование часто требует минимальных изменений в коде клиентов, тогда как **в RabbitMQ** масштабирование часто требует более активного участия разработчиков приложения.                                                                                                                                                                            |
|          | **В Kafka** производитель может явно указать целевую партицию, **в RabbitMQ** — создать несколько очередей по определённому алгоритму.                                                                                                                                                                                                                               |
|          | **В Kafka** сообщения с одинаковым ключом всегда попадают в одну и ту же партицию, **в RabbitMQ** — при публикации новых сообщений они распределяются между разными шардами, что может быть не всегда строго упорядочено.                                                                                                                                            |
