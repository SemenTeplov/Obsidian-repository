### Какие основные состояния потоков в Java?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **В Java потоки могут находиться в нескольких основных состояниях** в своём жизненном цикле. Эти состояния контролируются как JVM, так и операционной системой. Некоторые из них: **New**, **Runnable**, **Blocked** и **Terminated**.                                                                                                                                                                                                                                                                                                                |
| New         | **Поток создан, но ещё не запущен**. Это состояние возникает сразу после создания объекта Thread, но до вызова метода start(). В этом состоянии поток не готов к выполнению и не потребляет системные ресурсы.                                                                                                                                                                                                                                                                                                                                        |
| Runnable    | **После вызова метода start()** поток переходит в состояние **Runnable**. В этом состоянии поток считается готовым к выполнению и может быть выбран планировщиком потоков для выполнения на процессоре. Поток может фактически выполняться или ожидать своей очереди на выполнение.                                                                                                                                                                                                                                                                   |
| Blocked     | **Поток переходит в это состояние**, если он пытается войти в синхронизированный блок или метод, доступ к которому в данный момент удерживается другим потоком. Как только монитор освобождается, поток возвращается в состояние **Runnable**.<br>Также существует состояние **TIMED_WAITING** — похожее на состояние WAITING, но с указанным временем ожидания. Поток переходит в это состояние при вызове методов sleep() или join() с указанием таймаута. По истечении времени ожидания поток автоматически возвращается в состояние **Runnable**. |
| Terminated  | **Поток переходит в это состояние**, когда метод run() завершает своё выполнение либо из-за нормального завершения, либо из-за неперехваченного исключения. В этом состоянии поток больше не может быть перезапущен.<br>**Важно**: в любой момент времени поток может находиться только в одном состоянии. Эти состояния представляют собой состояния виртуальной машины, которые не отражают состояния потоков операционной системы.                                                                                                                 |

### Как создать новый поток в Java?

| Под темы           | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Через класс Thread | **Чтобы создать поток через наследование от класса Thread, нужно**:<br>1. Создать класс, расширяющий Thread.<br>2. Переопределить метод run() — здесь размещается код, который будет выполняться в отдельном потоке.<br>3. Создать экземпляр этого класса.<br>4. Вызвать метод start() для запуска потока.<br>**Важно**:<br>- Если вызвать run() напрямую, код выполнится в текущем потоке, а не в новом.<br>- Метод start() создаёт новый поток и затем вызывает run() уже в контексте этого нового потока.<br>Ограничение**: Java не поддерживает множественное наследование — если класс наследует Thread, он не может наследовать другие классы. |

### Что такое метод join() в контексте потоков и для чего он используется?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **Метод join() в контексте потоков** — это метод синхронизации, который **приостанавливает выполнение текущего потока, пока не завершится другой поток**.<br>**Используется для**:<br>- **Контроля над порядком исполнения потоков** , особенно когда между главным и дочерними потоками возникают зависимости по данным.<br>- **Контроля работы демон-потоков** — они обычно прекращают работу с завершением основной программы, но использование join() позволяет точно контролировать их работу до окончания и провести все необходимые операции перед финишированием программы.<br>- **Доступ к данным из разных потоков** — в ситуации одновременной загрузки данных различными потоками важно убедиться, что все сегменты данных были полностью получены перед обработкой, — в этом случае join() помогает избежать получения неполных данных. |
| Принцип работы | **В Java** — вызов join() образует стартовую последовательность, в которой каждый последующий участник начинает работу только после того, как завершит свою работу предыдущий. Если в программе используется несколько дочерних потоков, и надо, чтобы главный поток завершался после дочерних, то для каждого дочернего потока надо вызвать метод join().                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |

### Чем отличаются Runnable и Callable?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Интерфейсы Runnable и Callable в Java отличаются тем, как они предоставляют результат выполнения и обрабатывают исключения**. Оба интерфейса используются для создания многопоточных задач, которые могут быть выполнены другими потоками, но имеют разные особенности.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| Runnable    | **Определён в пакете java.lang**. Представляет простейший способ описания задачи для выполнения в отдельном потоке.<br>**Особенности**:<br>- Содержит единственный метод **void run()**, который не принимает аргументов и не возвращает результат.<br>- Метод run() содержит код, который будет выполняться в отдельном потоке.<br>- Когда объект Runnable передаётся в конструктор класса Thread, он становится исполняемым кодом этого потока.<br>- **Ограничения**:  <br>    - Не возвращает результат.<br>    - Не бросает проверяемые исключения (за исключением тех, которые являются подклассами RuntimeException).                                                                                                                                                                                                                                                             |
| Callable    | **Определён в пакете java.util.concurrent**. Представляет задачу, которая может быть выполнена несколькими потоками и возвращает результат.<br>**Особенности**:<br>- Имеет единственный метод **V call() throws Exception**, который возвращает значение типа V (обобщённый тип) и может выбрасывать исключения.<br>- Метод call() содержит код, который будет выполняться в отдельном потоке.<br>- Когда объект Callable передаётся в ExecutorService и запускается с помощью метода submit(), он возвращает объект **Future**, который представляет результат выполнения задачи. Объект Future позволяет получить результат, проверить статус задачи и отменить её выполнение.<br>**Ограничение**: в отличие от Runnable, экземпляр Callable не может быть передан в конструктор класса Thread — его нужно обернуть в объект Future (например, FutureTask) и передать в класс Thread. |

### Какие методы используются для приостановки и возобновления работы потока?

| Под темы      | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Приостановка  | Некоторые методы для приостановки работы потока в Java:<br>- **Thread.sleep()** — статический метод класса Thread, приостанавливает выполнение текущего потока на заданное время (миллисекунды или наносекунды). Во время выполнения система перестаёт выделять потоку процессорное время, распределяя его между другими потоками.<br>- **Thread.join()** — приостанавливает выполнение текущего потока до тех пор, пока не завершится другой поток. Например, если есть два потока t1 и t2, и написан вызов t1.join(), то t2 не начнёт работу, пока t1 не завершит свою.<br>- **wait()** — освобождает монитор и переводит вызывающий поток в состояние ожидания до тех пор, пока другой поток не вызовет метод notify().<br>**Важно**: у методов, приостанавливающих выполнение потока, есть особенность — если во время их выполнения будет вызван метод interrupt() этого потока, они, не дожидаясь конца времени ожидания, сгенерируют исключение InterruptedException. |
| Возобновление | Для возобновления работы потока в Java используются следующие методы:<br>- **notify()** — продолжает работу потока, у которого ранее был вызван метод wait().<br>- **notifyAll()** — возобновляет работу всех потоков, у которых ранее был вызван метод wait().<br>**Важно**: все эти методы вызываются только из синхронизированного контекста — синхронизированного блока или метода.<br>**Ранее** для приостановки и возобновления работы потока также использовались методы **suspend()** и **resume()**, но они были объявлены устаревшими, и их использование не рекомендуется. Вместо принудительного завершения потока применяется схема, в которой каждый поток сам ответственен за своё завершение. Поток может остановиться либо тогда, когда он закончит выполнение метода run(), (main() — для главного потока), либо по сигналу из другого потока.                                                                                                             |

### Чем отличается метод sleep() от wait()?

| Под темы      | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Метод sleep() | **Статический метод класса Thread**. **Временно приостанавливает выполнение текущего потока на указанный период времени** (измеряется в миллисекундах).<br>**Особенности**:<br>- Поток переходит в состояние TIMED_WAITING.<br>- Операционная система исключает поток из планировщика.<br>- По истечении времени поток возвращается в состояние RUNNABLE.<br>- **Не требует внешнего сигнала для пробуждения** — поток автоматически продолжит выполнение после истечения указанного времени ожидания. Однако поток в состоянии sleep() также можно прервать с помощью метода interrupt(), что приведёт к генерации исключения InterruptedException.<br>**Использование**: метод sleep() лучше всего применять в сценариях, когда нужно отложить выполнение потока на определённый период времени, ожидая ресурс. Часто используется для имитации длительной операции или временного интервала между задачами во время модульного тестирования.<br>**Важно**: метод sleep() не гарантирует точное время приостановки — это минимальное время, после которого поток может быть возобновлён. Реальное время может быть больше из-за загрузки системы, приоритетов потоков и других факторов.                                                                  |
| Метод wait()  | **Нестатический метод, определённый в классе Object**. **Освобождает блокировку объекта** и переводит текущий поток в состояние ожидания до тех пор, пока другой поток не вызовет методы notify() или notifyAll() на том же объекте-мониторе.<br>**Особенности**:<br>- Поток снимает блокировку, которую он удерживал, что позволяет другому потоку получить доступ к объекту и изменить его состояние.<br>- После того, как другой поток вызывает notify() или notifyAll(), текущий поток может продолжить своё выполнение.<br>**Использование**: метод wait() обычно используется для решения шаблона «производитель-потребитель» или других сценариев синхронизации.<br>**Требования к контексту вызова**: метод wait() должен вызываться только в синхронизированном контексте (synchronized-блок или метод). Вызывающий поток должен владеть монитором объекта, на котором вызывается wait(). При несоблюдении этих условий будет выброшено исключение IllegalMonitorStateException.<br>**Важно**: механизм wait/notify требует точной координации между потоками — если notify() будет вызван до wait(), то уведомление будет потеряно, и поток может остаться в состоянии ожидания навсегда, если не будет вызван ещё один notify() или notifyAll(). |

### Как работает метод interrupt() и в каких случаях он используется?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **Метод interrupt() класса Thread в Java работает, устанавливая флаг прерывания для потока**, и используется для прерывания работы потока. Принудительно останавливать поток не рекомендуется — это может привести к ошибке.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| Принцип работы | Метод interrupt() устанавливает **внутренний флаг прерывания потока** в true. Это указывает, был ли поток прерван. Проверить статус прерывания можно с помощью метода **isInterrupted** — он возвращает true, если поток был прерван, и false, если нет.<br>**Особенности работы в разных случаях**:<br>- **Если поток находится в состоянии ожидания** (например, выполняется метод sleep(), wait()) — вызов interrupt() прерывает ожидание, и выбрасывается исключение **InterruptedException**. В этом случае флаг прерывания сбрасывается.<br>- **Если поток не выполняет блокирующие операции** — прерывание не вызовет исключение. В таком случае поток должен сам регулярно проверять флаг прерывания с помощью isInterrupted(), чтобы корректно завершить выполнение.<br>**Важно**: прерывание не останавливает поток немедленно — это лишь сигнал, который поток может обработать. |
| Использование  | Метод interrupt() применяется в разных случаях, например:<br>- **Прерывание потока, который ждёт определённого события или выполнения условия**. Если это произошло, поток останавливается.<br>- **Прерывание потока, выполняющего блокирующую операцию**. Например, если поток засыпает на 1 секунду с помощью Thread.sleep(1000), через 3 секунды главный поток вызывает blockingThread.interrupt(), прерывая блокирующий поток.<br>- **Оповещение о завершении работы потока** — например, если поток должен работать вечно, но нужно отменить задание.                                                                                                                                                                                                                                                                                                                                  |

### Как реализовать безопасное взаимодействие между потоками при использовании общих ресурсов?

| Под темы                         | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| -------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Синхронизация (synchronized)     | **Ключевое слово synchronized** позволяет определить критические разделы кода или методы, к которым одновременно должен обращаться только один поток. Некоторые способы использования:<br>- **Синхронизированные методы** — объявляются с ключевым словом synchronized в сигнатуре. При вызове такого метода блокировка автоматически устанавливается на объект this (для нестатических методов) или на объект класса (для статических методов).<br>- **Синхронизированные блоки** — представляют блок кода, к которому одномоментно имеет доступ только один поток. Можно синхронизировать только часть метода, используя синхронизированные блоки. [<br>**Важно**: при использовании synchronized следует учитывать несколько принципов:[<br>- **Согласованность блокировок** — все операции, обращающиеся к общему ресурсу, должны использовать одну и ту же блокировку.<br>- **Гранулярность блокировок** — блокировать только то, что необходимо, на минимально возможное время.<br>- **Избегать вложенных блокировок** — они могут привести к взаимоблокировке (deadlock). |
| Блокировки (Locks)               | **Блокировки** — механизм синхронизации, который обеспечивает взаимное исключение для критических секций в многопоточной программе. Некоторые типы блокировок: <br>- **Exclusive Lock (ReentrantLock)** — только один поток может получить блокировку в один момент времени.<br>- **Shared Lock (ReadWriteLock)** — несколько потоков могут удерживать блокировку одновременно при определённых условиях.<br>**Организация блокировки**: для получения блокировки вызывается метод lock(), а после окончания работы с общими ресурсами — метод unlock(), который снимает блокировку. Важно оборачивать код в try-finally, чтобы ресурсы освободились даже в случае выброса исключения.                                                                                                                                                                                                                                                                                                                                                                                           |
| Атомарные операции (Atomic)      | **Атомарные классы** из пакета java.util.concurrent.atomic реализуют потокобезопасные операции с помощью механизма **compare-and-set (CAS)**. Например:<br>- **AtomicInteger** — предоставляет атомарные операции для целочисленных счётчиков, например, инкремент и декремент.<br>- **AtomicReference** — позволяет выполнять сложные атомарные операции, например, изменять ссылку на объект только если она указывает на ожидаемое значение.<br>**Особенности**:<br>- Атомарные операции выполняются как одна неделимая операция, что важно для предотвращения состояний гонки.<br>- Изменения, сделанные одним потоком, немедленно видны другим потокам.                                                                                                                                                                                                                                                                                                                                                                                                                     |
| Неизменяемые объекты (Immutable) | **Неизменяемые объекты** (классы, все поля которых являются final и инициализируются при создании) **автоматически потокобезопасны**, поскольку не могут быть изменены после создания. Некоторые преимущества:<br>- После создания объекты можно передавать другим потокам, и они всегда будут в актуальном состоянии.<br>- Не нужно проверять, не устарело ли состояние экземпляра и не модифицировал ли его другой поток.<br>**Важно**: следует различать действительно неизменяемые объекты и объекты с интерфейсом «только для чтения» — последние не являются в строгом смысле неизменяемыми и не могут быть использованы из разных потоков без предосторожностей.                                                                                                                                                                                                                                                                                                                                                                                                          |

### Какие стратегии можно использовать для управления состоянием потоков в многопоточной среде?

| Под темы      | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Синхронизация | **Синхронизация** позволяет контролировать доступ нескольких потоков к общим ресурсам (переменным, объектам, файлам). Некоторые стратегии:<br>- **Использование механизмов синхронизации**. Например:<br>    - **Мьютексы** — блокировки, которые гарантируют, что только один поток может работать с ресурсом в любой момент времени.<br>    - **Семафоры** — ограничивают количество потоков, которые могут одновременно работать с ресурсом.<br>    - **Ключевое слово synchronized** в Java — привязывает блок к определённому потоку, исключая доступ других потоков в этот момент.<br>- **Координация работы между потоками** с помощью методов wait(), notify() и notifyAll(). Например:<br>    - wait() заставляет текущий поток ожидать, пока другой поток не вызовет notify() или notifyAll() на том же объекте.<br>    - notify() пробуждает один случайно выбранный поток, который ожидает на этом объекте.<br>    - notifyAll() пробуждает все потоки, которые ожидают на этом объекте.<br>**Важно**: избыточная синхронизация или неэффективное распределение потоков могут снизить производительность, вызывая дополнительные накладные расходы, связанные с переключением контекста и ожиданием блокировок. |
| Планирование  | **Стратегии планирования** определяют, какие потоки планируются на выполнение. Некоторые подходы:<br>- **Алгоритмы с квантованием**. Каждому потоку предоставляется квант времени, в течение которого он может выполняться на процессоре. По истечении кванта операционная система переключает процессор на следующий поток в очереди.<br>- **Алгоритмы с приоритетами**. Каждому потоку назначается приоритет — целое число, обозначающее степень привилегированности потока. Операционная система при наличии нескольких готовых к выполнению потоков выбирает из них поток с наибольшим приоритетом.<br>- **Смешанные алгоритмы** — основаны на квантовании, но величина кванта и/или порядок выбора потока из очереди определяются приоритетами.<br>**Приоритет потока** может меняться в процессе выполнения, например, после завершения операции ввода-вывода могут увеличивать приоритет потока, чтобы дать ему возможность быстрее начать выполнение.                                                                                                                                                                                                                                                               |
| Пулы потоков  | **Пул потоков** — это управляемая коллекция потоков выполнения, которые ожидают назначения задач. Вместо создания нового потока для каждой задачи пул поддерживает группу доступных потоков и распределяет задачи между ними. Это уменьшает накладные расходы, связанные с созданием и уничтожением потоков.<br>**Некоторые особенности пулов потоков**:<br>- **Контроль количества активных потоков** — пул автоматически контролирует количество потоков, предотвращая истощение системных ресурсов.<br>- **Повторное использование потоков** — после завершения задачи поток возвращается в пул, что позволяет повторно использовать его для другой задачи.<br>- **Очередь задач** — пулы потоков могут использовать очередь задач для хранения задач, которые ожидают выполнения, что позволяет контролировать нагрузку на пул и управлять порядком выполнения задач.<br>**Когда стоит использовать пул потоков**: для задач, ограниченных вводом-выводом (I/O-bound), например, для работы с сетевыми запросами, операций с файлами, запросов к базам данных. Для задач, ограниченных процессором (CPU-bound), пул потоков менее эффективен.                                                                           |

### Каким образом можно реализовать паттерн «один производитель - много потребителей» в многопоточной программе на Java?

| Под темы                          | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| --------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                       | **Паттерн «один производитель — много потребителей» в многопоточной программе на Java можно реализовать с помощью очередей, каналов (pipes) или акторов (actors)**. Задача включает синхронизацию между потоками и потенциальную гонку между несколькими производителями и потребителями.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| С использованием очередей         | - **Использовать BlockingQueue**. Очередь сама «усыпляет» поток, если очередь пуста (для читателя) или переполнена (для писателя). Популярные реализации:<br>    - **ArrayBlockingQueue** — очередь фиксированного размера на массиве, честная (FIFO). <br>    - **LinkedBlockingQueue** — очередь на связном списке, имеет два лока (отдельно на take, отдельно на put), что повышает пропускную способность.<br>- **Использовать SynchronousQueue** — специальный тип BlockingQueue с нулевой ёмкостью, не хранит элементы, работает как место встречи двух потоков, где один передаёт объект прямо в руки другому. По умолчанию SynchronousQueue не гарантирует порядок доступа потоков, но есть режим «честной» очереди (FIFO — First In, First Out). <br>**Важно**: реализация завершения потоков зависит от конкретной ситуации. Одно из распространённых решений — добавление производителем в очередь особого элемента, который потребителем чётко будет распознаваться как конец добавления. |
| С использованием каналов (pipes)  | - **Использовать потоки ввода/вывода**. Например, PipedInputStream/PipedOutputStream — потоки ввода/вывода для передачи байтовых данных, PipedReader/PipedWriter — для передачи текстовой информации. Эти классы используются всегда в паре: либо поток ввода должен ссылаться на поток вывода, либо наоборот. Запись и чтение данных производятся в разных потоках.<br>**Важно**: использование трубы в одном потоке может привести к мёртвой блокировке. Если поток, производящий данные, уже не живёт, то трубопровод считается сломанным. Если поток назначения не успевает обработать получаемые данные, то поток-источник блокируется. Если поток назначения пытается прочитать данные, которых ещё нет, то он переходит в состояние ожидания.                                                                                                                                                                                                                                                  |
| С использованием акторов (actors) | - **Использовать асинхронный подход**. Результат выполнения задачи производителя передаётся напрямую потребителю и выполняется асинхронно, а производитель тем временем справляется со своей работой. Для этого можно использовать интерфейсы Supplier и Consumer, а для мелких задач — CompletableFuture.<br>**Важно**: при реализации паттерна «один производитель — много потребителей» необходимо соблюдать баланс по количеству потоков, так как каждый поток, кроме процессорного времени, запрашивает дополнительную память для своих нужд. Также переключение между потоками ресурсоёмко.                                                                                                                                                                                                                                                                                                                                                                                                     |

### Что такое valotile?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение    | **Volatile** — ключевое слово в программировании, которое **обозначает переменную, значение которой может быть изменено несколькими потоками одновременно**. Оно решает проблему кэширования значений в многопоточной среде: когда один поток изменяет переменную, а другой считывает её значение, без дополнительной синхронизации второй поток может продолжать видеть устаревшее значение из-за оптимизаций на уровне JVM, компилятора и процессора.<br>Некоторые свойства volatile:<br>- гарантирует, что операции чтения всегда вернут самое последнее значение переменной, записанное любым потоком;<br>- запрещает переупорядочивание операций чтения/записи относительно других volatile-операций;<br>- создаёт точки синхронизации, после которых все изменения, сделанные одним потоком, становятся видимыми для других потоков. |
| Принцип работы | Когда переменная объявлена как volatile, происходят следующие моменты:<br>- **Запись** в volatile-переменную всегда производится немедленно в основную память.<br>- **Чтение** volatile-переменной всегда происходит напрямую из основной памяти, аннулируя любое кэшированное значение.<br>- **volatile устанавливает отношение «happens-before»**: запись в volatile-переменную происходит после всех предыдущих чтений и записей в коде, чтение volatile-переменной — до всех последующих чтений и записей. Это означает, что volatile действует как «барьер памяти»: всё, что было до записи в volatile-переменную, гарантированно будет видно любому потоку, который после записи обратился к volatile-переменной.                                                                                                                    |
