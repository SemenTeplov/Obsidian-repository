| Определение                                   | Внедрение зависимостей — это процесс, при котором объекты определяют свои зависимости (то есть другие объекты, с которыми они работают) только через аргументы конструктора, аргументы фабричного метода или свойства, которые задаются для экземпляра объекта после его создания или возврата из фабричного метода. Затем контейнер внедряет эти зависимости при создании компонента. По сути, этот процесс противоположен (отсюда и название — «инверсия контроля») тому, когда сам компонент самостоятельно управляет созданием экземпляров или расположением своих зависимостей, используя прямое конструирование классов или шаблон проектирования «Локатор сервисов». |
| Внедрение зависимостей на основе конструктора | Внедрение зависимостей на основе конструктора осуществляется путем вызова контейнером конструктора с несколькими аргументами, каждый из которых представляет собой зависимость. Вызов статического фабричного метода с определенными аргументами для создания компонента практически эквивалентен вызову конструктора, и в этом обсуждении аргументы конструктора и статического фабричного метода рассматриваются одинаково.                                                                                                                                                                                                                                               |

%%Java%%
```
public class SimpleMovieLister { 
	// the SimpleMovieLister has a dependency on a MovieFinder 
	private final MovieFinder movieFinder; 
	
	// a constructor so that the Spring container can inject a MovieFinder 
	public SimpleMovieLister(MovieFinder movieFinder) { 
		this.movieFinder = movieFinder; 
	} 
	
	// business logic that actually uses the injected MovieFinder is omitted... }
```

%%Spring%%
```
// a constructor so that the Spring container can inject a MovieFinder 
class SimpleMovieLister(private val movieFinder: MovieFinder) { 
	// business logic that actually uses the injected MovieFinder is omitted... 
}
```

| Сопоставление аргументов конструктора | Сопоставление аргументов конструктора происходит с использованием типа аргумента. Если в аргументах конструктора определения компонента нет потенциальной неоднозначности, то порядок, в котором аргументы конструктора указаны в определении компонента, соответствует порядку, в котором эти аргументы передаются соответствующему конструктору при создании экземпляра компонента. |

%%Java%%
```
package x.y; 

public class ThingOne { 
	public ThingOne(ThingTwo thingTwo, ThingThree thingThree) { 
		// ... 
	} 
}
```

%%Kotlin%%
```
package x.y 

class ThingOne(thingTwo: ThingTwo, thingThree: ThingThree)
```

|     |                                                                                                                                                                                                                                                                                           |
| --- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
|     | Если предположить, что классы ThingTwo и ThingThree не связаны наследованием, потенциальной неоднозначности не возникает. Таким образом, следующая конфигурация работает корректно, и вам не нужно явно указывать индексы или типы аргументов конструктора в элементе <constructor-arg/>. |

%%XML%%
```
<beans> 
	<bean id="beanOne" class="x.y.ThingOne"> 
		<constructor-arg ref="beanTwo"/> 
		<constructor-arg ref="beanThree"/> 
	</bean> 
	
	<bean id="beanTwo" class="x.y.ThingTwo"/> 
	<bean id="beanThree" class="x.y.ThingThree"/> 
</beans>
```

|     |                                                                                                                                                                                                                                                                                        |
| --- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
|     | При ссылке на другой компонент тип известен, и сопоставление возможно (как в предыдущем примере). При использовании простого типа, например <value>true</value>, Spring не может определить тип значения и поэтому не может выполнить сопоставление по типу без дополнительной помощи. |

%%Java%%
```
package examples; 

public class ExampleBean { 
	// Number of years to calculate the Ultimate Answer 
	private final int years; 
	
	// The Answer to Life, the Universe, and Everything 
	private final String ultimateAnswer; 
	
	public ExampleBean(int years, String ultimateAnswer) { 
		this.years = years; this.ultimateAnswer = ultimateAnswer; 
	} 
}
```

%%Kotlin%%
```
package examples 

class ExampleBean( 
	private val years: Int, 
	
	// Number of years to calculate the Ultimate Answer 
	
	private val ultimateAnswer: String 
	
	// The Answer to Life, the Universe, and Everything )
```

|     |                                                                                                                                                                   |
| --- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
|     | В предыдущем примере контейнер может использовать сопоставление типов с простыми типами, если вы явно укажете тип аргумента конструктора с помощью атрибута type. |

%%XML%%
```
<bean id="exampleBean" class="examples.ExampleBean"> 
	<constructor-arg type="int" value="7500000"/> 
	<constructor-arg type="java.lang.String" value="42"/> 
</bean>
```

|     |                                                                             |
| --- | --------------------------------------------------------------------------- |
|     | С помощью атрибута index можно явно указать индекс аргументов конструктора. |

%%XML%%
```
<bean id="exampleBean" class="examples.ExampleBean"> 
	<constructor-arg index="0" value="7500000"/> 
	<constructor-arg index="1" value="42"/> 
</bean>
```

|     |                                                                                             |
| --- | ------------------------------------------------------------------------------------------- |
|     | Для устранения неоднозначности значений можно также использовать имя параметра конструктора |

%%XML%%
```
<bean id="exampleBean" class="examples.ExampleBean"> 
	<constructor-arg name="years" value="7500000"/> 
	<constructor-arg name="ultimateAnswer" value="42"/> 
</bean>
```

%%Java%%
```
package examples; 

public class ExampleBean { 
	// Fields omitted 
	@ConstructorProperties({"years", "ultimateAnswer"}) 
	public ExampleBean(int years, String ultimateAnswer) { 
		this.years = years; this.ultimateAnswer = ultimateAnswer; 
	} 
}
```

%%Kotlin%%
```
package examples 

class ExampleBean 
	@ConstructorProperties("years", "ultimateAnswer") 
	constructor(val years: Int, val ultimateAnswer: String)
```

| Внедрение зависимостей на основе сеттеров | Внедрение зависимостей на основе сеттеров осуществляется за счет того, что контейнер вызывает методы сеттеров для ваших бинов после вызова конструктора без аргументов или статического фабричного метода без аргументов для создания экземпляра бина. |

%%Java%%
```
public class SimpleMovieLister { 
	// the SimpleMovieLister has a dependency on the MovieFinder 
	private MovieFinder movieFinder; 
	
	// a setter method so that the Spring container can inject a MovieFinder 
	public void setMovieFinder(MovieFinder movieFinder) { 
		this.movieFinder = movieFinder; 
	} 
	
	// business logic that actually uses the injected MovieFinder is omitted... 
}
```

%%Kotlin%%
```
class SimpleMovieLister { 
	// a late-initialized property so that the Spring container can inject a MovieFinder 
	lateinit var movieFinder: MovieFinder 
	
	// business logic that actually uses the injected MovieFinder is omitted... 
}
```

| --- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
|     | ApplicationContext поддерживает внедрение зависимостей на основе конструкторов и сеттеров для управляемых им компонентов. Он также поддерживает внедрение зависимостей на основе сеттеров после того, как некоторые зависимости уже были внедрены через конструктор. Вы настраиваете зависимости в виде BeanDefinition, которые используете вместе с экземплярами PropertyEditor для преобразования свойств из одного формата в другой. Однако большинство пользователей Spring работают не с этими классами напрямую (то есть программно), а с XML-определениями бинов, аннотированными компонентами (то есть классами, аннотированными с помощью @Component, @Controller и т. д.) или методами @Bean в классах @Configuration на основе Java. Эти источники преобразуются во внутренние экземпляры BeanDefinition и используются для загрузки всего экземпляра контейнера Spring IoC. |

|                                 |                                                                                                                                                                                                                                                                                                                                         |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Процесс разрешения зависимостей | ApplicationContext создается и инициализируется с помощью метаданных конфигурации, описывающих все компоненты. Метаданные конфигурации можно задать с помощью XML, кода Java или аннотаций.                                                                                                                                             |
|                                 | Для каждого компонента его зависимости выражаются в виде свойств, аргументов конструктора или аргументов метода статической фабрики (если вы используете его вместо обычного конструктора). Эти зависимости передаются компоненту при его создании.                                                                                     |
|                                 | Каждое свойство или аргумент конструктора — это фактическое определение устанавливаемого значения или ссылка на другой компонент в контейнере.                                                                                                                                                                                          |
|                                 | Каждое свойство или аргумент конструктора, представляющий собой значение, преобразуется из указанного формата в фактический тип этого свойства или аргумента конструктора. По умолчанию Spring может преобразовывать значения, представленные в строковом формате, во все встроенные типы, такие как int, long, String, boolean и т. д. |
