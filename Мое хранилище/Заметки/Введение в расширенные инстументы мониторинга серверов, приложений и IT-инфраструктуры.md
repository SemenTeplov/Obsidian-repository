### Что такое мониторинг серверов и для чего он используется?

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение            | **Мониторинг серверов** — это непрерывный процесс сбора, анализа и визуализации данных о состоянии, производительности и доступности серверного оборудования, сетевого оборудования, сервисов и  приложений.<br>Это комплексная система инструментов и процедур, позволяющая администраторам в реальном времени отслеживать ключевые параметры работы IT‑инфраструктуры — как   физических серверов, так и виртуальных машин, облачных решений. |
| Для чего  используется | **Предотвращение простоев и сбоев**<br>    - раннее обнаружение проблем (перегрузка CPU, нехватка памяти, дискового пространства);<br>    - оперативное реагирование до возникновения критических ситуаций;<br>    - минимизация времени недоступности сервисов и финансовых потерь.                                                                                                                                                            |
|                        | **Оптимизация производительности**<br>    - выявление «узких мест» (bottlenecks) в работе системы;<br>    - балансировка нагрузки между ресурсами;<br>    - повышение эффективности использования оборудования; <br>    - снижение затрат на инфраструктуру за счёт рационального распределения ресурсов.                                                                                                                                       |
|                        | **Обеспечение безопасности**<br>    - обнаружение подозрительной активности (попытки подбора паролей, аномальный трафик);<br>    - выявление признаков атак (DDoS, несанкционированный доступ);<br>    - мониторинг журналов безопасности и событий;<br>    - предотвращение утечек данных.                                                                                                                                                     |
|                        | **Планирование ресурсов и масштабирования**<br>    - анализ трендов нагрузки за длительные периоды;<br>    - прогнозирование потребности в дополнительных мощностях;<br>    - обоснованное принятие решений о апгрейде оборудования или миграции в облако;<br>    - избежание аварийных ситуаций из‑за нехватки ресурсов.                                                                                                                       |
|                        | **Соблюдение договорных обязательств (SLA)**<br>    - объективный учёт времени бесперебойной работы (uptime);<br>    - контроль времени отклика сервисов;<br>    - предоставление отчётности клиентам и аудиторам;<br>    - подтверждение выполнения условий сервисных соглашений.                                                                                                                                                              |
|                        | **Улучшение качества обслуживания**<br>    - повышение доступности сервисов для пользователей;<br>    - сокращение времени реагирования на инциденты;<br>    - поддержание высокого уровня удовлетворённости клиентов.                                                                                                                                                                                                                          |
|                        | **Анализ и отчётность**<br>    - накопление исторических данных о работе системы;<br>    - выявление закономерностей и причин сбоев;<br>    - подготовка аналитических отчётов для руководства;<br>    - обоснование инвестиций в IT‑инфраструктуру.                                                                                                                                                                                            |

### Перечислите три базовых метрики, которые обычно мониторятся для сервера.

| Под темы                                       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                       |
| ---------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Нагрузка на  процессор  (CPU Load)             | - **Что показывает:** процент времени, в течение которого  процессор занят выполнением задач.<br>- **Почему важно:** перегрузка CPU (> 80 %) приводит к замедлению работы сервера и росту времени отклика.<br>- **Ключевые показатели:**<br>    - средняя нагрузка за 1/5/15 минут (_load average_);<br>    - текущий процент использования CPU.                                                                              |
| Использование  оперативной памяти  (RAM Usage) | - **Что показывает:** долю занятой оперативной памяти от общего объёма.<br>- **Почему важно:** нехватка RAM заставляет систему использовать SWAP (диск), что резко снижает производительность.<br>- **Критические пороги:**<br>    - > 75–85 % — сигнал о возможной нехватке ресурсов;<br>    - > 90 % — риск сбоев из‑за перегрузки (_OOM killer_ в Linux).                                                                  |
| Скорость  дисковой  подсистемы  (Disk I/O)     | - **Что показывает:** производительность чтения/записи данных на диски.<br>- **Почему важно:** медленные диски — частая причина «тормозов» сервера, особенно при работе с базами данных.<br>- **Ключевые метрики:**  <br>    - **IOPS** (операций ввода‑вывода в секунду): для HDD — 100–200, для SSD — тысячи; <br>    - **Latency** (время задержки ответа диска);<br>    - **Throughput** (пропускная способность в MB/s). |

### Какое программное обеспечение вы знаете для мониторинга IT-инфраструктуры?

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Zabbix                 | **Zabbix** — универсальная система мониторинга с открытым исходным кодом, которая позволяет отслеживать состояние и производительность серверов, сетевого оборудования, приложений и различных IT-сервисов в режиме реального времени. <br>**Ключевые возможности:**<br>- мониторинг серверов и рабочих станций (нагрузка на CPU, использование памяти, дисковое пространство, температура оборудования);<br>- сетевой мониторинг (скорость соединений, доступность серверов, задержки, состояние сетевых интерфейсов);<br>- отслеживание приложений и баз данных (количество пользователей, скорость отклика, количество запросов в секунду);<br>- контроль состояния операционных систем (работа процессов, критические обновления, появление/исчезновение файлов и процессов);<br>- бизнес-метрики и аналитика (количество регистраций пользователей, сумма заказов, конверсия и т. д.);<br>- автоматизация действий при инцидентах (перезагрузка процессов, очистка места на диске, перезапуск сервера);<br>- гибкие оповещения через email, SMS, мессенджеры (Telegram, Slack и др.);<br>- визуализация данных в виде графиков и дашбордов. <br>Zabbix масштабируется от небольших сетей до крупных корпоративных инфраструктур, поддерживает интеграцию через API и работает с различными базами данных (MySQL, PostgreSQL, Oracle и др.). |
| Nagios Core            | **Nagios Core** — открытое решение для мониторинга IT-инфраструктуры, включая сети, хосты, сервисы и приложения.<br>**Основные функции:**<br>- мониторинг сетевых служб (SMTP, POP3, HTTP, NNTP, PING и др.);<br>- контроль ресурсов хостов (загрузка процессора, использование диска и т. д.);<br>- поддержка плагинов для расширения возможностей мониторинга;<br>- параллельная проверка сервисов;<br>- определение иерархии сетевых хостов с помощью «родительских» хостов;<br>- уведомления при возникновении проблем и их устранении (через email, пейджер или пользовательские методы);<br>- поддержка обработки событий (event handlers) для проактивного решения проблем;<br>- автоматическая ротация логов. <br>Nagios Core работает на Linux и других UNIX-подобных системах, имеет веб-интерфейс для просмотра состояния сети, истории уведомлений и логов.                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| Prometheus + Grafana   | **Prometheus** — система мониторинга и оповещений с открытым исходным кодом, разработанная для современных облачных и контейнерных сред. **Grafana** — веб-приложение для визуализации данных, которое может работать с различными источниками данных, включая Prometheus. <br>**Особенности связки:**<br>- Prometheus собирает и хранит метрики в формате временных рядов, поддерживает язык запросов PromQL для анализа данных.<br>- Grafana преобразует сырые данные Prometheus в графики, диаграммы и дашборды, обеспечивая наглядную визуализацию. <br>- Prometheus поддерживает различные типы метрик: Counter (счётчики), Gauge (текущие значения), Histogram (группировка наблюдений). <br>- Возможно масштабирование системы через федерацию (локальные инстансы Prometheus собирают метрики, центральный инстанс агрегирует их) и шардинг. <br>Эта связка стала де-факто стандартом для мониторинга в современных дата-центрах.                                                                                                                                                                                                                                                                                                                                                                                                        |
| ManageEngine OpManager | **ManageEngine OpManager** — комплексное решение для мониторинга сетей и серверов.<br>**Ключевые возможности:**<br>- автоматическое обнаружение сетевых устройств и их картографирование;<br>- мониторинг физических и виртуальных серверов, WAN-мониторинг, анализ NetFlow;<br>- контроль хранилищ (RAID, ленточные библиотеки, коммутаторы fabrics);<br>- обнаружение сбоев с помощью реального мониторинга и анализа логов;<br>- интеграция с ServiceNow, REST API и другими системами;<br>- автоматизация рабочих процессов (перезапуск служб, резервное копирование конфигурации и т. д.);<br>- дашборды и отчёты для оптимизации использования ресурсов.<br>OpManager подходит для средних и крупных предприятий, поддерживает до 30 000 устройств.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| LogicMonitor           | **LogicMonitor** — SaaS-решение для комплексного мониторинга IT-инфраструктуры с использованием искусственного интеллекта.<br>**Функции:**<br>- автоматическое обнаружение ресурсов;<br>- мониторинг облачных сервисов, серверов, виртуальных машин, сетевых устройств;<br>- предиктивные предупреждения и аномалия-детекция на основе машинного обучения;<br>- анализ логов и корреляция с метриками;<br>- синтетический мониторинг (имитация пользовательских взаимодействий);<br>- интеграция с ServiceNow, PagerDuty, Slack, Teams и другими инструментами;<br>- дашборды и отчёты.<br>LogicMonitor ориентирован на средние и крупные организации, поддерживает более 3000 интеграций. <br>Выбор инструмента зависит от конкретных задач, масштаба инфраструктуры, бюджета и технических требований. При выборе стоит учитывать такие факторы, как поддержка нужных протоколов, возможность интеграции с другими системами, уровень автоматизации и доступность аналитики.                                                                                                                                                                                                                                                                                                                                                                   |

| Ключевые отличия | 1. **Уровень абстракции**<br>- **Серверы**: мониторинг «железа» и ОС — CPU, RAM, диск, сеть, температура, состояние RAID и т. п.<br>- **Приложения**: мониторинг на уровне ПО — запросы, сессии, транзакции, очереди, ошибки, время отклика, пулы соединений. <br>2. **Основные метрики**<br>- **Серверы**:<br>    - загрузка CPU (в %);<br>    - использование RAM (в % и МБ);<br>    - I/O диска (IOPS, latency);<br>    - сетевой трафик (Mbps, пакеты/с);<br>    - температура компонентов.<br>- **Приложения**:<br>    - количество запросов в секунду (RPS);<br>    - время обработки запроса (response time, latency);<br>    - число ошибок (HTTP 5xx, исключения);<br>    - длина очередей (message queues, DB connections);<br>    - количество активных сессий/пользователей;<br>    - потребление памяти _процессом_ (а не системой в целом);<br>    - время выполнения ключевых транзакций (например, оформление заказа)<br>3. **Цели мониторинга**<br>- **Серверы**: предотвратить аппаратные сбои, перегрев, исчерпание ресурсов; обеспечить стабильность инфраструктуры.<br>- **Приложения**: гарантировать качество пользовательского опыта, выявить баги и узкие места в коде, оптимизировать бизнес‑процессы.<br>4. **Инструменты и агенты**<br>- **Серверы**: чаще используют универсальные системы (Zabbix, Nagios, Prometheus) с агентами на ОС.  <br>- **Приложения**: требуют специализированных APM‑решений (Application Performance Monitoring), которые:<br>    - встраиваются в код (инструментирование);    <br>    - отслеживают вызовы между сервисами (tracing);    <br>    - анализируют логи и стеки ошибок;    <br>    - интегрируются с фреймворками (например, Spring Boot Actuator, OpenTelemetry).  <br>5. **Контекст ошибок**<br>- **Серверы**: ошибка обычно означает проблему с ресурсом (не хватает памяти, диск полон).<br>- **Приложения**: ошибка может быть логической (некорректный расчёт, тайм‑аут БД) даже при здоровых ресурсах. Важно понимать _где_ и _почему_ она возникла в коде.<br>6. **Масштабируемость и динамика**<br>- **Серверы**: параметры меняются относительно медленно (загрузка CPU за минуту).<br>- **Приложения**: метрики могут скачкообразно расти/падать за секунды (всплеск запросов, утечка памяти в процессе).<br>7. **Визуализация и отчётность**<br>- **Серверы**: дашборды с графиками ресурсов, алерты по порогам.<br>- **Приложения**: трейс‑диаграммы (call graphs), карты зависимостей сервисов, отчёты по транзакциям, анализ пользовательских сессий.<br>8. **Реакция на инциденты**<br>- **Серверы**: перезапуск сервиса, добавление ресурсов, замена оборудования.<br>- **Приложения**: деплой фиксов, оптимизация запросов, масштабирование микросервисов, откат версии. |

| Основные  методы  сбора  данных  в мониторинге  серверов | 1. **Агентский мониторинг**<br>    - **Суть**: на сервер устанавливается специальная  программа‑агент, которая собирает метрики и передаёт их в центральную систему.<br>    - **Что собирает**: загрузка CPU/RAM/диска, сетевые метрики, логи, статус служб, процессы.    <br>    - **Примеры инструментов**: Zabbix Agent, Datadog Agent, SolarWinds Agent.<br>    - **Плюсы**: высокая детализация, поддержка кастомных скриптов, низкая задержка.<br>    - **Минусы**: нагрузка на сервер, необходимость развёртывания и обновления агентов.<br>2. **Безагентный мониторинг (через стандартные протоколы)**<br>    - **SNMP** (Simple Network Management Protocol):<br>        - собирает данные о сетевых устройствах, серверах, СХД;<br>        - примеры: загрузка интерфейсов, температура, статус блоков питания;<br>        - инструменты: Cacti, Zabbix (SNMP-шаблоны).<br>    - **WMI** (Windows Management Instrumentation):<br>        - для серверов на Windows: процессы, службы, события, реестр;<br>        - инструменты: Nagios (с плагинами), PRTG.<br>    - **SSH/PowerShell/WinRM**:<br>        - выполнение команд удалённо (например, `df -h`, `Get-Process`);<br>        - инструменты: Ansible + мониторинг, custom scripts.<br>    - **Плюсы**: не требует установки ПО на сервере, простота развёртывания.<br>    - **Минусы**: меньшая детализация, зависимость  от настроек безопасности (firewall, учётные записи).<br>3. **Сбор логов (Log Collection)**<br>    - **Суть**: анализ файлов журналов (syslog, приложения, БД, веб‑серверы).<br>    - **Что собирают**: ошибки, предупреждения, запросы,  аутентификации, изменения конфигурации.<br>    - **Инструменты**:<br>        - централизованные системы: ELK Stack (Elasticsearch + Logstash + Kibana), Graylog;<br>        - агенты: Filebeat, Winlogbeat, NXLog.<br>    - **Плюсы**: глубокий анализ причин инцидентов, аудит безопасности.<br>    - **Минусы**: большой объём данных, необходимость настройки фильтров.<br>4. **HTTP/HTTPS‑проверки (Synthetic Monitoring)**<br>    - **Суть**: имитация запросов к сервисам (веб‑сервер, API, БД).<br>    - **Что проверяют**: доступность, время отклика, коды HTTP, содержимое страниц.<br>    - **Примеры**:<br>        - проверка статуса сервиса через `curl http://server/health`;<br>        - мониторинг API с валидацией JSON‑ответов.<br>    - **Инструменты**: Zabbix Web Scenario, Prometheus Blackbox Exporter, Pingdom.<br>    - **Плюсы**: оценка пользовательского опыта,  обнаружение проблем на уровне приложения.<br>    - **Минусы**: не отражает внутреннюю загрузку сервера.<br>5. **Сбор метрик через API сервисов**<br>    - **Суть**: запрос данных у встроенных API ОС или приложений<br>    - **Примеры**:<br>        - Linux: `/proc/meminfo`, `/sys/class/net/`, `systemd-journal`;<br>        - Docker: API для статистики контейнеров (`docker stats`);<br>        - Kubernetes: Metrics Server + Prometheus;<br>        - облачные платформы: AWS CloudWatch API, Azure Monitor API.<br>    - **Инструменты**: Prometheus (с экспортерами), custom scripts.<br>    - **Плюсы**: точность, доступ к специфичным  метрикам (например, очереди RabbitMQ).<br>    - **Минусы**: требует настройки аутентификации, может нагружать API.<br>6. **Сетевой мониторинг (NetFlow/sFlow, Packet Capture)**<br>    - **Суть**: анализ трафика между серверами и внешними узлами.<br>    - **Что собирают**: объём трафика, топология  соединений, аномалии (DDoS, сканеры).<br>    - **Инструменты**: <br>        - NetFlow‑коллекторы: SolarWinds NetFlow Traffic Analyzer, nfdump;<br>        - packet capture: Wireshark, tcpdump (для детального анализа).<br>    - **Плюсы**: выявление узких мест в сети, расследование инцидентов.<br>    - **Минусы**: высокие требования к хранилищу, сложность интерпретации.<br>7. **Мониторинг через виртуализацию/контейнеры**<br>    - **Суть**: сбор метрик от гипервизоров или оркестраторов.<br>    - **Примеры**:<br>        - VMware vSphere API (загрузка VM, хранилище);<br>        - Docker/containerd (использование CPU/памяти контейнерами);<br>        - Kubernetes (Pod status, resource requests/limits).<br>    - **Инструменты**: vRealize Operations, cAdvisor + Prometheus, Kubectl metrics.<br>    - **Плюсы**: видимость на уровне инфраструктуры и приложений.<br>    - **Минусы**: зависимость от платформы виртуализации.<br>8. **Пассивный мониторинг (на стороне сети)**<br>    - **Суть**: перехват и анализ трафика без вмешательства в работу сервера.<br>    - **Примеры**: SPAN‑порты, TAP‑устройства.<br>    - **Инструменты**: Wireshark, Suricata (IDS/IPS).<br>    - **Плюсы**: нулевая нагрузка на сервер.<br>    - **Минусы**: ограниченная детализация, требует сетевого оборудования. |

### Опишите, как можно использовать мониторинг для предотвращения сбоев в IT-инфраструктуре.

| Под темы                           | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Как мониторинг  предотвращает сбои | **Раннее обнаружение аномалий**<br>    - **Как работает:** система отслеживает метрики (загрузка  CPU, память, диск, сеть) и сравнивает их с базовыми значениями (_базелайнами_). При отклонении — сразу оповещает.<br>    - **Пример:** загрузка диска выросла с 30 % до 85 % за 10 минут. Система предупреждает: «Свободного места останется < 10 % через 2 часа». Админ успевает очистить логи или расширить хранилище.<br>    - **Инструменты:** Zabbix, Prometheus + Alertmanager, Datadog. |
|                                    | **Предиктивная аналитика (прогнозирование)**<br>    - **Как работает:** алгоритмы на основе исторических данных предсказывают перегрузки.<br>    - **Пример:** система видит, что каждую пятницу в 18:00 нагрузка на БД растёт на 200 %. За сутки до пика она отправляет уведомление: «Подготовьте дополнительные реплики».<br>    - **Инструменты:** Dynatrace, AppDynamics, кастомные ML‑модели на базе Prometheus.                                                                            |
|                                    | **Автоматическое реагирование**<br>    - **Как работает:** при достижении критических порогов система выполняет заранее заданные действия.<br>    - **Примеры:**<br>        - перезапуск зависшего сервиса;<br>        - масштабирование инстансов в облаке (если CPU > 90 % 5 мин);<br>        - отключение некритичных процессов при нехватке памяти.<br>    - **Инструменты:** Ansible + мониторинг, AWS Auto Scaling, Kubernetes Horizontal Pod Autoscaler.                                  |
|                                    | **Анализ логов и корреляция событий**<br>    - **Как работает:** сбор и анализ логов всех компонентов для выявления скрытых проблем.<br>    - **Пример:** система замечает серию предупреждений «DB  connection timeout» за последний час. Хотя сервер ещё работает, это сигнал: скоро начнутся ошибки 503. Админ проверяет пул соединений.<br>    - **Инструменты:** ELK Stack, Graylog, Splunk.                                                                                                |
|                                    | **Синтетический мониторинг (имитация пользователей)**<br>    - **Как работает:** регулярные запросы к ключевым сервисам (например, проверка API каждые 30 сек).<br>    - **Пример:** мониторинг показывает, что время ответа  API выросло с 200 мс до 2 с. Хотя пользователи ещё не жалуются, команда начинает расследование — выявляет утечку памяти в приложении.<br>    - **Инструменты:** Postman Monitor, Prometheus Blackbox Exporter.                                                     |
|                                    | **Контроль зависимостей и топологии**<br>    - **Как работает:** карта сервисов показывает, как сбой одного компонента повлияет на другие.<br>    - **Пример:** мониторинг фиксирует падение узла кэша.  Система предупреждает: «Через 10 мин нагрузка на БД вырастет в 5 раз». Команда заранее масштабирует БД.<br>    - **Инструменты:** Jaeger (распределённая трассировка), New Relic.                                                                                                       |
|                                    | **Отслеживание ресурсных лимитов**<br>    - **Как работает:** контроль квот (CPU/RAM/диск/соединения) с прогнозом исчерпания.<br>    - **Пример:** контейнер использует 95 % лимита памяти. Мониторинг предупреждает: «Через 40 мин будет OOM‑килл». Админ увеличивает лимит или оптимизирует код.<br>    - **Инструменты:** cAdvisor, Kubernetes Metrics Server.                                                                                                                                |
|                                    | **Мониторинг безопасности как профилактика сбоев**<br>    - **Как работает:** обнаружение атак (DDoS, брутфорс), которые могут перегрузить систему.<br>    - **Пример:** система фиксирует 1 000 запросов в секунду с одного IP. Блокирует его до того, как сервер упадёт.<br>    - **Инструменты:** Suricata, OSSEC, Cloudflare Rate Limiting.                                                                                                                                                  |
|                                    | **Анализ трендов и планирование мощностей**<br>    - **Как работает:** долгосрочный сбор метрик для прогнозирования потребности в ресурсах.<br>    - **Пример:** график показывает, что дисковое пространство растёт на 10 GB/неделю. За месяц до исчерпания система создаёт задачу: «Заказать дополнительный RAID».<br>    - **Инструменты:** Grafana + Prometheus, custom reporting.                                                                                                           |
|                                    | **Интеграция с процессами изменений (Change Management)**<br>    - **Как работает:** мониторинг сразу после деплоя или конфигурации проверяет, не вызвал ли изменение деградацию.<br>    - **Пример:** после обновления ПО время ответа API выросло на 300 %. Система откатывает версию автоматически.<br>    - **Инструменты:** Splunk IT Service Intelligence, PagerDuty.                                                                                                                      |

### Какие типы алертов (предупреждений) наиболее эффективны при мониторинге критически важных систем? Обоснуйте свой ответ.

| Под темы                                     | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| -------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Критические  алерты                          | Требуют немедленного вмешательства в любое время суток. Срабатывают при полном отказе сервиса, недоступности критически важных систем или ситуациях, которые могут привести к потере данных. <br>**Примеры:**<br>- недоступность ключевого сервиса (например, по данным внешнего мониторинга);<br>- полный отказ базы данных;<br>- критический сбой в работе критически важного приложения;<br>- резкое падение доступности инфраструктуры (например, потеря связи с основным маршрутизатором). <br>**Обоснование:** такие алерты предотвращают масштабные простои и потерю данных, что критично для непрерывности бизнеса. Они должны быть минимальными по количеству (в идеале — не более 1–2 в месяц для стабильной инфраструктуры).                                                                                                                                                                                                                                                                                       |
| Предупреждающие алерты                       | Сигнализируют о проблемах, которые могут привести к критической ситуации, если их не устранить в разумные сроки. <br>**Примеры:**<br>- заполнение диска до 85% (с дополнительным алертированием при достижении 5%);<br>- рост времени отклика приложения выше определённого порога (например, на 20% от нормы);<br>- увеличение количества ошибок (например, 4xx или 5xx ошибок в веб-сервере сверх установленного лимита);<br>- превышение нагрузки на CPU или память (например, CPU idle < 10% в течение 10 минут или CPU Load Average превышает количество доступных процессоров и не равно 0 в течение 5 минут); <br>- высокая нагрузка на дисковую подсистему (нагрузка на диск > 95% в течение 1 часа); <br>- аномальный скачок входящего или исходящего трафика, что может указывать на DDoS-атаку. <br>**Обоснование:** позволяют заранее выявить потенциальные проблемы и устранить их до того, как они приведут к сбою. Например, предупреждение о заполнении диска даёт время на очистку или расширение хранилища. |
| Алерты на основе динамических порогов        | Статические пороги часто неэффективны, так как нагрузка на системы имеет предсказуемые паттерны (например, в рабочее время и ночью она различается). Динамические пороги учитывают исторические данные и сезонные паттерны, адаптируясь к изменениям в нагрузке. <br>**Примеры:**<br>- использование процентилей вместо абсолютных значений (например, «CPU выше 95-го процентиля за последние 30 дней»);<br>- применение алгоритмов машинного обучения для выявления аномалий в поведении системы. <br>**Обоснование:** снижают количество ложных срабатываний и повышают точность оповещений, так как система учитывает нормальное поведение инфраструктуры.                                                                                                                                                                                                                                                                                                                                                                |
| Алерты с временными окнами                   | Кратковременные всплески метрик — нормальное явление для многих систем. Временные окна помогают отфильтровать такие флуктуации. <br>**Пример:** алерт срабатывает, только если условие выполняется в течение определённого времени (например, «CPU выше 80% в течение последних 5 минут»). <br>**Обоснование:** предотвращают ложные срабатывания из-за кратковременных сбоев или плановых задач.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| Коррелированные алерты                       | Современные IT-системы сложны и взаимосвязаны. Отказ одного компонента может вызвать каскад алертов от зависимых систем. Корреляция позволяет группировать связанные события и отправлять обобщённые уведомления. <br>**Пример:** вместо трёх отдельных сообщений приходит одно: «Обнаружена проблема с производительностью сервера приложений. Затронуты: CPU, время отклика, количество ошибок». <br>**Обоснование:** сокращает информационный шум и помогает быстрее определить корневую причину проблемы.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| Алерты на ключевые бизнес-метрики (SLI, SLO) | Построение алертинга основывается на концепциях SLI (Service Level Indicators), SLA (Service Level Agreement) и SLO (Service Level Objectives). Алерты настраивают на значения, которые находятся между SLO и Error Budget — когда поведение системы ещё можно назвать штатным, но если ничего не предпринять, есть риск выйти за пределы SLA. <br>**Пример:** алерт на превышение допустимого времени обработки запроса (latency) или доли ошибок в общем числе запросов.[]()<br>**Обоснование:** позволяет поддерживать качество обслуживания на уровне, согласованном с бизнес-требованиями, и избегать нарушений SLA.                                                                                                                                                                                                                                                                                                                                                                                                     |

### Какие стратегии можно использовать для масштабирования системы мониторинга в больших распределенных средах?

| Под темы           | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ключевые стратегии | Горизонтальное масштабирование (шардирование)<br>- **Суть:** добавление новых узлов мониторинга  для распределения нагрузки.<br>- **Как реализовать:**<br>    - развернуть несколько инстансов  сборщиков данных;<br>    - разделить зоны ответственности  по географическим регионам или сервисам;<br>    - использовать балансировщики  нагрузки между узлами.<br>- **Плюсы:** почти неограниченная  масштабируемость,  высокая доступность.<br>- **Инструменты:** Kubernetes для оркестрации,  HAProxy/NGINX  для балансировки. |
|                    | Вертикальное масштабирование<br>- **Суть:** наращивание ресурсов на  существующих узлах (CPU, RAM, диск).<br>- **Когда применять:** при умеренном росте нагрузки и  ограниченном числе мониторинговых точек.<br>- **Ограничения:** физический предел оборудования, риск  единой точки отказа.<br>- **Рекомендации:** заранее планировать апгрейды,  мониторить утилизацию ресурсов.                                                                                                                                                |
|                    | Гибридное масштабирование<br>- **Суть:** комбинация горизонтального  и вертикального подходов.<br>- **Реализация:**<br>    - критически важные компоненты  (база метрик) масштабировать вертикально;<br>    - второстепенные сервисы (агенты, парсеры) — горизонтально.<br>- **Плюс:** оптимальное соотношение затрат  и производительности.                                                                                                                                                                                       |
|                    | Автоматическое масштабирование (autoscaling)<br>- **Суть:** динамическое изменение  ресурсов по триггерам.<br>- **Триггеры:**<br>    - загрузка CPU > 80 % более 5 мин;<br>    - очередь сообщений > 10 000 элементов;<br>    - задержка обработки метрик > 30 сек.<br>- **Инструменты:** AWS Auto Scaling, Kubernetes  Horizontal Pod Autoscaler.<br>- **Плюс:** экономия ресурсов в периоды низкой нагрузки.                                                                                                                     |
|                    | Иерархическая архитектура мониторинга<br>- **Суть:** многоуровневая система с  локальными агрегаторами.<br>- **Уровни:**<br>    1. **Edge‑узлы** — собирают данные на местах  (в филиалах, облачных зонах).<br>    2. **Региональные агрегаторы** — обрабатывают данные по зонам.<br>    3. **Центральный хаб** — консолидирует информацию для аналитики.<br>- **Плюс:** снижение нагрузки на канал,  локальная обработка аварий.                                                                                                  |
|                    | Оптимизация хранения данных<br>- **Стратегии:**<br>    - **Ротация метрик:** хранение детальных данных за  последние 24 ч, агрегированных — за 30 дней.<br>    - **Партиционирование:** разделение базы по времени/сервисам.<br>    - **Сжатие данных:** использование колоночных  БД (ClickHouse, TimescaleDB).<br>- **Инструменты:** Prometheus + Thanos, InfluxDB.                                                                                                                                                              |
|                    | Фильтрация и агрегация на стороне агентов<br>- **Суть:** предварительная обработка  данных до отправки в центральную систему.<br>- **Что фильтровать:**<br>    - дублирующиеся события;<br>    - низкоприоритетные логи;<br>    - метрики вне пороговых значений.<br>- **Плюс:** сокращение трафика, снижение  нагрузки на хранилище.                                                                                                                                                                                              |
|                    | Использование потоковой обработки (streaming)<br>- **Суть:** анализ данных в реальном  времени без полного хранения.<br>- **Сценарии:**<br>    - обнаружение аномалий на лету;<br>    - триггеры для автомасштабирования;<br>    - оповещения о критических событиях.<br>- **Инструменты:** Apache Kafka, Apache  Flink, AWS Kinesis.                                                                                                                                                                                              |
|                    | Геораспределённая инфраструктура<br>- **Суть:** размещение узлов  мониторинга в разных регионах.<br>- **Плюсы:**<br>    - снижение задержки сбора данных;<br>    - устойчивость к региональным сбоям;<br>    - соответствие требованиям локализации данных.<br>- **Реализация:** мульти‑облачные развёртывания,  CDN для статических дашбордов.                                                                                                                                                                                    |
|                    | Модульная архитектура<br>- **Суть:** разделение системы на независимые компоненты.<br>- **Модули:**<br>    - сбор данных (агенты, безагентные сборщики);<br>    - обработка (агрегация, корреляция);<br>    - хранение (базы метрик, логи);<br>    - визуализация (дашборды, отчёты);<br>    - алертинг (триггеры, нотификации).<br>- **Плюс:** возможность масштабировать  отдельные компоненты независимо.                                                                                                                       |
