### Что такое JVM?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **JVM (Java Virtual Machine)** — **виртуальная машина Java**, основная часть исполняющей системы Java (Java Runtime Environment, JRE). Исполняет байт-код Java, предварительно созданный из исходного кода программы компилятором Java (javac).<br>JVM обеспечивает **независимость Java-программ от аппаратной платформы и операционной системы**. Это реализуется принципом WORA (write once, run anywhere) — «напиши один раз, запусти где угодно».                                                                                                                                                                                                                                                                                                                                                                                                    |
| Функции     | Некоторые функции JVM:<br>- **Загрузка классов**. Загрузчик классов (ClassLoader) загружает классы в память JVM.<br>- **Проверка байт-кода**. Перед выполнением программы байт-код проходит верификацию, которая позволяет определить, корректно ли была написана и скомпилирована программа на Java. <br>- **Управление памятью**. JVM автоматически управляет памятью: выделяет необходимый объём для объектов, отслеживает её использование, при наличии ненужных объектов освобождает память, предотвращает её утечки. <br>- **Оптимизация кода** во время выполнения (JIT-компиляция). JIT-компилятор преобразует байт-код в нативный машинный код, что повышает производительность.<br>- **Поддержка других языков программирования**. Например, некоторые из них изначально создавались для JVM (Groovy, Scala, Closure, Ceylon, Kotlin и другие). |

### В чем разница между JDK и JRE?

| Под темы                       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| JDK (Java Development Kit)     | **Предназначён для разработки приложений на языке Java**. Содержит JRE, а также дополнительные инструменты разработки. Некоторые компоненты JDK:<br>- Java-компилятор (javac) — преобразует исходный код .java в байт-код .class;<br>- отладчик (jdb) — позволяет искать и исправлять ошибки в программах;<br>- архиватор (jar) — создаёт архивы Java-приложений;<br>- генератор документации (javadoc) — формирует документацию по API;<br>- инструменты мониторинга и анализа производительности (jvisualvm, jconsole);<br>- заголовочные файлы и библиотеки для разработки нативных методов.<br>**Особенности**:<br>- JDK платформо-зависим, доступен для разных операционных систем: Windows, macOS, Linux, Solaris.<br>- В состав JDK не входит интегрированная среда разработки Java, поэтому разработчик, использующий только JDK, вынужден использовать внешний текстовый редактор и компилировать программы, используя утилиты командной строки.                        |
| JRE (Java Runtime Environment) | **Обеспечивает среду выполнения, необходимую для запуска Java-программ**. Не включает инструменты разработки, такие как компиляторы или отладчики.<br>**Компоненты JRE**:<br>- Java Virtual Machine (JVM) — виртуальная машина для выполнения байт-кода;<br>- библиотеки классов Java (Java Class Library) — набор стандартных классов и API;<br>- интеграционные компоненты для взаимодействия с операционной системой;<br>- инструменты базовой конфигурации и настройки среды выполнения.<br>**Особенности**:<br>- JRE достаточно для запуска скомпилированных Java-приложений, но не предоставляет возможностей для их создания или компиляции.<br>- Это минимальная установка для конечных пользователей, которым нужно только выполнять Java-программы.<br>**Важно**: JRE можно использовать отдельно от JDK, если нужно не разрабатывать, а только запускать приложения. В то же время JDK не применяется без JRE — программы без запуска и отладки создавать невозможно. |

### Какие основные компоненты JVM вы знаете?

| Под темы                                               | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Class Loader                                           | **Отвечает за загрузку классов** в память JVM. Некоторые функции загрузчика:<br>- **Загрузка** — поиск и чтение бинарных данных класса.<br>- **Связывание** — проверка, подготовка и разрешение символических ссылок.<br>- **Инициализация** — выполнение статических инициализаторов.<br>Java использует иерархическую систему загрузчиков классов:<br>- **Bootstrap ClassLoader** — загружает основные классы Java (rt.jar).<br>- **Extension ClassLoader** — загружает классы из стандартных расширений.<br>- **Application ClassLoader** — загружает классы из classpath приложения.<br>- **Пользовательские загрузчики** — могут быть определены разработчиком.                                                                                                                                                                                                                             |
| Область данных времени выполнения (Runtime Data Areas) | **Выделяются для хранения данных** во время выполнения программы. Некоторые области:<br>- **Куча** — разделяется между всеми потоками приложения, в ней хранятся объекты и массивы, созданные в процессе выполнения программы.<br>- **Стек** — JVM создаёт отдельный стек для каждого потока, в стеке хранится информация о вызовах методов, локальные переменные и промежуточные результаты вычислений.<br>- **Область методов** — разделяется между всеми потоками приложения, хранит данные класса, такие как имена методов, имена переменных и значения констант.<br>- **Пул констант** — структура данных в области методов, в которой хранятся константы, такие как строковые литералы, имена классов и имена методов, на которые ссылается байткод Java.<br>- **Регистры PC** — область памяти, в которой хранится адрес текущей исполняемой инструкции байткода Java для каждого потока. |
| Исполнительный движок (Execution Engine)               | **Исполняет байт-код** и преобразует его в машинные инструкции. Некоторые компоненты Execution Engine:<br>- **Interpreter** — читает инструкции байт-кода и выполняет их одну за другой.<br>- **Just-In-Time Compiler (JIT-компилятор)** — преобразует часто выполняемые части байт-кода в машинный код, который может выполняться напрямую процессором. Это ускоряет выполнение программы по сравнению с постоянной интерпретацией.<br>- **Garbage Collector** — управляет памятью JVM, автоматически освобождая её от объектов, которые больше не используются.                                                                                                                                                                                                                                                                                                                                |
| Система безопасности (Security Manager)                | **Управляет внешней границей песочницы JVM** — контролирует, как код, выполняющийся внутри JVM, может взаимодействовать с ресурсами за пределами JVM. Некоторые функции Security Manager:<br>- **Использование политики безопасности** — набора определённых разрешений для разных классов кода. Менеджер сравнивает действия, запрашиваемые приложениями, с политикой безопасности: если действие разрешено, он разрешает его, если нет — запрещает.<br>- **Настройка политики** — можно определять разрешения на основе местоположения кода или его подписи.<br>- **Проверка действий** — перед выполнением потенциально опасных операций Java API проверяет разрешение у менеджера безопасности.                                                                                                                                                                                              |

### Как JVM выполняет Java-программу?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **JVM (Java Virtual Machine, виртуальная машина Java) выполняет Java-программу, загружая скомпилированный байт-код и интерпретируя его построчно**.<br>Процесс запуска Java-программы включает несколько этапов:<br>1. **Написание исходного кода**. Программист создаёт файл с расширением .java.<br>2. **Компиляция**. Компилятор преобразует исходный код в байт-код — набор инструкций, которые не может выполнить процессор напрямую, но понимает JVM. Полученный файл с расширением .class содержит универсальный код, одинаково работающий на любой платформе.<br>3. **Загрузка**. Загрузчик классов (ClassLoader) загружает байт-код в память JVM.<br>4. **Верификация**. Проверяется корректность байт-кода.<br>5. **Исполнение**. JVM интерпретирует байт-код построчно.<br>Чтобы ускорить выполнение, в JVM встроен **JIT-компилятор (Just-In-Time)**. Он анализирует, какие части кода исполняются чаще всего, и переводит их в нативные инструкции процессора.<br>Во время выполнения программы JVM может динамически загружать новые классы, управлять потоками и обрабатывать исключения. Также JVM поддерживает многопоточность, что позволяет создавать и управлять несколькими потоками выполнения одновременно.<br>После завершения всех операций программа прекращает работу. JVM освобождает все задействованные ресурсы, завершает все потоки и очищает память. |

### Какие задачи выполняет сборщик мусора в JVM?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **Сборщик мусора (Garbage Collector, GC) в виртуальной машине Java (JVM) автоматически освобождает память, выделенную для объектов, которые больше не используются**. Объект считается «мусором», если на него нет активных ссылок.<br>Также GC может участвовать в **оптимизации распределения памяти**, например, путём сжатия оставшейся кучи для предотвращения фрагментации памяти.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| Принцип работы | Процесс сборки мусора в JVM происходит в несколько этапов:<br>1. **Маркировка** — определение живых объектов.<br>2. **Очистка** — освобождение памяти, занятой мёртвыми объектами.<br>3. **Компактификация (опционально)** — перемещение живых объектов для уменьшения фрагментации.<br>Сборщик мусора просматривает весь граф объектов в памяти, начиная с «корней» (например, стека вызова и глобальных ссылок) и следуя ссылкам на другие объекты.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| Алгоритмы      | В JVM существует несколько алгоритмов сборки мусора, каждый из которых имеет свои особенности и подходит для различных типов приложений:<br>- **Serial GC** — используется в системах с ограниченными ресурсами, таких как небольшие встроенные системы или приложения с низким потреблением памяти.<br>- **Parallel GC** — подходит для многопроцессорных систем, где важна производительность, и может значительно ускорить процесс сборки мусора.<br>- **CMS (Concurrent Mark Sweep)** — разработан для приложений, требующих меньших задержек, и работает, пытаясь минимизировать время простоя приложения.<br>- **G1 (Garbage-First)** — более современный и универсальный, предназначен для приложений с большим объёмом памяти и требованиями к низкой задержке.<br>- **ZGC (Z Garbage Collector) и Shenandoah** — экспериментальные алгоритмы, ориентированные на системы с очень большим объёмом памяти и стремлением к минимально возможным задержкам. |

### Какие типы сборщиков мусора вы знаете в JVM?

| Под темы                    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| --------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Serial (Последовательный)   | **Самая простая реализация GC**. Предназначена для небольших приложений, работающих в однопоточных средах.<br>**Принцип работы**: для сборки мусора используется один поток. В начале каждой сборки мусора работа основных потоков приложения останавливается и возобновляется только после окончания сборки.<br>**Особенности**:<br>- Подходит для клиентских приложений, которым не важны короткие паузы.<br>- Запуск сборщика приводит к событию «остановки мира», когда всё приложение приостанавливает работу.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Parallel (Параллельный)     | **Реализация GC по умолчанию**, также известна как сборщик пропускной способности. Предназначен для приложений со средними и большими наборами данных, которые выполняются на многопроцессорном или многопоточном оборудовании.<br>**Принцип работы**: для сборки мусора используется несколько потоков. Несколько потоков предназначены для малой сборки мусора в молодом поколении, единственный поток занят основной сборкой мусора в старшем поколении.<br>**Особенности**:<br>- Подходит для многопоточной среды, когда требуется завершить много задач и допустимы длительные паузы, например при выполнении пакетного задания.<br>- Запуск параллельного GC также вызывает «остановку мира», и приложение зависает.                                                                                                                                                                                                                                                                           |
| CMS (Concurrent Mark Sweep) | **Призван минимизировать время паузы в работе приложения**. Работает за счёт одновременной маркировки доступных объектов во время функционирования приложения. Фаза очистки, которая очищает недоступные объекты, также выполняется параллельно, что сокращает время паузы.<br>**Особенности**:<br>- Подходит для приложений, в которых низкая задержка является приоритетом по сравнению с максимальной пропускной способностью, например для интерактивных приложений.<br>- Несмотря на уменьшение времени паузы, может привносить накладные расходы из-за одновременного выполнения операций.                                                                                                                                                                                                                                                                                                                                                                                                     |
| G1                          | **Разработан для многопоточных приложений с большим доступным размером кучи** (более 4 ГБ).<br>**Принцип работы**: разбивает кучу на набор областей одинакового размера и использует несколько потоков для их сканирования. G1-сборщик определяет регионы с наибольшим количеством мусора и сначала выполняет сбор мусора в них.<br>**Особенности**:<br>- Освобождает пространство в основном с помощью эвакуации: живые объекты, найденные в выбранных для сбора областях памяти, копируются в новые области памяти, уплотняя их в процессе.<br>- Чередует две фазы: фаза только для молодого поколения и фаза восстановления пространства.<br>**Особенности**:<br>- Используется по умолчанию, можно явно включить его, указав -XX:+UseG1GC в командной строке.<br>- Не является сборщиком в реальном времени — пытается достичь установленных целевых значений времени паузы с высокой вероятностью в течение более длительного времени, но не всегда с абсолютной уверенностью для данной паузы. |

### Опишите процесс загрузки классов в JVM

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Процесс загрузки классов в виртуальной машине Java (JVM) динамический**: классы загружаются не заранее, а по требованию (ленивая загрузка). Это происходит с помощью **загрузчика классов (Classloader)**. Он отвечает за поиск файлов классов и их загрузку в память JVM.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| Этапы       | Процесс загрузки классов в JVM включает три этапа:<br>1. **Загрузка** — поиск и импорт бинарных данных для типа по его имени, создание класса или интерфейса из этого бинарного представления.<br>2. **Связывание (линковка)** — установка связей между классами и проверка их на наличие несоответствий или ошибок. Состоит из трёх этапов:<br>    - **Верификация** — JVM убеждается, что загруженные файлы классов соответствуют структуре и ограничениям, указанным в спецификации языка Java. Некорректные или вредоносные файлы классов на этом этапе отклоняются.<br>    - **Подготовка** — JVM инициализирует статические поля, методы и другие ресурсы, необходимые для выполнения класса. Присваивает статическим полям значения по умолчанию и выделяет под них память.<br>    - **Разрешение** — на этом этапе происходит разрешение символьных ссылок в файлах классов путём замены их на прямые ссылки, такие как адреса методов и смещения полей. Этот процесс выполняется динамически во время выполнения программы.<br>3. **Инициализация** — JVM запускает все статические блоки кода в классе и присваивает статическим полям начальные значения, указанные в файле класса. Инициализация происходит лениво — только при первом обращении к классу или его статическим членам. Если класс не используется, он может так и остаться неинициализированным. |

### Какие особенности HotSpot JVM вы можете назвать?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Некоторые особенности виртуальной машины Java HotSpot (JVM)**:<br>- **Технологии динамической компиляции JIT** и адаптивной оптимизации для повышения производительности.<br>- **Две версии виртуальной машины** — Client и Server. Client-версия настроена для лучшего выполнения приложений в клиентской среде, Server-версия — для максимальной скорости выполнения программы в серверной среде.<br>- **Поддержка нескольких сборщиков мусора**.<br>- **Поддержка флагов** для настройки работы виртуальной машины при запуске.                                                                                                                                                                    |
| Архитектура | HotSpot JVM состоит из трёх основных подсистем:<br>1. **Подсистема загрузчиков классов** — отвечает за нахождение и загрузку .class-файлов в память.<br>2. **Области данных времени выполнения** — управляют памятью, необходимой для выполнения программ.<br>3. **Механизм выполнения (Execution Engine)** — включает интерпретатор и компиляторы JIT, выполняющие байт-код, а также интегрированные сборщики мусора.<br>Во время исполнения программа использует несколько типов областей памяти, часть из которых общая для всех потоков, а часть — индивидуальна для каждого потока. Например, куча (heap) — основная область динамической памяти, где выделяются память под все объекты и массивы. |

### Как JVM управляет памятью, включая стек и кучу?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Виртуальная машина Java (JVM) автоматически управляет памятью, включая стек и кучу**. Разработчику не нужно явно выделять и освобождать память — за это отвечает подсистема JVM, известная как **Garbage Collector (сборщик мусора)**.<br>JVM выделяет различные области памяти (Runtime Data Areas) для различных типов хранения данных и манипулирования ими во время выполнения программы. Часть из них общая для всех потоков, часть — выделяется отдельно для каждого потока.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| Стек        | **Стек** используется для хранения данных о выполнении метода, включая локальные переменные, аргументы метода и адреса возврата. Некоторые особенности управления памятью в стеке: <br>- **Каждый поток имеет свой собственный стек**. Данные, хранящиеся в стеке, доступны только в области видимости соответствующего потока.<br>- **При вызове метода** создаётся новый фрейм стека, который содержит информацию для выполнения метода.<br>- **JVM выполняет код метода** с помощью локальных переменных и параметров, хранящихся в фрейме стека.<br>- **По завершении работы метода** фрейм стека удаляется, и управление передаётся обратно вызывающему методу.<br>- **Размер стека** обычно ограничен. Если программа попытается использовать больше памяти, чем доступно в стеке, это приведёт к ошибке переполнения стека.<br>- **Память для стека** не обязательно должна быть непрерывной — реализации JVM могут выделять и управлять памятью стека динамически в зависимости от платформы.                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| Куча        | **Куча (heap)** — это область памяти, где хранятся объекты. Некоторые особенности управления памятью в куче: <br>- **Объекты создаются в куче** с помощью ключевого слова new.<br>- **Куча разделена на несколько областей**:<br>    - **Young Generation** — здесь создаются новые объекты.<br>    - **Survivor Spaces (S0 и S1)** — сюда перемещаются объекты, пережившие сборку мусора в Eden.<br>    - **Old Generation** — для долгоживущих объектов.<br>    - **Permanent Generation / Metaspace** (в зависимости от версии JVM) — для метаданных классов.<br>- **Сборка мусора** — JVM автоматически освобождает память, занятую неиспользуемыми объектами. Процесс сборки мусора происходит в несколько этапов:<br>    - **Маркировка** — определение живых объектов.<br>    - **Очистка** — освобождение памяти, занятой мёртвыми объектами.<br>    - **Компактификация** (опционально) — перемещение живых объектов для уменьшения фрагментации.<br>- **Размер кучи** можно управлять с помощью флагов Xms (начальный размер) и Xmx (максимальный размер). Куча не передаётся виртуальной машине целиком — какая-то часть резервируется в качестве виртуального пространства, за счёт которого куча может в будущем расти.<br>**Важно**: поскольку доступ к объектам в куче может быть глобальным, для предотвращения проблем с несогласованностью данных в многопоточных приложениях необходимы механизмы синхронизации потоков. |
