### Что такое SQL инъекция?

| Под темы    | Заметка                                                                                                                                                                                                                                |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **SQL-инъекция (SQLi)** — способ взлома веб-приложений, при котором злоумышленник вставляет вредоносный SQL-код в запрос к базе данных. Это позволяет злоумышленникам получить неразрешённый доступ к данным, изменить их или удалить. |
|             | Уязвимости возникают, когда в приложении неадекватно фильтруется пользовательский ввод, и из-за этого возможно внедрять в запросы вредоносные данные.                                                                                  |

### Какие могут быть последствия использования SQL инъекций?

| Под темы    | Заметка                                                                                                                                                                                                                 |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Последствия | **Кража конфиденциальных данных** — злоумышленник может получить логины, пароли, адреса электронной почты, номера телефонов, данные банковских карт, паспортов.                                                         |
|             | **Удаление или подмена данных** — атакующий может выполнить команду DELETE, DROP TABLE или UPDATE, уничтожив критически важную информацию или исказив её.                                                               |
|             | **Получение контроля над сервером** — в некоторых случаях злоумышленник может получить доступ к серверу с размещёнными на нём приложениями, после чего сможет выполнять произвольный код, устанавливать вредоносное ПО. |
|             | **Нарушение работы сайта (DoS)** — зловредные запросы и циклические операции, запускаемые через SQL-инъекцию, потребляют много ресурсов и перегружают сервер базы данных.                                               |
|             | **Репутационные и юридические риски** — утечка персональных данных пользователей ведёт к потере доверия клиентов, судебным искам и штрафам по законам о защите данных.                                                  |

### В каких случаях приложение может быть уязвимо к SQL инъекциям?

| Под темы | Заметка                                                                                                                                     |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Причины  | **Ошибки разработчиков** — например, не проверка и не очистка передаваемых параметров (идентификаторов) перед использованием их в запросах. |
|          | **Недостаточная валидация** — поверхностная проверка входных данных.                                                                        |
|          | **Излишние права доступа к базе данных** — например, подключение с правами администратора для обычных операций.                             |

### Как можно предотвратить SQL инъекции при использовании динамически формируемых запросов?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                           |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Кодирование    | **Использовать параметризованные запросы (подготовленные операторы)**. Вместо конкатенации пользовательского ввода непосредственно в SQL-запрос данные передаются отдельно, что предотвращает интерпретацию ввода как части SQL-синтаксиса. Пример: ``SELECT * FROM users WHERE username = ? AND password = ?``.  |
|                | **Использовать хранимые процедуры** вместо динамических SQL-запросов. Хранимая процедура инкапсулирует SQL-логику и может помочь предотвратить SQL-инъекцию, ограничив доступ к структуре и функциональности базы данных.                                                                                         |
| Проверка ввода | **Валидировать и очищать входные данные**. Проверять данные на наличие вредоносных символов или шаблонов, удалять или экранировать их перед использованием в SQL-запросах.                                                                                                                                        |
| Права доступа  | **Следовать принципу наименьших привилегий** — предоставлять приложению минимально необходимые разрешения для пользователя базы данных.                                                                                                                                                                           |
| Мониторинг     | **Использовать межсетевые экраны веб-приложений (WAF)** — они анализируют входящие запросы и блокируют подозрительные шаблоны. WAF не заменяет безопасные методы кодирования, но может обеспечить дополнительный уровень защиты, особенно для устаревших приложений или когда исправление уязвимостей затруднено. |
|                | **Настроить логирование** всех ошибок БД и подозрительных вводов — по логам можно заметить попытки SQL-инъекций (например, регулярные ошибки синтаксиса, странные входные данные) и принять меры до того, как атака достигнет цели.                                                                               |

### Какие методы предотвращения SQL инъекций существуют в Java?

| Под темы                        | Заметка                                                                                                                                                                                                                           |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Prepared Statements             | **Позволяют отделить SQL-код от данных**. Вместо динамической вставки пользовательского ввода непосредственно в запрос используются параметризированные запросы. Это делает невозможным внедрение вредоносного кода.              |
| Input Validation                | **Проверяет пользовательский ввод** на соответствие ожидаемому формату и отсутствие вредоносных символов.                                                                                                                         |
| Stored Procedures               | **Позволяют инкапсулировать SQL-логику** на стороне базы данных, что снижает риск SQL-инъекций.                                                                                                                                   |
| Web Application Firewalls (WAF) | **Может автоматически блокировать подозрительные запросы**, включая попытки SQL-инъекций. WAF можно настроить на мониторинг входящего трафика, выявление подозрительных шаблонов и применение предварительно определённых правил. |

### Как работает PreparedStatement в Java в контексте предотвращения SQL инъекций?

| Под темы | Заметка                                                                                                                                                                                   |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **PreparedStatement в Java предотвращает SQL-инъекции, разделяя SQL-код и данные на уровне СУБД**. Это позволяет обрабатывать пользовательский ввод как данные, а не как исполняемый код. |

### Какие существуют специфичные техники обнаружения SQL инъекций при проведении тестирования на проникновение?

| Под темы      | Заметка                                                                                                                                                                                                                                                                                                                                                                                                               |
| ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ручные методы | **Добавление потенциально вредоносных SQL-запросов** в параметры запросов. Например, введение условий OR 1=1 или AND 1=1 для обхода аутентификации. После вставки подобных строк запрос отправляется на сервер, и анализируется реакция приложения. Если видны странные ошибки базы данных, неожиданные результаты или даже успешный вход в систему без правильных учётных данных, это может указывать на уязвимость. |
|               | **Проверка всех возможных точек входа**. Не только формы, но и URL-параметры, cookies и заголовки HTTP (User-Agent, Referer).                                                                                                                                                                                                                                                                                         |
|               | **Анализ ошибок и ответов сервера**. Важно обратить внимание на признаки, которые могут указывать на уязвимость: ошибки базы данных, изменённые или неожиданные данные в ответе, успешный вход в систему без правильных учётных данных.                                                                                                                                                                               |
| Инструменты   | **SQLMap** — автоматизированный инструмент с открытым исходным кодом для обнаружения и эксплуатации SQL-инъекций. Поддерживает различные типы SQL-инъекций (Boolean-based blind, Time-based blind и т. д.) и предоставляет детальные отчёты о найденных уязвимостях.                                                                                                                                                  |
|               | **Burp Suite** — инструмент для тестирования безопасности веб-приложений, включает встроенный сканер SQL-инъекций. Позволяет перехватывать запросы и модифицировать параметры на лету, что удобно для ручного исследования.                                                                                                                                                                                           |
|               | **OWASP ZAP** — инструмент для сканирования веб-приложений на уязвимости, включая SQL-инъекции. Имеет встроенные тесты на SQL-инъекции: пробует вставлять ' OR '1'='1 в поля, отправлять UNION SELECT и анализировать ответы.                                                                                                                                                                                         |

### Как можно использовать Content Security Policy для защиты от SQL инъекций?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение    | **Content Security Policy (CSP) можно использовать для защиты от SQL-инъекций**, ограничивая источники, из которых разрешено загружать контент на сайт. Это помогает предотвратить выполнение вредоносного кода, введённого злоумышленником, даже если он был вставлен на страницу.                                                                    |
| Принцип работы | CSP создаёт **список доверенных источников**, откуда разрешено загружать контент. Этот список включается в специальный HTTP-заголовок, который передаётся браузеру при каждом запросе страницы. Браузер проверяет, соответствует ли источник запрашиваемого ресурса этому списку. Если нет, ресурс блокируется, и попытка выполнения кода прерывается. |

### Какие существуют методы автоматизации обнаружения уязвимостей к SQL инъекциям в больших системах?

| Под темы            | Заметка                                                                                                                                                                                                                                             |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Статический анализ  | **SAST (Static Application Security Testing)** — проверяет исходный код приложения на наличие уязвимостей без запуска программы. Инструменты SAST ищут известные паттерны, например, небезопасную обработку пользовательских данных в SQL-запросах. |
| Динамический анализ | **DAST (Dynamic Application Security Testing)** — анализирует приложение в реальном времени, симулируя действия злоумышленника. Включает отправку запросов в приложение и наблюдение за его реакцией, чтобы выявить уязвимости.                     |
