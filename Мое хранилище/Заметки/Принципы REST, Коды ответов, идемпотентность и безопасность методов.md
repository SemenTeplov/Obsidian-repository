### Что такое REST?

| Под темы    | Заметка                                                                                                                                                                                                                                       |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **REST (Representational State Transfer, дословно — «передача состояния представления»)** — **архитектурный стиль взаимодействия компонентов распределённого приложения в сети**. Это набор правил для создания API на основе протокола HTTP. |
|             | Главная идея REST — представить все данные сервиса в виде ресурсов, у каждого из которых есть свой адрес (URL). Ресурсы могут быть, например, объектами, документами или коллекциями данных.                                                  |
|             | Если API соответствует принципам REST, его называют **RESTful API**. Термин REST предложил в 2000 году Рой Филдинг (один из создателей протокола HTTP) в своей диссертации.                                                                   |
| Принципы    | **Клиент-серверная архитектура** — клиент (обычно фронтенд или приложение) и сервер (бэкэнд, база данных) чётко разделены. Клиент отвечает за интерфейс пользователя, сервер — за хранение данных и логику.                                   |
|             | **Отсутствие состояния (Stateless)** — каждый запрос клиента не зависит от предыдущего и содержит всю информацию, необходимую для обработки на сервере. Сервер не хранит «контекст» между запросами.                                          |
|             | **Кэшируемость** — ответы сервера могут кэшироваться (сохраняться) на некоторое время, чтобы при повторных запросах не нагружать сервер повторной обработкой одинаковых данных.                                                               |
|             | **Единообразие интерфейса** — взаимодействие с разными ресурсами происходит единообразно, по общим правилам.                                                                                                                                  |

### Перечислите какие-либо коды ответов HTTP и их значения.

| Под темы        | Заметка                                                                                                                                                                                                                                                          |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение     | **Коды ответов HTTP** — трёхзначные числовые ответы сервера на запрос клиента, которые показывают, как сервер обработал запрос: успешно или с ошибкой. Коды делятся на пять классов: информационные, успешные, перенаправления, ошибки клиента и ошибки сервера. |
| Информационные  | **100 Continue** — сервер получил заголовки запроса, клиент может продолжать отправку тела запроса.                                                                                                                                                              |
|                 | **101 Switching Protocols** — сервер согласен переключиться на протокол, запрошенный клиентом (например, для WebSockets).                                                                                                                                        |
|                 | **102 Processing** — сервер получил и обрабатывает запрос, но ответ пока не готов.                                                                                                                                                                               |
|                 | **103 Early Hints** — сервер предварительно отправляет заголовки, чтобы клиент мог начать предзагрузку ресурсов.                                                                                                                                                 |
| Успешные        | **200 OK** — стандартный успешный ответ для большинства запросов.                                                                                                                                                                                                |
|                 | **201 Created** — ресурс был успешно создан (обычно используется после POST-запросов).                                                                                                                                                                           |
|                 | **202 Accepted** — запрос принят на обработку, но обработка ещё не завершена (асинхронные операции).                                                                                                                                                             |
|                 | **204 No Content** — запрос обработан успешно, но нет данных для возврата (часто используется в REST API).                                                                                                                                                       |
|                 | **206 Partial Content** — сервер отдаёт только часть ресурса (используется при частичных загрузках файлов).                                                                                                                                                      |
| Перенаправления | **301 Moved Permanently** — ресурс был окончательно перемещён по новому URI.                                                                                                                                                                                     |
|                 | **302 Found** — временное перенаправление (ранее называлось «Moved Temporarily»).                                                                                                                                                                                |
|                 | **304 Not Modified** — контент не изменился с момента последнего запроса (используется для кэширования).                                                                                                                                                         |
|                 | **307 Temporary Redirect** — временное перенаправление с сохранением метода HTTP-запроса.                                                                                                                                                                        |
|                 | **308 Permanent Redirect** — постоянное перенаправление с сохранением метода HTTP-запроса.                                                                                                                                                                       |
| Ошибки          | **400 Bad Request** — неверный синтаксис запроса или невалидные параметры.                                                                                                                                                                                       |
|                 | **401 Unauthorized** — требуется аутентификация пользователя.                                                                                                                                                                                                    |
|                 | **403 Forbidden** — доступ к ресурсу запрещён (даже при аутентификации).                                                                                                                                                                                         |
|                 | **404 Not Found** — запрашиваемый ресурс не найден на сервере.                                                                                                                                                                                                   |
|                 | **405 Method Not Allowed** — HTTP-метод не разрешён для данного ресурса.                                                                                                                                                                                         |
|                 | **429 Too Many Requests** — клиент отправил слишком много запросов (защита от DDoS).                                                                                                                                                                             |

### Какие HTTP методы считаются идемпотентными?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                        |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение | **Идемпотентными** считаются **корректно реализованные HTTP-методы**, при которых повторный идентичный запрос, отправленный один или несколько раз подряд, имеет один и тот же эффект, не изменяющий состояние сервера. Другими словами, идемпотентный метод при повторном выполнении не должен иметь никаких побочных эффектов, кроме сбора статистики или подобных операций. |
| Примеры     | **GET** — повторные GET-запросы к одной и той же сущности не изменяют её состояние на сервере, а только возвращают данные.                                                                                                                                                                                                                                                     |
|             | **PUT** — повторный PUT-запрос на одну и ту же сущность приводит к тому же результату, что и первый запрос.                                                                                                                                                                                                                                                                    |
|             | **DELETE** — повторные DELETE-запросы к одной и той же сущности не изменят состояния системы, если сущность уже была удалена.                                                                                                                                                                                                                                                  |
|             | **HEAD** — повторные HEAD-запросы не изменяют состояние сущности на сервере.                                                                                                                                                                                                                                                                                                   |
|             | **OPTIONS** — повторные OPTIONS-запросы не изменяют состояние сущности на сервере.                                                                                                                                                                                                                                                                                             |
|             | **TRACE** — повторные TRACE-запросы не изменяют состояние сущности на сервере.                                                                                                                                                                                                                                                                                                 |

### Как в контексте REST применяются принципы безопасности и идемпотентности?

| Под темы        | Заметка                                                                                                                                                                                                                                  |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение     | **Принципы безопасности и идемпотентности применяются в контексте REST API** через использование определённых HTTP-методов. Эти принципы важны для обеспечения безопасности данных и корректной работы клиент-серверного взаимодействия. |
| Безопасность    | **GET** — используется для чтения данных, не изменяет состояние сервера. Это позволяет кэшировать ответы, использовать предварительные запросы.                                                                                          |
|                 | **HEAD** и **OPTIONS** — также безопасные, не изменяют данные.                                                                                                                                                                           |
| Идемпотентность | **PUT** — предназначен для полного обновления ресурса по известному URI, сколько бы раз ни повторяли запрос, результат будет один.                                                                                                       |
|                 | **DELETE** — удаляет ресурс, удалить уже удалённый объект нельзя, повторный запрос не изменит ситуацию.                                                                                                                                  |

### Каково значение и использование кода ответа HTTP 204?

| Под темы | Заметка                                                                                                                                                                                                                                                |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ    | **Код ответа HTTP 204 («No Content»)** указывает, что **сервер успешно обработал запрос клиента, но не имеет контента для отправки обратно**.                                                                                                          |
|          | Другими словами, сервер говорит клиенту: «Всё прошло хорошо, но для вас нет ничего нового». В ответе передаются только заголовки без тела сообщения. Клиент не должен обновлять содержимое документа, но может применить к нему полученные метаданные. |

### В чем разница между методами PUT и PATCH в контексте REST?

| Под темы | Заметка                                                                                                                                                                                   |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Разница между методами PUT и PATCH в контексте REST заключается в том, как они применяют изменения**. Оба метода используются для обновления ресурсов на сервере, но имеют разные цели. |
| PUT      | Используется для полного обновления ресурса.                                                                                                                                              |
| PATCH    | Применяется для частичного обновления ресурса.                                                                                                                                            |

### Как REST использует принципы единообразия интерфейса и беспостредственного управления ресурсами?

| Под темы                              | Заметка                                                                                                                                                                                                                                                                                                                                                                                                 |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ                                 | **В архитектуре REST принципы единообразия интерфейса и непосредственного управления ресурсами (манипуляции с ресурсами) используются для упрощения взаимодействия между клиентами и серверами**. Эти принципы лежат в основе REST (Representational State Transfer — «передача состояния представления») — архитектурного стиля взаимодействия компонентов распределённой системы через протокол HTTP. |
| Единообразие интерфейса               | **Принцип единообразия интерфейса** в REST предполагает **унифицированный формат клиент-серверного взаимодействия**.                                                                                                                                                                                                                                                                                    |
| Непосредственное управление ресурсами | **Принцип непосредственного управления ресурсами** в REST реализуется через **использование стандартных методов HTTP** для выполнения операций с ресурсами.                                                                                                                                                                                                                                             |

### Какие существуют подходы к обработке версионирования в RESTful API?

| Под темы              | Заметка                                                                                                          |
| --------------------- | ---------------------------------------------------------------------------------------------------------------- |
| URI Versioning        | Номер версии включается в URI.                                                                                   |
| Media Type Versioning | **Номер версии включается в тип содержимого ответа**. Для указания версии используется заголовок «Content-Type». |
| Header Versioning     | Номер версии включается в заголовки HTTP запроса API.                                                            |

### Опишите, как в RESTful веб-сервисах реализуется кеширование и его влияние на производительность.

| Под темы                      | Заметка                                                                                                                                                                                                                                                                           |
| ----------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                   | **В RESTful веб-сервисах кеширование реализуется с помощью HTTP-заголовков**, которые указывают клиентам и прокси, можно ли кэшировать ответ и как долго. Цель — избегать повторной генерации данных и снижать нагрузку на сервер.                                                |
| Реализация                    | **Cache-Control**. Указывает клиентам и прокси, можно ли кэшировать ответ и как долго. Примеры: Cache-Control: no-cache — не кэшировать, Cache-Control: max-age=3600 — кэшировать на 1 час.                                                                                       |
|                               | **ETag (Entity Tag)**. Уникальный идентификатор версии ресурса. Клиент сохраняет ETag и при следующем запросе отправляет его в заголовке If-None-Match. Сервер сравнивает — если ETag не изменился, возвращает 304 Not Modified, без тела ответа.                                 |
|                               | **Last-Modified**. Альтернатива ETag: указывает время последнего изменения ресурса. Клиент отправляет If-Modified-Since, сервер возвращает 304 Not Modified, если данные не обновились.                                                                                           |
| Влияние на производительность | **Сокращает время ответа** — первый запрос клиента сохраняется в кеше для будущих запросов. Пока срок действия данных не истёк или не изменился, результаты API можно использовать снова и снова.                                                                                 |
|                               | **Снижает нагрузку на сервер** — кэширование действует как промежуточное программное обеспечение между клиентом и сервером: перехватывает запрос от клиента и обрабатывает данные запроса. Если данные ответа находятся в кеше, клиент может получить данные без участия сервера. |
|                               | **Повышает производительность приложения** — поскольку сервер освобождается от повторной обработки кэшированных данных, вместо этого он может выполнять другие операции.                                                                                                          |

### Объясните, что такое HATEOAS?

| Под темы    | Заметка                                                                                                                                                                                                                                                       |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **HATEOAS** (Hypermedia as the Engine of Application State, дословно «гипермедиа как движок для состояния приложения») — **архитектурные ограничения для REST-приложений**.                                                                                   |
|             | **Суть HATEOAS**: клиент взаимодействует с сетевым приложением, сервер которого обеспечивает динамический доступ через гипермедиа. При этом REST-клиенту не требуется заранее знать, как взаимодействовать с приложением или сервером за пределом гипермедиа. |
