### Что такое Maven и для чего он используется?

| Под темы                  | Заметка                                                                                                                                                                                                               |
| ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение               | **Maven** — это инструмент автоматизации сборки проектов с открытым исходным кодом, разработанный Apache Software Foundation. Изначально создан для Java‑разработки, но поддерживает и другие языки (C#, Ruby и др.). |
| Ключевые функции и задачи | **Управление зависимостями**                                                                                                                                                                                          |
|                           | Автоматически загружает нужные библиотеки из репозиториев                                                                                                                                                             |
|                           | Разрешает транзитивные зависимости (библиотеки, требуемые вашими зависимостями);                                                                                                                                      |
|                           | Устраняет конфликты версий;                                                                                                                                                                                           |
|                           | Поддерживает области видимости (например, зависимости только для тестов).                                                                                                                                             |
|                           | **Стандартный жизненный цикл сборки**                                                                                                                                                                                 |
|                           | - `$validate$` — проверка корректности проекта;                                                                                                                                                                       |
|                           | - `$compile$` — компиляция кода;                                                                                                                                                                                      |
|                           | - `$test$` — запуск тестов;                                                                                                                                                                                           |
|                           | - `$package$` — упаковка в JAR/WAR/EAR;                                                                                                                                                                               |
|                           | - `$verify$` — проверки качества;                                                                                                                                                                                     |
|                           | - `$install$` — размещение в локальном репозитории;                                                                                                                                                                   |
|                           | - `$deploy$` — публикация в удалённом репозитории.                                                                                                                                                                    |
|                           | **Репозитории**                                                                                                                                                                                                       |
|                           | - **локальный** (кэш на компьютере, папка `.m2`);                                                                                                                                                                     |
|                           | - **центральный** (публичное хранилище Maven Central);                                                                                                                                                                |
|                           | - **удалённые** (корпоративные или сторонние хранилища).                                                                                                                                                              |
|                           | **Плагины**                                                                                                                                                                                                           |
|                           | - `$maven-compiler-plugin$` (компиляция);                                                                                                                                                                             |
|                           | - `$maven-surefire-plugin$` (тестирование);                                                                                                                                                                           |
|                           | - `$spring-boot-maven-plugin$` (запуск Spring Boot‑приложений);                                                                                                                                                       |
|                           | - `$maven-javadoc-plugin$` (генерация документации) и др.                                                                                                                                                             |

### Какое основное преимущество Gradle по сравнению с Maven?

| Под темы                       | Заметка                                                                                                                                      |
| ------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ                          | Основное преимущество Gradle перед Maven —  **высокая гибкость и производительность** за счёт  ряда технологических и архитектурных решений. |
| Ключевые аспекты превосходства | **Гибкость конфигурации**                                                                                                                    |
|                                | - **Gradle**: использует Groovy или Kotlin DSL (Domain‑Specific Language) — это полноценный язык программирования внутри скрипта сборки.     |
|                                | - **Maven**: жёстко структурированный XML (`pom.xml`) без логики — только декларативное описание.                                            |
|                                | **Скорость сборки**                                                                                                                          |
|                                | - **Инкрементальные сборки** (`incremental builds`): Gradle анализирует изменения и пересоббирает _только затронутые части_ проекта.         |
|                                | - **Кэширование** (`build cache`): сохраняет результаты задач и повторно использует их при неизменных входных данных.                        |
|                                | - **Daemon‑процесс**: фоновый сервис, остающийся в памяти между запусками (снижает накладные расходы на инициализацию JVM).                  |
|                                | - **Параллельное выполнение**: поддерживает одновременный запуск задач и проектов на многоядерных процессорах.                               |
|                                | **Управление зависимостями**                                                                                                                 |
|                                | - Эффективнее разрешает конфликты в сложных микросервисных архитектурах.                                                                     |
|                                | - Поддерживает централизованное управление через _shared build scripts_.                                                                     |
|                                | - Быстрее загружает зависимости за счёт оптимизированного кэширования.                                                                       |
|                                | **Расширяемость**                                                                                                                            |
|                                | - Легко создавать **пользовательские задачи** и типы задач.                                                                                  |
|                                | - Интеграция с современными инструментами: Docker, Kubernetes, облачные платформы.                                                           |
|                                | - Поддержка _Configuration Cache_ для ускорения повторных запусков.                                                                          |
|                                | **Многопроектные сборки**                                                                                                                    |
|                                | - Встроенная поддержка `multi-project builds` без сложной настройки.                                                                         |
|                                | - Зависимости между подпроектами автоматически управляются.                                                                                  |
|                                | **Синтаксис и читаемость**                                                                                                                   |
|                                | - Gradle DSL лаконичнее XML.                                                                                                                 |

### Чем отличается Maven от Ant?

| Под темы                 | Заметка                                                                                                                                                                                                                                  |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Подход к конфигурации    | - **Ant**: _процедурный_ (императивный).  <br>Вы явно прописываете **каждую команду** и их порядок: <br>«скомпилируй это», «скопируй туда», «заархивируй вот так»_.  <br>Всё зависит от того, как вы опишете шаги в `build.xml`.         |
|                          | - **Maven**: _декларативный_.  <br>Вы описываете **что нужно получить** (модель проекта в `pom.xml`), а Maven сам решает, _как_ этого добиться, опираясь на встроенные правила.                                                          |
| Структура проекта        | - **Ant**: нет обязательных соглашений. Можно размещать исходники, ресурсы и скрипты где угодно — лишь бы вы явно указали пути в `build.xml`.                                                                                            |
|                          | - **Maven**: жёсткие соглашения (_convention over configuration_).  Стандартные папки: `src/main/java`, `src/test/java`, `target` и др. Если следовать им, многое работает «из коробки».                                                 |
| Управление зависимостями | **Ant**: нет встроенного механизма.                                                                                                                                                                                                      |
|                          | **Maven**: встроенная система зависимостей.                                                                                                                                                                                              |
| Жизненный цикл сборки    | **Ant**: нет предопределённого цикла.                                                                                                                                                                                                    |
|                          | - **Maven**: стандартный жизненный цикл с фазами:  <br>`validate` → `compile` → `test` → `package` →  `verify` → `install` → `deploy`. Каждая фаза включает набор задач по умолчанию; вы можете добавлять свои.                          |
| Скрипты и расширяемость  | **Ant**: XML‑скрипты (`build.xml`). Гибкость за счёт явного управления каждым шагом. Но скрипты часто получаются громоздкими и трудноподдерживаемыми. Плагины есть, но экосистема скромнее, чем у Maven.                                 |
|                          | **Maven**: XML‑POM (`pom.xml`) + плагины. Конфигурация короче за счёт соглашений. Огромная экосистема плагинов (сборка, тестирование, документация, деплой и т. п.). Можно настраивать поведение плагинов, но свобода меньше, чем в Ant. |
| Отчёты и интеграция      | **Ant**: не генерирует отчёты по умолчанию. Нужно подключать дополнительные инструменты.                                                                                                                                                 |
|                          | - **Maven**: встроенные отчёты (тестирование, покрытие, зависимости и др.) через плагины. Лучшая интеграция с IDE (IntelliJ, Eclipse) и CI/CD‑системами.                                                                                 |

### Как в Maven указать версию Java, используемую в проекте?

| Под темы         | Заметка                                                                                                                                                                                                                                       |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ            | В Maven версию Java задают через **плагин `maven- compiler-plugin`** — он отвечает за компиляцию кода.                                                                                                                                        |
| Основные способы | **Через свойства проекта** (простой и распространённый вариант) Указываются параметры `maven.compiler.source` и `maven.compiler.target`                                                                                                       |
|                  | **Через конфигурацию плагина** (более явный способ)  <br>В секции `<plugins>` прописываете `maven-compiler-plugin` и задаёте `source` и `target`                                                                                              |
|                  | **Для Java 9+** — через параметр `--release` (рекомендуется)  <br>Начиная с Java 9, лучше использовать `<release>` — он автоматически настраивает компиляцию для указанной версии платформы (заменяет `-source`, `-target` и `-bootclasspath` |

### Как в Gradle добавить зависимость из центрального репозитория Maven?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                    |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | Чтобы добавить зависимость из центрального репозитория  Maven в проект Gradle, нужно выполнить **два обязательных  шага** в файле `build.gradle` (или `build.gradle.kts` для Kotlin DSL).                                                                                                                                  |
| Шаги     | Указать репозиторий. Добавьте в блок `repositories` вызов `mavenCentral()` — это «алиас» для центрального репозитория Maven.                                                                                                                                                                                               |
|          | Добавить зависимость. В блоке `dependencies` укажите зависимость в формате:  <br>`{конфигурация} 'groupId:artifactId:version'`                                                                                                                                                                                             |
| Детали   | Конфигурации (scopes). `implementation` — зависимость доступна в коде и при сборке (аналог `compile` в Maven); `testImplementation` — только для тестов (аналог `test` в Maven); `runtimeOnly` — только на этапе выполнения (аналог `runtime` в Maven); `compileOnly` — только для компиляции (аналог `provided` в Maven). |
|          | Формат записи. Сокращённая форма `'groupId:artifactId:version'` эквивалентна развёрнутой.                                                                                                                                                                                                                                  |
|          | Обновление зависимостей. <br>    Gradle автоматически:<br>    - проверяет наличие библиотеки в локальном кэше; <br>    - при отсутствии — загружает из `mavenCentral()`;<br>    - разрешает транзитивные зависимости.                                                                                                      |
|          | Проверка доступности. Убедиться, что зависимость есть в центральном репозитории, можно на сайте                                                                                                                                                                                                                            |

### Как настроить кэширование зависимостей в Gradle для ускорения сборки проекта?

| Под темы                              | Заметка                                                                                                                                                |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ                                 | В Gradle кэширование зависимостей **встроено по умолчанию** и работает автоматически, но его можно тонко настроить для максимального ускорения сборки. |
| Как работает кэширование по умолчанию | - хранит скачанные зависимости в локальном кэше (`~/.gradle/caches/modules-2/files-2.1`);                                                              |
|                                       | - проверяет наличие артефакта в кэше перед загрузкой из репозитория;                                                                                   |
|                                       | - использует метаданные (хеши) для проверки актуальности.                                                                                              |
| Ключевые настройки для оптимизации    | Включите глобальное кэширование сборки                                                                                                                 |
|                                       | Настройте время хранения динамических версий                                                                                                           |
|                                       | Включите демон Gradle                                                                                                                                  |
|                                       | Параллельное выполнение задач                                                                                                                          |
|                                       | Конфигурирование по требованию                                                                                                                         |
|                                       | Оптимизируйте порядок репозиториев                                                                                                                     |
|                                       | Избегайте динамических версий                                                                                                                          |
|                                       | Ограничьте количество репозиториев                                                                                                                     |
|                                       | Используйте виртуальный репозиторий                                                                                                                    |
|                                       | Очищайте кэш при необходимости                                                                                                                         |

### Как использовать профили в Maven для различных сред разработки?

| Под темы                  | Заметка                                                                                                                                             |
| ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение               | Профили в Maven позволяют **настраивать сборку  проекта для разных сред** (разработка, тестирование,  продакшен) без изменения основного `pom.xml`. |
| Где определяются профили  | **`pom.xml`** проекта — локальные профили.                                                                                                          |
|                           | **`settings.xml`** пользователя (`~/.m2/settings.xml`) — пользовательские.                                                                          |
|                           | **`settings.xml`** глобальные (`%M2_HOME%/conf/settings.xml`) — системные.                                                                          |
| Способы активации профиля | Явно через командную строку                                                                                                                         |
|                           | По умолчанию. Профиль активируется, если не указан другой.                                                                                          |
|                           | По переменной среды                                                                                                                                 |
|                           | По наличию/отсутствию файла                                                                                                                         |
|                           | По ОС                                                                                                                                               |
