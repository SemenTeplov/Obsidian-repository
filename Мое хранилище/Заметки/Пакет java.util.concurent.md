### В чем основное преимущество использования классов из пакета java.util.concurrent по сравнению с традиционным способом работы с потоками в Java?

| Под темы                              | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                           | **Основное преимущество использования классов из пакета java.util.concurrent по сравнению с традиционным способом работы с потоками в Java** — **более гибкие инструменты для управления потоками и синхронизации ресурсов**. Это позволяет координировать выполнение потоков, обеспечивать безопасный доступ к общим ресурсам и предотвращать проблемы, связанные с параллельным выполнением кода.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| Традиционный способ работы с потоками | В Java для работы с потоками используются, например:<br>- **Класс Thread** — позволяет создавать потоки и работать с ними. Сами экземпляры класса не являются фактическими потоками, а скорее предоставляют интерфейс для управления потоками.<br>- **Ключевое слово synchronized** — используется для синхронизации потоков, определяет критические разделы кода или методы, к которым одновременно должен обращаться только один поток. Это гарантирует, что несколько потоков не будут мешать доступу друг друга к общим ресурсам.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| Классы из пакета java.util.concurrent | Пакет java.util.concurrent предлагает, например:<br>- **Классы-синхронизаторы** — позволяют координировать выполнение потоков, например, контролировать количество потоков, которым разрешён доступ к общему ресурсу одновременно. Например, класс **Semaphore** — счётчик, который управляет доступом к ресурсам.<br>- **Потокобезопасные коллекции** — предоставляют потокобезопасные реализации стандартных коллекций, таких как списки, множества и отображения, для использования в многопоточных приложениях. Например, класс **ConcurrentHashMap** — потокобезопасная реализация Map, обеспечивает высокую производительность при одновременном доступе из нескольких потоков.<br>- **Атомарные классы** — гарантируют, что определённые операции будут выполняться потокобезопасно, например операции инкремента и декремента, обновления и добавления значения (add).<br>- **Пулы потоков** — управление группой потоков, которые переиспользуются для выполнения задач. Это позволяет избежать создания большого количества потоков, которые могут привести к недостатку памяти, и ограничить максимальное количество одновременно выполняемых потоков. |

### Какой класс из пакета java.util.concurrent используется для блокировок, которые позволяют более гибкое управление, чем синхронизированные блоки или методы?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **ReentrantLock, Semaphore, CountDownLatch и CyclicBarrier** — классы из пакета java.util.concurrent, которые используются для управления блокировками, позволяющими более гибкое управление, чем синхронизированные блоки или методы.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| ReentrantLock  | **Реализует интерфейс Lock** и предоставляет более гибкий механизм синхронизации потоков по сравнению с ключевым словом synchronized. Некоторые особенности:<br>- **Рекурсивное получение блокировки** — один поток может получать блокировку несколько раз, каждое получение должно быть связано с соответствующим освобождением.<br>- **Явное управление блокировками** — в отличие от synchronized, ReentrantLock требует ручного получения и освобождения блокировки с помощью методов lock() и unlock().<br>- **Прерываемость** — потоки, ожидающие блокировку, могут быть прерваны.<br>- **Поддержка неблокирующих попыток захвата блокировки** — это полезно для предотвращения взаимных блокировок в сложных системах, где несколько ресурсов должны быть заблокированы одновременно.<br>- **Возможность использования условных переменных (Condition)** — этот механизм предоставляет гранулярный контроль над ожидающими потоками. |
| Semaphore      | **Реализует шаблон синхронизации «семафор»**, который ограничивает одновременный доступ к общему ресурсу нескольким потокам с помощью счётчика. Некоторые особенности:<br>- Если значение счётчика больше нуля, поток получает доступ к ресурсу, при этом счётчик уменьшается на единицу.<br>- После окончания работы с ресурсом поток освобождает семафор, и счётчик увеличивается на единицу.<br>- Если счётчик равен нулю, поток блокируется и ждёт, пока не получит разрешение от семафора.<br>- В конструктор класса передаётся количество потоков, которому семафор будет разрешать одновременно использовать заданный ресурс.                                                                                                                                                                                                                                                                                                         |
| CountDownLatch | **Позволяет одному или нескольким потокам ждать, пока не завершится определённое количество операций, выполняющихся в других потоках**. Некоторые особенности: <br>- В конструктор CountDownLatch (CountDownLatch(int count)) передаётся количество операций, которое должно быть выполнено, чтобы замок «отпустил» заблокированные потоки.<br>- Блокировка потоков снимается с помощью счётчика: любой действующий поток при выполнении определённой операции уменьшает значение счётчика.<br>- Когда счётчик достигает 0, все ожидающие потоки разблокируются и продолжают выполняться.<br>- Важно, что счётчик CountDownLatch — одноразовый, его нельзя сбросить.                                                                                                                                                                                                                                                                         |
| CyclicBarrier  | **Позволяет группе потоков ждать друг друга, пока все они не достигнут общей точки барьера**. Некоторые особенности:<br>- Барьер называется циклическим, потому что его можно использовать после освобождения ожидающих потоков.<br>- Поддерживает необязательную команду Runnable, которая запускается один раз за точку барьера, после прибытия последнего потока в группе, но перед освобождением любых потоков.<br>- Используется в программах, где есть фиксированное количество потоков, которые должны периодически ждать друг друга.<br>- Если поток покидает точку барьера преждевременно из-за прерывания, неудачи или времени ожидания, все другие потоки, ожидающие в этой точке, также покидают её ненормально.                                                                                                                                                                                                                 |

### Какой класс из пакета java.util.concurrent позволяет выполнить задачу в пуле потоков и вернуть результат в будущем?

| Под темы                 | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение              | **ExecutorService** — класс из пакета java.util.concurrent, который позволяет выполнить задачу в пуле потоков и вернуть результат в будущем.<br>Этот интерфейс расширяет интерфейс Executor, предоставляя методы управления завершением работы и создания объекта Future для отслеживания хода выполнения асинхронных задач.<br>Также в пакете есть классы, которые реализуют интерфейс ExecutorService, например **ThreadPoolExecutor** и **ScheduledExecutorService**                                                                                                                                                                                                                                                                                                                                                                                                                        |
| Executor                 | **Executor** — простой интерфейс с единственным методом execute(Runnable command). Он предназначен для выполнения переданного объекта Runnable, но не предоставляет методов для управления и контроля выполнения задач.<br>Интерфейс ExecutorService расширяет Executor, предоставляя более продвинутый способ управления выполнением задач и включая методы управления жизненным циклом потока.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ThreadPoolExecutor       | **ThreadPoolExecutor** — класс, который выполняет каждую отправленную задачу, используя один из, возможно, нескольких объединённых потоков.<br>Некоторые особенности работы:<br>- Автоматически регулирует размер пула в соответствии с ограничениями, установленными параметрами corePoolSize и maximumPoolSize.<br>- Когда новая задача отправляется, если запущено меньше потоков corePoolSize, создаётся новый поток для обработки запроса, даже если другие рабочие потоки простаивают.<br>- Если запущено меньше потоков maximumPoolSize, новый поток будет создан для обработки запроса только если очередь заполнена.<br>Класс предоставляет множество настраиваемых параметров и возможностей расширения, но программистам рекомендуется использовать фабричные методы Executors, которые предварительно настраивают параметры для наиболее распространённых сценариев использования. |
| ScheduledExecutorService | **ScheduledExecutorService** — интерфейс, который позволяет планировать выполнение задач после заданной задержки или периодически.<br>Некоторые особенности работы:<br>- Методы schedule создают задачи с различными задержками и возвращают объект задачи, который можно использовать для отмены или проверки выполнения.<br>- Методы scheduleAtFixedRate и scheduleWithFixedDelay создают и выполняют задачи, которые выполняются периодически до отмены.<br>Интерфейс ScheduledExecutorService реализует класс **ScheduledThreadPoolExecutor**.<br>**Важно**: все методы расписания принимают относительные задержки и периоды в качестве аргументов, а не абсолютное время или даты.                                                                                                                                                                                                       |

### Как работает ConcurrentHashMap и в чем его основное отличие от Hashtable и synchronizedMap?

| Под темы          | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ----------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ConcurrentHashMap | **Работает, разделяя карту на сегменты** и блокируя только часть при обновлении. Это позволяет:<br>- **Одновременно работать с разными частями карты** — например, в версиях до Java 8 карта разделялась на фиксированное число сегментов (по умолчанию 16), каждый со своим замком.<br>- **Обеспечивать безопасную итерацию** без необходимости внешней синхронизации. Итераторы работают по принципу fail-safe, отражая состояние карты на момент создания итератора и не выбрасывая ConcurrentModificationException.<br>**Особенности**:<br>- Не допускает использование null ни в качестве ключа, ни в качестве значения — попытка вставить null приведёт к NullPointerException.<br>- Поддерживает только ограниченный набор методов, доступных для безопасного доступа из нескольких потоков — putIfAbsent(), remove(), replace() и т. д..<br>- Уровень параллельности по умолчанию — 16, но может быть изменён во время создания ConcurrentHashMap. |
| Hashtable         | **Использует синхронизацию на уровне всей таблицы** и блокирует все операции. Это может привести к замедлению работы системы в случае большой конкуренции за доступ к таблице.<br>**Особенности**:<br>- Не позволяет использовать null в качестве ключа или значения.<br>- Использует метод hashCode() для определения, к какому сегменту должна быть назначена пара ключ-значение.<br>- Для разрешения коллизий (когда разные объекты имеют одинаковый хэш-код) использует массив списков.                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| synchronizedMap   | **Создаёт обёртку над обычной Map**, синхронизируя все операции через один общий монитор. Это гарантирует потокобезопасность, но существенно ограничивает возможности параллельного доступа.<br>**Особенности**:<br>- Каждый публичный метод synchronizedMap обёрнут в блок synchronized, который использует общий объект-монитор.<br>- В каждый момент времени с картой может работать только один поток, все остальные будут ожидать освобождения блокировки.<br>- Предоставляет итератор, который не является потокобезопасным — при итерации необходимо внешнее синхронизирование на объекте карты.<br>- Полностью поддерживает null в качестве ключей и значений (если базовая Map это позволяет).                                                                                                                                                                                                                                                    |

### Что такое CyclicBarrier и в каких сценариях его использование наиболее целесообразно?

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение            | **CyclicBarrier** — синхронизирующий механизм из библиотеки java.util.concurrent в Java, который позволяет нескольким потокам остановиться в определённой точке выполнения (барьере) до тех пор, пока все потоки не достигнут этой точки. После того, как все потоки достигают барьера, они могут продолжить выполнение.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| Принцип работы         | - **CyclicBarrier создаётся с числом потоков** (или «участников»), которые должны достичь барьера. Пока не все участники достигнут барьера, они будут ожидать.<br>- **Каждый поток вызывает метод await()** на объекте CyclicBarrier — это указывает, что данный поток достиг барьера и должен ждать остальных.<br>- **Как только все потоки вызовут этот метод**, барьер «открывается», и все потоки продолжают выполнение.<br>- **После того, как все потоки проходят через барьер**, он может быть использован повторно, поэтому и называется «циклическим». Это удобно для сценариев, где есть многократные синхронизации потоков на разных этапах выполнения программы.<br>- **CyclicBarrier может быть настроен так**, чтобы выполнять определённое действие (заданный Runnable) один раз, когда все потоки достигают барьера. Это полезно для выполнения дополнительной работы после завершения синхронизации. |
| Сценарии использования | **Сценарий, где приложение нужно читать 3 файла с помощью 3 потоков**, анализировать прочитанные строки, и только после чтения и анализа всех трёх файлов приложение должно вызвать другой поток для дальнейшей обработки. В этом сценарии можно использовать CyclicBarrier и предоставить Runnable-действие, которое будет выполняться, когда все потоки достигнут барьера.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |

### Как работает CopyOnWriteArrayList?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **CopyOnWriteArrayList** — потокобезопасный вариант ArrayList в Java, который использует подход «копирование при записи» (copy-on-write).                                                                                                                                                                                                                                                                                                                                                 |
| Принцип работы | - **Из-за дополнительного шага создания нового резервного массива** каждый раз при обновлении списка производительность хуже, чем у ArrayList.<br>- **Операции чтения** не приводят к снижению производительности, и оба класса выполняют их одинаково.<br>**Важно**: CopyOnWriteArrayList полезен только в определённых случаях, когда количество операций записи мало по сравнению с количеством операций чтения, иначе это может вызвать проблемы с памятью при большом объёме данных. |

### Какие есть виды ThreadPool?

| Под темы  | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Fixed     | **Имеет фиксированное количество потоков**. Некоторые особенности работы:<br>- Максимум n потоков активны для обработки задач.<br>- Если передано более n задач, они удерживаются в очереди, пока потоки не освободятся.<br>- Если в работе одного из потоков произойдёт сбой, на место сломанного создаётся новый поток.<br>- Любой поток из пула активен, пока пул не закрыт.<br>Для создания такого пула используется метод Executors.newFixedThreadPool(int n).                                                                                                                                                                                                                                                                                                                                      |
| Cached    | **Динамически регулирует количество потоков** в зависимости от нагрузки. Новые потоки создаются по необходимости, а idle-потоки автоматически завершаются.<br>**Подходит** для коротких асинхронных задач. Однако есть и ограничения: если создать слишком много потоков, это может привести к исчерпанию ресурсов. Также не рекомендуется использовать такой пул для задач с непредсказуемым временем выполнения, например, задач ввода-вывода. <br>Для создания кэшированного пула используется метод Executors.newCachedThreadPool().                                                                                                                                                                                                                                                                 |
| Scheduled | **Позволяет планировать задачи** для выполнения с задержкой или периодически. Например, для пакетных заданий, которые должны запускаться раз в день, или обновлений в реальном времени, отправляемых пользователям с интервалами.<br>**Реализуется** с помощью метода Executors.newScheduledThreadPool(int corePoolSize). Параметр corePoolSize указывает количество потоков, которые должны оставаться в пуле, даже если они неактивны.<br>**Особенности**:<br>- Методы для планирования задач: schedule(Runnable command, long delay, TimeUnit unit), scheduleAtFixedRate(Runnable command, long initialDelay, long period, TimeUnit unit).<br>- Есть перегруженный вариант метода, который принимает объект типа ThreadFactory.                                                                       |
| Fork/Join | **Основан на алгоритме «Разделяй и властвуй»**: каждая задача разбивается на подзадачи, затем они выполняются параллельно, и как только все из них завершаются, происходит объединение результатов.<br>**Особенности**:<br>- Каждый рабочий поток имеет собственную двухстороннюю рабочую очередь (deque) типа WorkQueue.<br>- Используется алгоритм work-stealing: когда поток становится idle, он пытается «красть» задачи у других потоков, которые всё ещё заняты своей работой.<br>- В Java есть предопределённый общий ForkJoinPool, который можно создать, вызвав ForkJoinPool.commonPool().<br>**Реализуется** с помощью класса ForkJoinPool. По умолчанию количество потоков в пуле равно количеству доступных процессоров в системе, но можно создать свой пул, указав, сколько потоков нужно. |

### Что такое Atomic?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **Atomic в Java** — это набор классов, которые поддерживают **потокобезопасное программирование без блокировки** для отдельных переменных. Такие классы в совокупности называются **атомными переменными**.<br>Пакет, содержащий атомарные классы, — **java.util.concurrent.atomic**.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| Принцип работы | Атомарные переменные предназначены для **атомарного обновления**. Это означает, что их операции (такие как увеличение, уменьшение или сравнение и установка значений) выполняются как единый, неделимый шаг. Это гарантирует, что ни один другой поток не сможет наблюдать переменную в промежуточном состоянии.<br>Атомарность важна для параллельного программирования, поскольку помогает предотвратить **«состояние гонки»** (race condition) — ошибку, при которой несколько потоков пытаются одновременно читать и изменять одни и те же данные. Например, операция приращения (count++) состоит из трёх отдельных этапов: чтения текущего значения, увеличения значения на единицу и записи обновлённого значения обратно в count. В многопоточной среде поток может быть вытеснен после операции чтения и до операции записи, затем другой поток может прочитать то же самое (старое) значение count, увеличить его и записать обратно. Когда первый поток возобновляет выполнение и записывает своё значение обратно в count, он перезаписывает значение из второго потока. В результате, несмотря на то, что count++ был вызван дважды, count увеличился только один раз. |


### Какие есть виды Atomic?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Виды     | Некоторые часто используемые атомарные классы в Java:<br>- **AtomicInteger** — предоставляет атомарные операции (инкремент, декремент, добавление и т. д.) на целочисленных значениях без синхронизации.<br>- **AtomicLong** — поддерживает атомарные операции на значениях типа long для потокобезопасных числовых вычислений.<br>- **AtomicBoolean** — представляет логическое значение, которое может обновляться атомарно для безопасных операций на основе флагов.<br>- **AtomicReference** — позволяет атомарное чтение и запись ссылок на объекты, полезно для непримитивных данных.<br>Также существуют классы **AtomicIntegerArray**, **AtomicLongArray** и **AtomicReferenceArray**, которые расширяют поддержку атомарных операций для массивов этих типов. |

### За счет какого алгоритма работают Atomic?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение    | **Атомарные классы (Atomic) в Java работают за счёт использования механизма compare-and-set (CAS)**. Этот механизм обеспечивает атомарность операций (таких как увеличение, уменьшение или сравнение и установка значений) без использования блокировок.<br>CAS — атомарная инструкция, которую поддерживает большинство современных процессоров.                                                                                                                                                |
| Принцип работы | Принцип работы CAS заключается в оптимистичном подходе к синхронизации:<br>1. Запоминается текущее значение переменной.<br>2. Выполняются вычисления нового значения.<br>3. Проверяется, что текущее значение не изменилось другим потоком.<br>4. Если не изменилось — атомарно заменяется на новое значение.<br>5. Если изменилось — операция повторяется с начала.<br>Это позволяет объединить три шага — получение значения, сравнение и его обновление — в одну операцию на машинном уровне. |

### Что такое ThreadPool?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **ThreadPool** — это механизм управления **пулом потоков** в программировании.<br>ThreadPool представляет собой набор предварительно созданных потоков, которые могут выполнять задачи асинхронно. Вместо создания нового потока для каждой задачи ThreadPool позволяет повторно использовать существующие потоки из пула.<br>Основное назначение пула потоков — обратить «сырую» многопоточную обработку в удобный инструмент эффективного решения трудоёмких задач, допускающих разбиение на независимые кусочки. |

### Как можно использовать BlockingQueue для реализации асинхронного очередного доступа к ресурсам?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **BlockingQueue (блокирующая очередь) в Java можно использовать для реализации асинхронного доступа к ресурсам** (например, для создания producer-consumer систем). Этот интерфейс расширяет интерфейс Queue и включает методы, блокирующие выполнение потока при попытке добавления или удаления элементов из пустой или полной очереди.<br>BlockingQueue полезен в серверных приложениях, где нужно обрабатывать большое количество задач асинхронно: очереди сообщений, задачи от пользователей, логирование, обработка файлов                                                                                                                                                                                                                                                                                            |
| Принцип работы | - **Когда очередь пуста**, поток, который пытается из неё что-то достать, блокируется и ждёт, пока не появится новый элемент.<br>- **Если очередь заполнена**, поток, который пытается что-то добавить, тоже блокируется.<br>Некоторые методы BlockingQueue, которые используются для работы с ресурсами:<br>- **put()** — добавляет элемент, блокируется, если очередь заполнена;<br>- **take()** — извлекает элемент, блокируется, если очередь пуста;<br>- **offer()** — добавляет элемент, возвращает false, если очередь заполнена;<br>- **poll()** — извлекает элемент, возвращает null, если очередь пуста;<br>- **offer(timeout) и poll(timeout)** — версии с таймаутом.<br>Реализации методов BlockingQueue потокобезопасны (синхронизированы), поэтому писать в очередь и читать из неё можно в несколько потоков. |

### Какие особенности и преимущества ScheduledExecutorService по сравнению с обычными ExecutorService?

| Под темы                 | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение              | **ScheduledExecutorService в Java отличается от обычного ExecutorService возможностью планировать выполнение задач по расписанию**. Это подинтерфейс ExecutorService, определённый в пакете java.util.concurrent.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ExecutorService          | **ExecutorService** — интерфейс, который расширяет интерфейс Executor, предоставляя методы управления завершением работы и создания объекта Future для отслеживания хода выполнения асинхронных задач. Некоторые особенности ExecutorService:<br>- **Метод submit** принимает объекты Runnable и Callable, возвращает объект Future, который можно использовать для получения результата или отмены задачи.<br>- **Методы invokeAny и invokeAll** выполняют массовое выполнение: запускают коллекцию задач и ждут, пока хотя бы одна или все не завершатся.<br>- **Методы для управления завершением работы**. Например, метод shutdown() позволяет ранее отправленные задачи выполнить перед завершением, а метод shutdownNow() предотвращает запуск ожидающих задач и пытается остановить текущие.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ScheduledExecutorService | **ScheduledExecutorService** добавляет методы, которые позволяют планировать выполнение задач после заданной задержки или с периодичностью. Некоторые особенности ScheduledExecutorService:<br>- **Метод schedule** создаёт и выполняет однократное действие, которое выполняется после заданной задержки.<br>- **Метод scheduleAtFixedRate** создаёт и выполняет периодическое действие, которое выполняется вначале после заданной начальной задержки, а затем с заданным периодом.<br>- **Метод scheduleWithFixedDelay** создаёт и выполняет периодическое действие, которое выполняется сначала после данной первоначальной задержки, а затем с заданной задержкой между завершением одного выполнения и началом следующего.<br>**Преимущества ScheduledExecutorService**:<br>- **Возможность запускать задачи с условием истечения некоторого времени** или периодичности выполнения. Например, для выполнения «сервисных» периодических задач, которые должны быть выполнены вне зависимости от работы основного функционала программы.<br>- **Возможность настраивать интервал** или время, на которое выполнение будет отложено. Интервалом может быть время между двумя последовательными запусками или время между окончанием одного выполнения и началом другого.<br>- **Возможность отменять запланированные задачи** с помощью возвращённого объекта ScheduledFuture.<br>**Важно**: все методы schedule принимают относительные задержки и периоды в качестве аргументов, а не абсолютные времена или даты. |

### В чем заключается принцип работы класса Phaser и как его функциональность может быть использована для координации фаз выполнения многопоточных задач?

| Под темы             | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение          | **Класс Phaser в Java** — это синхронизационный механизм, который позволяет координировать выполнение потоков на определённых фазах. Он является частью пакета java.util.concurrent и был введён в Java 7.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| Принцип работы       | Phaser разделяет выполнение на несколько фаз, каждая из которых представляет собой точку синхронизации. **Потоки должны ждать в конце каждой фазы**, пока все зарегистрированные потоки не достигнут этой фазы. Когда все потоки прибыли, Phaser переходит к следующей фазе.<br>**Особенности**:<br>- Каждая фаза имеет номер, который начинается с нуля и увеличивается, когда все стороны достигают Phaser.<br>- Количество сторон-участников жёстко не задано и может меняться: поток может регистрироваться в качестве участника и отменять своё участие.<br>- Участник не обязан ожидать, пока все остальные участники соберутся на барьере — чтобы продолжить свою работу, достаточно сообщить о своём прибытии. |
| Пример использования | **Функциональность класса Phaser может быть использована для координации фаз выполнения многопоточных задач** в многопоточном приложении. Например, можно запустить несколько потоков, которые выполняют разные фазы, и Phaser будет ждать, пока все участники не завершат выполнение фазы. После завершения последней фазы происходит отмена регистрации всех участников с помощью метода arriveAndDeregister(). <br>**Важно**: при использовании Phaser в сложных сценариях нужно быть осторожным, чтобы избежать тупиков. Например, если поток держит блокировку во время ожидания на Phaser, и другие потоки нуждаются в этой блокировке, это может привести к ситуации тупика.                                    |

### В чем разница между CyclicBarrier и CountDownLatch?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| CyclicBarrier  | **Позволяет всем потокам ждать друг друга в определённый момент**. Ни один из потоков не может пройти через эту точку, пока все потоки её не достигнут.<br>**Особенности**:<br>- Позволяет повторно использовать барьер после каждого прохождения группы потоков через него.<br>- Выполняется сразу же после того, как заданное количество потоков вызовет метод await(), блокируя дальнейшее выполнение до достижения барьера.<br>- Поддерживает опциональное действие, которое выполняется после того, как все потоки достигли барьера. Это может быть полезно для согласования состояния между потоками. <br>- Можно сбросить барьер вызовом метода reset() — после сброса его можно использовать снова для ожидания новой группы потоков.<br>**Применение**: CyclicBarrier используется, когда разные потоки обрабатывают часть вычисления, и когда все потоки завершили выполнение, результат нужно объединить в родительском потоке.                                                                  |
| CountDownLatch | **Ориентирован на одноразовое ожидание фиксированного количества событий**. После того, как заданное количество вызовов метода countDown() будет выполнено, все ожидающие потоки будут разблокированы.<br>**Особенности**:<br>- Не предоставляет возможности переключиться в ожидающем потоке после вызова countDown() — разблокированные потоки могут продолжить своё выполнение незамедлительно.<br>- Не имеет понятия об участниках — просто ждёт завершения фиксированного количества операций.<br>- Не предоставляет метод для сброса состояния — однажды достигнуто установленное количество событий, оно не может быть сброшено для повторного использования.<br>**Применение**: CountDownLatch подходит для случаев, когда основному потоку необходимо дождаться завершения ряда связанных задач в других потоках, прежде чем продолжить выполнение своих задач. Например: запуск сервера только после того, как все сервисы готовы, запуск тестов после того, как все компоненты инициализированы. |

### Что такое ThreadLocal?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **ThreadLocal** — это специальный класс в Java, который **даёт каждому потоку отдельную копию переменной**. Это позволяет:<br>- **Изолировать данные** в рамках каждого потока, не влияя на копии других потоков.<br>- **Избегать гонок за ресурсами**.<br>Внутри JVM каждый объект Thread содержит хеш-карту **ThreadLocalMap**, которая служит контейнером для хранения переменных ThreadLocal. Ключами в этой карте являются ссылки на объекты ThreadLocal, а значения — это то, что хранится в каждом конкретном потоке.<br>**Некоторые методы класса ThreadLocal:**<br>- **get** — возвращает значение локальной переменной текущего потока.<br>- **set** — устанавливает значение локальной переменной для текущего потока.<br>- **remove** — удаляет значение локальной переменной текущего потока.<br>- **withInitial** — дополнительный фабричный метод, который устанавливает начальное значение get() и set().<br>**ThreadLocal полезен**, когда нужно поддерживать состояние для каждого потока без сложной синхронизации. Например, для хранения сессий пользователей, контекстов транзакций или других данных, которые нужно изолировать от других потоков.<br>**Однако у ThreadLocal есть и ограничения**. Некоторые из них:<br>- **Утечки памяти** из-за неправильного использования. Если забыть вызвать remove(), данные останутся в памяти даже после того, как поток завершит свою основную работу.<br>- **Проблемы с пулами потоков**. В пуле потоки многократно переиспользуются, и если не очищать переменные, данные из одного потока могут случайно попасть в другой, что приведёт к неожиданным багам и утечкам памяти.<br>- **Проблемы с производительностью**. Когда много потоков, каждый из которых создаёт собственную копию данных в ThreadLocal, это может привести к дополнительным накладным расходам на память. |
