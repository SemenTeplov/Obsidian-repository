### Что такое SOAP и REST в контексте веб-сервисов?

| Под темы    | Заметка                                                                                                                                                                                                                                    |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| SOAP        | **SOAP** — **протокол обмена данными** в виде XML. Определяет правила взаимодействия: как формировать сообщения, обрабатывать ошибки, управлять сессиями.                                                                                  |
| Особенности | Использует исключительно XML для форматирования сообщений, независимо от транспортного протокола.                                                                                                                                          |
|             | Может работать поверх HTTP, SMTP, TCP и даже JMS.                                                                                                                                                                                          |
|             | Поддерживает сложные операции: вызов удалённых процедур (RPC) и обмен документами.                                                                                                                                                         |
|             | Имеет встроенные механизмы обработки ошибок и повторных попыток через стандарт WS-ReliableMessaging.                                                                                                                                       |
| REST        | **REST** — **архитектурный стиль** для создания масштабируемых и гибких веб-сервисов. Основная идея — представление ресурсов (данных, объектов) через унифицированные интерфейсы, что упрощает взаимодействие между клиентами и серверами. |
| Особенности | **Клиент-сервер** — разделение клиентской и серверной частей системы: клиент управляет пользовательским интерфейсом, сервер — бизнес-логикой и данными.                                                                                    |
|             | **Отсутствие состояния (Stateless)** — каждый запрос от клиента должен содержать всю информацию, необходимую для его обработки. Сервер не сохраняет состояние сеанса между запросами, что повышает отказоустойчивость и масштабируемость.  |
|             | **Кэширование** — ответы сервера должны быть помечены как кэшируемые или не кэшируемые. Это позволяет снизить нагрузку на сервер и ускорить работу клиентов за счёт локального хранения данных.                                            |
|             | **Единообразие интерфейса** — идентификация ресурсов (через URI, например, /products/123), манипуляции с ресурсами через представления (например, JSON или XML).                                                                           |

### В каких случаях предпочтительнее использовать SOAP, а в каких — REST?

| Под темы | Заметка                                                                                                                |
| -------- | ---------------------------------------------------------------------------------------------------------------------- |
| SOAP     | Интеграция в корпоративные системы, где уже используется SOAP-инфраструктура;                                          |
|          | Проект связан с финансовыми, медицинскими, страховыми сервисами, где критична безопасность и строгие политики доступа; |
|          | Работа идёт с устаревшими платформами или сторонними системами, которые поддерживают только SOAP;                      |
|          | Важна гарантированная доставка сообщений, а также обработка ошибок на уровне протокола.                                |
| REST     | Публичные API — если создаётся API, который будет использоваться широким кругом разработчиков;                         |
|          | Мобильные приложения — где скорость и эффективность использования трафика критичны;                                    |
|          | Веб-сервисы с высокой нагрузкой — благодаря поддержке кэширования и простоте масштабирования;                          |
|          | Микросервисная архитектура — где необходима лёгкая коммуникация между сервисами.                                       |

### В чем основное различие между SOAP и REST?

| Под темы | Заметка                                                                                                                                                                                          |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| SOAP     | Протокол обмена сообщениями в виде XML.                                                                                                                                                          |
|          | **Строгая структура** — сообщения передаются в формате XML, что обеспечивает строгую типизацию и согласованную структуру сообщений.                                                              |
|          | **Поддержка различных протоколов** — SOAP может работать через HTTP, SMTP, TCP или UDP.                                                                                                          |
|          | **Встроенная поддержка безопасности** — например, стандарты WS-Security, которые позволяют шифровать сообщения, управлять доступом, подписывать данные.                                          |
|          | **Надёжность доставки сообщений** — SOAP поддерживает механизмы, гарантирующие доставку сообщений даже в случае сбоев.                                                                           |
|          | **Поддержка сложных операций** — SOAP хорошо справляется со сложными сценариями, такими как цепочки вызовов, транзакции или подтверждения.                                                       |
| REST     | Архитектурный стиль, используемый при разработке распределённых систем, веб-сервисов и API                                                                                                       |
|          | **Использование стандартных HTTP-методов** — GET (получение ресурса), POST (создание нового ресурса), PUT (обновление ресурса) и DELETE (удаление ресурса).                                      |
|          | **Отсутствие состояния (Statelessness)** — каждый запрос от клиента к серверу должен содержать всю информацию, необходимую для его обработки. Сервер не хранит информацию о предыдущих запросах. |
|          | **Кешируемость** — ответы от сервера могут быть закешированы, если это указано, что позволяет уменьшить нагрузку на сервер и ускорить время отклика.                                             |
|          | **Единообразие интерфейсов** — использование стандартизированных методов HTTP и соглашений о структуре URL позволяет легко взаимодействовать с системой и изменять её компоненты.                |

### В чем основные преимущества GraphQL перед REST?

| Под темы | Заметка                                                                                                                                                                                                                                                                           |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| GraphQL  | **Гибкость запросов** — клиент получает ровно те данные, которые запросил, без лишней информации. Например, с помощью GraphQL можно одним запросом получить профиль пользователя, его посты и список друзей, в REST для этого пришлось бы выполнить несколько отдельных запросов. |
|          | **Единая точка входа** — вместо множества эндпоинтов используется один URL для всех операций.                                                                                                                                                                                     |
|          | **Предотвращение проблемы N+1 запросов** — получение связанных данных в одном запросе.                                                                                                                                                                                            |
|          | **Сильная типизация** — схема GraphQL обеспечивает контракт между клиентом и сервером.                                                                                                                                                                                            |
|          | **Возможность получать разные наборы данных** через один и тот же эндпоинт. Например, мобильное приложение может запросить минимальный набор полей для экономии трафика, а веб-версия — получить расширенную информацию.                                                          |
|          | **Встроенные механизмы валидации запросов и обработки ошибок** — при запросе несуществующих полей или неправильных аргументов сервер вернёт понятное сообщение об ошибке, а не просто код HTTP 400 или 500.                                                                       |
| REST     | **Простота и логичность** — благодаря использованию стандартных методов HTTP интерфейсы легко понимать и быстро внедрять.                                                                                                                                                         |
|          | **Гибкость и масштабируемость** — REST не требует хранения состояния, что позволяет без усилий обрабатывать большой объём параллельных запросов, особенно в облачных или микросервисных архитектурах.                                                                             |
|          | **Поддержка разных форматов** — REST не ограничивается одним способом передачи данных, можно использовать JSON, xml, YAML и другие форматы.                                                                                                                                       |
|          | **Универсальность** — REST API легко интегрируется с браузерами, мобильными платформами, внешними сервисами и даже IoT-устройствами.                                                                                                                                              |
|          | **Лёгкое тестирование** — большинство HTTP-клиентов и любой тестировщик REST-интерфейсов позволяют быстро проверить работу API без сложной настройки.                                                                                                                             |

### Какие форматы данных обычно используются в REST и SOAP сервисах?

| Под темы   | Заметка                                                                                                                                                                                                                                          |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| REST       | REST не привязан к конкретному формату.                                                                                                                                                                                                          |
| Использует | **JSON (JavaScript Object Notation)** — стал де-факто стандартом из-за своей компактности и удобства использования в JavaScript.                                                                                                                 |
|            | **XML** — до сих пор используется в некоторых корпоративных и устаревших системах, особенно в связке с протоколом SOAP. Однако из-за многословности и сложности парсинга уступил место JSON в большинстве новых проектов, использующих REST API. |
|            | **Текстовые данные** и даже бинарные форматы.                                                                                                                                                                                                    |
| SOAP       | **SOAP использует исключительно XML** для форматирования сообщений, независимо от транспортного протокола.                                                                                                                                       |

### Каковы основные требования к безопасности для SOAP и REST сервисов?

| Под темы | Заметка                                                                                                                                                                                                                                              |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| SOAP     | **SOAP** ориентирован на безопасность **на уровне сообщений**. Основной стандарт безопасности — **WS-Security**. Он добавляет поддержку для шифрования сообщений, цифровых подписей и использования токенов безопасности, таких как SAML и Kerberos. |
| REST     | **REST** не имеет встроенного стандарта безопасности, как WS-Security, но активно использует **безопасность на уровне транспортного слоя и заголовков HTTP**.                                                                                        |

### Как происходит обмен сообщениями между клиентом и сервером в REST и в SOAP?

| Под темы | Заметка                                                                                                                                                                                                                                        |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| REST     | **В REST** обмен сообщениями осуществляется **без сохранения состояния**. Каждое сообщение самодостаточное и содержит всю информацию, необходимую для его обработки. Сервер не хранит результаты предыдущих сессий с клиентскими приложениями. |
| SOAP     | **В SOAP** сообщения передаются **в формате XML**.                                                                                                                                                                                             |

### Как обрабатываются ошибки в GraphQL?

| Под темы    | Заметка                                                                                                                                                                                   |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ       | **Ошибки в GraphQL обрабатываются путём возврата массива ошибок в поле errors ответа**. Это позволяет серверу и клиенту эффективно взаимодействовать даже в случае возникновения проблем. |
| Виды ошибок | **Синтаксические** — возникают при разборе запроса (например, пропущенные скобки, неправильные токены).                                                                                   |
|             | **Ошибки валидации** — возникают, когда запрос не соответствует схеме (несуществующие поля, пропущенные аргументы).                                                                       |
|             | **Ошибки выполнения (runtime errors)** — возникают при обработке запроса на сервере.                                                                                                      |
|             | **Ошибки аутентификации и авторизации** — возникают, когда пользователь не аутентифицирован или у него нет прав на выполнение запроса.                                                    |
|             | **Ошибки сети (network errors)** — связаны с проблемами передачи данных.                                                                                                                  |

### Какие существуют спецификации безопасности для SOAP и как они реализуются?

| Под темы             | Заметка                                                                                                                                                                                                                                          |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ                | **Для протокола SOAP (Simple Object Access Protocol) существуют спецификации безопасности**, которые расширяют базовый функционал протокола. К ним относятся **WS-Security**, **WS-ReliableMessaging**, **WS-AtomicTransaction** и **WS-Trust**. |
| WS-Security          | Обеспечивает безопасность SOAP-сообщений.                                                                                                                                                                                                        |
| WS-ReliableMessaging | **Определяет механизм надёжной доставки сообщений** между конечными точками веб-сервисов.                                                                                                                                                        |
| WS-AtomicTransaction | **Определяет тип координации атомарных транзакций** для веб-служб SOAP.                                                                                                                                                                          |
| WS-Trust             | **Расширяет WS-Security** — предоставляет механизм для запроса и выдачи токенов безопасности, а также для управления отношениями доверия между клиентами и поставщиками веб-сервисов.                                                            |

### Какие существуют методы оптимизации производительности REST сервисов?

| Под темы              | Заметка                                                                                                                                                                                                                                     |
| --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Кэширование           | **Кэширование на стороне клиента** — ответы хранятся в кэше (например, в браузере), чтобы избежать повторяющихся запросов на сервер. Для управления поведением кэширования используются HTTP-заголовки: Cache-Control, ETag, Last-Modified. |
|                       | **Кэширование на стороне сервера** — часто используемые ресурсы хранятся в памяти или других системах кэширования (например, Redis, Memcached). Это снижает потребность в дорогостоящих запросах к базе данных или ресурсоёмких операциях.  |
|                       | **Кэширование на уровне приложения** — минимизирует избыточные вычисления и дорогостоящие операции. Этот метод может потребовать специальной логики в приложении для поддержания целостности и актуальности кэша.                           |
| Пагинация             | **Помогает разбивать большие объёмы данных** на меньшие, более управляемые части — «страницы». Это снижает объём передаваемых данных по сети, что улучшает время отклика API с точки зрения клиента.                                        |
| Сжатие данных         | **Сжатие HTTP-ответов** — например, с помощью заголовков Accept-Encoding (клиент указывает алгоритмы сжатия) и Content-Encoding (сервер информирует клиента о методе сжатия).                                                               |
|                       | **Сжатие только для определённых маршрутов** — в больших приложениях иногда нужно сжимать данные только для определённых маршрутов.                                                                                                         |
|                       | **Кэширование сжатых ответов** — сжатые ответы можно хранить в кэше для ускорения повторных запросов.                                                                                                                                       |
| Асинхронная обработка | **Использование очередей** — сервер помещает задачу в очередь, отдельный воркер обрабатывает её и сохраняет результат, доступный через REST.                                                                                                |
|                       | **Паттерн «Polling with Exponential Backoff»** — клиент периодически проверяет статус задачи, увеличивая интервал между запросами после каждого неудачного опроса.                                                                          |
|                       | **Long polling** — клиент отправляет запрос, и сервер удерживает соединение открытым до тех пор, пока данные не будут готовы.                                                                                                               |

### В чем состоят сложности миграции с SOAP на REST и как их можно преодолеть?

| Под темы  | Заметка                                                                                                                                                                                                                          |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Сложности | **Различие в архитектуре**. SOAP — протокол со строгими стандартами (XML, WSDL, WS-Security), REST — архитектурный стиль, гибкий и простой, не требует формальных контрактов.                                                    |
|           | **Преобразование форматов данных**. SOAP использует XML, который более подробный и сложный по сравнению с облегчённым форматом JSON, обычно используемым в REST-сервисах.                                                        |
|           | **Переопределение бизнес-логики**. Например, если большинство операций SOAP-сервисов зависят от состояния, может потребоваться перепроектировать клиентское приложение, чтобы не полагаться на состояние сервера.                |
|           | **Несовместимость протоколов**. REST в основном работает через HTTP, а SOAP может поддерживать другие протоколы. Во время преобразования может потеряться информация о других протоколах из-за несовместимости с REST-форматами. |
| Решения   | **Использовать API-шлюз или адаптер**. Он переводит запросы REST в форматы, совместимые с SOAP, и наоборот, обеспечивая обратную совместимость во время перехода.                                                                |
|           | **Поэтапная миграция**. Например, можно строить обёртку REST-API вокруг существующих SOAP-сервисов, чтобы клиенты REST могли вызывать новый API, а SOAP-сервисы оставались неизменными.                                          |
|           | **Документирование REST-API**. Например, можно использовать OpenAPI или Swagger, чтобы упростить интеграцию с современными системами.                                                                                            |
|           | **Проведение интеграционного тестирования**. Нужно убедиться, что REST-API точно отражает функциональность оригинального SOAP-API и соответствует принципам REST.                                                                |
