### Что такое поток в контексте Java?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Поток (thread) в контексте Java** — это независимая последовательность выполнения инструкций в рамках одной программы. Потоки позволяют выполнять несколько задач одновременно, используя возможности многоядерных процессоров.<br>**Основные характеристики потока**:<br>- Выполняется в рамках одного процесса и использует его память.<br>- У каждого потока есть своё выполнение.<br>- Имеет доступ к общей памяти.<br>**Зачем нужны потоки**: для увеличения производительности на многоядерных процессорах, улучшения отзывчивости программ (например, в GUI-приложении можно выполнять долгие операции в фоновом потоке, чтобы интерфейс не зависал), асинхронной обработки (например, чтение/запись файлов, работа с сетью). |
| Виды        | В Java есть два типа потоков:<br>1. **Платформенные**. Это тяжёлые потоки, которые выполняются напрямую операционной системой. Каждый Java-поток соответствует одному потоку ядра (kernel thread) — это модель «one-to-one». Планирование, переключение контекста и управление полностью осуществляются операционной системой.<br>2. **Виртуальные** (user-mode threads, легковесные потоки). Управляются полностью JVM, без участия ОС в планировании. Можно создавать миллионы виртуальных потоков, не исчерпывая ресурсы. Они не привязаны напрямую к потокам ядра — вместо этого множество виртуальных потоков разделяют небольшой пул carrier threads — это обычные платформенные потоки.                                         |

### Как создать поток в Java?

| Под темы                 | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Через класс Thread       | **Чтобы создать поток через наследование от класса Thread, нужно**:<br>1. Создать класс, расширяющий Thread.<br>2. Переопределить метод run() — здесь размещается код, который будет выполняться в отдельном потоке.<br>3. Создать экземпляр этого класса.<br>4. Вызвать метод start() для запуска потока.                                                                                                                                                                                 |
| Через интерфейс Runnable | **Интерфейс Runnable** содержит метод run() — в нём находится точка входа и логика исполняемого потока. **Чтобы создать поток, нужно**:<br>1. Определить реализацию интерфейса Runnable (например, в виде лямбда-выражения).<br>2. Создать объект Thread с помощью Runnable: var t = new Thread(task). Класс Thread имеет несколько конструкторов, в данном случае используется тот, который принимает объект Runnable.<br>3. Запустить поток Thread с помощью метода start():  t.start(). |

### Чем отличается метод run() от метода start() в контексте потоков?

| Под темы      | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение   | Метод run() и метод start() в контексте потоков в Java отличаются тем, что start() создаёт новый поток и вызывает run() в этом потоке, а run() выполняет код последовательно в текущем потоке.                                                                                                                                                                                                                                                                                                                                                                    |
| Метод run()   | **run()** — это метод, содержащий код, который будет выполнен в потоке. Он является частью интерфейса Runnable или может быть переопределён в классе, расширяющем Thread.<br>**Особенности**:<br>- Если вызвать run() напрямую, без вызова start(), код будет выполнен в текущем потоке, как обычный метод, и новый поток не будет создан.<br>- Обычно метод run() используется для определения задачи (тела) потока, а не для запуска самого потока.<br>- Проблемы, возникающие при вызове run(), легче отслеживать и отлаживать в контексте вызывающего потока. |
| Метод start() | **start()** — метод класса Thread, который создаёт новый поток и вызывает метод run() в этом новом потоке.<br>**Особенности**:<br>- Вызов start() инициирует параллельное выполнение потока.<br>- Метод start() выполняет подготовительные действия, такие как выделение системных ресурсов.<br>- Попытка вызвать start() более одного раза на одном и том же экземпляре потока приведёт к исключению IllegalThreadStateException.<br>**Важно**: нельзя вызывать run() напрямую — если сделать так, то метод run() выполнится в текущем потоке, а не в новом.     |

### В чем разница между конкуренцией и параллелизмом?

| Под темы                     | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                  | **Разница между конкурентностью (concurrency) и параллелизмом (parallelism) заключается в подходе к выполнению задач**. На первый взгляд эти термины кажутся синонимами, описывающими способность системы выполнять несколько задач одновременно, но за ними скрываются фундаментально разные концепции управления процессами.                                                                                                                                                                                                                                                                                                                                                                                          |
| Конкурентность (Concurrency) | **Конкурентность** — это свойство системы структурировать программу как набор независимо выполняющихся задач. **Важно, что задачи не обязательно выполняются в один и тот же момент времени** — система может переключаться между ними так быстро, что создаётся иллюзия одновременности.<br>**Некоторые особенности конкурентности:**<br>- Независимость от аппаратного обеспечения — конкурентную программу можно запустить и на одноядерном процессоре.<br>- Эффективность при I/O-операциях — конкурентность особенно полезна, когда задачи часто ожидают внешних событий (например, ответа от сервера или чтения файла). Пока одна задача ждёт, система может переключиться на другую и выполнять полезную работу. |
| Параллелизм (Parallelism)    | **Параллелизм** — это фактическое, физическое выполнение нескольких задач или частей одной задачи в один и тот же момент времени. **Для реализации параллелизма необходимо аппаратное обеспечение** с несколькими вычислительными блоками, например, многоядерный процессор или несколько процессоров в системе.<br>**Главная цель параллелизма** — ускорить вычисления. Большая и сложная задача разбивается на более мелкие, независимые подзадачи, которые распределяются между доступными ядрами процессора. Каждое ядро решает свою часть головоломки, и в итоге общее время выполнения значительно сокращается.                                                                                                   |

### Как принудительно запустить поток?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **В Java нет способа принудительно запустить поток**.<br>Управление запуском потока полностью контролируется JVM (Java Virtual Machine). Когда вызывается метод start() для объекта класса Thread, JVM решает, когда и как запустить этот поток. Точное время запуска зависит от планировщика потоков в JVM, который определяет, когда и как долго каждый поток будет выполняться в рамках доступного процессорного времени. |

### Что такое монитор?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение    | **Монитор в Java** — это механизм синхронизации, который управляет одновременным доступом к объекту. Он гарантирует, что **только один поток может выполнять критическую секцию** (блок кода, который обращается к общим ресурсам) одновременно, в то время как другие потоки ожидают.<br>Каждый объект в Java имеет ассоциированный с ним монитор.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| Принцип работы | Когда поток входит в синхронизированный блок, он:<br>1. Запрашивает монитор объекта.<br>2. Если монитор доступен, он получает его и выполняет синхронизированный код.<br>3. Если другой поток уже удерживает монитор, новый поток ожидает, пока монитор не будет освобождён.<br>4. После выполнения поток освобождает монитор, позволяя другим потокам получить его.<br>**Ключевые концепции мониторов**:<br>- **Взаимное исключение** — только один поток может выполнять синхронизированный раздел кода одновременно. Пример: только один поток может обновлять общую переменную одновременно.<br>- **Координация потоков** — позволяет потокам взаимодействовать друг с другом с помощью методов: wait() (поток освобождает монитор и ждёт), notify() (пробуждает один ожидающий поток), notifyAll() (пробуждает все ожидающие потоки). |
| Реализация     | В Java монитор реализован с помощью **ключевого слова synchronized**. Этот оператор предваряет блок кода или метод, который подлежит синхронизации.<br>**Некоторые особенности реализации**:<br>- Только методы и блоки могут быть синхронизированы, но не переменные и классы.<br>- Если метод, в котором содержится критически важная «многопоточная» логика, статический, синхронизация осуществляется по классу.<br>- После окончания работы блока кода монитор объекта освобождается и становится доступным для других потоков.                                                                                                                                                                                                                                                                                                       |

### Что такое потоки демоны?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение | **Демон-потоки (daemon threads) в Java** — это особый тип потоков, которые работают на заднем плане и предназначены для выполнения вспомогательных задач.<br>**Ключевое отличие демон-потоков от обычных (пользовательских) потоков** заключается в их поведении при завершении программы: когда все пользовательские потоки завершают работу, JVM останавливается, не дожидаясь завершения оставшихся демон-потоков.<br>**Демон-потоки часто называют «потоками-служителями»** — их основная задача — обслуживание пользовательских потоков. Они не блокируют завершение программы, а тихо умирают вместе с ней.<br>**Демон-потоки полезны для выполнения задач, которые должны работать в фоновом режиме**, таких как мониторинг ресурсов системы, сборка мусора или выполнение периодических очисток.<br>**Важно**: демон-потоки наследуют свой статус от создавшего их потока. Если поток-создатель является демоном, то и созданный поток по умолчанию будет демоном, если не указать иное.<br>**Некоторые встроенные демон-потоки в JVM**: Garbage Collector — сборщик мусора, автоматически очищающий неиспользуемые объекты, Finalizer — отвечает за вызов финализаторов объектов перед удалением, Reference Handler — обрабатывает ссылки различных типов (weak, soft, phantom), Signal Dispatcher — управляет сигналами для JVM. |

### Как работает механизм синхронизации потоков в Java?

| Под темы              | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение           | **Механизм синхронизации потоков в Java работает с помощью ключевого слова synchronized**. Оно помечает метод или блок кода, чтобы сделать их потокобезопасными. Синхронизация обеспечивает **взаимное исключение**: только один поток может выполнять код в определённый момент времени.<br>Механизм тесно связан с концепцией мониторов объектов. Каждый объект в Java имеет связанный с ним монитор, который служит для координации доступа потоков к этому объекту. Монитор состоит из блокировки (lock) и очереди ожидания (wait set).<br>Синхронизация может осуществляться на разных уровнях: для методов, блоков или переменных, а также с использованием интерфейса Condition.                                                                                                                                                 |
| Методы                | **Синхронизированные методы** объявляются с ключевым словом synchronized в сигнатуре. При вызове такого метода блокировка автоматически устанавливается на объект this (для нестатических методов) или на объект класса (для статических методов).<br>**Некоторые особенности работы**:<br>- Блокировка распространяется на весь метод, что может снизить производительность.<br>- Если поток пытается зайти в синхронизированный метод, а монитор уже захвачен, поток блокируется по монитору объекта.<br>- Потоки, вызывающие нестатические синхронизированные методы одного и того же класса, будут блокировать друг друга только если они вызваны для одного объекта.<br>- Потоки, вызывающие статические синхронизированные методы одного класса, будут всегда блокировать друг друга — они блокируются по монитору Class-объекта. |
| Блоки                 | **Синхронизированный блок** — это блок кода, к которому одномоментно имеет доступ только один поток. Форма: synchronized (obj) { // критическая секция }.<br>**Особенности работы**:<br>- Когда поток входит в синхронизированный блок, он получает монитор указанного объекта. <br>- Любой другой поток, пытающийся войти в блок, синхронизированный на том же объекте, будет заблокирован до освобождения монитора.<br>- После окончания работы блока монитор объекта освобождается и становится доступным для других потоков.<br>**Важно**: синхронизация происходит не по самой ссылке, а по тому объекту, на который она ссылается, поэтому в synchronized можно передавать только ненулевые ссылки.                                                                                                                               |
| Переменные (volatile) | **Ключевое слово volatile** предлагает механизм синхронизации доступа к полю экземпляра без блокировки. **Особенности работы**:<br>- Запись в volatile-переменную всегда производится немедленно в основную память.<br>- Чтение volatile-переменной всегда происходит напрямую из основной памяти, аннулируя любое кэшированное значение.<br>- volatile устанавливает отношение «happens-before»: запись в volatile-переменную происходит после всех предыдущих чтений и записей в коде, чтение — до всех последующих чтений и записей.<br>**Однако volatile не гарантирует атомарность операций**. Например, операция counter++ разбивается на три шага: чтение, инкрементация и запись, и между ними другой поток может изменить значение, что приведёт к потере данных.                                                              |
| Condition             | **Интерфейс Condition** работает вместе с объектом Lock для предоставления более гибкого механизма ожидания и уведомления между потоками. **Особенности**:<br>- Поток не может взаимодействовать с Condition и его методами, если они не удерживаются с помощью Lock.<br>- Методы await(), signal() и signalAll() используются для управления ожиданием и возобновлением работы потоков.<br>**Важно**: выбор правильного механизма синхронизации зависит от множества факторов: требуемой семантики доступа, необходимости атомарных операций, количества конкурирующих потоков и ожидаемой производительности. Универсального решения не существует.                                                                                                                                                                                   |

### Что такое взаимная блокировка (deadlock) и как ее избежать?

| Под темы<br>           | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение            | **Взаимная блокировка (deadlock)** — ситуация в многозадачной среде или СУБД, при которой несколько процессов находятся в состоянии ожидания ресурсов, занятых друг другом, и ни один из них не может продолжать своё выполнение. В итоге работа всей программы останавливается или затормаживается — та не выполняет свои функции.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| Причины возникновения  | Для возникновения взаимной блокировки нужны специфические условия одновременного выполнения нескольких процессов. Некоторые из них:<br>- **Условие взаимного исключения** — каждый ресурс в данный момент или отдан ровно одному процессу, или доступен.<br>- **Условие удержания и ожидания** — процессы, в данный момент удерживающие полученные ранее ресурсы, могут запрашивать новые ресурсы.<br>- **Условие отсутствия принудительной выгрузки ресурса** — у процесса нельзя принудительным образом забрать ранее полученные ресурсы, процесс, владеющий ими, должен сам освободить ресурсы.<br>- **Условие циклического ожидания** — должна существовать круговая последовательность из двух и более процессов, каждый из которых ждёт доступа к ресурсу, удерживаемому следующим членом последовательности.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| Методы обнаружения     | Поиск взаимных блокировок осуществляется путём построения и анализа **графа ожидания**. В графе узлами отмечаются процессы и объекты, блокировки — рёбрами, направленными от узла, соответствующего захваченному объекту, к узлу, соответствующему захватившему его процессу, ожидания — рёбрами, направленными от узла, соответствующего ожидающему процессу, к узлу, соответствующему ожидаемому объекту. Цикл в графе ожидания соответствует взаимной блокировке. Существует специальный алгоритм поиска циклов в графе.<br>Некоторые инструменты разработчиков и языки программирования умеют находить взаимные блокировки автоматически. Например, Java Virtual Machine создаёт дампы, где описывает, какие ресурсы заняты, а какие свободны, и на основе дампов обнаруживает взаимоблокировки и информирует о них.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Способы предотвращения | Некоторые методы предотвращения взаимной блокировки:<br>- **Разработка иерархии блокировок**. Между блокировками устанавливается отношение сравнения и вводится правило о запрете захвата «большей» блокировки в состоянии, когда уже захвачена «меньшая». Если процессу нужно несколько блокировок, ему нужно всегда начинать с самой «большой» — предварительно освободив все захваченные «меньшие», если такие есть — и затем в нисходящем порядке.<br>- **Ранжирование ресурсов**. Каждому блокировке процесса задаётся приоритет: процессы обязаны начинать с самых «больших» блокировок, а перед тем как обратиться к ним, освобождать все «меньшие».<br>- **Проверка возможности блокировки**. Программа сама проверяет, можно ли получить доступ к ресурсам или это вызовет взаимную блокировку.<br>- **Игнорирование блокировок**. Это простое, но опасное решение: не блокировать ресурсы, когда с ними работает какой-то процесс. В таком случае взаимоблокировок не возникает, но без блокировок нельзя обеспечить сохранность данных, поэтому игнорирование применять не рекомендуется.<br>Практически об устранении взаимных блокировок надо заботиться ещё на этапе проектирования системы — это единственный более-менее надёжный способ борьбы с ними. В крайнем случае, когда основная концепция не допускает возможности избежать взаимных блокировок, следует хотя бы строить все запросы ресурсов так, чтобы такие блокировки безболезненно снимались. |

### Какие преимущества и недостатки использования ExecutorService для управления потоками?

| Под темы     | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение  | **ExecutorService** — интерфейс в пакете Java java.util.concurrent, который предоставляет способ управления пулом потоков для выполнения задач. Он обеспечивает более высокий уровень абстракции по сравнению с базовым интерфейсом Executor, позволяя лучше контролировать выполнение задач.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| Преимущества | - **Повторное использование потоков в пуле** — это снижает накладные расходы по сравнению с созданием новых потоков для каждой задачи.<br>- **Ограничение количества потоков**, используемых для группы задач, — это позволяет избежать нехватки ресурсов и повысить общую производительность системы.<br>- **Управление рабочими очередями** для управления потоком задач, что может уменьшить конкуренцию и повысить скорость реагирования.<br>- **Управление выполнением задач**: запуск, остановка и завершение задач.<br>- **Получение результатов задач** — возможность получить результаты выполнения задач, а также обрабатывать их результаты или ошибки.<br>- **Контроль над пулом потоков** — ExecutorService обеспечивает удобный доступ к пулу потоков, который может быть настроен для определённых потребностей приложения.<br>- **Возможность передавать экземпляр ExecutorService по кругу** — чтобы различные части программы могли отправлять ему задачи, при этом управление происходит в одном месте. <br>В Java интерфейс ExecutorService имеет несколько реализаций, например ThreadPoolExecutor, ScheduledThreadPoolExecutor и ForkJoinPool. |
| Недостатки   | - **Необходимость явно завершать работу исполнителя** — в интерфейсе ExecutorService есть методы shutdown() (ждёт завершения запущенных задач) и shutdownNow() (останавливает исполнитель немедленно). Если не завершить работу, JVM не сможет завершиться.<br>- **Потребление памяти** — объекты потоков занимают много памяти, и если создавать потоки для каждой асинхронной задачи, это может привести к ошибке OutOfMemoryError. В ExecutorService с одним потоком потоки используются повторно, что снижает потребление памяти.<br>- **Ограничения по количеству потоков** — например, в фиксированном пуле потоков (ThreadPoolExecutor) нельзя посылать задачи, если там нет места.<br>**Рекомендуется**: всегда закрывать пулы потоков — иначе приложение может не завершиться. Также стоит тестировать многопоточные приложения под нагрузкой — многопоточные баги проявляются только при конкуренции.                                                                                                                                                                                                                                                      |

### Какие стратегии и подходы используются для оптимизации производительности при работе с потоками?

| Под темы                | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| В операционных системах | **Управление потоками**. Позволяет эффективно использовать параллелизм, обеспечиваемый современными операционными системами и многоядерными процессорами. Одновременный запуск нескольких потоков на разных ядрах процессора сокращает общее время выполнения приложения. Некоторые стратегии:<br>- **Использование пулов потоков** — заранее создаётся фиксированное количество потоков, которые ожидают задачи из общей очереди. Это минимизирует накладные расходы на создание и уничтожение потоков.<br>- **Синхронизация потоков** — организация доступа к данным и совместного использования ресурсов между потоками. Используются механизмы синхронизации, например, мьютексы (блокировки) и семафоры.<br>- **Установка приоритетов потоков** — из двух потоков, готовых к выполнению, выбирается тот, у кого больше приоритет. Однако настройку приоритета следует выполнять осторожно, учитывая поведение других потоков в системе.<br>- **Мониторинг производительности потоков** — постоянный мониторинг помогает выявить и устранить потенциальные проблемы на ранних этапах.                                                                                    |
| В программировании      | - **Многопоточность** — разделение программы на параллельно выполняющиеся задачи. Позволяет эффективно использовать ресурсы процессора и ускорить выполнение вычислительно интенсивных задач. Некоторые подходы:<br>    - **Распараллеливание задач** — многие операции можно разделить на независимые подзадачи, которые можно выполнять параллельно.<br>    - **Использование конвейера** — задача разбивается на последовательные этапы, каждый из которых выполняется в отдельном потоке. Данные передаются от этапа к этапу через буферы. Этот паттерн эффективен, когда разные этапы обработки используют разные ресурсы системы (например, CPU и I/O).<br>    - **Work Stealing («Кража работы»)** — каждый поток имеет свою очередь задач, но при её опустошении может «украсть» задачи у других потоков. Этот паттерн обеспечивает балансировку нагрузки и масштабируемость, особенно для рекурсивных алгоритмов типа «разделяй и властвуй».<br>**Важно**: избыточная синхронизация или неэффективное распределение потоков могут снизить производительность, вызывая дополнительные накладные расходы, связанные с переключением контекста и ожиданием блокировок. |
| В базах данных          | - **Параллельная обработка** — одновременное выполнение нескольких задач или операций в базе данных для повышения производительности и сокращения времени обработки данных. Некоторые способы параллельной обработки:<br>    - **Параллельное выполнение запросов** — запросы разделяются на части и выполняются параллельно, что особенно эффективно для сложных и объёмных запросов.<br>    - **Параллельное индексирование** — индексы создаются и обновляются параллельно, чтобы ускорить доступ к данным и оптимизировать выполнение запросов.<br>    - **Параллельное разделение транзакций** — транзакции разделяются между несколькими потоками, что позволяет более эффективно управлять изменениями в базе данных.<br>- **Кэширование** — часто используемые запросы сохраняются в кэше, что делает доступ к ним практически мгновенным. Это полезно для данных, которые редко изменяются, но часто запрашиваются.<br>- **Кластеризация данных** — данные группируются по определённому критерию (например, по значению поля), что упрощает логическое понимание структуры данных и ускоряет процесс чтения.                                                       |
| В сетевых приложениях   | - **Управление трафиком** — алгоритмы управляют потоком данных в сети, что может помочь снизить задержку и увеличить пропускную способность. Например, метод WRED (Weighted Random Early Detection).<br>- **Использование QoS** — технологии, которая позволяет определять приоритеты трафика и распределять полосу пропускания. Это уменьшает перегрузку сети, когда объём трафика превышает доступную пропускную способность. QoS классифицирует трафик по различным категориям, которым присваиваются определённые приоритеты.<br>- **Алгоритмы очередей** — различным классам трафика присваиваются уровни приоритета в порядке передачи пакетов из очередей, что позволяет сети эффективнее распределять ресурсы в периоды перегруженности.<br>- **Регулярный мониторинг и анализ производительности сети** — сетевые администраторы используют инструменты мониторинга для отслеживания моделей трафика, определения точек перегрузки и внесения корректировок в политику <br>**Выбор стратегии** зависит от уникальных требований и характеристик системы — каждый метод имеет свои преимущества и недостатки.                                                        |

### Какие особенности и преимущества ForkJoinPool по сравнению с обычными потоками?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **ForkJoinPool в Java имеет особенности, которые отличаются от обычных потоков, и преимущества, связанные с распараллеливанием задач**. Некоторые особенности: <br>- **Алгоритм «кражи работы» (work-stealing)**. В традиционном пуле потоков, когда поток завершает назначенную задачу, он либо остаётся без работы, либо ему назначают новую задачу планировщиком пула. В ForkJoinPool idle-потоки активно ищут и «крадут» задачи у других потоков, которые могут быть перегружены или ждать. Это динамическое перераспределение задач оптимизирует использование ресурсов.<br>- **Управление задачами на нескольких процессорах или ядрах**. ForkJoinPool intelligently распределяет задачи по доступным потокам, что обеспечивает равномерное распределение нагрузки.<br>- **Рекурсивное выполнение задач**. Фреймворк автоматически разбивает задачи на управляемые единицы, что позволяет эффективно параллелизовать обработку, особенно в случаях, где природа задачи соответствует подходу «разделяй и властвуй».<br>- **Динамическое создание и выполнение задач**. Задачи в ForkJoinPool могут динамически создавать новые задачи по необходимости, что полезно для алгоритмов, которые требуют создания дополнительных подзадач во время выполнения.<br>- **Синхронное и асинхронное выполнение задач**. Хотя ForkJoinPool в основном работает синхронно, он также позволяет обрабатывать задачи асинхронно, что позволяет контролировать выполнение задач и синхронизацию. |
| Принцип работы | ForkJoinPool основан на алгоритме «Разделяй и властвуй»: каждая задача разбивается на подзадачи, затем они выполняются параллельно, и как только все из них завершаются, происходит объединение результатов.<br>Каждый рабочий поток имеет собственную двухстороннюю рабочую очередь (deque), что позволяет сканировать доступные для выполнения подзадачи. Поток блокируется только тогда, когда нет доступных для выполнения подзадач.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |

### Как можно реализовать механизм ожидания завершения нескольких потоков?

| Под темы        | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| --------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ExecutorService | **Метод invokeAll()** класса ExecutorService позволяет дождаться завершения всей группы задач. Он отправляет на выполнение все задачи из коллекции объектов Callable, блокирует их выполнение до завершения всех задач и возвращает список объектов Future, которые представляют результаты всех задач, в том же порядке, в котором они были отправлены. <br>**Чтобы сделать ожидание управляемым**, можно прекратить приём новых задач с помощью метода shutdown() и затем использовать awaitTermination(). Этот метод блокирует поток, пока все задачи не будут выполнены или не будет достигнут указанный тайм-аут.<br>**Недостаток метода invokeAll()** в некоторых ситуациях — он ждёт завершения всех задач, даже если какие-то из них уже завершились с ошибкой.                                                                                                 |
| CountDownLatch  | **CountDownLatch** позволяет потокам ожидать завершение определённых операций, выполняемых другими потоками, прежде чем переходить к следующему этапу работы.<br>**Работа**: CountDownLatch создаётся с значением счётчика. Каждый рабочий поток выполняет свою задачу и вызывает countDown(). Ожидающий поток вызывает await() и блокируется. Когда счётчик достигает нуля, ожидающий поток возобновляет выполнение.<br>**Особенности**:<br>- CountDownLatch используется для однократной синхронизации.<br>- После обнуления счётчика он не может быть сброшен.<br>- Если в блоке try возникли исключения, countDown() нужно вызывать в блоке finally, чтобы гарантировать выполнение синхронизации.                                                                                                                                                                  |
| CyclicBarrier   | **CyclicBarrier** позволяет группе потоков одновременно достичь определённой точки (барьера) и продолжить выполнение своей работы.<br>**Работа**: после завершения своего выполнения потоки вызывают метод await() и ждут, пока другие потоки достигнут барьера. Когда все потоки, которые вызвали await(), достигнут барьера и общее количество потоков равно указанному, барьер позволяет потокам продолжить.<br>**Особенности**:<br>- CyclicBarrier можно использовать несколько раз.<br>- Барьер может быть настроен с действием, которое выполняется после того, как все потоки достигнут барьера. Это действие можно использовать для объединения результатов потоков, ожидающих на барьере.<br>- Барьер ломается, когда любой из ожидающих потоков покидает барьер. Это происходит, когда один или более ожидающих потоков прерван или время ожидания завершено. |
