### Что такое Spring Cache и зачем он используется?

| Под темы                  | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение               | **Spring Cache** — это модуль фреймворка Spring, предоставляющий _абстракцию кэширования  для Java‑приложений. Это не реализация кэша, а набор  инструментов для прозрачного добавления  кэширования в существующее Spring‑приложение с  минимальным вмешательством в код.<br>Ключевые интерфейсы абстракции:<br>- `org.springframework.cache.Cache` — определяет базовые операции (get, put, evict, clear) для  работы с кэшем как с хранилищем пар «ключ‑значение»;<br>- `org.springframework.cache.CacheManager` — позволяет  получать регионы кэша (экземпляры `Cache`). |
| Как работает Spring Cache | Механизм применяет кэширование к методам Java:<br>1. При вызове метода Spring проверяет, выполнялся ли он ранее с такими же аргументами.<br>2. Если да — возвращает результат из кэша без повторного вызова метода.<br>3. Если нет — выполняет метод, сохраняет результат в кэш и возвращает его.<br>**Важно:** подход работает только для методов, гарантированно возвращающих один и тот же  результат для заданных аргументов.                                                                                                                                            |
| Зачем используется        | Основные цели применения:<br>- **Ускорение работы приложения** — снижение времени отклика за счёт повторного  использования результатов.<br>- **Снижение нагрузки на БД** — уменьшение числа запросов к базе данных.<br>- **Оптимизация ресурсоёмких операций** — кеширование результатов интенсивных  вычислений или операций ввода‑вывода.<br>- **Повышение масштабируемости** — более эффективное использование ресурсов  при росте нагрузки.                                                                                                                             |

### Какие аннотации предоставляет Spring Cache из коробки?

| Под темы     | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| @Cacheable   | - **Назначение**: кэширует результат метода при первом вызове.  При последующих вызовах с теми же аргументами  возвращает значение из кэша, минуя выполнение метода.<br>- **Ключевые атрибуты**:<br>    - `cacheNames` (или `value`) — имя кэша (обязательно);<br>    - `key` — ключ для хранения (по умолчанию  генерируется автоматически);<br>    - `condition` — условие применения  кэширования (SpEL‑выражение);<br>    - `unless` — условие исключения из  кэша (SpEL‑выражение). |
| @CachePut    | - **Назначение**: принудительно обновляет  кэш, выполняя  метод _всегда_ (даже если значение уже есть в кэше).  Используется для актуализации данных.<br>- **Особенности**: не проверяет кэш перед вызовом метода — всегда  выполняет логику и сохраняет результат.                                                                                                                                                                                                                      |
| @CacheEvict  | - **Назначение**: удаляет данные из кэша. Используется при изменении/удалении сущностей.<br>- **Ключевые атрибуты**:<br>    - `allEntries` — если `true`, очищает весь кэш  (а не только по ключу);<br>    - `beforeInvocation` — если `true`, удаление  происходит _до_ вызова метода.                                                                                                                                                                                                  |
| @Caching     | - **Назначение**: комбинирует несколько операций кэширования для  одного метода (например, `@Cacheable` + `@CacheEvict`).<br>- **Используется**, когда нужно применить несколько аннотаций одного  типа (например, два разных кэша).                                                                                                                                                                                                                                                     |
| @CacheConfig | - **Назначение**: задаёт общие настройки кэширования _на уровне класса_  (чтобы не дублировать их в каждом методе).<br>- **Ключевые атрибуты**:<br>    - `cacheNames` — имена кэшей по умолчанию;<br>    - `keyGenerator` — генератор ключей;<br>    - `cacheManager` — менеджер кэша;<br>    - `cacheResolver` — резолвер кэша.                                                                                                                                                         |

### В каких случаях имеет смысл использовать кэширование в приложении?

| Под темы                     | Заметка                                                                                                                                                                                                                                                          |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Основные  случаи  применения | **Частые чтения одних и тех же данных**<br>    - Данные запрашиваются многократно,  но обновляются редко (справочники, настройки, метаданные).<br>    - Пример: каталог товаров в интернет‑магазине,  список стран/регионов, тарифы услуг.                       |
|                              | **Ресурсоёмкие вычисления**<br>    - Результаты сложных расчётов (агрегация,  машинное обучение, парсинг) сохраняются, чтобы не  повторять их при каждом запросе.<br>    - Пример: прогноз погоды на основе множества  параметров, расчёт оптимального маршрута. |
|                              | **Медленные внешние зависимости**<br>    - Задержки при обращении к БД, API сторонних  сервисов, файловой системе.<br>    - Пример: запрос к удалённому сервису геолокации,  чтение большого файла с диска.                                                      |
|                              | **Высокая нагрузка на базу данных**<br>    - Множество одинаковых запросов к БД создаёт  «горячие точки». Кэширование снижает число обращений.<br>    - Пример: топ‑10 популярных статей на новостном сайте.                                                     |
|                              | **Повторные запросы от пользователей**<br>    - Один и тот же пользователь или группа  пользователей часто запрашивают идентичные данные.<br>    - Пример: профиль пользователя, история  заказов, лента рекомендаций.                                           |
|                              | **Данные с предсказуемым шаблоном доступа**<br>    - Известно, какие данные будут востребованы  (например, «хиты» за сутки).<br>    - Пример: кэш популярных видео на  стриминговой платформе.                                                                   |
|                              | **Необходимость работы в офлайн‑режиме**<br>    - Приложение должно функционировать без  постоянного подключения к сети.  <br>    - Пример: карты местности, локальный кэш новостей.                                                                             |
|                              | **Снижение затрат на инфраструктуру**<br>    - Уменьшение числа запросов к платным API  или дорогим сервисам.<br>    - Пример: кэширование ответов платёжного шлюза.                                                                                             |

Как включить и настроить Spring Cache в Spring Boot-приложении?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Шаги     | Добавление зависимости<br>Сначала нужно добавить в проект зависимость `spring-boot-starter-cache`. Она транзитивно включает модули `spring-context-support` и `spring-context`, которые предоставляют классы и интерфейсы для работы с кэшированием.                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
|          | Чтобы активировать кэширование, нужно добавить аннотацию `@EnableCaching` к любому классу конфигурации или к классу, аннотированному `@SpringBootApplication`. После этого Spring начнёт искать методы с аннотациями кэширования (например, `@Cacheable`, `@CachePut`, `@CacheEvict`) и перехватывать их вызовы.                                                                                                                                                                                                                                                                                                                                                                                                             |
|          | Настройка провайдера кэша<br>По умолчанию Spring Boot использует простой in-memory кэш на основе `ConcurrentHashMap`. Это удобно для разработки и тестирования, но в продакшене обычно требуется более мощный провайдер (например, Redis, EhCache, Caffeine, Hazelcast и др.).                                                                                                                                                                                                                                                                                                                                                                                                                                               |
|          | Настройка через свойства<br>Некоторые параметры кэширования можно задать через файл `application.properties` или `application.yml`:<br>- `spring.cache.type` — тип кэша (например, `redis`, `caffeine` и т. д.). Если не указан, Spring Boot автоматически определит его на основе доступных провайдеров. <br>- `spring.cache.cache-names` — список имён кэшей, которые будут созданы при запуске. <br>- Провайдер-специфичные настройки, например:<br>    - для Redis: `spring.cache.redis.time-to-live` (время жизни записи в миллисекундах), `spring.cache.redis.use-key-prefix` (использовать префикс ключа); <br>    - для Caffeine: `spring.cache.caffeine.spec` (например, `maximumSize=500, expireAfterAccess=60s`). |
|          | Использование аннотаций<br>После включения кэширования можно использовать аннотации для управления кэшем:<br>- `@Cacheable` — кэширует результат метода. При последующих вызовах с теми же аргументами возвращается значение из кэша, а не выполняется метод.<br>- `@CachePut` — обновляет кэш, выполняя метод всегда (даже если значение уже есть в кэше).<br>- `@CacheEvict` — удаляет данные из кэша.<br>- `@Caching` — объединяет несколько операций кэширования для одного метода.<br>- `@CacheConfig` — задаёт общие настройки кэширования на уровне класса.                                                                                                                                                           |
|          | Отключение кэширования<br>Чтобы отключить кэширование в любой среде, можно использовать свойство `spring.cache.type=none` в конфигурационном файле (например, `application-dev.properties`).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |

### Как работает аннотация @Cacheable, и как определить ключ кэша?

| Под темы                    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                       |
| --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Как работает  `@Cacheable`  | Аннотация `@Cacheable` указывает Spring, что **результат  метода нужно кэшировать**. При последующих  вызовах метода с теми же аргументами Spring вернёт  значение из кэша, не выполняя метод повторно.                                                                                                                                                                                                                       |
|                             | **Механизм работы:**<br>1. При вызове метода Spring проверяет, есть ли  в кэше запись с соответствующим ключом.<br>2. Если запись найдена (`cache hit`) — возвращает сохранённое значение.<br>3. Если записи нет (`cache miss`) — выполняет метод, сохраняет результат в кэш  и возвращает его.<br>**Обязательный параметр** — `value` (или  `cacheNames`) — имя кэша, куда будут помещаться  результаты.                     |
| Как определяется  ключ кэша | Ключ кэша — это идентификатор, по которому Spring  находит сохранённое значение. По умолчанию Spring  использует **встроенный `KeyGenerator`** на основе следующих правил:<br>- **Нет параметров** → возвращается `SimpleKey.EMPTY`. <br>- **Один параметр** → используется сам параметр как ключ.<br>- **Несколько параметров** → создаётся `SimpleKey`, содержащий все параметры  (на основе их `hashCode()` и `equals()`). |
| Настройка ключа  через SpEL | Для гибкого управления ключом используйте атрибут `key` с выражениями **Spring Expression Language (SpEL)**.                                                                                                                                                                                                                                                                                                                  |

### Чем отличается @CachePut от @Cacheable и @CacheEvict?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| @Cacheable  | **Основная задача:** _избежать выполнения метода_, если результат уже есть в кэше.<br>**Как работает:**<br>1. При вызове метода Spring сначала  проверяет кэш по ключу.<br>2. Если значение найдено — возвращает его **без вызова метода**.<br>3. Если значения нет — выполняет метод, сохраняет  результат в кэш и возвращает его.<br>**Когда применять:** для методов с **дорогостоящими вычислениями** или **частыми чтениями**, где результат стабилен при  одинаковых параметрах. |
| @CachePut   | **Основная задача:** _обязательно выполнить метод_ и **обновить кэш** новым значением.<br>**Как работает:**<br>1. Метод **всегда выполняется** (даже если значение уже есть в кэше).<br>2. Результат сохраняется в кэш (перезаписывает существующее значение).<br>**Когда применять:** при **обновлении данных**, когда нужно гарантировать выполнение логики метода и актуализацию кэша.                                                                                              |
| @CacheEvict | **Основная задача:** _удалить данные из кэша_ (очистить кэш).<br>**Как работает:**<br>1. Удаляет запись по указанному ключу (или весь кэш при `allEntries = true`).<br>2. Может выполняться **до** или **после** метода (`beforeInvocation`).<br>**Когда применять:** при **удалении данных** или когда нужно принудительно обновить кэш.                                                                                                                                              |

### Как настроить использование внешних систем кэширования, таких как Redis или Caffeine, в Spring Cache?

| Под темы           | Заметка                                                                                                                                                                                                                                                                |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Настройка Redis    | **Добавьте зависимости.** Для работы с Redis в Spring Boot обычно используется стартер `spring-boot-starter-data-redis`. По умолчанию он использует клиент Lettuce, но можно переключиться на Jedis, исключив зависимость Lettuce и добавив Jedis.                     |
|                    | Настройте свойства подключения в `application.properties` или `application.yml`                                                                                                                                                                                        |
|                    | **Создайте конфигурацию кэша.** Можно использовать автоконфигурацию Spring Boot, которая создаст `RedisCacheManager` по умолчанию, если Redis настроен. Для тонкой настройки создайте собственный класс конфигурации с аннотацией `@Configuration` и `@EnableCaching`. |
|                    | **Используйте аннотации кэширования** (`@Cacheable`, `@CachePut`, `@CacheEvict`) в методах сервисов, как обычно.                                                                                                                                                       |
| Настройка Caffeine | **Добавьте зависимости:**<br>- `spring-boot-starter-cache` для поддержки кэширования в Spring Boot;<br>- `caffeine` — сама библиотека Caffeine.                                                                                                                        |
|                    | **Настройте конфигурацию.** Есть два способа: через Java-код или через свойства.                                                                                                                                                                                       |
|                    | **Используйте аннотации кэширования** в методах сервисов.                                                                                                                                                                                                              |

### Какие существуют подходы к инвалидации кэша в распределённой среде?

| Под темы                                     | Заметка                                                                                                                                                                                                                                                                                                                                   |
| -------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Time‑to‑Live (TTL)                           | **Суть:** автоматическое удаление записей по истечении  заданного времени.<br>**Плюсы:**<br>- простота настройки;<br>- минимальная нагрузка на систему;<br>- не требует сложной координации между узлами<br>**Минусы:**<br>- возможны устаревшие данные до истечения TTL;<br>- не подходит для критически актуальных данных.              |
|                                              | Ручная инвалидация (Manual Invalidation)<br>**Суть:** явное удаление/обновление кэша при изменении данных в источнике.<br>**Плюсы:**<br>- полный контроль над актуальностью;<br>- мгновенное обновление.<br>**Минусы:**<br>- риск пропустить инвалидацию;<br>- требует синхронизации между сервисами.                                     |
| Write‑Through                                | **Суть:** данные сначала записываются в кэш, затем в  основное хранилище.<br>**Плюсы:**<br>- гарантированная актуальность кэша;<br>- атомарность операций.<br>**Минусы:**<br>- замедление записи (двойная операция);<br>- риск потери данных при сбое хранилища.<br>**Схема:**<br>1. `cache.set(key, value)`<br>2. `database.save(value)` |
| Read‑Through / Cache‑Aside                   | **Суть:** при отсутствии данных в кэше они загружаются  из источника и сохраняются.<br>**Плюсы:**<br>- минимизация кэш‑промахов;<br>- гибкость.<br>**Минусы:**<br>- первая загрузка медленнее;<br>- возможны несогласованные чтения.                                                                                                      |
| Event‑Driven  Invalidation                   | **Суть:** использование событий (через очереди сообщений)  для синхронизации кэша.<br>**Плюсы:**<br>- актуальность данных в реальном времени;<br>- масштабируемость.<br>**Минусы:**<br>- сложность реализации;<br>- накладные расходы на обработку событий.                                                                               |
| Версионирование  ключей (Key Versioning)     | **Суть:** добавление версии к ключу кэша при  обновлении данных.<br>**Плюсы:**<br>- мгновенная инвалидация на всех узлах;<br>- отсутствие конфликтов.<br>**Минусы:**<br>- увеличение объёма данных;<br>- необходимость управления версиями.<br>**Формат ключа:**  <br>`entity:id:v{version}` → `user:123:v2`                              |
| Каскадная  инвалидация                       | **Суть:** удаление связанных записей при изменении  основной сущности.<br>**Пример:**  <br>При обновлении пользователя → инвалидировать:<br>- `user:123`<br>- `user_orders:123`<br>- `user_profile:123`                                                                                                                                   |
| Проактивное  обновление  (Proactive Refresh) | **Суть:** фоновая перезагрузка данных до истечения TTL.<br>**Плюсы:**<br>- отсутствие «холодных» запросов;<br>- плавная нагрузка.<br>**Минусы:**<br>- избыточные операции;<br>- сложность настройки.                                                                                                                                      |

### Какие есть проблемы многопоточности при использовании кэша, и как их избежать (например, cache stampede)?

| Под темы                                    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Основные  проблемы  многопоточности  в кэше | Cache Stampede (эффект «стада»)<br>**Суть:** при одновременном запросе отсутствующих  в кэше данных множество потоков выполняют одну  и ту же ресурсоёмкую операцию.<br>**Пример:**<br>- кэш пуст;<br>- 100 запросов одновременно ищут один ключ;<br>- все 100 потоков идут в БД и выполняют  тяжёлый запрос;<br>- система перегружается.<br>**Решение:**<br>- **Блокировка по ключу:** только один поток выполняет загрузку, остальные ждут.<br>- **Проактивное обновление:** фоновая перезагрузка данных до истечения TTL.<br>- **Ленивая инвалидация:** возврат устаревших данных + фоновая загрузка новых.<br>- **Семафоры/мьютексы** на уровне приложения. |
|                                             | Аномалии чтения (Stale Reads)<br>**Суть:** потоки читают устаревшие данные из‑за задержки инвалидации.<br>**Пример:**<br>- поток A обновляет данные в БД;<br>- поток B читает из кэша старые данные (инвалидация ещё не прошла).<br>**Решение:**<br>- **Write-through:** запись сначала в кэш, потом в хранилище.<br>- **Версионирование ключей:** `key:v1` → `key:v2`.<br>- **Строгая синхронизация** через распределённые блокировки.                                                                                                                                                                                                                         |
|                                             | Потеря обновлений (Lost Updates)<br>**Суть:** параллельные записи приводят к потере изменений.<br>**Пример:**<br>- поток 1 читает `x = 5` из кэша;<br>- поток 2 читает `x = 5` из кэша;<br>- поток 1 записывает `x = 6`;<br>- поток 2 записывает `x = 6` (потеряно изменение потока 1).<br>**Решение:**<br>- **Оптимистическая блокировка:** проверка версии при записи.<br>- **Атомарные операции** (Redis `INCR`, `CAS`).<br>- **Очереди обновлений** с гарантированной последовательностью.                                                                                                                                                                  |
|                                             | Состояние гонки при инвалидации<br>**Суть:** несколько потоков пытаются обновить/удалить один ключ одновременно.<br>**Пример:**<br>- поток A удаляет ключ;<br>- поток B пытается прочитать тот же ключ и загружает данные заново;<br>- поток C тоже пытается загрузить данные.<br>**Решение:**<br>- **Распределённые блокировки** (Redis Redlock, ZooKeeper).<br>- **Идемпотентные операции:** повторные запросы не меняют результат.<br>- **Дедупликация запросов** (объединение одинаковых запросов).                                                                                                                                                         |
|                                             | Переполнение памяти (Memory Bloat)<br>**Суть:** одновременное заполнение кэша большим объёмом данных.<br>**Пример:**<br>- множество потоков загружают разные данные;<br>- кэш быстро растёт и потребляет всю память.<br>**Решение:**<br>- **Ограничение размера кэша** (LRU, LFU).<br>- **Эвакуация по TTL** (автоматическое удаление старых записей).<br>- **Мониторинг использования памяти.**                                                                                                                                                                                                                                                                |

### Как разделить данные между разными серверами?(consistent hashing) Что такое "прогревание кеша"?

| Под темы                                          | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Consistent  Hashing  (согласованное  хеширование) | **Суть:** алгоритм распределения данных по серверам,  минимизирующий переразмещение при изменении числа узлов.<br>Как работает<br>1. **Создаётся хеш‑кольцо** (hash ring) — виртуальное пространство ключей (например, от 0 до 2³² − 1).<br>2. **Серверы размещаются на кольце** — их идентификаторы (IP, имя)  хешируются и сопоставляются с позицией на кольце.<br>3. **Данные распределяются по  принципу «ближайший сервер по часовой стрелке»**:<br>    - Ключ хешируется → попадает в точку на кольце.<br>    - Ищем первый сервер, расположенный по  часовой стрелке_ от этой точки.<br>    - Данные хранятся на этом сервере<br>Преимущества<br>- **Минимальное переразмещение данных** при добавлении/удалении сервера (переезжают только ключи между соседними узлами).<br>- **Равномерное распределение** (при использовании виртуальных узлов — _vnodes_).<br>- **Масштабируемость** — легко добавлять/удалять серверы без глобальной перестройки.                                   |
| Прогревание  кэша (Cache  Warming)                | **Суть:** заблаговременная загрузка данных в кэш _до_ того, как они потребуются пользователям.<br>Зачем нужно<br>- **Избежать cache miss** при первом запросе (снижение латентности).<br>- **Снизить нагрузку** на БД/сервисы в момент пиковых запросов.<br>- **Обеспечить плавный старт** после перезапуска сервиса.<br>Способы прогревания<br>1. **Автоматическое** (на основе логов/статистики):<br>    - Анализировать популярные запросы за последние N часов.<br>    - Загружать топ‑K ключей в кэш при старте.<br>2. **Ручное** (через скрипты/админ‑панель):<br>    - Вызвать API для загрузки ключевых данных. <br>    - Использовать CLI‑утилиты.<br>3. **По расписанию** (cron‑задачи):<br>    - Каждые 5 мин обновлять кэш для «горячих» данных.<br>4. **На основе событий** (например, после деплоя):<br>    - Триггернуть прогревание через CI/CD‑пайплайн.<br>5. **Ленивое прогревание** (фоновый поток):<br>    - Постепенно загружать данные в кэш, не блокируя основной поток. |
