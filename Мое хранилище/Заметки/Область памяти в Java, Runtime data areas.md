### Что такое куча (Heap) в контексте JVM?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Куча (Heap) в контексте виртуальной машины Java (JVM)** — это область памяти, где **объекты динамически выделяются и хранятся во время выполнения программы**. В отличие от стека, который хранит локальные переменные и примитивные типы данных, куча используется для хранения объектов и массивов. <br>**Что хранит куча**:<br>- объекты, созданные с помощью оператора new;<br>- поля объектов, включая примитивные типы данных и ссылки на другие объекты;<br>- массивы, независимо от того, содержат ли они примитивные типы или объекты;<br>- статические переменные, поскольку они принадлежат классу, а не конкретному экземпляру;<br>- пул строк — строки в Java хранятся в специальном пуле строк в куче, чтобы повысить эффективность памяти.<br>**Особенности кучи**:<br>- общая для всех потоков — объекты, созданные в любом потоке, размещаются в куче и могут быть доступны для всех других потоков;<br>- не передаётся виртуальной машине целиком — какая-то часть резервируется в качестве виртуального пространства, за счёт которого куча может в будущем расти. |

### Какие типы памяти существуют в Java Virtual Machine (JVM)?

| Под темы            | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Куча (Heap)         | **Область памяти, где хранятся объекты и массивы**, созданные во время выполнения программы. Некоторые особенности:<br>- Динамическое выделение — объекты создаются и уничтожаются динамически.<br>- Автоматическое управление — управляется сборщиком мусора (Garbage Collector).<br>- Общая память — объекты, созданные в любом потоке, размещаются в куче и могут быть доступны для всех других потоков.<br>Куча делится на несколько поколений для эффективного управления памятью и сборки мусора:<br>- **Молодое поколение (Young Generation)** — содержит только что созданные объекты, разделено на область Эдема (Eden space) и области выживших (Survivor spaces).<br>- **Старое поколение (Old Generation)** — хранит долгоживущие объекты, реже подвергается сборке мусора.                                                                             |
| Стек (Stack)        | **Каждый поток выполнения в JVM имеет свой собственный стек** — **стек времени выполнения**. Он используется для хранения локальных переменных, вызовов методов и других данных, связанных с выполнением потока. Некоторые особенности:<br>- Память для стека не должна быть непрерывной — реализации JVM могут выделять и управлять памятью динамически в зависимости от платформы.<br>- После всех вызовов методов стек очищается и уничтожается JVM.<br>- Данные стека специфичны для потока, что обеспечивает безопасность локальных переменных.                                                                                                                                                                                                                                                                                                                |
| Метод (Method Area) | **Область памяти, где хранятся данные о классах, методах, статических переменных** и других элементах, необходимых для выполнения программы. Некоторые особенности:<br>- Хранит метаданные классов (имена, поля, методы), константный пул (строки, ссылки на методы и поля), байткод методов, статические переменные.<br>- Общая для всех потоков — классы и их метаданные общие для всех экземпляров класса.<br>- После загрузки классов область становится доступной только для чтения — структурная информация классов обычно не изменяется во время выполнения.<br>В старых версиях Java (до Java 7) область Method Area была частью пространства PermGen (Permanent Generation), но с Java 8 её заменил **Metaspace** — более гибкая и динамически масштабируемая область памяти.                                                                              |
| Нативная (Native)   | **Область памяти, выделенная за пределами кучи Java** и используемая JVM. Также известна как **память вне кучи**. Некоторые особенности:<br>- Данные в нативной памяти находятся за пределами JVM, для чтения и записи данных требуется сериализация.<br>- Из-за размещения за пределами JVM нативная память не освобождается сборщиком мусора.<br>- В нативной памяти JVM хранит стеки потоков, внутренние структуры данных и файлы, отображаемые в памяти.<br>Некоторые области нативной памяти в JVM:<br>- **Metaspace** — используется для хранения метаданных классов, заменяет пространство PermGen в Java 8 и позже.<br>- **Code Cache** — область, где JVM хранит скомпилированный код, используемый компилятором Just-In-Time (JIT).<br>- **String Pool** — нативная таблица фиксированного размера для хранения строк, используемых в оптимизации памяти. |

### Опишите, что такое стек (Stack) в JVM и для чего он используется

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **Стек (Stack) в виртуальной машине Java (JVM)** — это область памяти, в которой хранятся **локальные переменные и информация о вызовах методов**.<br>**Использование стека**:<br>- **Хранение локальных переменных** — примитивных типов данных (int, float, boolean и т. д.) и ссылок на объекты, которые объявлены внутри метода.<br>- **Хранение ссылок на объекты** — ссылок на объекты, которые хранятся в куче (Heap).<br>- **Хранение промежуточных результатов вычислений** — например, результатов арифметических операций, операций сравнения, которые используются в выражениях и конструкциях управления.<br>- **Хранение информации о вызове методов** — каждый раз, когда вызывается метод, создаётся новый «фрейм стека», который хранит информацию о состоянии вызова метода: параметры, локальные переменные, адрес возврата и так далее.<br>**Особенности стека**:<br>- Каждый поток в JVM имеет свой стек.<br>- Данные, хранящиеся в стеке, доступны только в области видимости соответствующего потока. В результате для доступа к стековой памяти не требуется синхронизация потоков. |
| Устройство     | Стек организован в виде последовательности **стековых фреймов** (Stack Frames). Каждый фрейм соответствует вызову метода. **Некоторые компоненты фрейма**:<br>- **Локальные переменные** — массив переменных, где хранятся все локальные переменные и их значения.<br>- **Стек операндов** — стек последним-вошёл-первым-вышел (last-in-first-out, LIFO), который действует как рабочая область среды выполнения для любых промежуточных операций.<br>- **Данные фрейма** — здесь хранятся все символы, соответствующие методу, а также информация о блоке catch на случай исключений.<br>**Важно**: память для стека не должна быть непрерывной — реализации JVM могут выделять и управлять памятью стека динамически, в зависимости от платформы.                                                                                                                                                                                                                                                                                                                                                         |
| Принцип работы | Стек работает по принципу **LIFO** (Last In, First Out), то есть последний добавленный элемент будет первым извлечённым. **Когда поток вызывает метод**, на стек добавляется новый стековый фрейм. **Когда метод завершает выполнение**, соответствующий стековый фрейм удаляется из стека.<br>**Важно**: этот процесс повторяется для каждого вызова метода, включая рекурсивные и вложенные вызовы.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |

### Как работает сборка мусора (Garbage Collection) в куче JVM?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **Сборка мусора (Garbage Collection) в куче виртуальной машины Java (JVM) работает автоматически**. Цель — освободить память, уничтожив неиспользуемые объекты. Объект подлежит удалению, если он нигде не используется, то есть на него больше нет ссылок.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| Принцип работы | Стандартная реализация сборки мусора включает три этапа:<br>1. **Пометка объектов как живых**. Сборщик мусора идентифицирует все живые объекты в памяти путём обхода графа объектов. Когда GC посещает объект, он помечает его как доступный и, следовательно, живой. Все объекты, недоступные из корней GC, рассматриваются как кандидаты на сбор мусора.<br>2. **Зачистка мёртвых объектов**. После фазы разметки пространство памяти занято либо живыми (посещёнными), либо мёртвыми (не посещёнными) объектами. Фаза зачистки освобождает фрагменты памяти, которые содержат эти мёртвые объекты.<br>3. **Компактное расположение оставшихся объектов в памяти**. Мёртвые объекты, которые были удалены во время предыдущей фазы, не обязательно находились рядом друг с другом, поэтому память может быть фрагментированной. Память можно уплотнить: оставшиеся объекты располагаются в непрерывном блоке в начале кучи. Процесс уплотнения облегчает последовательное выделение памяти для новых объектов.<br>Некоторые типы сборки мусора:<br>- **Малая сборка (minor collection)** — происходит при заполнении областей молодого поколения. Большинство молодых объектов умирают и подлежат удалению, но если какие-то из них всё ещё используются на момент GC, они перемещаются в область Выживших 0 или 1, а оттуда в Хранилище.<br>- **Основная сборка (major collection)** — происходит при заполнении Хранилища. Основная сборка обычно занимает больше времени из-за большего количества объектов. |

### Каково назначение памяти Metaspace в JVM?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Metaspace в JVM (Java Virtual Machine) предназначен для хранения метаданных классов** и другой отражательной информации.<br>В этой области хранятся, например:<br>- **Метаданные классов**: имена классов, методы, поля, аннотации и другая информация, необходимая для динамических функций Java, включая отражение и информацию о типах во время выполнения.<br>- **Статические переменные**: статические поля, объявленные в классах, которые доступны всем экземплярам этого класса.<br>- **Пул констант времени выполнения**: содержит литералы и ссылки, используемые во время выполнения. Для строковых литералов здесь хранятся ссылки (фактические значения — в строковом пуле в Heap), а другие литералы (например, числовые и логические) — непосредственно как значения.<br>Metaspace заменил область PermGen (Permanent Generation) в Java 8 и более поздних версиях. В отличие от PermGen, который имел фиксированный размер, Metaspace динамически выделяет память из операционной системы по необходимости. |

### Объясните разницу между Young и Old (Tenured) поколениями в куче JVM

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Young Generation** — это область памяти для новых, вновь создаваемых объектов. Большая часть объектов в этом поколении живёт недолго (итераторы, временные объекты, используемые внутри методов и т. п.) и удаляется при выполнении сборок мусора.<br>**Old (Tenured) Generation** — здесь скапливаются долгоживущие объекты (крупные высокоуровневые объекты, синглтоны, менеджеры ресурсов и проч.). Как правило, эта область больше, чем область для Young Generation.<br>После незначительной сборки мусора, когда устаревшие объекты достигают определённого порога возраста (по умолчанию порог современных JVM установлен на 15 циклов сборки мусора), они вместе с объектами-долгожителями переходят из молодого поколения в старое.<br>Таким образом, **Young Generation содержит короткоживущие объекты, а Old (Tenured) Generation — объекты, которые будут жить долго**. |

### Какие стратегии сборки мусора (Garbage Collection) существуют в JVM и в чем их особенности?

| Под темы                    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Serial (Последовательный)   | **Особенности**:<br>- Использует один поток для выполнения всей работы по сборке мусора.<br>- В начале каждой сборки мусора работа основных потоков приложения останавливается и возобновляется только после окончания сборки.<br>- Подходит для приложений с небольшим объёмом данных и не требовательных к задержкам.<br>- **Недостатки**: длительные паузы при сборке мусора, особенно если речь идёт о больших объёмах данных.<br>**Достоинства**: низкие требования к ресурсам, подходит для программ с маленькой кучей.                                                                                                                                                                                                                                                                                                                                             |
| Parallel (Параллельный)     | **Особенности**:<br>- Использует несколько потоков для ускорения сборки мусора.<br>- Подходит для приложений со средними и большими наборами данных, которые выполняются на многопроцессорном или многопоточном оборудовании.[<br>- Несколько потоков предназначены для малой сборки мусора в молодом поколении, единственный поток занят основной сборкой мусора в старшем поколении.<br>- **Достоинства**: возможность автоматической подстройки под требуемые параметры производительности и меньшие паузы на время сборок.<br>- **Недостатки**: определённая фрагментация памяти.<br>**Применение**: Parallel GC подходит для приложений, которые нуждаются в высокой производительности и могут терпеть некоторые паузы, например, для систем пакетной обработки.                                                                                                    |
| CMS (Concurrent Mark Sweep) | Особенности:<br>- Минимизирует время паузы в работе приложения за счёт одновременной маркировки доступных объектов во время функционирования приложения.<br>- Фаза очистки, которая очищает недоступные объекты, также выполняется параллельно, что сокращает время паузы.<br>- Подходит для приложений, в которых низкая задержка является приоритетом по сравнению с максимальной пропускной способностью, например для интерактивных приложений.<br>**Недостатки**: несмотря на уменьшение времени паузы, может привносить накладные расходы из-за одновременного выполнения операций. Кроме того, сборщик CMS сталкивается с проблемами фрагментации, что может привести к необходимости периодической полной сборки мусора.<br>**Устарел**: с Java 9 CMS не используется, в Java 14 сборщик полностью удалён.                                                        |
| G1 (Garbage-First)          | **Особенности**:<br>- Делит кучу на области и отдаёт предпочтение сбору в областях с наибольшим количеством мусора (отсюда «Garbage-First» — «сначала мусор»).<br>- Стремится обеспечить высокую пропускную способность и предсказуемое время отклика.<br>- Разработан для приложений с большой кучей, где важны как пропускная способность, так и низкая задержка.<br>**Достоинства**: обеспечивает более предсказуемое время паузы по сравнению с CMS и эффективно справляется с фрагментацией памяти.<br>**Недостатки**: для достижения оптимальной производительности может потребоваться более тонкая настройка флагов и конфигураций JVM.<br>**Применение**: G1 особенно полезен для серверных приложений, работающих на многоядерных процессорах.<br>**Важно**: выбор стратегии зависит от специфики приложения и требований к его производительности и задержкам. |

### Что такое Direct Memory в контексте JVM и как его использование влияет на производительность приложения?

| Под темы                      | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение                   | **Direct Memory** (также называется памятью вне кучи) — это область памяти, выделенная за пределами управляемой кучи JVM. В отличие от объектов, хранящихся в куче, Direct Memory по умолчанию не управляется сборщиком мусора (GC) JVM. Вместо этого она управляется с помощью нативных API операционной системы, например, malloc/free в системах, похожих на Unix.<br>В Java Direct Memory обычно доступна через экземпляры прямого ByteBuffer, созданные с помощью метода ByteBuffer.allocateDirect(int capacity)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Принцип работы                | Direct Memory используется для операций, которые требуют взаимодействия с нативным кодом или операций ввода-вывода. Например:<br>- чтение или запись данных из файлов;<br>- работа с сетевыми сокетами;<br>- использование определённых API, например, каналов NIO.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Использование                 | **Некоторые преимущества Direct Memory**:<br>- **Отсутствие накладных расходов GC** — прямая память не сканируется сборщиком мусора, что снижает время паузы для больших наборов данных.<br>- **Эффективность ввода-вывода** — прямые буферы сопоставляются непосредственно с нативными буферами ввода-вывода, что устраняет проблему «двойной копии».<br>- **Не ограничена размером кучи** — можно превышать лимиты, установленные параметром -Xmx (контролируется параметром -XX:MaxDirectMemorySize). Это полезно для больших кэшей в памяти (например, баз данных).<br>**Некоторые недостатки Direct Memory**:<br>- **Риски ручного жизненного цикла** — если поток «очиститель» не запускается (например, из-за сильных ссылок), возникают утечки памяти.<br>- **Более медленное выделение и освобождение** — вызовы памяти на уровне ОС (например, malloc, free) медленнее, чем выделение в куче.<br>**Рекомендуется избегать Direct Memory в следующих случаях**:<br>- **Размеры данных небольшие** — накладные расходы на выделение прямой памяти могут перевешивать преимущества избежания копии.<br>- **Буферы короткие** — частое выделение и освобождение прямых буферов медленнее, чем буферов в куче (из-за системных вызовов).<br>- **Управление памятью подвержено ошибкам** — неправильное обращение с прямыми буферами может привести к утечкам памяти или ошибкам OutOfMemoryError. |
| Влияние на производительность | **В Java 17** есть параметр -XX:MaxDirectMemorySize, который позволяет указать максимальное количество памяти, которое может быть выделено для прямых буферов. Можно указать максимальное количество в байтах, килобайтах, мегабайтах или гигабайтах. <br>**По умолчанию** максимальный размер прямой памяти ограничен максимальным размером кучи (-Xmx). Если не указать -XX:MaxDirectMemorySize, максимальный размер прямой памяти по умолчанию равен тому же значению, что и максимальный размер кучи (-Xmx).<br>**Важно**: выделение слишком много прямой памяти может повлиять на ресурсы системы, потенциально приводя к ошибкам из-за нехватки памяти или снижению производительности.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |

### Как JVM обрабатывает intern-строки в памяти и как это влияет на производительность приложения?

| Под темы                      | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                   | **JVM обрабатывает intern-строки (уникальные строковые литералы) в памяти с помощью механизма интернирования** — хранения только одной копии каждой уникальной строки в пуле (String Pool). Это позволяет оптимизировать работу с строками, особенно при работе с большим количеством повторяющихся строк.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| Принцип работы                | Когда создаётся строка в коде, JVM сначала проверяет, есть ли уже такая строка в пуле. Если есть — возвращает ссылку на существующий объект, если нет — создаёт новый.<br>**Некоторые особенности интернирования:**<br>- **Автоматическое интернирование** — строковые литералы (например, «Hello») автоматически добавляются в пул.<br>- **Явное интернирование** — можно использовать метод intern() — он проверяет, существует ли строка в пуле, и если нет, добавляет её в пул и возвращает ссылку.<br>**Расположение String Pool**: до Java 7 пул располагался в области PermGen, которая имела фиксированный размер и не подвергалась сборке мусора. С Java 7 пул переехал в основную кучу (Heap), что сделало его более гибким и расширяемым.                                                                                                                                                                                                  |
| Влияние на производительность | **Интернирование позволяет:**<br>- **Снизить потребление памяти** — вместо создания нескольких объектов с одинаковым содержимым Java использует существующие. Это особенно полезно в приложениях, которые используют много одинаковых строк, например, в парсерах, обработчиках текста или программах, связанных с базами данных.<br>- **Ускорить сравнение** — поскольку интернированные строки используются повторно, можно выполнять более быстрое сравнение по ссылкам (\==) вместо сравнения на основе содержимого (equals()).<br>**Однако интернирование имеет и ограничения**:<br>- **Злоупотреблять им не стоит** — пул хранится в heap, и при массовом использовании можно получить обратный эффект — ошибку OutOfMemoryError.<br>- **Интернирование наиболее полезно** при работе с неизменяемыми и повторяющимися строками — для динамически генерируемых или изменяемых строк преимущества интернирования могут быть менее значительными. |
