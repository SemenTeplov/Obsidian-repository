Какие фреймворки используются для модульного тестирования в Java?

| Под темы    | Заметка                                                                                                                                                                                                            |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| JUnit       | **Распространённый фреймворк** для модульного тестирования (юнит-тестирования) Java-приложений.                                                                                                                    |
| Возможности | **Аннотации** для обозначения различных частей тестов. Например, @Test — обозначает метод, который выполняет тест, @BeforeEach — указывает метод, который должен быть выполнен перед каждым тестом.                |
|             | **Ассерты (Assertions)** для проверки результатов тестов: assertEquals(expected, actual) — проверяет, равны ли ожидаемое и фактическое значения, assertTrue(condition) — проверяет, что условие возвращает истину. |
|             | **Тестовые наборы** — позволяет объединять несколько тестов в наборы, что упрощает управление большими проектами.                                                                                                  |
|             | **Тестирование исключений** — поддерживает проверку того, что при выполнении кода выбрасываются ожидаемые исключения.                                                                                              |
|             | **Параметризованные тесты** — с помощью JUnit можно создавать тесты с разными входными параметрами, что помогает проверять поведение программы в различных условиях.                                               |
| TestNG      | **Изначально создавался как улучшенная версия JUnit**, предлагая расширенную функциональность для тестирования.                                                                                                    |
| Возможности | **Поддержка зависимостей между тестами** — можно гарантировать, что конкретные тесты выполняются перед другими.                                                                                                    |
|             | **Гибкое группирование тестов** — тесты можно упорядочить и сгруппировать по приоритетности.                                                                                                                       |
|             | **Многопоточное выполнение** — тестовый метод можно запускать в нескольких потоках.                                                                                                                                |
| Mockito     | **Фреймворк для создания заглушек (моков, stubs)** при тестировании. Позволяет имитировать поведение внешних зависимостей, чтобы тестировать единицы кода в изоляции.                                              |
| Возможности | **Создание заглушек** для классов и интерфейсов.                                                                                                                                                                   |
|             | **Проверка вызова метода** и значений передаваемых методу параметров.                                                                                                                                              |
|             | **Использование концепции «частичной заглушки»** — заглушка создаётся на класс с определением поведения, требуемого для некоторых методов класса.                                                                  |
|             | **Проверка вызова метода с задержкой** — фреймворк позволяет проверить вызов определённого метода в течение заданного времени.                                                                                     |
| Spock       | **Среда тестирования и спецификации** для приложений Java и Groovy.                                                                                                                                                |
| Возможности | **Конструкции для написания спецификаций** — класс, описывающий тестовый сценарий, фичи-тесты — методы, проверяющие конкретное поведение кода, блоки — структурные элементы, разделяющие этапы тестирования.       |
|             | **Параметризованные тесты** — Spock позволяет создавать тесты с разными входными данными с помощью блока where.                                                                                                    |
|             | **Создание объектов-заглушек и моков** для тестирования взаимодействия между компонентами: Mock — для проверки вызова методов, Stub — для предоставления предопределённых ответов.                                 |
|             | **Проверка на исключения** — проверка на исключения реализуется с помощью блока then.                                                                                                                              |

### Что такое JUnit и какие преимущества он предлагает для тестирования Java приложений?

| Под темы     | Заметка                                                                                                                                                                                                                                                           |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение  | **JUnit** — фреймворк для написания и выполнения модульных тестов в языке программирования Java. Он позволяет разработчикам создавать независимые тесты для отдельных компонентов программы, что облегчает поиск и устранение ошибок на ранних этапах разработки. |
| Преимущества | **Автоматизация** — тесты могут быть выполнены автоматически после каждого изменения кода, что помогает быстро находить ошибки.                                                                                                                                   |
|              | **Повышение надёжности** — модульные тесты позволяют проверять функциональность каждого компонента программы, что снижает вероятность возникновения ошибок в финальной версии.                                                                                    |
|              | **Упрощение поддержки кода** — тесты помогают выявлять изменения, которые могут нарушить работу программы при добавлении нового функционала.                                                                                                                      |
|              | **Лёгкость в использовании** — с помощью простых аннотаций и инструментов JUnit можно быстро организовать тестирование любых частей программы.                                                                                                                    |

|                | Позволяет запускать тесты, написанные в JUnit 3, 4 и 5, одновременно через платформу JUnit 5.                                                                                                              |

### Опишите процесс настройки и запуска тестов с использованием JUnit и какие основные аннотации используются в этом процессе.

| Под темы          | Заметка                                                                                                                                                                                                                                      |
| ----------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ             | **Процесс настройки и запуска тестов с использованием JUnit** включает создание тестовых классов, инициализацию данных, использование аннотаций для указания жизненного цикла тестов и выполнения определённых действий до или после тестов. |
| Процесс настройки | **Создание тестовых классов**. Разработчик размещает аннотации перед методами в тестовом классе, указывая фреймворку, как именно их следует обрабатывать.                                                                                    |
|                   | **Инициализация данных**. Например, если есть несколько тестов, которые требуют одних и тех же начальных условий, можно обозначить метод с аннотацией @Before, задав необходимые условия тестирования один раз.                              |
|                   | **Организация тестов**. Тесты могут быть организованы в test suites (связки тестов). Например, в JUnit 5 есть аннотации для создания наборов тестов.                                                                                         |
| Аннотации         | **@Test** — означает, что метод является тестовым.                                                                                                                                                                                           |
|                   | **@Before** — выполняется перед каждым тестовым методом, используется для настройки окружения.                                                                                                                                               |
|                   | **@After** — выполняется после каждого тестового метода, используется для освобождения ресурсов.                                                                                                                                             |
|                   | **@BeforeClass** — выполняется один раз перед всеми тестами в классе, метод, помеченный этой аннотацией, должен быть статическим.                                                                                                            |
|                   | **@AfterClass** — выполняется один раз после всех тестов в классе, используется для задач очистки, таких как закрытие файлов или соединений.                                                                                                 |
|                   | **@Test(timeout)** — гарантирует, что тест не займёт слишком много времени: если он превышает указанное время (в миллисекундах), тест падает.                                                                                                |

### Какие существуют виды тестирования ПО и для каких целей они используются?

| Под темы           | Заметка                                                                                                                                                                                                         |
| ------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ              | **Существует несколько видов тестирования программного обеспечения (ПО)**, которые направлены на проверку разных аспектов: функционального, производительности, безопасности и совместимости.                   |
| Функциональное     | **Цель** — проверить, насколько корректно приложение выполняет заявленные функции.                                                                                                                              |
| Цели               | Выявить ошибки и несоответствия в функциональности на ранних стадиях разработки;                                                                                                                                |
|                    | Убедиться, что каждая опция системы выполняется согласно требованиям, заданным заказчиком или разработчиком;                                                                                                    |
|                    | Улучшить пользовательский опыт — гарантировать интуитивно понятный интерфейс и корректное поведение программы при различных сценариях использования.                                                            |
| Производительности | **Цель** — определить, как быстро работает вычислительная система или её часть под определённой нагрузкой.                                                                                                      |
| Цели               | Выявить и устранить узкие места производительности в программном приложении;                                                                                                                                    |
|                    | Оценить время ответа приложения при различных условиях нагрузки;                                                                                                                                                |
|                    | Сравнить производительность приложения с конкурентами или аналогичными решениями.                                                                                                                               |
| Безопасности       | **Цель** — выявить уязвимости, которые могут быть использованы злоумышленниками для несанкционированного доступа к данным или нанесения ущерба системе.                                                         |
| Цели               | Обнаружить слабые места до их эксплуатации злоумышленниками;                                                                                                                                                    |
|                    | Устранить обнаруженные уязвимости и улучшить общую безопасность продукта.                                                                                                                                       |
| Совместимости      | **Цель** — обеспечить корректную работу и взаимодействие программных приложений с разными типами оборудования, операционными системами, веб- и мобильными браузерами, сетями и другими программными элементами. |
| Цели               | Определить и зафиксировать любые проблемы совместимости, которые могут привести к сбоям в работе продукта;                                                                                                      |
|                    | Повысить общую стабильность продукта, так как тестирование гарантирует, что все его компоненты будут работать без сбоев в любой среде;                                                                          |
|                    | Найти потенциальные угрозы безопасности — если продукт несовместим с определёнными системами или сетями, злоумышленники могут использовать систему и получить доступ к конфиденциальным данным.                 |

### Какие стратегии можно применить для тестирования микросервисов?

| Под темы                    | Заметка                                                                                                                                                                                                                                                               |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Юнит-тестирование           | Проверяет отдельные функции и методы внутри микросервиса.                                                                                                                                                                                                             |
| Cтратегии                   | **Изолированное тестирование** — тесты проверяют только одну функцию или метод, без зависимости от других частей системы.                                                                                                                                             |
|                             | **Использование моков** — они позволяют изолировать тестируемый код, заменяя настоящие зависимости на поддельные.                                                                                                                                                     |
|                             | **Проверка граничных случаев** — некорректных данных, крайних значений.                                                                                                                                                                                               |
|                             | **Автоматизация** — юнит-тесты могут быть автоматически запущены в рамках CI/CD-процессов.                                                                                                                                                                            |
| Интеграционное тестирование | Проверяет взаимодействие между несколькими микросервисами или их компонентами.                                                                                                                                                                                        |
| Cтратегии                   | **Имитация зависимостей** — часто требуется использование заглушек и моков для имитации работы зависимых сервисов.                                                                                                                                                    |
|                             | **Создание сценариев тестирования** — они покрывают цепочки взаимодействий, например, для интернет-магазина — сценарий оформления заказа: от добавления товара в корзину до оплаты и уведомления о доставке.                                                          |
|                             | **Проверка обработки данных с разных сторон** — иногда полезно провести тесты, меняя точку взаимодействия, например, если сервис А отправляет запрос в сервис Б, начать проверку с одного сервиса, затем проверить, как другой сервис обрабатывает полученные данные. |
|                             | **Создание негативных сценариев** — тесты с нестандартными данными или некорректными значениями, чтобы убедиться, что система корректно обрабатывает ошибки.                                                                                                          |
| Контрактное тестирование    | Гарантирует, что интерфейсы взаимодействующих микросервисов соответствуют заранее определённым контрактам.                                                                                                                                                            |
| Cтратегии                   | **Определение контракта** — потребитель описывает, какие запросы он отправляет и какие ответы или события ожидает получить от поставщика.                                                                                                                             |
|                             | **Запуск тестов на основе контракта** — сервис-поставщик запускает у себя тесты, в которых повторяет запросы из контракта к своему API и проверяет, что ответы совпадают с ожиданиями потребителя.                                                                    |
|                             | **Раннее обнаружение проблем** — контракты служат своеобразным предварительным соглашением между командами, любое несоответствие выявляется на этапе CI/CD, ещё до среды интеграции.                                                                                  |
|                             | **Изоляция тестов** — потребитель и провайдер тестируются независимо.                                                                                                                                                                                                 |
| Нагрузочное тестирование    | Оценивает производительность и устойчивость системы при высокой нагрузке.                                                                                                                                                                                             |
| Cтратегии                   | **Создание реалистичных тестовых сценариев** — нужно воссоздавать реальные потоки пользователей, сочетая разные действия и типы запросов.                                                                                                                             |
|                             | **Определение целей и метрик производительности** — перед началом тестирования необходимо чётко определить цели, например, измерить время отклика, подтвердить устойчивость при пиковых нагрузках.                                                                    |
|                             | **Фокус на важных микросервисах** — лучше определить микросервисы с наибольшими рисками и начинать их нагрузочное тестирование на ранних этапах цикла разработки.                                                                                                     |
|                             | **Одновременное нагрузочное тестирование сгруппированных микросервисов** — можно достичь путём параллельного выполнения реалистичных нагрузочных тестов нескольких микросервисов.                                                                                     |
|                             | **Использование инструментов** — понадобятся инструменты, способные эффективно моделировать реальные сценарии нагрузки, например, Gatling, который может генерировать огромные объёмы HTTP-трафика.                                                                   |

### Как реализовать параметризованные тесты в JUnit 5?

| Под темы            | Заметка                                                                                                                                                                        |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ               | **Параметризованные тесты в JUnit 5 реализуются с помощью аннотации @ParameterizedTest**. Она позволяет запустить один и тот же тест несколько раз с разными входными данными. |
| Передача параметров | **Аннотация @ValueSource** — указывает, что тест будет запущен с набором строк, перечисленных в массиве.                                                                       |
|                     | **Аннотация @EnumSource** — передаёт в метод все значения определённого перечисления.                                                                                          |
|                     | **Аннотация @MethodSource** — позволяет передавать в тестовый метод параметры, создаваемые программным кодом с помощью статического метода.                                    |

### Какие особенности тестирования реактивных приложений и какие инструменты для этого используются?

| Под темы    | Заметка                                                                                                                                                                                                                                                                        |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ       | **Тестирование реактивных приложений** имеет особенности, связанные с асинхронной природой реактивного кода и работой с потоками данных. Для этого используются специальные инструменты, которые учитывают эти особенности.                                                    |
| Особенности | **Асинхронные и ленивые потоки** — результат не приходит «сразу». Нужна проверка подписки, значений, ошибок и завершения.                                                                                                                                                      |
|             | **Невозможность вызывать методы классов напрямую** — вместо этого используется API реактивных потоков.                                                                                                                                                                         |
|             | **Сложность отладки реактивных потоков** — в то время как в синхронной и блокирующей системе трассировка стека ошибок указывает на первопричину проблемы, в потоке асинхронного реактора ошибка регистрируется у подписчика, но была вызвана у оператора при обработке потока. |
|             | **Необходимость протестировать конвейер потоковой обработки** в трёх случаях: когда поток завершается нормально, когда поток завершается с ошибкой и когда поток отменяется.                                                                                                   |
