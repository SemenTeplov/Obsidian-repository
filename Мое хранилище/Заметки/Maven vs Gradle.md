### Что такое системы сборки в контексте разработки программного обеспечения?

| Под темы                      | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| ----------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ                         | Системы сборки в разработке программного обеспечения — это **специализированные программы для автоматизации процесса преобразования исходного кода в исполняемые файлы** (программы, библиотеки, конфигурационные файлы).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |

Позволяет генерировать версии кода для разных целевых платформ.<br>    <br>6. **Управление версиями**. Поддерживает разные версии компонентов и их совместимость. |

| Основные отличия | Основные отличия Maven и Gradle:<br><br>1. **Синтаксис и формат конфигурационных файлов**<br>    <br>- **Maven**: использует XML‑формат (_pom.xml_). Строгая, иерархическая структура с чётко определёнными секциями.<br>    <br>- **Gradle**: применяет DSL на основе Groovy или Kotlin (_build.gradle_ / _build.gradle.kts_). Более лаконичный и гибкий синтаксис, позволяющий писать программный код.<br>    <br>2. **Подход к конфигурации**<br>    <br>- **Maven**: «конвенция над конфигурацией» — жёсткая предопределённая структура проекта. Для типовых задач достаточно указать минимальные настройки.<br>    <br>- **Gradle**: максимальная гибкость — можно описывать любые сценарии сборки через код, включая кастомные задачи и условия.<br>    <br>3. **Управление зависимостями**<br>    <br>- **Maven**: централизованный репозиторий (Maven Central), строгие правила версионирования. Зависимости объявляются в _pom.xml_.<br>    <br>- **Gradle**: поддерживает те же репозитории, но добавляет гибкие правила замены зависимостей, кастомные конфигурации и варианты фич (_feature variants_).<br>    <br>4. **Скорость сборки**<br>    <br>- **Maven**: по умолчанию пересобирает модули последовательно. Для параллельной сборки требуется явная настройка (`-T`).<br>    <br>- **Gradle**: встроенная поддержка инкрементальной сборки (пересобирает только изменённые части) и параллельного выполнения задач. Есть кэш задач (_Build Cache_), ускоряющий повторные сборки.<br>    <br>5. **Кастомизация и плагины**<br>    <br>- **Maven**: плагины пишутся на Java, требуют глубокого понимания API. Добавление нестандартной логики сложно.<br>    <br>- **Gradle**: позволяет писать кастомные задачи прямо в скрипте сборки (на Groovy/Kotlin). Поддержка внешних скриптов и плагинов на Java/Kotlin.<br>    <br>6. **Отладка и мониторинг**<br>    <br>- **Maven**: базовый режим отладки, плагины типа _Surefire_ для тестов.<br>    <br>- **Gradle**: инструмент _Build Scan_ — интерактивный веб‑интерфейс с детальной аналитикой сборки (время задач, загруженные зависимости, ошибки).<br>    <br>7. **Интеграция с экосистемой**<br>    <br>- **Maven**: широкая поддержка в IDE (Eclipse, IntelliJ), CI/CD‑системах. Стандарт для Java‑проектов.<br>    <br>- **Gradle**: официальная система сборки для Android (Android Studio). Лучшая интеграция с Kotlin. Поддерживает мультипроектные сборки.<br>    <br>8. **Кривая обучения**<br>    <br>- **Maven**: проще для новичков — XML понятен, структура предсказуема.<br>    <br>- **Gradle**: требует знаний Groovy/Kotlin и понимания DSL, но даёт больше возможностей для сложных сценариев.<br>    <br>9. **Типичные сферы применения**<br>    <br>- **Maven**: небольшие и средние Java‑проекты, корпоративные среды с жёсткими стандартами.<br>    <br>- **Gradle**: крупные мультимодульные проекты, Android‑разработка, CI/CD с частыми итерациями, проекты на Kotlin.<br>    <br>10. **Производительность**<br>    <br>- **Maven**: может быть медленнее на больших проектах из‑за отсутствия инкрементальной сборки по умолчанию.<br>    <br>- **Gradle**: значительно быстрее за счёт кэширования, параллелизма и инкрементальных сборок (например, сокращение времени сборки с 32 до 9 минут в enterprise‑проектах). |

### Какова основная функция файла pom.xml в Maven?

| Под темы                | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ                   | Файл **pom.xml** (Project Object Model) — центральный  конфигурационный файл в проекте Maven. Его основная  функция — **описывать структуру, зависимости и настройки проекта** для автоматизации сборки, тестирования, упаковки и  развёртывания приложения.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| Ключевые задачи pom.xml | 1. **Определение метаданных проекта**  <br>    Задаются уникальные идентификаторы:<br>    - `<groupId>` — обычно домен организации в обратном порядке (например, `com.mycompany`).<br>    - `<artifactId>` — имя проекта или модуля.    <br>    - `<version>` — версия проекта (может включать суффикс `-SNAPSHOT` для промежуточных сборок).   <br>    - `<name>` — отображаемое имя артефакта.        <br>    - `<url>` — ссылка на сайт проекта.<br>        <br>2. **Управление зависимостями**  <br>    В секции `<dependencies>` перечисляются все внешние библиотеки, необходимые проекту. Для каждой указываются:   <br>    - `groupId`, `artifactId`, `version` — уникальные идентификаторы зависимости.      <br>    - `<scope>` — область применения (например, `test` для библиотек тестирования).    <br>    Maven автоматически:   <br>    - загружает зависимости из репозиториев;      <br>    - обрабатывает транзитивные зависимости (библиотеки, нужные самим зависимостям).<br>        <br>3. **Настройка процесса сборки**  <br>    Секция `<build>` определяет:    <br>    - плагины (например, `spring-boot-maven-plugin` для Spring Boot);        <br>    - директории ресурсов (`<resources>`);        <br>    - цели сборки (JAR, WAR и др. через `<packaging>`).<br>        <br>4. **Определение свойств проекта**  <br>    В `<properties>` задаются переменные (например, версия Java), которые можно повторно использовать в конфигурации:    <br>  <br>5. **Конфигурация профилей сборки**  <br>    Секция `<profiles>` позволяет определять разные настройки для сред (разработка, продакшен) и активировать их через команды типа:<br>    <br>6. **Управление плагинами**  <br>    В `<pluginManagement>` указываются версии и конфигурации плагинов, чтобы обеспечить их согласованную работу в мультимодульных проектах. |

| Ответ                                    | Gradle обеспечивает **заметный прирост  производительности** по сравнению с Maven за  счёт ряда архитектурных и технологических  решений. | Ключевые преимущества производительности | 1. - **Инкрементальная сборка**      <br>        - **Gradle**: анализирует входные/выходные данные каждой задачи и выполняет пересборку **только при изменении зависимостей**. Неизменённые части проекта пропускаются.          <br>        - **Maven**: по умолчанию перекомпилирует модули целиком, даже если изменился один файл (без внешних кэшей).          <br>    - **Кэширование сборок (Build Cache)**        <br>        - Gradle сохраняет результаты выполнения задач в локальном или удалённом кэше. При повторных запусках использует готовыеартефакты, если входные данные не изменились.            <br>        - В Maven аналогичное кэширование требует сторонних плагинов и настраивается сложнее.            <br>    - **Параллельное выполнение задач**        <br>        - Gradle поддерживает **параллельную сборку проектов** (`--parallel`) и **параллельное выполнение задач** внутри проекта. Эффективно использует многоядерные процессоры.            <br>        - Maven (с версии 3.0) тоже имеет флаг `--threads`, но управление JVM‑экземплярами менее оптимизировано, что может приводить к избыточному потреблению памяти.            <br>    - **Gradle Daemon**        <br>        - Фоновый процесс, остающийся в памяти между запусками. Снижает накладные расходы на:            <br>            - инициализацию JVM;                <br>            - загрузку классов;                <br>            - разбор конфигураций.                <br>        - В Maven каждый запуск начинается «с нуля», что замедляет частые сборки.           <br>    - **Оптимизированное управление зависимостями**<br>        - Gradle эффективнее разрешает транзитивные зависимости и проверяет их актуальность.    <br>        - Поддерживает **вариативные зависимости** (feature variants), что сокращает избыточные загрузки.        <br>    - **Up‑to‑date checks**        <br>        - Перед выполнением задачи Gradle проверяет, нужны ли её входные данные. Если файлы не изменились, задача пропускается.          <br>        - В Maven подобные проверки реализованы частично и требуют ручной настройки.           <br>    - **Конфигурация кэша (Configuration Cache)**        <br>        - Gradle кэширует результаты конфигурации сборки (например, разрешение плагинов), ускоряя старт процесса.           <br>        - В Maven конфигурация пересчитывается при каждом запуске.            <br>    - **Оптимизация для CI/CD**   <br>        - В пайплайнах Gradle сокращает время сборки за счёт:     <br>            - повторного использования кэша между запусками;        <br>            - инкрементальных изменений в коде;     <br>            - параллельной обработки модулей. |

### Опишите, как Gradle использует Groovy (или Kotlin) для конфигурации сборки.

| Под темы           | Заметка                                                                                                                                                                                                                                                                                                                                                                                        |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ              | Gradle использует **Groovy DSL** (в файлах `build.gradle`) или  **Kotlin DSL** (в файлах `build.gradle.kts`) как _ предметно‑ориентированный язык_ для описания  конфигурации сборки. Это не просто скрипты  —  это структурированный способ задать:<br>- зависимости проекта;    <br>- задачи (tasks);<br>- плагины;<br>- настройки компиляции и упаковки.                                    |
| Ключевые механизмы | 1. **Файлы конфигурации** <br>    - `build.gradle` — для Groovy DSL.   <br>    - `build.gradle.kts` — для Kotlin DSL.       <br>    Gradle автоматически находит и исполняет их при запуске команд (`gradle build`, `gradle assemble` и др.).<br>    <br>2. **Декларативный синтаксис с элементами программирования**  <br>    Вы описываете сборку через вызовы методов, а не через XML‑теги. |

### В чем заключается принцип Convention over Configuration, и как он применяется в Maven?

| Под темы                | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Суть                    | **Convention over Configuration** («соглашения вместо конфигурации») — это архитектурный принцип, согласно которому система по умолчанию использует **разумные,  предсказуемые настройки**, избавляя разработчика от избыточной ручной конфигурации.<br>Ключевые идеи:<br>- Система предполагает «очевидные» значения для  типовых параметров.<br>- Разработчик конфигурирует только то, что отклоняется от  стандарта.<br>- Уменьшается объём рутинной настройки, снижается порог  входа.<br>- Повышается консистентность проектов (все следуют одним соглашениям).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| Как Maven реализует CoC | Maven применяет этот принцип на всех уровнях: от структуры  проекта до жизненного цикла сборки.<br>1. Стандартная структура директорий<br>Maven «знает», где искать исходники, ресурсы и тесты, **без явного указания в `pom.xml`**:<br>- `src/main/java` — Java‑исходники; <br>- `src/main/resources` — ресурсы (конфиги, шаблоны);<br>- `src/test/java` — тестовые классы; <br>- `src/test/resources` — ресурсы для тестов; <br>- `target/` — выходная директория (классы, JAR, отчёты).<br>2. Типовые артефакты по умолчанию<br>- Если не указано иное, проект собирается в **JAR**. <br>- Для веб‑приложений достаточно сменить `<packaging>war</packaging>` — Maven сам поймёт, где размещать `web.xml` (`src/main/webapp`).<br>3. Жизненный цикл сборки<br>Maven определяет **стандартный порядок фаз** (`validate` → `compile` → `test` → `package` → `install` → `deploy`), который работает «из коробки». Вам не нужно описывать, когда компилировать, когда запускать тесты и т. п.<br>4. Плагины с предустановленными настройками<br>Основные плагины (например, `maven-compiler-plugin`, `maven-surefire-plugin`) имеют **дефолтные параметры**:<br>- Версия Java: обычно `1.5` (можно переопределить).<br>- Расположение исходников: как указано выше.<br>- Запуск тестов: автоматически находит классы с тестами.<br>5. Управление зависимостями<br>- Групповой ID, артефакт и версия в `pom.xml` однозначно идентифицируют зависимость.<br>- Maven сам скачивает их из репозитория и размещает в `~/.m2/repository`.<br>- Транзитивные зависимости разрешаются автоматически. |

### Как в Gradle реализовать кастомную задачу с использованием Groovy или Kotlin?

| Под темы         | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| ---------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ            | В Gradle кастомную задачу можно создать **прямо в  `build.gradle`/`build.gradle.kts`** или  **вынести в отдельный класс** (в `buildSrc/`).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Ключевые моменты | 1. **Регистрация задачи:**<br>    - Groovy: `task имя { ... }` или `task имя(type: Тип)  { ... }`.  <br>    - Kotlin: `tasks.register<Тип>("имя") { ... }`.<br>        <br>2. **Действия задачи:**<br>    - `doFirst` — выполнить в начале.<br>    - `doLast` — выполнить в конце.  <br>    - `@TaskAction` (в классе) — основное действие.<br>        <br>3. **Зависимости:**<br>    - `dependsOn` — указать, что задача зависит от других.<br>        <br>4. **Ввод/вывод:**<br>    - Используйте аннотации `@Input`, `@OutputFile` и др. для инкрементальной сборки.<br>        <br>5. **Параметры:**<br>    - Через свойства проекта (`-P`).<br>    - Через поля класса задачи (в `buildSrc`).<br>        <br>6. **Переиспользование:**<br>    - Для сложных задач создавайте классы в `buildSrc/`.<br>    - Это обеспечивает типобезопасность и повторное использование. |
