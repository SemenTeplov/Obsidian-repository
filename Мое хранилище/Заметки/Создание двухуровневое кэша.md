### Что такое кэширование данных и для чего оно используется?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **Кэширование данных** (кэш) — это процесс временного сохранения копий данных, чтобы повторно не запрашивать или не вычислять их каждый раз заново. **Цель кэширования** — сократить время доступа к данным, снизить нагрузку на сервер и улучшить отклик системы.<br>Кэширование используется в разных сферах, например:<br>- **В веб-разработке** — для временного хранения данных, таких как изображения или CSS-стили, чтобы ускорить загрузку веб-страниц.<br>- **В игровой индустрии** — для хранения текстур, моделей персонажей и других ресурсов, чтобы уменьшить время загрузки игры и повысить плавность геймплея.<br>- **В финансовом секторе** — для быстрого доступа к финансовым данным, курсам валют и другой важной информации, которая должна быть доступна в реальном времени.<br>- **В медицинских исследованиях** — для хранения и быстрого доступа к результатам анализов, диагностическим данным и исследованиям. |
| Принцип работы | Процесс кэширования включает несколько этапов:<br>1. **Первый запрос** — когда система впервые запрашивает определённые данные, кэш не содержит их копии, система получает информацию из основного источника (медленного).<br>2. **Сохранение в кэше** — полученные данные сохраняются в кэше для будущего использования.<br>3. **Последующие запросы** — при повторном запросе тех же данных система сначала проверяет кэш. Если данные найдены в кэше, они возвращаются без обращения к медленному источнику — это «кэш-хит». Если данных в кэше нет, происходит «кэш-мисс», и система обращается к основному источнику.<br>4. **Обновление кэша** — периодически кэш проверяет актуальность хранимых данных и обновляет их при необходимости.                                                                                                                                                                                         |

### Какие основные типы кэшей вы знаете?

| Под темы              | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Локальный (in-memory) | Хранится в памяти приложения. Для реализации используются структуры данных, например HashMap, или специализированные библиотеки, например Guava Cache или Caffeine. Некоторые особенности:<br>- Простейший кэш можно реализовать с помощью HashMap, который хранит пары «ключ-значение» и обеспечивает быстрый доступ к данным.<br>- Для многопоточных приложений лучше использовать ConcurrentHashMap, который обеспечивает потокобезопасность без необходимости внешней синхронизации.<br>- Каждая реализация должна иметь стратегию удаления устаревших записей — будь то LRU (Least Recently Used), LFU (Least Frequently Used), FIFO или их комбинации.                                                                                                    |
| Распределенный        | Хранится вне приложения, доступен для нескольких экземпляров приложения. Для реализации используются библиотеки и платформы, например Ehcache, Apache Ignite, Oracle Coherence, Infinispan, VMware GemFire или Redis. Некоторые особенности:<br>- Данные распределяются между узлами автоматически, с поддержкой репликации для надёжности.<br>- Например, Hazelcast использует консистентное хеширование для распределения данных, что обеспечивает равномерное распределение нагрузки и минимальные перемещения данных при изменении состава кластера.<br>- Каждый раз, когда обновляется или удаляется информация, хранящаяся в локальном кэше на одной машине, нужно обновлять кэши в памяти на всех машинах, которые входят в состав распределённого кэша. |
| База данных           | Кэширование может применяться на уровне базы данных для хранения часто запрашиваемых результатов запросов. Например, в технологии Hibernate есть два уровня кэширования:<br>1. **Кэш первого уровня** — хранит объекты, которые в настоящее время используются в конкретной сессии Hibernate. Когда объект загружается или обновляется впервые в сессии, он хранится в этом кэше. Любой последующий запрос на получение того же объекта в той же сессии обслуживается из кэша, избегая запроса к базе данных.<br>2. **Кэш второго уровня** — общий кэш для нескольких сессий. Хранит данные, часто используемые в разных сессиях, снижая запросы к базе данных. В отличие от кэша первого уровня, кэш второго уровня необязателен и должен быть явно включён.   |
| HTTP                  | Для веб-приложений Java поддерживает механизмы кэширования HTTP с использованием заголовков, например Cache-Control и ETag. Эти заголовки позволяют браузерам и веб-серверам кэшировать ресурсы, например изображения, стилиheets и скрипты, снижая необходимость повторно загружать их при последующих запросах.<br>Также в Java есть механизм кэширования на уровне приложения, например, в Spring, который позволяет интегрировать кэширование в приложения, аннотируя методы аннотациями кэширования, такими как @Cacheable, @CachePut и @CacheEvict.                                                                                                                                                                                                       |

### Как работает двухуровневый кэш?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Принцип работы | **В Java двухуровневый кэш может быть реализован с помощью разных библиотек и фреймворков**. Например:<br>- **Guava Cache** — библиотека для кэширования данных в памяти.<br>- **Hibernate** — фреймворк, который использует кэш второго уровня (L2 cache) — глобальный кэш на уровне SessionFactory. В отличие от кэша первого уровня (L1 cache), L2 cache разделяется между несколькими сессиями: все сессии, созданные одной фабрикой, могут читать и записывать в единое кэш-хранилище.<br>- **Распределённые системы кэширования** (Redis, Memcached) — для реализации второго уровня, который доступен для всех экземпляров приложения.   |
| Реализация     | **В некоторых случаях двухуровневый кэш активируется автоматически** — например, в Hibernate. **В других случаях** его нужно включить явным образом и настроить подходящий провайдер. Например, в Hibernate L2-кэш не активируется автоматически, его нужно включить в настройках (например, в application.properties или persistence.xml).<br>**Важно**:<br>- Чтение из кэша второго уровня происходит только если нужный объект не был найден в кэше первого уровня.<br>- При обновлении данных приложение обновляет базу данных и инвалидирует соответствующие данные в обоих кэшах, чтобы данные оставались согласованными на всех уровнях. |

### Какие стратегии инвалидации кэша вы знаете?

| Под темы                            | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ----------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Time-based (по времени)             | **Кэш автоматически сбрасывается после определённого времени** (Time-To-Live, TTL). После истечения времени данные считаются устаревшими и удаляются из кэша. Последующие запросы за этими данными приводят к miss в кэше, что вызывает загрузку свежих данных из основного источника.<br>**Принцип**: данные считаются действительными в течение фиксированного периода, независимо от фактических изменений.<br>**Преимущества**: простота реализации, подходит для данных, которые не меняются часто.<br>**Недостатки**: если время истечения установлено слишком долго или слишком коротко, это может привести к использованию устаревших данных или ненужным обновлениям.           |
| Key-based (по ключу)                | **С каждым элементом кэша ассоциируется уникальный ключ**. При изменении исходных данных (например, удалении или обновлении) ключ инвалидируется, и кэшированные данные удаляются или обновляются.<br>**Преимущества**: гарантирует, что всегда используются последние данные, эффективна для данных, которые меняются часто.<br>**Недостатки**: реализация сложнее, чем стратегия по времени, может потребоваться дополнительное пространство для ключей.                                                                                                                                                                                                                               |
| Write-through (с записью)           | **Данные записываются в кэш и основное хранилище данных одновременно**. Это обеспечивает согласованность кэша с базой данных.<br>**Пример**: в каталоге товаров в интернет-магазине при обновлении цены продукта новая цена записывается в кэш, а также в базу данных. Это синхронизирует оба слоя: при запросе цены пользователи получают последнее значение непосредственно из кэша.<br>**Недостатки**: операция записи относительно медленная, потому что вызов записи должен обновить как кеш, так и основное хранилище данных. В производственных системах для обеспечения согласованности используют поддержку транзакций или механизмы повторной записи.                          |
| Write-behind (с отложенной записью) | **Данные сначала записываются в кеш, а затем асинхронно — в основное хранилище**. Это улучшает производительность записи, но вводит сложность.<br>**Пример**: в системе профиля пользователя при обновлении имени пользователя новое имя записывается в кеш, а фоновый процесс записывает изменение в базу данных через короткую задержку. Это обеспечивает быстрый ответ пользователю, но в базе данных на короткий период будут устаревшие данные.<br>**Недостатки**: если сервер с кешем выйдет из строя до синхронизации с БД, данные могут быть утеряны. Поэтому Write-Behind обычно используют с персистентным кешем и механизмами повторной записи для обеспечения долговечности. |

### Какие преимущества предоставляет двухуровневый кэш по сравнению с одноуровневым?

| Под темы          | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Одноуровневый кэш | **Обеспечивает быстрый доступ к часто используемым данным**. Например, в Hibernate кэш первого уровня (Level 1 cache, L1) реализован на уровне сессии. Некоторые особенности:<br>- Привязан к объекту сессии, не может быть общим для нескольких потоков.<br>- Гарантирует повторяемость чтения сущностей на уровне приложения: независимо от того, сколько раз сущность загружается, вызывающей стороне возвращается одна и та же управляемая ссылка на сущность.<br>- Не должен использоваться совместно несколькими потоками, чтобы не ставить под угрозу производительность.                                                                                                                                                                                                                                                                                                                                                                            |
| Двухуровневый кэш | **Сочетает два уровня кэша** — локальный (в памяти) и удалённый (распределённый или общий):<br>- **Локальный кэш (первый уровень)** — обеспечивает моментальный ответ для часто запрашиваемых данных. Располагается на том же сервере, где работает приложение (чаще всего — в оперативной памяти).<br>- **Удалённый кэш (второй уровень)** — используется для данных, которых нет в локальном кэше, но которые нужно отдавать быстрее, чем из базы данных. Позволяет хранить больше данных и делиться ими между серверами.<br>**Принцип работы**: при загрузке объекта сначала проверяется локальный кэш, а затем — при отсутствии объекта там — обращается к удалённому кэшу. Если объект найден во втором уровне, его состояние берётся из памяти, минуя базу данных. Если же и во втором уровне кэша записи нет (кэш-промах), тогда выполняется SQL-запрос к БД, а полученные данные затем сохраняются в оба кэша — локальный и удалённый — на будущее. |
| Сравнение         | **Преимущества двухуровневого кэширования**:<br>- **Быстрый доступ** — локальный кэш обеспечивает моментальный ответ для часто запрашиваемых данных.<br>- **Масштабируемость** — удалённый кэш позволяет хранить больше данных и делиться ими между серверами.<br>- **Снижение нагрузки на базу данных** — большинство запросов обслуживается из кэша, уменьшая число обращений к БД.<br>- **Отказоустойчивость** — если локальный кэш выйдет из строя, данные будут получены из удалённого кэша.<br>**Недостатки двухуровневого кэширования**:<br>- **Сложность управления** — требуется синхронизация между двумя уровнями кэша.<br>- **Проблема устаревших данных** — если данные обновились в одном кэше, но не в другом, клиент может получить старую версию.<br>- **Задержки при работе с удалённым кэшем** — запросы ко второму уровню выполняются медленнее из-за сетевых задержек.                                                                 |

### Как вы можете реализовать двухуровневый кэш в Java?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| В Spring | **Первый уровень кэша** — локальный кэш в оперативной памяти, специфичный для одного экземпляра приложения. Для его реализации можно использовать библиотеки кэширования, например Caffeine, Guava, Ehcache. Некоторые особенности первого уровня:<br>- локален для одного экземпляра приложения;<br>- быстрый доступ из-за хранения в памяти;<br>- неперсистентный, обычно ограничен памятью приложения;<br>- не разделяет данные между несколькими экземплярами.<br>**Второй уровень кэша** — распределённый кэш, который позволяет нескольким экземплярам приложения делиться кэшированными данными. Для его реализации можно использовать внешние системы кэширования, например Redis, Memcached, Hazelcast. Некоторые особенности второго уровня:<br>- доступен для всех экземпляров приложения;<br>- может быть постоянным, обеспечивая доступность данных даже после перезапуска;<br>- поддерживает большие объёмы данных по сравнению с локальными кэшами;<br>- требует сетевого доступа, что может вводить задержки по сравнению с локальными кэшами.<br>**Важно**: Spring не управляет данными между несколькими кэшами, даже если они объявлены для одного метода. Поэтому нужно обновлять кэш первого уровня при каждом обращении к нему. |

### Какие механизмы синхронизации данных между уровнями кэша существуют?

| Под темы                     | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ---------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| В памяти (in-memory)         | Некоторые механизмы синхронизации данных между уровнями кэша в памяти в Java:<br>- **Протокол когерентности кэшей**. Позволяет синхронизировать изменения данных в кэшах между собой и с основной памятью. Например, в многопроцессорных системах есть общая шина, к которой подключены кэши всех ядер, и любые запросы отправляются на шину, и видны всем подключённым к шине сразу.<br>- **Механизм синхронизации локальной памяти нитей** — «happens before» (дословно «случилось перед»). В коде есть правила, при наступлении которых память синхронизируется — обновляется до актуального состояния. Например, если мютекс был освобождён и снова захвачен, то перед захватом обязательно выполнится синхронизация памяти.<br>- **Использование модификатора volatile**. Данные, которые могут храниться только в памяти, обращаются не в кэш процессора за значением, а напрямую в память. Это позволяет избежать гонок данных.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| Распределённые (distributed) | Некоторые механизмы синхронизации данных между уровнями кэша в распределённых системах в Java:<br>- **Стратегия «запись сквозь» (write-through)**. При изменении данных они одновременно обновляются и в кеше, и в основном хранилище. Это обеспечивает высокую согласованность, но замедляет операции записи.<br>- **Стратегия «запись позади» (write-behind)**. Данные сначала пишутся в кеш, а затем асинхронно обновляются в основном хранилище. Это ускоряет отклик, но создаёт риск потери данных при сбое.<br>- **Стратегия «запись в обход» (write-around)**. Данные пишутся непосредственно в хранилище, минуя кеш. Кеш заполняется только при чтении. Это хорошо для данных, которые редко повторно запрашиваются.<br>- **Стратегия инвалидации**. При изменении данных соответствующие записи в кеше помечаются как недействительные. Это простой подход, но может привести к периодам с устаревшими данными.<br>- **Репликация данных**. Например, в системе распределённого кэширования Hazelcast данные распределяются между узлами автоматически, с поддержкой репликации для надёжности. По умолчанию Hazelcast создаёт одну копию (бэкап) каждого элемента данных на другом узле кластера, что обеспечивает отказоустойчивость: если узел, содержащий оригинальные данные, выходит из строя, копия становится новым источником правды.<br>- **Синхронизация кэшей на отдельных узлах**. В некоторых случаях для поддержания консистентности данных достаточно при любом изменении значения в кэше удалять эту запись на всех узлах. Для этого можно использовать EventListener, который отправляет сообщения в топик при удалении записи, и каждый узел слушает события из очереди, связанной с общим топиком. |

### Какие проблемы могут возникнуть при использовании двухуровневого кэша и как их можно решить?

| Проблемы | Некоторые проблемы, которые могут возникнуть при использовании двухуровневого кэша:<br>- **Сложность управления** — требуется синхронизация между двумя уровнями кэша.<br>- **Проблема устаревших данных** — если данные обновились в одном кэше, но не в другом, клиент может получить старую версию.<br>- **Задержки при работе с удалённым кэшем** — запросы ко второму уровню выполняются медленнее из-за сетевых задержек.<br>- **Сложность поддержки транзакционности** — в распределённом кэше тяжелее обеспечить ACID, особенно если не использовать транзакционную стратегию. Можно столкнуться с гонками: узел А обновил значение и разослал инвалидацию, но узел B в это же время прочитал старое значение до применения инвалидации.                                                                                                                                                                                                                                                                                                                                          |
| Решения  | Для решения проблем, возникающих при использовании двухуровневого кэша, можно использовать следующие методы:<br>- **Использовать стратегии кэширования** — например, стратегию READ_WRITE с «мягкими блокировками», которая гарантирует, что параллельные транзакции не читают устаревшие данные.<br>- **Реализовывать внешнее уведомление об изменениях** — если обновление произошло не через контекст текущего кэша, механизмы второго уровня не узнают об изменении.<br>- **Установить относительно небольшой TTL (время жизни) для записей** — чтобы даже без уведомлений кэш сам периодически сбрасывал старые данные и подтягивал свежие из БД. Это не даёт строгой консистентности, но ограничивает «время протухания» данных.<br>- **Избегать избыточного кэширования** — не нужно кэшировать всё, не стоит вкладывать кэши друг в друга (кешировать результат, который уже был взят из кэша).<br>- **Тестировать под нагрузкой** — кэширование может изменить характер нагрузки (например, снизить число запросов к БД, но увеличить потребление памяти и CPU на сериализацию). |

### Как реализовать автоматическое управление размером кэша на обоих уровнях?

| Под темы                   | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| -------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Первый уровень кэширования | **Кэш первого уровня** (кэш сессии) **активируется автоматически** и не подлежит отключению. Он задействуется для временного хранения данных, активно используемых в течение текущей сессии. Некоторые особенности реализации: <br>- **Автоматическое включение объекта** в кэш при создании через методы save(), persist() или saveOrUpdate().<br>- **Отслеживание модификаций** сущности в течение сессии: обновление (методы update() или merge()) — Hibernate проверяет наличие сущности в кэше, если она присутствует, её состояние обновляется в кэше, в противном случае сущность загружается из базы данных, обновляется и кэшируется.<br>- **Удаление объекта** — через метод delete() объект удаляется из кэша, что обеспечивает консистентность данных в рамках сессии.<br>**Рекомендации**:<br>- Рекомендуется регулярно закрывать и возобновлять сессии, чтобы предотвратить чрезмерное накопление данных.<br>- В случаях, когда производительность или потребление памяти становятся проблемой, кэш можно очистить с помощью метода session.clear() (хотя это приведёт к потере всех несохранённых изменений).<br>- Метод session.evict(Object entity) позволяет удалить из кэша конкретный объект сущности, который больше не планируется использовать в текущей сессии. |
| Второй уровень кэширования | **Кэш второго уровня** **не является обязательным** и может быть настроен с учётом специфических потребностей приложения. Он предназначен для сохранения объектов сущностей между различными сессиями. Некоторые особенности реализации:<br>- **Конфигурационные настройки** — выбор и настройка поставщика кэширования, определение политик сериализации и управления данными, регулировка параметров производительности и масштабируемости.<br>- **Синхронизация состояния объекта** — когда объект сущности создаётся или обновляется, его состояние автоматически синхронизируется с кэшем второго уровня, обеспечивая актуальность данных.<br>- **Ограничение количества хранимых объектов** — например, можно создать в конфигурации кэша два региона: один с краткосрочным хранением объектов, другой с долгосрочным, и отправлять часто изменяющиеся объекты в первый регион, а все остальные — во второй.                                                                                                                                                                                                                                                                                                                                                                      |
