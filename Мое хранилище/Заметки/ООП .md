### Какие четыре основных принципа объектно-ориентированного программирования?

| Под темы     | Заметка                                                                                                                                                                                                                                                                             |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение  | **Абстракция, инкапсуляция, наследование и полиморфизм** — четыре основных принципа объектно-ориентированного программирования (ООП). Вместе они определяют, как создаются, организуются и взаимодействуют объекты в программе.                                                     |
| Абстракция   | **Абстракция** позволяет выделить в объекте только важные характеристики и скрыть детали, которые не имеют значения для работы остальной программы. Через абстракцию программист показывает только те свойства и методы, которые действительно нужны для взаимодействия с объектом. |
| Инкапсуляция | **Инкапсуляция** означает, что все данные и функции, необходимые для работы объекта, сосредоточены внутри него самого. Объект сам управляет своим состоянием и своими действиями.                                                                                                   |
| Наследование | **Наследование** позволяет создавать новые классы на основе существующих. Новый класс автоматически получает все свойства и методы своего родителя, но при необходимости может добавлять новые характеристики или изменять поведение унаследованных методов.                        |
| Полиморфизм  | **Полиморфизм** позволяет объектам разных классов реагировать на одни и те же команды по-своему. Это значит, что методы с одинаковыми именами могут работать по-разному в зависимости от объекта, к которому они применяются.                                                       |

### Что такое инкапсуляция в Java?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Инкапсуляция в Java** — это процесс объединения состояния объекта (полей или атрибутов) и методов (функций), которые работают с этим состоянием, в единое целое, часто называемое классом. Это одна из четырёх фундаментальных концепций объектно-ориентированного программирования (ООП), наряду с наследованием, полиморфизмом и абстракцией.                                                                                                                                                                                                                                                                                  |
| Принцип     | **Ключевой принцип инкапсуляции** — **внутренние детали объекта должны быть скрыты от внешнего мира**. Это предотвращает несанкционированный доступ к данным объекта и манипулирование ими.<br>Некоторые преимущества инкапсуляции:<br>- **Контроль целостности данных** — предотвращает установку недопустимых значений через проверки в сеттерах.<br>- **Гибкость реализации** — позволяет изменять внутреннюю реализацию без влияния на внешний код.<br>- **Упрощение интерфейса** — предоставляет только необходимые для взаимодействия методы.<br>- **Повышение безопасности** — ограничивает доступ к чувствительным данным. |

### Для чего используется ключевое слово 'extends' в Java?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **Ключевое слово extends в Java используется для обозначения наследования** — механизма, который позволяет одному классу перенимать свойства и методы другого.<br>Класс, который наследует свойства, называется **подклассом** (дочерним классом, производным классом), а класс, от которого наследуются свойства, — **суперклассом** (родительским классом, базовым классом).<br>Наследование реализует отношение типа «is-a» («является»: класс-наследник — специализированная версия родительского класса). |
| Принцип работы | При использовании extends подкласс получает доступ ко всем открытым свойствам и методам суперкласса. **Некоторые особенности наследования**:<br>- Подкласс не наследует закрытые поля и методы своего суперкласса, поскольку они недоступны за пределами класса, в котором они определены.<br>- Подкласс может расширять суперкласс, добавляя свои собственные поля и методы.<br>- Конструкторы не наследуются, но конструктор потомка обязан вызвать конструктор родителя (явно через super() или неявно).    |

### Как различаются абстрактные классы и интерфейсы в Java?

| Под темы          | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| ----------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Абстрактный класс | **Абстрактный класс** — это класс, который **не может быть инстанцирован** (нельзя создать объект этого класса напрямую). Он служит основой для других классов.<br>**Особенности**:<br>- Может содержать как абстрактные методы (без реализации), так и методы с реализацией.<br>- Может определять поля (переменные экземпляра) и методы для управления этими полями.<br>- Может содержать конструкторы, которые будут вызываться при создании объектов подклассов.<br>- Методы и переменные могут иметь любые модификаторы доступа (private, protected, public).<br>**Использование**: абстрактные классы применяются для создания обобщённых конструкций, которые позволяют определять общие характеристики для множества подклассов. Например, можно создать абстрактный класс «Человек» с полями имя и возраст (так как все люди имеют это свойство) и методами, возвращающими эти значения.                                                                                                                                                                                                                                                                                                                             |
| Интерфейс         | **Интерфейс** — это специальный тип, который **определяет набор абстрактных методов** (методов без реализации), которые должны быть реализованы классами, имплементирующими этот интерфейс.<br>**Особенности**:<br>- По умолчанию все методы публичные и абстрактные.<br>- В интерфейсах могут быть методы с реализацией по умолчанию (default), статические методы и методы с ключевым словом private для внутренних нужд.<br>- Переменные в интерфейсе по умолчанию являются public, static и final.<br>**Использование**: интерфейсы предпочтительны, когда необходимо задать общий контракт для множества разных классов.<br>**Некоторые различия между интерфейсом и абстрактным классом**:<br>- **Множественное наследование**: класс может реализовывать несколько интерфейсов, но наследоваться может только от одного абстрактного класса.<br>- **Методы с реализацией**: интерфейс до Java 8 не может содержать методы с реализацией, тогда как абстрактный класс может. С Java 8 интерфейсы могут иметь методы с реализацией (default и static методы).<br>- **Переменные**: интерфейс может содержать только статические и финальные переменные, а абстрактный класс может содержать любые переменные экземпляра. |

### Что такое полиморфизм в контексте Java?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Полиморфизм в контексте Java** — это принцип объектно-ориентированного программирования (ООП), который позволяет работать с объектами разных типов через общий контракт, получая корректное поведение без переписывания клиентского кода. То есть: один интерфейс — много реализаций, одна точка вызова — разная логика в зависимости от конкретного объекта.<br>Полиморфизм помогает писать код, который не «привязан» к деталям реализации: клиентский слой знает контракт, а не внутреннее устройство.                                                                                                                                                                                                                                                               |
| Виды        | В Java полиморфизм реализуется двумя основными способами:<br>1. **Статический (полиморфизм времени компиляции)** — определяется на этапе компиляции через перегрузку методов. Позволяет определить несколько методов с одинаковым именем, но разными параметрами в одном классе. Компилятор Java сам определяет, какой именно метод вызывать, основываясь на типах аргументов и их количестве.<br>2. **Динамический (полиморфизм времени выполнения)** — определяется во время выполнения программы через наследование и переопределение методов. Метод родительского класса может быть переопределён в дочернем классе, и при вызове метода через ссылку на родительский класс будет выполняться версия метода из того класса, к которому объект фактически принадлежит. |

### Как работает механизм перегрузки и переопределения методов в Java?

| Под темы        | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Перегрузка      | **Перегрузка методов (method overloading)** — это возможность определить несколько методов с одинаковым именем, но разными параметрами в одном классе.<br>**Принцип работы**: компилятор Java определяет, какой именно метод вызывать, основываясь на количестве, типе и порядке аргументов, переданных при вызове. Это происходит в два этапа:<br>1. **Точное соответствие** — сначала ищется метод с точным соответствием типов аргументов.<br>2. **Автоматическое расширение типов** — если точного соответствия нет, компилятор пробует применить автоматическое преобразование типов.<br>**Правила**:<br>- Перегруженные методы должны отличаться количеством и/или типами параметров.<br>- Возвращаемое значение не учитывается — нельзя перегружать методы, отличающиеся только типом возвращаемого значения.<br>- Модификаторы доступа не влияют — перегрузка не зависит от public, private или protected модификаторов.<br>- Исключения не учитываются — список throws-исключений не влияет на перегрузку.                                                                                 |
| Переопределение | **Переопределение методов (method overriding)** — это возможность дочернему классу предоставить собственную реализацию метода, унаследованного от родительского класса.<br>**Принцип работы**: когда метод переопределён, JVM во время выполнения определяет, какая именно реализация должна быть вызвана, основываясь на фактическом типе объекта.<br>**Правила**:<br>- Сигнатура метода (имя и список параметров) в подклассе должна точно совпадать с сигнатурой в суперклассе.<br>- Тип возвращаемого значения должен быть тем же или подтипом возвращаемого типа родительского метода (ковариантный возвращаемый тип).<br>- Уровень доступа не может быть более ограничительным, чем у переопределяемого метода.<br>- Подкласс не может генерировать новые или более широкие проверяемые исключения.<br>**Особенности**:<br>- Static методы не могут быть переопределены — для них используется скрытие методов (method hiding).<br>- Private методы не видны подклассам и поэтому не могут быть переопределены.<br>- Конструкторы не могут быть переопределены, поскольку они не наследуются. |

### В чем разница между статическим и динамическим связыванием в Java?

| Под темы                               | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Статическое связывание (Early Binding) | **Происходит во время компиляции**. Компилятор определяет, какой метод будет вызван, на основе типа переменной или ссылки на объект, которая содержит метод. Если тип переменной или ссылки заранее известен, то компилятор может точно определить, какой метод будет вызван, и связать его с этой переменной или ссылкой. <br>**Некоторые особенности статического связывания:**<br>- Используется для разрешения перегруженных методов.<br>- Приватные, статические и терминальные методы разрешаются при помощи статического связывания, поскольку их нельзя переопределять.<br>- Для обнаружения нужного метода используется тип ссылочной переменной. |
| Динамическое связывание (Late Binding) | **Происходит во время выполнения**. Компилятор не может точно определить, какой метод будет вызван, и связь происходит только во время выполнения программы.<br>**Некоторые особенности динамического связывания:**<br>- Определяет, какой метод будет вызван, на основе фактического типа объекта, на который ссылается переменная или ссылка.<br>- Если тип объекта не известен заранее, то компилятор не может точно определить, какой метод будет вызван, и связь происходит только во время выполнения программы.                                                                                                                                     |

### Как принципы SOLID влияют на архитектуру и дизайн программного обеспечения в ООП?

| Под темы                                   | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Принцип единственной ответственности (SRP) | **Означает, что каждый класс должен отвечать за одну задачу**. Меняться класс должен только по одной причине: изменилась логика в рамках его ответственности. <br>**Влияние на архитектуру**:<br>- **Формирование модульного дизайна**. Разделяя обязанности на более мелкие, сфокусированные компоненты, команды могут быстрее вносить изменения, снижать количество ошибок. Например, в системе управления пользователями класс, который одновременно отвечает за аутентификацию и сохраняет данные в базу, становится узким местом — если потребуется обновление безопасности, это может привести к непредвиденным последствиям в несвязанных частях системы. Поместив аутентификацию в отдельный класс Authenticator, а сохранение данных — в UserRepository, создаются чёткие границы, которые упрощают тестирование, отладку и масштабирование.<br>- **Избегание ошибки «Божественный класс»** — когда один класс обременён слишком большим количеством обязанностей. Соблюдение SRP позволяет избегать этого антипаттерна.                                                                                                          |
| Открытости — закрытости (OCP)              | **Гласит: программные компоненты должны быть открыты для расширения, но закрыты для изменения**. Добавление новой функциональности не должно требовать изменения существующих классов или функций. Вместо этого новые возможности добавляются через расширение (наследование, внедрение интерфейсов, композицию и другие механизмы). <br>**Влияние на архитектуру**:<br>- **Снижение рисков и стоимости изменений**. Когда новые требования или бизнес-изменения приводят к минимальным модификациям существующего кода, это снижает вероятность ошибок, упрощает тестирование, поддержку и развитие системы в целом.<br>- **Облегчение внедрения новых функций**. Например, в системе обработки платежей: вместо изменения существующего кода при добавлении нового способа оплаты (например, криптовалюты) можно создать новый класс, реализующий уже существующий интерфейс платежа.<br>- **Использование интерфейсов** для реализации разных способов оплаты — это позволяет процессору оплаты работать с интерфейсом, не завися от конкретных реализаций.                                                                             |
| Подстановки Барбары Лисков (LSP)           | **Утверждает, что поведение подклассов должно быть совместимо с поведением их суперклассов**. Другими словами, объекты подтипов должны быть заменяемыми экземплярами своих супертипов без изменения ожидаемого поведения программы.<br>**Влияние на архитектуру**:<br>- **Обеспечение корректной работы полиморфизма** — подклассы должны полностью сохранять поведение суперклассов, а не просто расширять или изменять их. Нарушение этого принципа может привести к неожиданным ошибкам при использовании наследования.<br>- **Устранение проблемы «квадрат/прямоугольник»** — например, если класс Square наследуется от Rectangle, но переопределяет методы setHeight() и setWidth() так, чтобы они всегда делали высоту равной ширине, это нарушает LSP — ожидается, что при вызове setHeight() и setWidth() объекта Rectangle сначала будет изменяться одно измерение, а потом другое, но в случае Square эти методы нарушают это ожидание. Чтобы исправить это, можно пересмотреть архитектуру классов так, чтобы Square не наследовался от Rectangle, или выделить общий интерфейс для обоих классов и разработать их независимо. |
| Инверсии зависимостей (DIP)                | **Помогает создавать гибкие, слабосвязанные системы** путём изменения направления зависимости — от конкретных реализаций к абстрактным контрактам.<br>**Суть принципа**: модули верхнего уровня (бизнес-логика, основные компоненты системы) не должны зависеть от модулей нижнего уровня (конкретных реализаций, деталей). Вместо этого они должны зависеть от абстракций (интерфейсов, абстрактных классов).<br>**Влияние на архитектуру**:<br>- **Уменьшение жёстких связей между компонентами** — это делает код более гибким и удобным для изменений. Например, если класс DataService жёстко привязан к MySQLDatabase, с DIP он работает не с конкретной MySQLDatabase, а с абстракцией Database — можно легко заменить базу данных, не меняя код DataService.<br>- **Упрощение тестирования** — DIP позволяет заменять реальные зависимости моками или заглушками, которые следуют тому же интерфейсу.<br>- **Разделение ответственности** — каждый модуль выполняет одну задачу и взаимодействует через контракты.                                                                                                                 |

### Что такое принцип инверсии зависимостей и как он реализуется в Java?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                              |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Принцип инверсии зависимостей (Dependency Inversion Principle, DIP)** — один из пяти принципов SOLID в объектно-ориентированном программировании. Он помогает создавать гибкие, слабосвязанные системы путём изменения направления зависимости — от конкретных реализаций к абстрактным контрактам.                                                                                                                |
| Суть        | **Модули верхнего уровня** (бизнес-логика, основные компоненты системы) **не должны зависеть от модулей нижнего уровня** (конкретных реализаций, деталей). Вместо этого они должны зависеть от **абстракций** (интерфейсов, абстрактных классов).<br>**Абстракции не должны зависеть от деталей**, а наоборот — **детали** (конкретные реализации) должны зависеть от абстракций.                                    |
| Правила     | Некоторые правила реализации DIP:<br>- **Взаимодействие между классами** должно быть реализовано через интерфейсы или абстрактные классы.<br>- **Типами всех членов классов** должны быть интерфейсы или абстрактные классы.<br>- **Классы, являющиеся конечными реализациями**, не должны расширяться (или должны быть финальными).<br>- **Методы** не должны перекрываться при наследовании (или быть финальными). |
