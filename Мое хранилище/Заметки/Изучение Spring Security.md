### Что такое Spring Security и для чего он используется?

| Под темы          | Заметка                                                                                                                                                                                                                                      |
| ----------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение       | Spring Security — это **фреймворк на Java/Jakarta EE** для реализации механизмов безопасности в  приложениях, созданных на базе Spring Framework.                                                                                            |
| Основные  функции | **Аутентификация** (_Authentication_) — проверка личности пользователя:<br>    - формы входа с логином и паролем;<br>    - Basic Authentication;<br>    - OAuth2 и OpenID Connect (социальные входы);<br>    - JWT (JSON Web Token) для API. |
|                   | **Авторизация** (_Authorization_) — определение прав доступа пользователя:<br>    - управление ролями (RBAC — Role‑Based Access Control);<br>    - политика доступа к веб‑страницам, методам и данным.                                       |
|                   | **Защита ресурсов** — контроль доступа к различным элементам приложения:<br>    - разграничение публичных и защищённых URL;<br>    - ограничение доступа по ролям;<br>    - поддержка SSL/TLS для защиты передачи данных.                    |

### Как включить Spring Security в вашем Spring Boot приложении?

| Под темы   | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ      | Чтобы включить Spring Security в Spring Boot приложении, необходимо выполнить несколько шагов: добавить зависимость, настроить конфигурацию и при необходимости настроить пользователей и роли.                                                                                                                                                                                                                                                                                    |
| Выполнение | Добавление зависимости<br>Добавьте в проект зависимость `spring-boot-starter-security`. Это можно сделать в файле `pom.xml` (для Maven) или `build.gradle` (для Gradle).                                                                                                                                                                                                                                                                                                           |
|            | Базовая настройка<br>По умолчанию Spring Security автоматически настраивается и защищает все endpoints приложения. При запуске приложения в логах будет выведен случайно сгенерированный пароль для пользователя `user`. Однако для гибкой настройки потребуется создать класс конфигурации.<br>Создайте класс конфигурации с аннотацией `@Configuration` и `@EnableWebSecurity`. В этом классе можно настроить правила доступа к ресурсам, форму входа, выход и другие параметры. |
|            | Настройка пользователей и ролей (опционально)<br>Если требуется настроить конкретных пользователей и роли, можно использовать `InMemoryUserDetailsManager` или интегрироваться с базой данных.                                                                                                                                                                                                                                                                                     |

### Какие основные компоненты архитектуры Spring Security вы знаете?

| Под темы                          | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                            |
| --------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Основные  компоненты  архитектуры | **Security Filter Chain (Цепочка фильтров безопасности)**<br>    - точка входа для всех HTTP‑запросов;<br>    - набор фильтров, обрабатывающих запросы  последовательно;<br>    - отвечает за аутентификацию, авторизацию,  CSRF‑защиту,  управление сессиями; <br>    - позволяет создавать кастомные фильтры с  собственной  логикой.                                                                                            |
|                                   | **AuthenticationManager (Менеджер аутентификации)**<br>    - центральный компонент, управляющий  процессом аутентификации;<br>    - получает объект аутентификации от фильтра;<br>    - делегирует запрос подходящим AuthenticationProvider;<br>    - возвращает аутентифицированный объект  при успешной проверке.                                                                                                                |
|                                   | **AuthenticationProvider (Поставщик аутентификации)**<br>    - непосредственно проверяет учётные  данные пользователя;<br>    - поддерживает разные механизмы  (БД, LDAP, JWT, OAuth2);<br>    - примеры реализаций:<br>        - `DaoAuthenticationProvider` (база данных + UserDetailsService);<br>        - `JWTAuthenticationProvider` (проверка JWT‑токенов);<br>        - кастомные провайдеры для нестандартных  сценариев. |
|                                   | **UserDetailsService**<br>    - загружает данные пользователя (обычно из БД);<br>    - реализует метод  `loadUserByUsername(String username)`;<br>    - возвращает объект `UserDetails`, содержащий:<br>        - логин;<br>        - зашифрованный пароль;<br>        - роли/разрешения (`GrantedAuthority`);<br>        - статусы аккаунта (активен, не заблокирован и т. п.).                                                   |
|                                   | **PasswordEncoder (Кодировщик паролей)**<br>    - обеспечивает безопасное хранение и проверку  паролей;<br>    - преобразует пароли в криптографические хэши;<br>    - встроенные реализации:<br>        - `BCryptPasswordEncoder`;<br>        - `SCryptPasswordEncoder`;<br>        - `Pbkdf2PasswordEncoder`.                                                                                                                    |
|                                   | **SecurityContextHolder**<br>    - хранит `SecurityContext` для текущего запроса/потока;<br>    - содержит объект `Authentication`, включающий:<br>        - `Principal` (идентифицированный пользователь);<br>        - `Credentials` (пароль/токен);<br>        - `Authorities` (роли и разрешения).                                                                                                                             |
|                                   | **AccessDecisionManager (Менеджер принятия  решений об доступе)**<br>    - определяет, разрешён ли доступ к ресурсу на основе:<br>        - ролей пользователя;<br>        - настроенных правил безопасности;<br>        - атрибутов запроса.                                                                                                                                                                                      |
|                                   | **SecurityContext**<br>    - объект, хранящий информацию об аутентификации  текущего пользователя;<br>    - доступен через  `SecurityContextHolder.getContext()`;<br>    - сохраняется в сессии или передаётся через  токены (JWT).                                                                                                                                                                                                |
|                                   | **HttpSecurity**<br>    - DSL‑интерфейс для конфигурирования  правил безопасности;<br>    - позволяет:<br>        - определять защищённые URL;<br>        - настраивать формы входа;<br>        - включать/отключать защиту от CSRF, CORS и др.                                                                                                                                                                                    |
|                                   | **Authentication (Объект аутентификации)**<br>    - представляет учётные данные пользователя;<br>    - реализуется через классы вроде  `UsernamePasswordAuthenticationToken`;<br>    - содержит:<br>        - `Principal` (логин/идентификатор);<br>        - `Credentials` (пароль/токен);<br>        - `Authorities` (роли).                                                                                                     |

| Основные методы  ограничения доступа | В DSL‑интерфейсе `HttpSecurity`  используются методы:<br>- `.permitAll()` — доступ без аутентификации;<br>- `.authenticated()` — требуется аутентификация;<br>- `.hasRole(String role)` — доступ только для  указанной роли (с префиксом `ROLE_`);<br>- `.hasAuthority(String authority)` — доступ по полномочию (без префикса);<br>- `.hasAnyRole(String... roles)` — доступ для любой из перечисленных ролей;<br>- `.denyAll()` — полный запрет доступа; <br>- `.access(String expression)` — сложные выражения SpEL. |

### Как настроить многомерную аутентификацию (Multi-Factor Authentication, MFA) с использованием Spring Security?

| Под темы                                                | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                                             | В Spring Security 7 реализована встроенная поддержка многофакторной аутентификации (MFA) через аннотацию `@EnableMultiFactorAuthentication`. Этот подход позволяет требовать несколько факторов аутентификации (например, пароль и одноразовый токен) для доступа к защищённым ресурсам. Система автоматически управляет процессом завершения недостающих факторов и перенаправляет пользователя при необходимости.                                                                                                 |
| Основные принципы работы                                | Spring Security использует концепцию `FactorGrantedAuthority` для отслеживания выполненных факторов аутентификации. Когда пользователь вводит пароль, к его аутентификации добавляется полномочие `FACTOR_PASSWORD`. После ввода одноразового токена (OTT) добавляется `FACTOR_OTT`. Правила авторизации могут требовать наличия нескольких таких полномочий для доступа к ресурсам. <br>Если пользователь завершил только один фактор, Spring Security автоматически перенаправит его для завершения недостающего. |
| Настройка с помощью `@EnableMu ltiFactorAuthentication` | **Добавьте зависимость Spring Security** в проект, если она ещё не добавлена.                                                                                                                                                                                                                                                                                                                                                                                                                                       |
|                                                         | **Настройте конфигурацию безопасности**, используя аннотацию `@EnableMultiFactorAuthentication`. Укажите необходимые факторы в параметре `authorities`. Эта аннотация указывает, что для доступа к защищённым ресурсам требуются оба фактора: пароль и одноразовый токен.                                                                                                                                                                                                                                           |
|                                                         | **Настройте цепочку безопасности (SecurityFilterChain)**. Добавьте механизмы аутентификации для каждого фактора. Здесь `.formLogin(withDefaults())` настраивает стандартную форму ввода пароля, а `.oneTimeTokenLogin(withDefaults())` — механизм ввода одноразового токена.                                                                                                                                                                                                                                        |
|                                                         | **Настройте сервис одноразовых токенов** (`OneTimeTokenService`), если требуется кастомизация генерации и отправки токенов. Например, можно настроить отправку токена по email или SMS.                                                                                                                                                                                                                                                                                                                             |

### Как использовать Spring Security для защиты REST API с помощью OAuth2?

| Под темы                                     | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение                                  | Для защиты REST API с помощью OAuth 2.0 в Spring Security можно использовать **OAuth 2.0 Resource Server**. Этот подход позволяет проверять токены доступа, полученные от сервера авторизации, и обеспечивать доступ к ресурсам только для аутентифицированных пользователей.                                                                                                                                                                                                                                                                                |
| Основные компоненты OAuth 2.0                | В OAuth 2.0 выделяют четыре роли:<br>1. **Владелец ресурса** — конечный пользователь, который предоставляет доступ к своим данным.<br>2. **Сервер авторизации** — система, которая проверяет подлинность пользователя и выдаёт токен доступа после успешной аутентификации.<br>3. **Клиент** — приложение (мобильное, веб- или настольное), которое запрашивает доступ к API.<br>4. **Сервер ресурсов** — сервер, на котором размещены защищённые ресурсы. Он проверяет токен доступа и предоставляет доступ к ресурсам, если аутентификация прошла успешно. |
| Настройка сервера ресурсов (Resource Server) | Для защиты REST API нужно настроить сервер ресурсов, который будет проверять токены доступа. Обычно для этого используется JWT (JSON Web Token), но Spring Security также поддерживает непрозрачные токены.                                                                                                                                                                                                                                                                                                                                                  |
|                                              | Зависимости<br>Добавьте в проект зависимость `spring-boot-starter-oauth2-resource-server`                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
|                                              | Конфигурация<br>Настройте SecurityFilterChain с использованием `oauth2ResourceServer()`                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
|                                              | Настройка JwtDecoder<br>Если используется JWT, нужно настроить `JwtDecoder` для проверки подписи токена.                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
|                                              | Настройка через свойства приложения<br>В `application.yml` можно указать параметры для JWT                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
|                                              | Проверка ролей и областей (scopes)<br>Чтобы ограничить доступ к определённым ресурсам по ролям или областям, можно использовать `AuthorizationManager` или аннотации.                                                                                                                                                                                                                                                                                                                                                                                        |

### Как настроить собственный фильтр безопасности в Spring Security и в каких случаях это может быть необходимо?

| Под темы                                        | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| ----------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Настройка собственного  фильтра безопасности    | Создание фильтра<br>Реализуйте интерфейс  `jakarta.servlet.Filter` (для Spring Boot 3+) или `javax.servlet.Filter`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
|                                                 | Регистрация фильтра в цепочке безопасности<br>Добавьте фильтр в конфигурацию `SecurityFilterChain`. **Ключевые методы:**<br>- `addFilterBefore(filter, beforeFilter)` — добавляет фильтр перед указанным;<br>- `addFilterAfter(filter, afterFilter)` — после указанного;<br>- `addFilterAt(filter, atFilter)` — на конкретную позицию.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
|                                                 | Тонкая настройка позиции фильтра<br>Чтобы точно контролировать позицию фильтра в цепочке, используйте `addFilterAt()`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Когда необходимо  создавать собственный  фильтр | 1. **Проверка нестандартных заголовков**  <br>    Например, API‑ключи в `X‑API‑Key`, токены в `Authorization‑V2` и т. п.<br>2. **Интеграция с внешними системами аутентификации**  <br>    Если используется сторонний сервис идентификации (не OAuth2/OpenID Connect).<br>3. **IP‑фильтрация и гео‑ограничения**  <br>    Проверка IP‑адреса клиента, блокировка по регионам.<br>4. **Rate limiting (ограничение частоты запросов)**  <br>    Контроль количества запросов от одного пользователя/IP.<br>5. **Логирование и аудит**  <br>    Фиксация деталей запросов (IP, время, параметры) для анализа безопасности.<br>6. **Проверка цифровых подписей**  <br>    Валидация подписи запроса (например, для API с электронной подписью).<br>7. **Кастомная сессия‑менеджмент**  <br>    Альтернативная реализация хранения сессии (в Redis, базе данных).<br>8. **Многофакторная аутентификация (MFA)**  <br>    Дополнительная проверка второго фактора (SMS‑код, биометрия).<br>9. **Динамические правила доступа**  <br>    Доступ на основе сложных условий (время суток, статус аккаунта и т. д.).<br>10. **Интеграция со внутренними сервисами**  <br>    Проверка токенов/прав через внутренние микросервисы. |
