### Что такое монолитная архитектура?

| Под темы                 | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение              | **Монолитная архитектура** — это подход к построению программного  обеспечения, при котором **все компоненты и модули приложения  тесно связаны и работают как единое целое**.                                                                                                                                                                                                                                                                                                                                                    |
| Ключевые  характеристики | - **Единое хранилище кода и структур данных**: вся кодовая  база находится в одном репозитории.<br>- **Тесная взаимосвязь компонентов**: изменения в одном  модуле часто требуют обновлений в других.<br>- **Унифицированное развёртывание**: приложение  разворачивается и управляется как единый объект (в одном контейнере или среде).<br>- **Общая база данных**: обычно используется  реляционная СУБД с множеством таблиц.<br>- **Единый технологический стек**: все части системы  работают на одних и тех же технологиях. |


### Какие основные преимущества монолитной архитектуры?

| Под темы              | Заметка                                                                                                                                                                                                                                       |
| --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Основные преимущества | **Простота разработки**.<br>- единая кодовая база упрощает работу команды;<br>- разработчикам легче  ориентироваться в структуре приложения;<br>- нет необходимости настраивать  взаимодействие между сервисами и работать с API.             |
|                       | **Быстрый старт проекта**.<br>- возможность оперативно выпустить  первые версии продукта;<br>- наличие готовых решений  «из коробки» для типовых задач;<br>- минимальная необходимость в доработках  при стандартных сценариях использования. |
|                       | **Упрощённое развёртывание**.<br>- приложение разворачивается одной операцией;<br>- не требуется управлять множеством сервисов;<br>- снижены требования к инфраструктуре  и навыкам DevOps.                                                   |
|                       | **Лёгкость тестирования**.<br>- все компоненты тестируются в единой среде;<br>- проще проводить интеграционные  и функциональные тесты;<br>- нет необходимости проверять  взаимодействие между распределёнными сервисами.                     |
|                       | **Экономичность**.<br>- ниже затраты на инфраструктуру (один сервер/контейнер);<br>- меньше трудозатрат на поддержку и мониторинг;<br>- сокращены операционные расходы  по сравнению с микросервисами.                                        |
|                       | **Высокая производительность для  малых нагрузок**.<br>- отсутствие сетевых вызовов между компонентами;<br>- минимальные задержки при обмене данными;<br>- эффективная работа с единой базой данных.                                          |
|                       | **Удобство для небольших команд**.<br>- подходит для стартапов и  проектов с ограниченными ресурсами;<br>- упрощает координацию внутри команды;<br>- снижает порог входа для новых разработчиков.                                             |
|                       | **Естественная согласованность данных**.<br>- единая база данных гарантирует  целостность информации;<br>- нет проблем с распределёнными транзакциями;<br>- упрощены операции чтения/записи.                                                  |
|                       | **Упрощённая отладка**.<br>- трассировка ошибок не требует  анализа распределённых вызовов;<br>- легче локализовать проблемы в коде;<br>- меньше зависимостей между компонентами.                                                             |
|                       | **Скорость внесения изменений** (для небольших систем).<br>- правки вносятся в одном месте;<br>- не требуется синхронизация между сервисами;<br>- быстрое тестирование изменений  в целостной среде.                                          |

### В каких случаях предпочтительно использовать монолитную архитектуру?

| Под темы        | Заметка                                                                                                                                                                                                                                                                                                    |
| --------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Предпочтительно | **Старт проекта и MVP (минимально жизнеспособный продукт)**<br>- нужно как можно быстрее выйти на рынок с  базовой версией продукта;<br>- цель — проверить идею, собрать обратную  связь, подтвердить спрос;<br>- ресурсы (время, бюджет, команда) ограничены.                                             |
|                 | **Небольшие и средние проекты с простой логикой**<br>- функционал невелик и чётко определён;<br>- нет множества разнородных бизнес‑процессов,  требующих изоляции;<br>- достаточно одного технологического стека  и единой базы данных.                                                                    |
|                 | **Команды малого размера (3–7 человек)**<br>- вся разработка сосредоточена в одной  кодовой базе — проще координировать работу;<br>- новички быстрее вникают в код;<br>- не требуется сложная инфраструктура и  DevOps‑практики для управления множеством сервисов.                                        |
|                 | **Низкие и предсказуемые нагрузки**<br>- число пользователей и запросов стабильно и невелико;<br>- не требуется горизонтальное масштабирование  отдельных компонентов;<br>- один сервер/контейнер справляется с текущей нагрузкой.                                                                         |
|                 | **Краткосрочные проекты и внутренние инструменты**<br>- приложение нужно на ограниченный срок  (например, для пилотного этапа);<br>- это внутренний сервис компании с узкой аудиторией (50–100 пользователей);<br>- долгосрочное развитие и добавление  функционала не планируются.                        |
|                 | **Ограниченная инфраструктура и бюджет**<br>- нет ресурсов на настройку контейнеризации,  оркестрации, мониторинга микросервисов;<br>- требуется минимум серверов и административных усилий;<br>- важно снизить операционные затраты.                                                                      |
|                 | **Типовые задачи с готовыми решениями**<br>- проект опирается на «решения из коробки»  (CMS, фреймворки с батарейками);<br>- доработок мало, и они локальны;<br>- не нужно интегрировать разнородные  технологии или сторонние сервисы.                                                                    |
|                 | **Высокая целостность данных и транзакций**<br>- критична атомарность операций  (например, финансовые транзакции);<br>- проще обеспечить ACID‑свойства в единой  БД, чем в распределённой системе;<br>- нет необходимости в событийно‑ориентированной  архитектуре или CDC.                                |
|                 | **Проекты с жёсткими сроками и минимальными рисками**<br>- важнее выпустить работающий продукт сейчас,  чем закладывать гибкость на будущее;<br>- изменения функционала ожидаются редко и невелики;<br>- команда предпочитает стабильность и простоту поддержки.                                           |
|                 | **Отсутствие экспертизы в микросервисах**<br>- у команды нет опыта проектирования,  развёртывания и мониторинга распределённых систем;<br>- риски ошибок при переходе на микросервисы  превышают выгоды;<br>- проще освоить один стек и одну кодовую базу,  чем управлять множеством независимых сервисов. |

### Какие недостатки монолитной архитектуры могут повлиять на процесс разработки?

| Под темы   | Заметка                                                                                                                                                                                                                                                                                                                                                     |
| ---------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Недостатки | **Ограниченная масштабируемость**<br>- невозможно масштабировать отдельные  компоненты — приходится увеличивать ресурсы для всего приложения;<br>- даже если нагрузка растёт только на один модуль  (например, обработку изображений), масштабировать нужно весь монолит;<br>- это ведёт к неоправданным затратам и  неэффективному использованию ресурсов. |
|            | **Сложность внесения изменений**<br>- любое изменение требует перекомпиляции  и полного тестирования всего приложения;<br>- высок риск «побочных эффектов»: правка в  одном месте может сломать функционал в другом;<br>- замедляется цикл разработки — даже мелкие  фичи требуют полного редеплоя.                                                         |
|            | **Затяжной процесс развёртывания**<br>- деплой всего монолита занимает больше  времени, чем обновление отдельного сервиса;<br>- увеличивается время простоя при обновлениях;<br>- сложнее внедрять практику CI/CD с частыми релизами.                                                                                                                       |
|            | **Жёсткая связность компонентов**<br>- модули сильно зависят друг от друга, что затрудняет рефакторинг;<br>- сложно выделить и переиспользовать отдельные части кода;<br>- новые разработчики тратят больше времени на понимание архитектуры.                                                                                                               |
|            | **Негибкий технологический стек**<br>- вся система привязана к одному языку/фреймворку/СУБД;<br>- трудно внедрять новые технологии для отдельных задач;<br>- обновления стека требуют переработки всего приложения.                                                                                                                                         |
|            | **Риск единой точки отказа**<br>- ошибка в одном модуле может привести к падению всего приложения;<br>- сложнее обеспечивать отказоустойчивость;<br>- диагностика проблем требует анализа всей  системы, а не изолированного сервиса.                                                                                                                       |
|            | **Усложнение командной работы**<br>- при росте команды возрастает число конфликтов при мердже кода;<br>- труднее распределять задачи между разработчиками;<br>- снижается скорость параллельной разработки.                                                                                                                                                 |
|            | **Проблемы с тестированием**<br>- интеграционные тесты требуют запуска всего приложения;<br>- дольше выполняется полный цикл тестирования;<br>- сложнее изолировать тесты для отдельных модулей.                                                                                                                                                            |
|            | **Ограниченные возможности оптимизации**<br>- нельзя применять разные стратегии  кэширования для разных компонентов;<br>- сложно настраивать производительность отдельных функций;<br>- оптимизация одного модуля может ухудшить работу других.                                                                                                             |
|            | **Трудности с эволюцией системы**<br>- с ростом кодовой базы усложняется поддержка и развитие;<br>- накапливается технический долг из‑за  невозможности поэтапного рефакторинга;<br>- переход на микросервисную архитектуру в будущем  требует масштабных переделок.                                                                                        |

### Как монолитная архитектура влияет на масштабирование приложения?

| Под темы                  | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Проблемы  масштабирования | **Вертикальное масштабирование как единственный вариант**<br>- нельзя увеличить мощность только «узкого»  места (например, модуля обработки изображений);<br>- приходится наращивать ресурсы всего  приложения (CPU, RAM, диск);<br>- это ведёт к:<br>    - избыточным затратам (оплачиваются  ресурсы для всех компонентов, даже не нагруженных);<br>    - технологическим ограничениям (сервер  нельзя бесконечно «утолщать»). |
|                           | **Зависимость всех компонентов от общей базы данных**<br>- рост нагрузки на любой модуль увеличивает нагрузку на БД;<br>- сложно внедрить шардирование или реплики  без переделки всего монолита;<br>- «горячие» таблицы становятся бутылочным горлышком.                                                                                                                                                                        |
|                           | **Невозможность независимого развёртывания частей**<br>- даже мелкое обновление требует деплоя всего приложения;<br>- увеличивается время простоя;<br>- затруднено внедрение CI/CD с частыми релизами.                                                                                                                                                                                                                           |
|                           | **Нарастание технической сложности**<br>- с ростом кодовой базы усложняется:<br>    - понимание взаимосвязей между модулями;<br>    - поиск и исправление ошибок;<br>    - рефакторинг.<br>- новые фичи требуют всё больше времени на интеграцию.                                                                                                                                                                                |
|                           | **Ограничения технологического стека**<br>- весь монолит привязан к одному языку/фреймворку/СУБД;<br>- нельзя выбрать оптимальные технологии  для отдельных задач (например, использовать Go для высоконагруженных сервисов, а Python — для ML);<br>- обновления стека требуют полной перекомпиляции и тестирования.                                                                                                             |
|                           | **Риски единой точки отказа**<br>- ошибка в одном модуле может обрушить всё приложение;<br>- сложно реализовать отказоустойчивость на уровне отдельных функций;<br>- диагностика проблем требует анализа всей системы.                                                                                                                                                                                                           |
|                           | **Затруднённое параллельное развитие**<br>- при росте команды:<br>    - учащаются конфликты при мердже кода;<br>    - сложнее распределять задачи;<br>    - снижается скорость разработки.<br>- нет возможности делить команду по «сервисным» зонам ответственности.                                                                                                                                                             |
|                           | **Проблемы с тестированием при росте нагрузки**<br>- интеграционные тесты требуют запуска всего приложения;<br>- увеличивается время полного цикла тестирования;<br>- сложно симулировать нагрузку на отдельные модули.                                                                                                                                                                                                          |

### Какие стратегии можно использовать для разделения монолитной архитектуры на более мелкие компоненты?

| Под темы           | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ключевые стратегии | **Поэтапное выделение микросервисов (Strangler Fig Pattern)**<br>- Сохраняется работающий монолит,  параллельно создаётся новый микросервис.<br>- Маршрутизация трафика переключается  между старой и новой реализацией.<br>- Постепенная миграция функционала:  сначала — некритичные модули, затем — основные.<br>- **Плюсы:** минимизация рисков, возможность отката.<br>- **Минусы:** временное усложнение системы  (двойная поддержка). |
|                    | **Декомпозиция по бизнес‑доменам (Domain‑Driven Design, DDD)**<br>- Анализ предметной области и выделение  чётких границ сервисов (например: «Заказы», «Платежи», «Пользователи»).<br>- Каждый сервис отвечает за свою бизнес‑функцию  и имеет изолированную базу данных.<br>- **Плюсы:** логичная структура, слабая связанность.<br>- **Минусы:** требует глубокого понимания бизнес‑процессов.                                             |
|                    | **Разделение по техническим слоям**<br>- Выделение отдельных компонентов: API‑слой,  бизнес‑логика, хранилище данных.<br>- Пример: отделение веб‑интерфейса от серверной части.<br>- **Плюсы:** простота реализации, быстрая отдача.<br>- **Минусы:** риск «распределённого монолита»  (сервисы остаются тесно связанными).                                                                                                                  |
|                    | **Выделение «горячих» точек нагрузки**<br>- Идентификация модулей с высокой  нагрузкой (например, обработка изображений).<br>- Изоляция этих компонентов в отдельные  сервисы с независимым масштабированием.<br>- **Плюсы:** быстрое решение проблем производительности.<br>- **Минусы:** не затрагивает архитектуру в целом.                                                                                                               |
|                    | **Создание общих библиотек (Shared Libraries)**<br>- Вынесение повторяющегося кода  (аутентификация, логирование) в отдельные модули.<br>- Использование через зависимости, а не копирование.<br>- **Плюсы:** уменьшение дублирования, стандартизация.<br>- **Минусы:** риск сильной связанности, если библиотеки часто меняются.                                                                                                            |
|                    | **Событияно‑ориентированная декомпозиция**<br>- Замена синхронных вызовов на асинхронные  сообщения (через Kafka, RabbitMQ).<br>- Сервисы реагируют на события, а не вызывают друг друга напрямую.<br>- **Плюсы:** отказоустойчивость, слабая связанность.<br>- **Минусы:** усложнение отладки, необходимость обработки ошибок.                                                                                                              |
|                    | **Разделение данных (Database per Service)**<br>- Каждый микросервис получает собственную базу данных.<br>- Отказ от общей БД монолита.<br>- **Плюсы:** независимость развёртывания, изоляция данных.<br>- **Минусы:** сложность обеспечения  согласованности (требуется Saga, CQRS).                                                                                                                                                        |
|                    | **API‑фасад (API Gateway)**<br>- Введение шлюза для маршрутизации  запросов между монолитом и новыми сервисами.<br>- Единая точка входа для клиентов.<br>- **Плюсы:** прозрачное переключение, контроль доступа.<br>- **Минусы:** дополнительная точка отказа.                                                                                                                                                                               |
|                    | **Контейнеризация и оркестрация**<br>- Упаковка сервисов в контейнеры (Docker).<br>- Управление через Kubernetes/Nomad.<br>- **Плюсы:** стандартизация развёртывания,  автоматическое масштабирование.<br>- **Минусы:** требует DevOps‑экспертизы.                                                                                                                                                                                           |
|                    | **Инкрементальная миграция данных**<br>- Постепенный перенос данных из общей БД в отдельные хранилища.<br>- Синхронизация через CDC (Change Data Capture) или события.<br>- **Плюсы:** минимизация простоя.<br>- **Минусы:** временные накладные расходы на синхронизацию.                                                                                                                                                                   |

### Какие подходы к мониторингу и управлению производительностью наиболее эффективны для монолитных архитектур?

| Под темы                             | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Мониторинг метрик производительности | **Сбор и анализ ключевых показателей:**<br>- **Время отклика (response time)** — измерение задержки между запросом пользователя и получением ответа. Помогает выявить медленные операции и узкие места в обработке запросов.<br>- **Загрузка CPU и памяти** — отслеживание использования процессорных ресурсов и оперативной памяти. Позволяет определить, какие части приложения потребляют больше всего ресурсов.<br>- **Использование дискового пространства и скорость ввода-вывода (I/O)** — контроль за операциями чтения/записи на диск, что важно для приложений с интенсивным взаимодействием с базой данных.<br>- **Количество одновременных запросов (конcurrency)** — помогает оценить нагрузку на систему и её способность справляться с пиками трафика.<br>- **Ошибки и исключения** — сбор логов для анализа сбоев и аномалий.<br>Для сбора метрик можно использовать инструменты вроде Prometheus, Grafana, Zabbix или встроенные возможности APM-систем (Application Performance Monitoring). |
| Оптимизация базы данных              | **Кэширование данных** — использование кэша (например, Redis или Memcached) для часто запрашиваемых данных, чтобы снизить нагрузку на базу данных и ускорить время отклика. <br>**Оптимизация запросов** — анализ медленных запросов с помощью инструментов профилирования (например, EXPLAIN в SQL), добавление индексов, рефакторинг сложных запросов.<br>**Шарддинг и репликация** — если возможно, разделение данных на несколько серверов (шарды) или использование реплик для распределения нагрузки чтения. Однако в монолитной архитектуре это может быть сложно реализовать без глубокой переработки системы.                                                                                                                                                                                                                                                                                                                                                                                         |
| Управление нагрузкой                 | **Балансировка нагрузки** — распределение входящих запросов между несколькими экземплярами монолита (если возможно горизонтальное масштабирование). Это помогает равномерно распределить нагрузку и избежать перегрузки одного сервера. <br>**Ограничение запросов** — внедрение ограничителей скорости (rate limiting) для предотвращения перегрузки системы при резких скачках трафика.<br>**Очереди задач** — использование систем вроде RabbitMQ или Kafka для асинхронной обработки фоновых задач (например, отправка писем, обработка медиафайлов), что снижает нагрузку на основную часть приложения.                                                                                                                                                                                                                                                                                                                                                                                                   |
| Профилирование и анализ кода         | **Профилировщики** — инструменты вроде Java VisualVM, Xdebug или встроенные профилировщики в IDE помогают выявить «медленные» участки кода и неэффективные алгоритмы.<br>**Статический анализ кода** — проверка кода на предмет потенциальных проблем, например, утечек памяти или неоптимальных конструкций.<br>**Анализ зависимостей** — выявление тесно связанных модулей, которые могут усложнять оптимизацию. Инструменты вроде Dependency-Wise или ручные диаграммы зависимостей помогают визуализировать связи между компонентами.                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| Управление ресурсами сервера         | **Вертикальное масштабирование** — увеличение мощности сервера (CPU, RAM, диск), если это технически возможно и экономически оправдано. Однако у этого подхода есть ограничения.<br>**Оптимизация настроек сервера и ПО** — настройка параметров JVM (для Java-приложений), веб-сервера, базы данных под конкретную нагрузку.<br>**Мониторинг состояния оборудования** — отслеживание температуры, нагрузки на диски, состояния сетевых интерфейсов, чтобы предотвратить аппаратные сбои.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| Логирование и трассировка            | **Структурированное логирование** — использование централизованных систем логирования (например, ELK Stack, Graylog) для анализа ошибок и аномалий.<br>**Распределённая трассировка** — инструменты вроде Jaeger или Zipkin помогают отслеживать путь запроса через систему и выявлять задержки на конкретных этапах обработки.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| Автоматизация и CI/CD                | **Тестирование производительности в CI/CD** — включение нагрузочного тестирования в процесс разработки для раннего выявления проблем.<br>**Мониторинг в продакшене** — непрерывный сбор метрик и их анализ в режиме реального времени, чтобы оперативно реагировать на деградации производительности.<br>**Алгоритмы самооптимизации** — в некоторых случаях можно реализовать динамическую настройку параметров приложения (например, размер пула соединений с БД) в зависимости от текущей нагрузки.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |

### Какие технологии и инструменты лучше всего подходят для рефакторинга монолитной архитектуры?

| Под темы                                     | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Инструменты для анализа кода и зависимостей  | - **Статические анализаторы кода** (например, SonarLint, SpotBugs, ReSharper) помогают выявить проблемные участки кода, дубликаты, нарушения стандартов кодирования и потенциальные ошибки. <br>- **Инструменты для визуализации зависимостей** позволяют наглядно увидеть связи между модулями и компонентами системы. Это помогает определить границы будущих сервисов и минимизировать связанность.<br>- **Deptrac** — библиотека для проверки соблюдения границ модулей в коде. Она позволяет определить модули и контролировать, что они не используют классы других модулей напрямую.                                                                                          |
| Технологии для декомпозиции и модульности    | - **Доменно-ориентированное проектирование (DDD)** помогает определить границы сервисов на основе бизнес-доменов. Принцип единого языка (Ubiquitous Language) в DDD способствует согласованности терминологии между бизнесом и разработкой. <br>- **Модульный монолит** — промежуточный этап рефакторинга, при котором монолит разбивается на автономные модули с чёткими границами. Каждый модуль может иметь собственную базу данных и взаимодействовать с другими модулями через service client. <br>- **Spring Modulith** — инструмент для Java-приложений, который помогает организовать модульную структуру монолита и упростить его последующую трансформацию в микросервисы. |
| Средства для коммуникации между компонентами | - **REST, gRPC, GraphQL** — протоколы для взаимодействия между сервисами. Выбор зависит от конкретных требований к производительности, типу данных и других факторов. <br>- **Очереди сообщений** (Kafka, RabbitMQ) используются для асинхронной коммуникации между компонентами. Позволяют реализовать событийно-ориентированную архитектуру, где сервисы реагируют на события, а не вызывают друг друга напрямую. <br>- **API-шлюз (API Gateway)** обеспечивает единую точку входа для клиентов, маршрутизирует запросы к соответствующим сервисам, может выполнять агрегацию ответов и решать сквозные задачи (аутентификация, авторизация и т. д.).                              |
| Инструменты для управления данными           | - **Системы управления базами данных (СУБД)**. При переходе к микросервисам каждый сервис может использовать отдельное хранилище данных. Для работы с реляционными базами данных часто применяют ORM-инструменты (например, Sequelize). <br>- **Кэширование** (Redis, Memcached) помогает снизить нагрузку на базу данных и ускорить время отклика.                                                                                                                                                                                                                                                                                                                                  |
| Оркестрация и контейнеризация                | - **Docker** позволяет упаковывать сервисы в контейнеры, обеспечивая изолированную среду выполнения. <br>- **Kubernetes** используется для оркестрации контейнеров, управления их развёртыванием, масштабированием и обновлением.                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Паттерны для упрощения рефакторинга          | - **Strangler Fig Pattern** — постепенное замещение функционала монолита новыми микросервисами. Трафик переключается между старой и новой реализацией поэтапно. <br>- **Anti-Corruption Layer** изолирует устаревшие подсистемы, позволяя взаимодействовать с ними через дополнительный уровень, который преобразует данные и вызовы. <br>- **Saga** координирует распределённые транзакции в микросервисной архитектуре. Подходит для систем, управляемых событиями, но не рекомендуется при использовании баз данных SQL и в системах с циклическими зависимостями.                                                                                                                |
| Инструменты для тестирования и мониторинга   | - **Системы для написания тестов** (JUnit, pytest и др.) помогают убедиться, что после рефакторинга функционал работает корректно. <br>- **Инструменты для мониторинга и логирования** (Prometheus, Grafana, ELK Stack) необходимы для отслеживания состояния системы в процессе миграции и после неё. <br>- **Распределённая трассировка** (Jaeger, Zipkin) позволяет отслеживать путь запроса через систему и выявлять задержки на конкретных этапах обработки.                                                                                                                                                                                                                    |

### Какие основные вызовы и проблемы связаны с миграцией с монолитной архитектуры на микросервисную?

| Под темы                    | Заметка                                                                                                                                                                                                                                                                                                     |
| --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ключевые вызовы  и проблемы | **Недостаточное планирование**<br>- отсутствие чёткой стратегии и  поэтапного плана;<br>- недооценка объёма работ и сроков;<br>- не проработаны критерии успеха  и показатели эффективности.                                                                                                                |
|                             | **Неправильное разделение на сервисы**<br>- получение «распределённого  монолита» — сервисов с сильной связанностью;<br>- чрезмерная фрагментация (слишком мелкие  сервисы, увеличивающие накладные расходы);<br>- размытые границы сервисов, нарушающие  принцип единой ответственности (SRP).             |
|                             | **Проблемы интеграции и коммуникации**<br>- сложности с выбором протоколов  (REST, gRPC, GraphQL) и форматов данных;<br>- задержки и ошибки при асинхронной коммуникации;<br>- необходимость внедрения API Gateway,  очередей (Kafka, RabbitMQ), паттернов Circuit Breaker, Retry.                          |
|                             | **Управление данными и согласованностью**<br>- трудности при разделении единой БД  монолита на отдельные хранилища;<br>- риск потери согласованности данных  (необходимость eventual consistency, Saga, CQRS);<br>- проблемы с миграцией и синхронизацией  данных между сервисами.                          |
|                             | **Оркестрация и инфраструктура**<br>- сложность настройки контейнеризации  (Docker) и оркестрации (Kubernetes);<br>- необходимость автоматизации CI/CD  для множества сервисов;<br>- рост числа точек отказа и требований к мониторингу.                                                                    |
|                             | **Мониторинг и логирование**<br>- разрозненные логи и метрики по множеству сервисов;<br>- сложности трассировки запросов через несколько микросервисов;<br>- потребность в инструментах распределённой  трассировки (Jaeger, Zipkin) и централизованного логирования (ELK Stack).                           |
|                             | **Организационные и культурные изменения**<br>- необходимость перехода к DevOps-практикам и Agile;<br>- перестройка команд (кросс‑функциональные  команды, принцип «you build it, you run it»);<br>- обучение разработчиков новым технологиям и подходам;<br>- сопротивление изменениям со стороны команды. |
|                             | **Тестирование и качество**<br>- увеличение сложности тестирования из‑за распределённости системы;<br>- необходимость тестирования интеграции, отказоустойчивости, нагрузки;<br>- обеспечение покрытия тестами множества сервисов.                                                                          |
|                             | **Поддержка legacy‑кода и параллельная работа**<br>- необходимость поддерживать монолит во время миграции;<br>- синхронизация изменений между монолитом и новыми сервисами;<br>- риски конфликтов при одновременном развитии двух архитектур.                                                               |
|                             | **Экономические и ресурсные ограничения**<br>- высокие затраты на инфраструктуру (серверы, сети, хранилища);<br>- потребность в квалифицированных специалистах  (архитекторы, DevOps, системные аналитики);<br>- увеличение операционных расходов на поддержку множества сервисов.                          |
