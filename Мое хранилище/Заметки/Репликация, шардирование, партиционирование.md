### Что такое репликация в контексте реляционных баз данных?

| Под темы      | Заметка                                                                                                                                                                                                                                                                                                                                                      |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение   | **Репликация в контексте реляционных баз данных** — это процесс копирования и синхронизации данных между несколькими серверами или инстансами базы данных, чтобы все они содержали одинаковую актуальную информацию. По сути, это создание «зеркал» основной базы, которые могут быть использованы для чтения, резервирования или аварийного восстановления. |
| Необходимость | **Отказоустойчивость** — если основной сервер выходит из строя, резервная копия (реплика) берёт на себя его функции.                                                                                                                                                                                                                                         |
|               | **Масштабирование** — можно распределить нагрузку: одни серверы обрабатывают записи, другие — чтения.                                                                                                                                                                                                                                                        |
|               | **Резервное копирование в реальном времени** — в отличие от классического бэкапа, репликация поддерживает копии постоянно актуальными.                                                                                                                                                                                                                       |

### Почему шардирование может быть полезным для больших баз данных?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                          |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ    | **Шардирование (шардинг) полезно для больших баз данных, так как позволяет эффективно распределить нагрузку и ускорить обработку запросов**. Это метод горизонтального масштабирования, при котором данные распределяются по нескольким серверам (шардам). Каждый шард хранит только часть общей информации, что позволяет распределять нагрузку |

### Каковы основные цели партиционирования данных?

| Под темы | Заметка                                                                                                                  |
| -------- | ------------------------------------------------------------------------------------------------------------------------ |
| Цели     | **Ускорение запросов**. Индексы применяются только к нужным разделам, уменьшая объём сканирования.                       |
|          | **Облегчённое управление**. Можно легко удалять или архивировать старые данные без блокировки основной таблицы.          |
|          | **Балансировка нагрузки**. Разделение нагрузки между разделами при хешировании.                                          |
|          | **Эффективное резервное копирование**. Можно делать бэкап только отдельных секций.                                       |
|          | **Улучшение производительности аналитических отчётов**. База данных обрабатывает только нужные данные, а не весь массив. |

### Какие существуют типы репликации в реляционных базах данных и в чем их различия?

| Под темы           | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| По принципу работы | **Синхронная** — изменения данных сразу же применяются ко всем копиям. Транзакция считается завершённой только после того, как она успешно зафиксируется на всех серверах. Это гарантирует полную согласованность данных, но требует больше ресурсов и времени на выполнение операций. Пример — банковские транзакции, для которых важна полная синхронизация данных.                                                                                                                       |
|                    | **Асинхронная** — изменения сначала вносятся в основную базу данных, а затем передаются на реплики с некоторой задержкой. Транзакция считается завершённой сразу после её сохранения на основном сервере. Это снижает нагрузку на систему, так как операции выполняются асинхронно, но есть риск того, что некоторые данные могут временно отличаться друг от друга до полной синхронизации. Пример — социальные сети, асинхронная репликация применяется для быстрого обновления контента. |
| По топологии       | **Master–slave (ведущий–ведомый)** — запись выполняется в одном узле (master), а чтение можно распределить между многими репликами (slaves).                                                                                                                                                                                                                                                                                                                                                |
|                    | **Master–master (активная–активная)** — несколько серверов могут одновременно принимать изменения. Такая схема требует продуманного механизма разрешения конфликтов.                                                                                                                                                                                                                                                                                                                        |
|                    | **Многоуровневая (каскадная)** — одна реплика может сама выступать источником для других узлов. Это удобно при работе с распределёнными системами.                                                                                                                                                                                                                                                                                                                                          |
| По механизму       | **Физическая** — копируются целые блоки или тома данных. Это быстрый способ, который работает на уровне файлов и дисковых массивов.                                                                                                                                                                                                                                                                                                                                                         |
|                    | **Логическая** — переносятся отдельные изменения — записи, строки таблиц или SQL-запросы. Такой подход гибче, позволяет реплицировать только часть информации и даже между разными СУБД.                                                                                                                                                                                                                                                                                                    |

### Какие проблемы могут возникнуть при шардировании базы данных и как их можно решить?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                              |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Проблемы | **Неравномерное распределение данных**. Плохо выбранный ключ шардирования (столбец, используемый для определения, к какому шарду относится строка) может привести к появлению «горячих точек» (перегруженных шардов) и снижению производительности. Например, шардирование базы данных пользователей по первой букве имени пользователя может привести к неравномерному распределению, если некоторые буквы встречаются чаще других. |
|          | **Сложность межшардовых запросов и транзакций**. Запросы, затрагивающие данные из нескольких шардов, могут быть сложными и медленными. Транзакции, охватывающие несколько шардов, требуют управления распределёнными транзакциями, что может быть сложно реализовать и поддерживать.                                                                                                                                                 |
|          | **Операционные издержки**. Управление шардированной системой баз данных требует больших операционных издержек, чем управление единой базой данных. Необходимо отслеживать состояние и производительность каждого шарда, обрабатывать сбои шардов, а также выполнять резервное копирование и восстановление на нескольких серверах.                                                                                                   |
|          | **Сложность поддержания согласованности данных** на нескольких шардах, особенно в распределённой среде. Необходимо реализовать стратегии для обеспечения согласованности и точности данных на всех шардах.                                                                                                                                                                                                                           |
| Решения  | **Выбрать правильный ключ шардирования**. Учитывать шаблоны доступа к данным и типы запросов.                                                                                                                                                                                                                                                                                                                                        |
|          | **Регулярно мониторить распределение данных** и при необходимости перераспределять шарды, чтобы избежать неравномерных нагрузок по мере роста данных.                                                                                                                                                                                                                                                                                |
|          | **Автоматизировать логику шардирования** — использовать инструменты или встроенные функции базы данных, которые автоматически распределяют данные и обрабатывают шардирование, чтобы поддерживать баланс на всех шардах.                                                                                                                                                                                                             |
|          | **Реализовывать стратегии для обеспечения согласованности данных** на нескольких шардах, особенно в распределённой среде.                                                                                                                                                                                                                                                                                                            |
|          | **Использовать инструменты для обработки сбоев шардов** — это поможет предотвратить сбои в работе, когда некоторые шарды становятся недоступными.                                                                                                                                                                                                                                                                                    |

### Какие существуют стратегии партиционирования таблиц в реляционных базах данных?

| Под темы       | Заметка                                                                                                                                                                                                                             |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Горизонтальное | **Разделение таблицы на партиции, основанные на строках**. Каждая партиция содержит определённое количество строк. Такой подход используется, когда таблица имеет большой объём данных и необходима более эффективная обработка.    |
| Вертикальное   | **Разделение таблицы на партиции, основанные на столбцах**. Каждая партиция содержит определённый набор столбцов. Такой подход может быть полезен, когда таблица имеет много столбцов и только некоторые из них часто используются. |
| Гибридное      | **Объединяет горизонтальное и вертикальное партиционирование**. База данных разбивается на партиции и горизонтально, и вертикально.                                                                                                 |

### Опишите процесс настройки мастер-слейв репликации в реляционной базе данных.

| Под темы             | Заметка                                                                                                                                                                                                                                                                                                                                             |
| -------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение          | **Настройка репликации master-slave в реляционной базе данных (например, MySQL) включает настройку мастер-сервера и реплик-сервера**. Репликация — это тиражирование изменений данных с главного сервера (мастера) на один или несколько зависимых серверов (реплик). Изменения, происходящие на мастере, повторяются на репликах (но не наоборот). |
| Настройка на мастере | **Включить бинарное логирование (binlog)** — это основа для отправки изменений данных с мастера на реплики. Если бинарное логирование не включено, репликация не будет возможна.                                                                                                                                                                    |
|                      | **Настроить уникальный идентификатор сервера (server-id)** — это помогает идентифицировать отдельные серверы в группе репликации.                                                                                                                                                                                                                   |
|                      | **Указать, какую базу реплицировать** (опционально) — если баз несколько, для каждой необходима отдельная строка с параметром binlog_do_db.                                                                                                                                                                                                         |
|                      | **Перезапустить MySQL** — изменения должны вступить в силу.                                                                                                                                                                                                                                                                                         |
| Настройка на реплике | **Создать пользователя для репликации** — он будет использоваться репликами для подключения к мастеру и чтения бинарных логов. Новому пользователю требуются привилегии REPLICATION SLAVE, которые позволяют ему читать бинарные логи с мастера.                                                                                                    |
|                      | **Проверить статус бинарного лога мастера** — перед настройкой реплики необходимо знать точные координаты в бинарном логе мастера, с которых реплика должна начать репликацию. Для этого нужно выполнить команду SHOW MASTER STATUS — вывод покажет значения File и Position.                                                                       |
|                      | **Настроить параметры для подключения к мастеру** — например, указать IP-адрес, имя пользователя и пароль, а также имя файла лога и позицию, с которой нужно начинать репликацию.                                                                                                                                                                   |
|                      | **Запустить процесс репликации** — выполнить команду START SLAVE. Это запустит два потока: поток I/O, который получает бинарные логи с мастера, и поток SQL, который выполняет события из логов.                                                                                                                                                    |

### В чем заключается проблема горячих точек при шардировании и какие подходы могут помочь в ее решении?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                        |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Проблема «горячих точек» (перегруженных шардов) при шардировании баз данных** заключается в **неравномерном распределении данных** из-за неправильного выбора ключа шардирования (столбца, по которому система решает, к какому шарду относится строка). Это приводит к перегрузке отдельных шардов, что снижает производительность системы. |
| Проблема    | **Перегрузка шарда** из-за перекоса выбранного ключа. Например, шардирование базы данных пользователей по первой букве имени пользователя может привести к неравномерному распределению, если некоторые буквы встречаются чаще других.                                                                                                         |
|             | **Простаивание других шардов** после завершения диапазонов, на которые разбиты данные.                                                                                                                                                                                                                                                         |
| Подходы     | **Использовать хешированное шардирование** или композитный ключ, включающий несколько полей, что обеспечивает более равномерное распределение данных.                                                                                                                                                                                          |
|             | **Проводить ребалансировку** — перераспределить часть данных на другие шарды, когда определённый шард перегружен.                                                                                                                                                                                                                              |
|             | **Временно увеличить мощность сервера** (scale-up).                                                                                                                                                                                                                                                                                            |
|             | **Изменить ключ шардирования**, чтобы нагрузка распределялась равномернее.                                                                                                                                                                                                                                                                     |

### Как партиционирование данных может влиять на производительность запросов в реляционных базах данных и при каких условиях это эффект наиболее заметен?

| Под темы | Заметка                                                                                                                                                                                  |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Способы  | **Ускорить выполнение запросов** — оптимизаторы сканируют ограниченное число строк в таблице вместо чтения всего содержимого.                                                            |
|          | **Сократить объём данных**, обрабатываемых запросом, — база данных применяет фильтр только к тем партициям, которые содержат необходимые данные.                                         |
| Заметно  | **Большая таблица** — партиционирование полезно для таблиц с миллионами строк. Партиционирование небольших таблиц (тысячами строк и меньше) малоэффективно.                              |
|          | **Запросы фильтруют данные по ключу партиции** (например, по дате или категории). Если запрос затрагивает сразу несколько разделов, выигрыш в производительности может быть минимальным. |
