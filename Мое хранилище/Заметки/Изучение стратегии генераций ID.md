### Какие вы знаете стратегии генерации ID, предоставляемые JPA?

| Под темы    | Заметка                                                                                                                                                                                                                                                       |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| AUTO        | **Используется по умолчанию**. Позволяет провайдеру JPA выбрать подходящую стратегию на основе диалекта базы данных и типа данных столбца ID.                                                                                                                 |
| Особенности | Если тип ID — числовой (Long, Integer), провайдер определяет стратегию на основе генератора последовательности или таблицы.                                                                                                                                   |
|             | Если тип ID — UUID, провайдер использует внутренний генератор UUIDv4.                                                                                                                                                                                         |
|             | Если используется провайдер Hibernate, он выбирает стратегию на основе диалекта базы данных. Для большинства популярных баз данных выбирается SEQUENCE.                                                                                                       |
| IDENTITY    | **Использует встроенный в БД тип данных столбца identity** для генерации значения первичного ключа. База данных автоматически генерирует новое значение при каждой операции вставки.                                                                          |
| Особенности | Нужно, чтобы СУБД поддерживала такую функциональность (например, в старых версиях PostgreSQL это не работало).                                                                                                                                                |
|             | При таком подходе сначала происходит вставка, и лишь затем появляется значение ID, что может быть менее предпочтительным, чем когда сначала Hibernate генерирует ID, а затем производит вставку.                                                              |
| SEQUENCE    | **Использует последовательность** — специальный объект БД для генерации уникальных значений.                                                                                                                                                                  |
| Особенности | Требует дополнительных запросов SELECT, чтобы получить следующее значение из последовательности.                                                                                                                                                              |
|             | Если не указана дополнительная информация, Hibernate запрашивает следующее значение из стандартной последовательности.                                                                                                                                        |
|             | Можно настроить процесс генерации последовательности с помощью аннотации @SequenceGenerator.                                                                                                                                                                  |
| TABLE       | **Для генерации уникального значения используется отдельная таблица**, которая эмулирует последовательность. Когда требуется новое значение, JPA-провайдер блокирует строку таблицы, обновляет хранящееся там значение и возвращает его обратно в приложение. |
| Особенности | Эта стратегия **редко используется** — она требует использования пессимистических замков, которые ставят все транзакции в последовательный порядок.                                                                                                           |
|             | Можно использовать аннотацию @TableGenerator, чтобы указать таблицу базы данных, которую Hibernate будет использовать для симуляции последовательности.                                                                                                       |

### Что такое стратегия генерации ID в контексте JPA?

| Под темы    | Заметка                                                                                                                                                                                                                 |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **В контексте JPA (Java Persistence API) есть несколько стратегий генерации ID** — значений для первичного ключа сущности. Они определяются аннотацией @GeneratedValue и могут быть AUTO, IDENTITY, SEQUENCE или TABLE. |
|             | Выбор стратегии зависит от типа данных столбца ID и возможностей базы данных.                                                                                                                                           |

### Для чего используется аннотация @GeneratedValue в JPA?

| Под темы    | Заметка                                                                                                                                                                                                                                                                |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Аннотация @GeneratedValue в JPA (Jakarta Persistence API) используется для автоматической генерации значений первичных ключей** в таблицах базы данных. Обычно аннотация применяется в сочетании с аннотацией @Id, которая помечает поле как идентификатор сущности. |

### Как работает стратегия AUTO для генерации ID в JPA?

| Под темы    | Заметка                                                                                                                                                                                                                                                          |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Стратегия AUTO для генерации ID в JPA (Java Persistence API) работает, позволяя провайдеру JPA выбрать подходящую стратегию** на основе конфигурации базы данных и типа поля ID. Это значение указано в аннотации @GeneratedValue над идентификационным полем. |
| Принцип     | **Если СУБД поддерживает последовательности**, Hibernate создаёт отдельный сиквенс для этой сущности.                                                                                                                                                            |
|             | **Если последовательности не поддерживаются**, Hibernate создаёт специальную таблицу, в которой будет хранить номер последнего созданного ID, и использует эту таблицу в качестве аналога сиквенса.                                                              |

### В чем разница между стратегиями SEQUENCE и TABLE для генерации ID в JPA?

| Под темы    | Заметка                                                                                                                                                                                                                                                                           |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ       | **Разница между стратегиями SEQUENCE и TABLE для генерации ID в JPA (Hibernate) заключается в том, как генерируются уникальные значения**. Эти стратегии используются в аннотации @GeneratedValue для атрибута первичного ключа.                                                  |
| SEQUENCE    | **Использует отдельную последовательность (sequence) в базе данных** для генерации уникальных значений для первичного ключа. При необходимости получить новое значение приложение запрашивает его у этой последовательности, которая затем возвращает следующий уникальный номер. |
| Особенности | Значение присваивается до сохранения сущности в БД.                                                                                                                                                                                                                               |
|             | Обеспечивает возможность пакетного (batch) сохранения данных — приложению не надо перезапрашивать из базы значение ID после сохранения каждой записи.                                                                                                                             |
|             | Для тонкой настройки генерации значений ID и параметров генерируемой последовательности используется аннотация @SequenceGenerator.                                                                                                                                                |
| TABLE       | **Для генерации уникального значения используется отдельная таблица, которая эмулирует последовательность**. Когда требуется новое значение, JPA-провайдер блокирует строку таблицы, обновляет хранящееся там значение и возвращает его обратно в приложение.                     |
| Особенности | Hibernate создаёт специальную таблицу, в которой будет содержаться последний сгенерированный ID.                                                                                                                                                                                  |
|             | Когда понадобится следующий номер ID, последний номер будет увеличен на единицу и записан в таблицу как последний.                                                                                                                                                                |

### Как настроить кастомный генератор ID в JPA, используя @GenericGenerator?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                            |
| -------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ          | **Аннотация @GenericGenerator в JPA (например, в Hibernate) позволяет настроить кастомный генератор ID**. Это позволяет использовать стратегии генерации ID, не предусмотренные стандартными типами, определёнными в JPA.                                                                                                                                                                                                          |
| Параметры      | **name** — задаёт имя генератора. Это имя указывается затем в аннотации @GeneratedValue(generator = «CUSTOM_GENERATOR»), если нужно, чтобы класс предметной области использовал этот генератор.                                                                                                                                                                                                                                    |
|                | **strategy** — определяет способ генерирования ID. Стратегия указывается либо через полное имя класса стратегии (например, strategy = «org.hibernate.id.enhanced.SequenceStyleGenerator»), либо через альяс (strategy = «enhanced-sequence»).                                                                                                                                                                                      |
|                | **Создать аннотацию** можно над id-полем (прямо под @GeneratedValue), а также над классом или над пакетом в файле настроек пакета package-info.java. Чаще всего, если конкретный генератор будет использоваться только одной сущностью, аннотацию с настройками генератора ставят над id-полем. Если генератор предназначен для использования несколькими сущностями, то его помещают в настройки пакета в файл package-info.java. |
| Принцип работы | **Кастомный генератор** — это класс, который реализует интерфейс IdentifierGenerator. В интерфейсе есть один метод — generate(), который отвечает за генерирование ID.                                                                                                                                                                                                                                                             |
|                | **Важно**: если не указаны ни тип генератора, ни стратегия, Hibernate просит диалект решить подходящую стратегию — это эквивалентно использованию AUTO в JPA.                                                                                                                                                                                                                                                                      |

### Какие преимущества и недостатки у стратегии IDENTITY с точки зрения производительности в JPA?

| Под темы     | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Преимущества | **Эффективность с точки зрения базы данных** — столбцы с автоинкрементом хорошо оптимизированы и не требуют дополнительных операторов.                                                                                                                                                                                                                                                                                  |
|              | **Надёжность генерации идентификаторов** — процесс генерации управляется в одном месте — базой данных.                                                                                                                                                                                                                                                                                                                  |
|              | **Простота использования** — стратегия проста в использовании, позволяет надёжно получать уникальные ID вне зависимости от клиентов, использующих БД.                                                                                                                                                                                                                                                                   |
| Рекомендации | Рекомендуется использовать IDENTITY для случаев, когда сохраняется небольшой объём данных или когда есть риск, что одна и та же база данных используется несколькими приложениями, которые могут туда писать данные.                                                                                                                                                                                                    |
| Недостатки   | **Невозможность использовать пакетное сохранение данных** — Hibernate отключает пакетные вставки (batching inserts) для объектов, использующих генератор IDENTITY. Это противоречит идеологии Hibernate — стратегии «транзакционная запись-после» (transactional write-behind), согласно которой он пытается максимально отложить сброс данных в БД из контекста персистентности, чтобы не делать много обращений к БД. |
|              | **Производительность обычных операций вставки** может быть чуть ниже, чем при использовании последовательностей.                                                                                                                                                                                                                                                                                                        |
|              | **Процесс инкремента (получения следующего) первичного ключа** происходит вне текущей выполняемой транзакции, поэтому откат транзакции может в конечном итоге обнулить уже присвоенные значения (могут возникнуть пропуски значений).                                                                                                                                                                                   |
|              | **Стратегия IDENTITY не работает при модели наследования TABLE_PER_CLASS** — может быть несколько объектов подкласса, имеющих один и тот же идентификатор, и запрос базового класса приведёт к получению объектов с одним и тем же идентификатором (даже если они принадлежат разным типам).                                                                                                                            |

### Как стратегия SEQUENCE генерации ID влияет на возможность оптимистичной блокировки в JPA?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                   |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ          | **Стратегия SEQUENCE генерации ID в JPA может влиять на возможность оптимистичной блокировки** из-за особенностей работы последовательности и возможных проблем, связанных с ней.                                                                                                                                                                                                                         |
| Принцип работы | **Последовательность** используется для генерации уникального значения ID перед вставкой данных в базу данных. Это обеспечивает возможность пакетного (batch) сохранения данных, так как приложению не нужно перезапрашивать из базы значение ID после сохранения каждой записи.                                                                                                                          |
|                | **Оптимистичная блокировка** использует поле версии в сущностях для контроля изменений. Во время изменения сущности поле версии автоматически увеличивается, и это поле используется в условии WHERE для проверки, изменилась ли сущность. Если происходит одновременное обновление, возникает **OptimisticLockException** — исключение, из-за которого активная транзакция всегда отмечается для отката. |
| Проблемы       | **Использование только одной последовательности** по умолчанию. JPA-провайдер использует только эту последовательность для всех операторов INSERT, сгенерированных для сущностей с настройками стратегии SEQUENCE.                                                                                                                                                                                        |
|                | - **Исчерпанию значений в последовательности** — в большинстве БД максимальное количество значений последовательности — 2^63-1, и исчерпать такое количество значений возможно, если приложение генерирует огромное количество данных.                                                                                                                                                                    |
|                | - **Конфликтам оптимистичной блокировки** — если изменения не перекрываются, и две транзакции изменяют только разные столбцы, это может привести к тому, что текущая транзакция пытается изменить данные, которые уже были изменены другой транзакцией.                                                                                                                                                   |

### В каких случаях предпочтительно использовать стратегию TABLE для генерации ID и почему?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                   |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Случаи      | **Если база данных не поддерживает последовательности** (например, MySQL). В этом случае JPA-провайдер может использовать стратегию TABLE, чтобы обеспечить уникальность ID.                                                                                                                                              |
|             | **В приложении, которое не часто создаёт много сущностей** (например, не более 100 записей в секунду) и не разделяет базу данных с другими приложениями. В этом случае можно использовать любую стратегию генерации ID, так как генерация ID обычно случается только однажды — во время сохранения сущности в базу данных |
| Плюсы       | **Работа даже когда последовательности не нативно поддерживаются**. JPA предлагает альтернативный генератор, который работает даже когда последовательности не поддерживаются.                                                                                                                                            |
|             | **Возможность указать таблицу базы данных**, которая будет использоваться для эмуляции последовательности. Для этого можно использовать аннотацию @TableGenerator.                                                                                                                                                        |
| Отступление | **Однако**: стратегия TABLE не рекомендуется для приложений, которые важны производительность и масштабируемость.                                                                                                                                                                                                         |
| Минусы      | **Значительное влияние на производительность**. В отличие от столбцов IDENTITY и последовательностей, которые могут увеличивать последовательность в одном запросе, генератор TABLE требует значительных затрат на производительность.                                                                                    |
|             | **Использование сериализуемого выполнения** (блокировка на уровне строки). Это может мешать масштабируемости.                                                                                                                                                                                                             |
|             | **Затраты на транзакционные блокировки на уровне строки**. Чтобы предотвратить два одновременных соединения от получения одного значения ID, используется блокировка на уровне строки, которая может быть транзакционной и выпускаться только при окончании текущей транзакции.                                           |
