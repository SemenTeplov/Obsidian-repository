### Какова основная цель использования Maven и Gradle в проектах Java?

| Под темы        | Заметка                                                                                                                                                                                                                                                                                             |
| --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ           | Основная цель использования **Maven** и **Gradle** в Java‑проектах — **автоматизация процесса сборки программного обеспечения** и управление зависимостями.                                                                                                                                         |
| Ключевые задачи | **Управление зависимостями**  <br>    - Автоматически подключают внешние библиотеки (например, Spring, Logback). <br>    - Контролируют версии зависимостей, предотвращая конфликты.  <br>    - Загружают библиотеки из центральных репозиториев (Maven Central) и сохраняют в локальном хранилище. |
|                 | **Сборка проекта**<br>    - Компилируют исходный код.<br>    - Упаковывают приложение в дистрибутивные форматы (JAR, WAR, ZIP).<br>    - Обрабатывают ресурсы (конфигурационные файлы, шаблоны).                                                                                                    |
|                 | **Тестирование**<br>    - Запускают юнит‑тесты и интеграционные тесты.<br>    - Формируют отчёты о результатах тестирования.                                                                                                                                                                        |
|                 | **Генерация документации**<br>    - Создают Javadoc и другие виды проектной документации.                                                                                                                                                                                                           |
|                 | **Стандартизация процессов**<br>    - Обеспечивают единообразие структуры проекта и этапов сборки.<br>    - Упрощают совместную работу в команде.                                                                                                                                                   |

### Какие файлы конфигурации используются Maven и Gradle?

| Под темы<br> | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                         |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Maven        | Основной конфигурационный файл — **`pom.xml`** (Project Object Model).<br>**Где находится:** в корневой директории проекта.<br>**Что содержит:**<br>- метаданные проекта (`groupId`, `artifactId`, `version`);<br>- список зависимостей (`<dependencies>`);<br>- настройки плагинов (`<build><plugins>`);<br>- репозитории для загрузки библиотек;<br>- профили сборки;<br>- свойства проекта (кодировка, версия Java и т. п.). |
| Gradle       | Основные конфигурационные файлы (находятся в корневой директории):<br>**`build.gradle`** (или `build.gradle.kts` для Kotlin DSL) — главный файл сборки.<br>**Что содержит:**<br>    - подключаемые плагины;<br>    - репозитории; <br>    - зависимости;<br>    - настройки проекта;<br>    - кастомные задачи сборки.                                                                                                          |
|              | **`settings.gradle`** (или `settings.gradle.kts`) — файл настроек мультипроектной сборки.<br>**Что содержит:**<br>- имя корневого проекта;<br>- список подпроектов (модулей);<br>- настройки инициализации.                                                                                                                                                                                                                     |
|              | **`gradle.properties`** — файл свойств (опциональный).<br>**Что содержит:** пары «ключ‑значение» для настройки:<br>- параметров JVM для Gradle;<br>- версий зависимостей;<br>- путей к репозиториям;<br>- других глобальных настроек.                                                                                                                                                                                           |

### Какое преимущество использования многомодульных проектов?

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                                  |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ключевые  преимущества | **Чёткое разделение ответственности**<br>    - Каждый модуль реализует конкретную  функциональность (например: `core` — бизнес‑логика, `web` — веб‑интерфейс, `api` — REST‑сервисы).<br>    - Уменьшается «спагетти‑код»: разработчики работают в чётко очерченных границах.<br>    - Проще понимать и поддерживать код. |
|                        | **Повторное использование кода**<br>    - Общие компоненты (утилиты, модели, сервисы) выделяются в отдельные модули.<br>    - Эти модули подключаются как зависимости в других частях системы.  <br>    - Исключается дублирование кода.                                                                                 |
|                        | **Ускорение сборки**<br>    - При изменении кода в одном модуле пересборка затрагивает только его и зависимые модули.<br>    - Независимые модули можно собирать параллельно.<br>    - В Gradle — инкрементальная сборка (пересобираются только изменённые части).                                                       |
|                        | **Гибкое управление зависимостями**<br>    - Зависимости объявляются на уровне модуля, а не всего проекта.<br>    - Можно использовать разные версии библиотек в разных модулях (при необходимости).<br>    - Проще выявлять и устранять «конфликты зависимостей».                                                       |
|                        | **Упрощённое тестирование**<br>    - Тесты запускаются для конкретного модуля, а не всей системы.<br>    - Можно тестировать модули изолированно (моки/заглушки для зависимостей).<br>    - Быстрее выявляются ошибки в конкретных компонентах.                                                                          |
|                        | **Масштабируемость команды**<br>    - Разные команды могут работать над разными модулями параллельно.<br>    - Чёткие интерфейсы между модулями уменьшают конфликты при слиянии кода.<br>    - Новички быстрее вникают в ограниченный контекст модуля.                                                                   |
|                        | **Гибкость развёртывания**<br>    - Можно развёртывать только изменённые модули (не всю систему).  <br>    - Модули могут иметь разные стратегии развёртывания (например, `web` — в контейнере, `batch` — как cron‑задача).<br>    - Проще реализовывать микросервисные архитектуры.                                     |
|                        | **Улучшенная поддерживаемость**<br>    - Изменения в одном модуле меньше влияют на другие (при грамотном проектировании).   <br>    - Проще проводить рефакторинг: можно переписывать модуль поэтапно.<br>    - Легче оценивать влияние изменений (через граф зависимостей).                                             |
|                        | **Стандартизация структуры**<br>    - Единые правила для всех модулей (например, расположение тестов, ресурсов).<br>    - Упрощается настройка CI/CD‑пайплайнов.<br>    - Новые разработчики быстрее адаптируются.                                                                                                       |
|                        | **Контроль видимости**<br>    - В Java 9+ можно использовать `module-info.java` для явного управления экспортом пакетов.<br>    - Ограничивается доступ к «внутренним» классам модулей.<br>    - Повышается безопасность и устойчивость архитектуры.                                                                     |

### Как в Maven добавить зависимость от другого модуля в многомодульном проекте?

| Под темы   | Заметка                                                                                                                                                                                                                         |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ      | Чтобы добавить зависимость от другого модуля в  многомодульном проекте Maven, нужно **прописать эту  зависимость в `pom.xml` целевого модуля** — так же, как  и любую внешнюю библиотеку.                                       |
| Инструкция | **Убедитесь в корректной структуре проекта**  <br>Многомодульный проект должен иметь:<br>- родительский `pom.xml` (в корневой директории);<br>- поддиректории для каждого модуля;<br>- в каждой поддиректории — свой `pom.xml`. |
|            | **Проверьте `groupId`, `artifactId` и `version`  модуля‑источника**  <br>В `pom.xml` модуля, от которого зависит другой модуль  (например, `module-a`), должны быть заданы                                                      |
|            | **Добавьте зависимость в `pom.xml` зависимого  модуля**  <br>В `pom.xml` модуля, который использует другой модуль  (например, `module-b`), добавьте блок `<dependency>`                                                         |
|            | **Убедитесь, что родительский `pom.xml` перечисляет  модули**  <br>В корневом `pom.xml` должен быть раздел `<modules>`                                                                                                          |
|            | **Соберите проект**. Maven автоматически:<br>- соберёт `module-a`;<br>- поместит его в локальный репозиторий;<br>- использует его как зависимость для `module-b`.                                                               |

### Как в Gradle задать зависимость между модулями в многомодульном проекте?

| Под темы   | Заметка                                                                                                                                                                                                                                                                                                    |
| ---------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ      | В Gradle зависимость между модулями задаётся **в файле `build.gradle` (или `build.gradle.kts`) целевого модуля** через функцию `project(':путь_к_модулю')`.                                                                                                                                                |
| Инструкция | **Определите структуру проекта**  <br>Убедитесь, что проект имеет типичную многомодульную структуру.                                                                                                                                                                                                       |
|            | **Укажите модули в `settings.gradle`**  <br>В корневом файле `settings.gradle` перечислите все модули.                                                                                                                                                                                                     |
|            | **Добавьте зависимость в `build.gradle` целевого модуля**  <br>В `build.gradle` модуля, который нуждается в другом модуле (например, `module‑b`), укажите зависимость.                                                                                                                                     |
|            | **Выберите конфигурацию зависимости**  <br>Вместо `implementation` можно использовать другие конфигурации:<br>- `api` — экспортирует зависимости транзитивно (полезен для библиотек);<br>- `compileOnly` — только для компиляции (не включается в runtime);<br>- `testImplementation` — только для тестов. |
|            | **Соберите проект**                                                                                                                                                                                                                                                                                        |

### Какие есть способы управления версиями зависимостей в многомодульных проектах на Maven и Gradle?

| Под темы | Заметка                                                                                                                                                                                                                                                  |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Maven    | **Секция `<dependencyManagement>` в родительском `pom.xml`**  <br>Центральный механизм управления версиями. Позволяет:<br>- задать версии зависимостей единообразно для всех модулей;<br>- переопределять версии в дочерних `pom.xml` при необходимости. |
|          | **Свойства (`<properties>`)**  <br>Вынесение версий в переменные для удобства обновления.                                                                                                                                                                |
|          | **Импорт BOM‑файлов (Bill of Materials)**  <br>Использование готовых наборов согласованных версий.                                                                                                                                                       |
| Gradle   | **Версия‑каталог (`version catalog`, `libs.versions.toml`)** _(Gradle 7+)_  <br>Современный подход: централизованное управление версиями в отдельном файле.                                                                                              |
|          | **Расширения проекта (`ext`)**  <br>Определение версий в корневом `build.gradle`.                                                                                                                                                                        |
|          | **Объект‑контейнер версий**  <br>Создание отдельного объекта для версий/                                                                                                                                                                                 |
|          | **Плагин `gradle-versions-plugin`**  <br>Позволяет проверять актуальность версий зависимостей.                                                                                                                                                           |
|          | **Блоки `platform` и `enforcedPlatform`**  <br>Для управления версиями через BOM‑файлы.                                                                                                                                                                  |

### Как в Gradle настроить многомодульный проект для использования различных версий одной зависимости в разных модулях?

| Под темы  | Заметка                                                                                                                                                                                          |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ     | В Gradle можно **намеренно разрешить разные версии  одной зависимости в разных модулях** , но важно понимать последствия (риск конфликтов в  рантайме). Ниже — способы настройки и рекомендации. |
| Механизмы | **Явное указание версий в каждом модуле**  <br>В `build.gradle` каждого модуля прописываете нужную версию без централизованного управления                                                       |
|           | **Отключение принудительного выравнивания версий**  <br>По умолчанию Gradle пытается привести все версии к единой (via _dependency resolution rules_).                                           |
|           | **Использование `enforcedPlatform` для контроля**  <br>Если нужно зафиксировать набор версий, но разрешить исключения                                                                            |
|           | **Правила разрешения зависимостей (Resolution Rules)**  <br>Гибкий способ задать логику выбора версии                                                                                            |
|           | **Изоляция зависимостей через конфигурации**  <br>Создайте отдельные конфигурации для разных версий                                                                                              |

### В чем состоит отличие принципа работы Incremental Build между Maven и Gradle и как это влияет на производительность сборки?

| Под темы                                       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ---------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Принцип работы инкрементальной сборки в Maven  | Maven поддерживает инкрементальную сборку частично, в основном через механизмы, встроенные в отдельные плагины. Основные аспекты:<br><br>- **Сравнение временных меток**. Плагины, такие как `maven-compiler-plugin`, проверяют временные метки исходных файлов и их скомпилированных аналогов в директории `target`. Если исходный файл новее, он перекомпилируется. unrepo.com +1<br>- **Ограничения**. Maven не имеет единого механизма для всех фаз жизненного цикла. Например, фаза `test` по умолчанию запускает все тесты, даже если изменился только один тестовый класс. Также Maven не учитывает удаление исходных файлов, что может привести к использованию устаревших классов. javathinking.com +1<br>- **Зависимости**. При изменении зависимостей Maven может потребовать полной перестройки модуля, что снижает эффективность инкрементальной сборки. <br><br>Для улучшения инкрементальной сборки в Maven можно использовать сторонние плагины, например, `incr-build-plugin`, который пропускает сборку модуля, если не обнаружено изменений в `pom.xml`, директории `src` и зависимостях. Однако этот плагин имеет ограничения, например, не учитывает фазы жизненного цикла Maven и требует явного включения через параметр `-DenableIncrementalBuild`. |
| Принцип работы инкрементальной сборки в Gradle | Gradle реализует инкрементальную сборку более гибко и эффективно:<br><br>- **Анализ входов и выходов задач**. Gradle определяет, изменились ли входы (исходные файлы, зависимости) или выходы (сгенерированные файлы) задачи с момента последней сборки. Если нет, задача помечается как `UP-TO-DATE` и пропускается. docs.gradle.org +1<br>- **Инкрементальная компиляция**. Gradle анализирует зависимости между файлами и перекомпилирует только те, которые изменились или зависят от изменённых файлов. <br>- **Поддержка annotation processing**. С версии Gradle 4.7 инкрементальная компиляция совместима с обработкой аннотаций (APT), а для KAPT поддержка появилась с Kotlin 1.3.30. Это позволяет избегать полной перекомпиляции при изменении аннотаций. <br>- **Build cache**. Gradle кэширует результаты задач, что позволяет повторно использовать их в последующих сборках, даже если они выполнялись на другом компьютере или в другой среде.                                                                                                                                                                                                                                                                                                             |
| Влияние на производительность                  | - В Maven инкрементальная сборка может быть менее эффективной из-за ограниченного механизма и необходимости полной перестройки при изменении зависимостей. Это приводит к более длительному времени сборки, особенно в крупных проектах. ssl-team.com +1<br>- Gradle демонстрирует значительное ускорение за счёт инкрементальной сборки, кэширования и параллелизма. В тестах Gradle показывал сокращение времени сборки на 30–60% по сравнению с Maven, особенно при повторных сборках и в CI/CD-пайплайнах.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |

### Как настроить многомодульный проект в Maven для автоматического запуска unit-тестов в зависимых модулях при изменении кода?

| Под темы                | Заметка                                                                                                                                                                                                                                                                                                                                                                        |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ                   | В Maven **нет встроенной автоматической системы**  отслеживания изменений кода с последующим запуском  тестов только в затронутых модулях. Однако можно  настроить **эффективный рабочий процесс**, сочетающий стандартные механизмы Maven с внешними  инструментами или скриптами.                                                                                            |
| Базовые механизмы Maven | 1. **Зависимости между модулями**  <br>    Maven автоматически определяет порядок сборки модулей на основе зависимостей (`<dependency>` в `pom.xml`). Если модуль‑А зависит от модуля‑Б, то:<br>    - модуль‑Б соберётся первым;<br>    - при изменении кода в модуле‑Б пересоберётся и модуль‑А.<br>2. **Фаза `test`**  <br>    По умолчанию тесты запускаются при выполнении |
