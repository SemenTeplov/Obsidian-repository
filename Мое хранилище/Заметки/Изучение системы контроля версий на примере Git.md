### Что такое система контроля версий и для чего она используется?

| Под темы             | Заметка                                                                                                                                                                                                                                                                                       |
| -------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение          | **Система контроля версий (СКВ, VCS)** — это инструмент, используемый разработчиками программного обеспечения для управления изменениями в исходном коде и других файловых ресурсах.                                                                                                          |
| СКВ используется для | **Отслеживания изменений** в файлах проекта. Можно узнать, когда, кем и зачем был создан, удалён или отредактирован какой-либо файл. Это облегчает поиск ошибок и отладку кода, помогает избежать путаницы и конфликтов.                                                                      |
|                      | **Восстановления предыдущих версий** кода. Это может быть необходимо в случае ошибки или для сравнения с текущей версией                                                                                                                                                                      |
|                      | **Слияния изменений** — когда несколько разработчиков работают над одним проектом, СКВ позволяет сливать их изменения в одну общую версию. Это упрощает совместную работу, поскольку каждый разработчик может работать над своей частью проекта, а затем объединить изменения без конфликтов. |
|                      | **Контроля доступа** — СКВ позволяет определить, кто имеет доступ к проекту и какие права у разных пользователей. Это обеспечивает безопасность и конфиденциальность проекта, поскольку только авторизованные лица могут вносить изменения и просматривать код.                               |
|                      | **Поддержки параллельной разработки** — СКВ позволяет разработчикам работать над разными функциями и исправлениями одновременно, не мешая друг другу. Это достигается через создание отдельных веток, которые затем сливаются с основной версией проекта.                                     |

### Какова основная цель использования Git?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Основная цель использования Git** — **сохранить полный «поток снимков» проекта**, чтобы можно было:<br>- отслеживать историю изменений и быстро находить, кто и что изменил;<br>- восстанавливать предыдущие версии файлов, если в новой версии обнаружились ошибки или проблемы;<br>- безопасно экспериментировать, создавая ветки проекта, не боясь повредить основную версию;<br>- объединять изменения от нескольких участников без путаницы и конфликтов. |

### Как создать новый репозиторий в Git?

| Под темы                                            | Заметка                                                                                                                                                                                                                                                                                                        |
| --------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ                                               | **Команда git init** создаёт новый репозиторий Git в текущей директории. При выполнении:<br>- Создаётся скрытая папка .git в корне проекта.<br>- В папке размещаются все служебные файлы Git: объекты, конфигурация, история.<br>- Директория превращается в репозиторий, который готов отслеживать изменения. |
| Чтобы создать репозиторий в новом проекте           | 1. Создать папку проекта: mkdir my-project.<br>2. Перейти в папку: cd my-project.<br>3. Инициализировать репозиторий: git init.                                                                                                                                                                                |
| Для создания репозитория в уже существующем проекте | 1. Перейти в папку с уже созданными файлами: cd existing-project.<br>2. Инициализировать репозиторий: git init.<br>3. Добавить все файлы в отслеживание: git add ..<br>4. Создать первый коммит: git commit -m "Initial commit".                                                                               |

### Какие есть состояния файлов в Git и что они обозначают?

| Под темы                 | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Modified (изменён)       | **Означает, что в файл внесли изменения, но ещё не зафиксировали их в Git**. Изменения могут быть в виде добавления, изменения или удаления содержимого файла.<br>Файлы в этом состоянии находятся в рабочей директории и представляют текущее состояние проекта, которое ещё не было зафиксировано в Git.                                                                                                                                                |
| Staged (индексирован)    | **Означает, что изменённый файл в его текущей версии отмечен для добавления в снимок следующего коммита**. <br>Это состояние возникает, когда файл был перемещён в область индексированных файлов (Staging Area) — область между рабочей директорией и директорией Git. В этой области хранятся файлы, которые готовы к коммиту.<br>Индексирование изменений выполняется с помощью команды git add.                                                       |
| Committed (зафиксирован) | **Означает, что данные надёжно хранятся в локальной базе данных**. <br>Это состояние возникает, когда файлы в состоянии «индексированные» перемещаются в состояние «зафиксированные» после выполнения коммита с помощью команды git commit. <br>**Важно**: файл может быть одновременно в состоянии «изменён» и «подготовлен», если версия в рабочей директории новее, чем в области подготовленных файлов, которая, в свою очередь, новее версии в HEAD. |

### Как переключиться на другую ветку в Git и создать новую ветку?

| Под темы | Заметка                                                                                                                                                                                |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Команда git checkout** позволяет переключиться с одной ветки на другую. Синтаксис: git checkout branch_name.<br>**Команда git branch** создаёт новую ветку с именем new_branch_name. |

### Как объединить две ветки с помощью команды merge в Git?

| Под темы | Заметка                                                                                                                                                                                                                                                                                               |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Команда git merge в Git позволяет объединить изменения из одной ветки в другую**. Git принимает два указателя на коммиты (обычно последние в ветке) и находит общий для них родительский коммит. Затем создаётся **коммит слияния**, в котором объединяются изменения из обеих последовательностей. |

### Опишите процесс решения конфликтов при слиянии веток в Git.

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ       | **Конфликты при слиянии веток в Git** возникают, когда система не может автоматически объединить изменения из двух веток. В таких случаях Git помечает конфликтующие участки в затронутых файлах специальными маркерами и предлагает их разрешить вручную.                                                                                                                                                                                                                                                                                                                                                         |
| Инструменты | Для разрешения конфликтов в Git используются, например:<br>- **Команда git status** — перечисляет все файлы с конфликтами при слиянии.<br>- **Команда git diff** — показывает различия между изменениями в разных ветках. <br>- **Команда git log --merge** — создаёт журнал со списком конфликтов коммитов между ветками, для которых выполняется слияние.<br>- **Инструменты визуального сравнения** — например, git mergetool — для сложных конфликтов в больших файлах. <br>- **Команда git merge --abort** — прерывает слияние, а ветка возвращается к состоянию, в котором она находилась до начала слияния. |

### Как работает система хранения Git, основанная на снимках, в отличие от других систем контроля версий?

| Под темы | Заметка                                                                                                                                                                                                                                                                       |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | Каждый раз при создании коммита (сохранении состояния проекта) система делает снимок всех файлов в проекте и сохраняет ссылку на этот снимок. Если файлы не изменились с момента предыдущего коммита, Git не дублирует их, а просто создаёт ссылку на уже сохранённую версию. |

### Как можно отменить коммит, уже отправленный в удаленный репозиторий, не удаляя историю коммитов?

| Под темы | Заметка                                                                                                                                                                                                                                           |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **В Git для отмены коммита, уже отправленного в удалённый репозиторий, без удаления истории коммитов, можно использовать команду git revert**. Она создаёт новый коммит, который отменяет изменения предыдущего, и сохраняет историю репозитория. |
