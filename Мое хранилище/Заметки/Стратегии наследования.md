### Что такое стратегия наследования в контексте JPA / Hibernate?

| Под темы  | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| --------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ     | **Стратегия наследования в контексте JPA (Java Persistence API) и Hibernate** — это подход к моделированию отношений наследования между объектами и соответствующими им таблицами базы данных. Наследование — это отношение «есть-а», при котором подкласс наследует свойства и поведение от суперкласса. Однако реляционные базы данных не поддерживают наследование, и JPA и Hibernate поддерживают специальные стратегии, чтобы отобразить иерархию наследования в одну или несколько таблиц базы данных. |
| JPA       | **Single Table** — все классы в иерархии наследования отображаются в одну таблицу базы данных. Для различения экземпляров подклассов используется столбец-дискриминатор.                                                                                                                                                                                                                                                                                                                                     |
|           | **Joined Table** — каждый класс в иерархии отображается в свою таблицу. Таблица базового класса содержит общие атрибуты, а таблица каждого подкласса — атрибуты, специфические для этого подкласса.                                                                                                                                                                                                                                                                                                          |
|           | **Table Per Class** — каждый конкретный класс в иерархии отображается в свою таблицу. В таблице включены все свойства класса и его суперклассов.                                                                                                                                                                                                                                                                                                                                                             |
| Hibernate | **Single Table** — создаёт одну таблицу для каждой иерархии классов. По умолчанию JPA выбирает эту стратегию, если не указать её явно.                                                                                                                                                                                                                                                                                                                                                                       |
|           | **Joined Table** — каждый класс в иерархии отображается в свою таблицу. Единственный столбец, который повторно появляется во всех таблицах, — идентификатор, который используется для соединения таблиц при необходимости.                                                                                                                                                                                                                                                                                   |
|           | **Table per Class** — все свойства класса находятся в его таблице, поэтому не требуется соединение.                                                                                                                                                                                                                                                                                                                                                                                                          |

### Каковы основные стратегии наследования, поддерживаемые JPA?

| Под темы        | Заметка                                                                                                                                                                                                                                                         |
| --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Single Table    | **Все классы в иерархии наследования отображаются в одну таблицу**. Для идентификации типа каждой строки используется специальная колонка (discriminator column).                                                                                               |
| Особенности     | По умолчанию значения колонки формируются из имён классов, но можно их поменять в аннотации у каждого класса-наследника.                                                                                                                                        |
|                 | В таблице появляется дополнительная колонка, в которой записываются значения, определяющие принадлежность к классу.                                                                                                                                             |
| Joined Table    | **Каждый класс в иерархии отображается в свою таблицу**. Таблица базового класса содержит общие атрибуты, а таблица каждого класса-наследника — атрибуты, специфические для этого класса-наследника.                                                            |
| Особенности     | Столбец первичного ключа в таблице подкласса служит внешним ключом первичного ключа таблицы суперкласса.                                                                                                                                                        |
|                 | В таблице родительского класса добавляется столбец DiscriminatorColumn с DiscriminatorValue для определения типа наследника.                                                                                                                                    |
| Table Per Class | **Каждый конкретный класс в иерархии отображается в свою таблицу**. Во всех таблицах подклассов хранятся все поля этого класса плюс те, которые унаследованы от суперкласса.                                                                                    |
| Особенности     | Если нужно получить отдельную сущность, то можно обратиться к соответствующей таблице (не нужно искать во всех таблицах).                                                                                                                                       |
|                 | **Минусы**: плохая поддержка полиморфизма, так как для выборки всех классов иерархии потребуется большое количество отдельных sql-запросов для каждой таблицы-наследника или использование UNION-запроса для соединения таблиц всех наследников в одну таблицу. |
|                 | **Повторение одних и тех же атрибутов в таблицах**.                                                                                                                                                                                                             |

### Как указать стратегию наследования в JPA с помощью аннотаций?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                   |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **В JPA стратегию наследования для иерархии классов сущностей указывают с помощью аннотации @Inheritance**. Она позволяет настроить один из типов отображения: **SINGLE_TABLE**, **JOINED** или **TABLE_PER_CLASS**. Также для некоторых стратегий (например, с дискриминатором) используют аннотации **@DiscriminatorColumn** и **@DiscriminatorValue**. |
|          | Аннотация @Inheritance применяется к корневому классу иерархии сущностей. Если не указана стратегия, по умолчанию применяется стратегия SINGLE_TABLE.                                                                                                                                                                                                     |

### Какие преимущества и недостатки имеет стратегия SINGLE_TABLE при использовании в JPA / Hibernate?

| Под темы     | Заметка                                                                                                                                                                                                                               |
| ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Преимущества | **Эффективность полиморфных запросов** — все атрибуты каждой сущности хранятся в одной таблице, и запрос не требует дополнительных предложений JOIN.                                                                                  |
|              | **Быстрота чтения и записи** — так как используется только одна таблица.                                                                                                                                                              |
|              | **Возможность использовать полиморфные ассоциации** — например, отобразить ассоциации между классами, которые имеют общие поля, но отличаются поведением.                                                                             |
| Недостатки   | **Ограничение целостности данных** — так как все атрибуты подклассов расположены в одной таблице, **ограничения NOT NULL не допускаются** для столбцов, принадлежащих подклассам. Это может создавать проблемы с целостностью данных. |
|              | **Необходимость определения имени столбца-дискриминатора** — можно определить имя с помощью аннотации @DiscriminatorColumn на суперклассе, или Hibernate использует столбец DTYPE по умолчанию.                                       |
|              | **Ограничение на использование ограничений NOT NULL** — это может быть критическим, особенно если нормализация и ограничение NOT NULL являются критическими.                                                                          |

### Как работает стратегия JOINED в контексте наследования сущностей в JPA / Hibernate?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                        |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ          | **Стратегия JOINED в контексте наследования сущностей в JPA/Hibernate** (стратегия «соединения») **отображает иерархию классов в отдельные таблицы**. Корневой класс иерархии представлен отдельной таблицей, а каждый наследник имеет свою таблицу, в которой отображены только поля этого класса-наследника. |
| Принцип работы | **Таблица подкласса** не содержит столбцы для полей, унаследованных от родительского класса, за исключением поля для первичного ключа @Id, который должен быть определён только в родительской таблице.                                                                                                        |
|                | **Столбец первичного ключа** в таблице подкласса служит внешним ключом первичного ключа таблицы суперкласса.                                                                                                                                                                                                   |
|                | **В таблице родительского класса** добавляется столбец DiscriminatorColumn с DiscriminatorValue для определения типа наследника.                                                                                                                                                                               |
|                | **При создании экземпляров подклассов** выполняется одна или несколько операций соединения таблиц.                                                                                                                                                                                                             |

### В чем различие между стратегиями TABLE_PER_CLASS и SINGLE_TABLE в JPA?

| Под темы        | Заметка                                                                                                                                                                                                                                                                                                                                                                                       |
| --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| TABLE_PER_CLASS | **Создаёт отдельную таблицу для каждого конкретного класса сущностей**. Каждый класс-наследник имеет свою таблицу. Во всех таблицах подклассов хранятся все поля этого класса плюс те, которые унаследованы от суперкласса                                                                                                                                                                    |
|                 | Если нужно получить отдельную сущность, можно обратиться к соответствующей таблице (не нужно искать во всех таблицах).                                                                                                                                                                                                                                                                        |
| SINGLE_TABLE    | **Создаёт одну таблицу на всю иерархию классов**. Например, если есть entity Employee с классами-потомками FullTimeEmployee и PartTimeEmployee, то при такой стратегии все FullTimeEmployee и PartTimeEmployee записываются в таблицу Employee, и при этом в таблице появляется дополнительная колонка с именем DTYPE, в которой записываются значения, определяющие принадлежность к классу. |
|                 | Эта стратегия обеспечивает хорошую поддержку полиморфных отношений между сущностями и запросами, которые охватывают всю иерархию классов сущностей.                                                                                                                                                                                                                                           |
|                 | По умолчанию эти значения формируются из имён классов, но можно их поменять в аннотации у каждого класса-наследника.                                                                                                                                                                                                                                                                          |

### Какие особенности реализации стратегии наследования TABLE_PER_CLASS в Hibernate по сравнению с другими стратегиями?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                   |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Стратегия наследования TABLE_PER_CLASS в Hibernate** создаёт **отдельную таблицу для каждого класса в иерархии наследования**. В отличие от стратегии SINGLE_TABLE, здесь для каждого подкласса создаётся своя таблица со всеми полями, включая унаследованные от родительского класса. |
| Особенности | Позволяет избежать NULL-значений для неиспользуемых полей.                                                                                                                                                                                                                                |
|             | Подходит для небольших, несложных иерархий, где нет большого количества подклассов и запросов.                                                                                                                                                                                            |
|             | При запросе данных о родительском классе Hibernate объединяет таблицы всех подклассов с помощью UNION.                                                                                                                                                                                    |

### Как стратегия наследования влияет на производительность запросов в JPA / Hibernate?

| Под темы          | Заметка                                                                                                                                                                                                                                                                                                                                  |
| ----------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Mapped Superclass | **Не влияет на производительность** — свойства суперкласса просто включаются в таблицы подклассов.                                                                                                                                                                                                                                       |
| Ограничения       | **Не поддерживает полиморфные запросы** — запрос к суперклассу выполняется в виде нескольких выборок SQL, по одному для каждого конкретного подкласса.                                                                                                                                                                                   |
|                   | **Не поддерживает полиморфные ассоциации** — если все подклассы сопоставлены с разными таблицами, полиморфная связь с их суперклассом не может быть представлена как простая связь с внешним ключом.                                                                                                                                     |
| Table per Class   | **Может привести к снижению производительности** — для выборки всех классов иерархии потребуется большое количество отдельных SQL-запросов для каждой таблицы-наследника или использование UNION-запроса для соединения таблиц всех наследников в одну таблицу. Это может быть особенно важно для больших иерархий или сложных запросов. |
|                   | Также при TABLE_PER_CLASS не работает стратегия генератора первичных ключей IDENTITY, поскольку может быть несколько объектов подкласса, имеющих один и тот же идентификатор, и запрос базового класса приведёт к получению объектов с одним и тем же идентификатором (даже если они принадлежат разным типам).                          |
| Joined            | **Требует выполнения одной или нескольких операций соединения таблиц** при создании экземпляров подклассов сущностей. В глубоких иерархиях классов это может привести к недопустимому снижению производительности                                                                                                                        |
|                   | Также запросы, которые покрывают всю иерархию классов, требуют операций соединения между таблицами подклассов, что также снижает производительность.                                                                                                                                                                                     |
| Explicit          | **Не влияет на производительность** — аннотация @Polymorphism(type = PolymorphismType.EXPLICIT) не влияет на то, чтобы некий класс иерархии возвращался, когда делается запрос по базовому классу.                                                                                                                                       |

### Как реализовать кастомную стратегию наследования в JPA / Hibernate, не предусмотренную стандартом?

| Под темы | Заметка                                                                                                                                                                                                                                                                            |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **В JPA и Hibernate можно реализовать кастомные стратегии наследования, не предусмотренные стандартом**, с помощью аннотаций @Inheritance и @DiscriminatorColumn. Например, можно использовать стратегии single table, joined table или стратегию с использованием дискриминатора. |
