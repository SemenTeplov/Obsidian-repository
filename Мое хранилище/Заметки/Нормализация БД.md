### Что такое нормализация базы данных?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                    |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение | **Нормализация базы данных** — это процесс организации данных в базе данных, направленный на уменьшение избыточности и улучшение целостности информации. Суть подхода — **разделение больших таблиц на более мелкие** и установление связей между ними. Данные распределяются по отдельным тематическим блокам, которые затем соединяются друг с другом с помощью уникальных меток-ключей. |
| Принципы    | **Избыточность данных** — уменьшение дублирования для экономии места и повышения эффективности.                                                                                                                                                                                                                                                                                            |
|             | **Целостность данных** — обеспечение точности и согласованности данных в базе.                                                                                                                                                                                                                                                                                                             |
|             | **Аномалии данных** — избегание проблем при изменении данных, таких как аномалии вставки, обновления и удаления.                                                                                                                                                                                                                                                                           |

### Какие цели преследует нормализация БД?

| Под темы | Заметка                                                                                                                                                                      |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Цели     | **Устранение дублирования** — одни и те же записи хранятся в нескольких местах базы, что увеличивает объём хранимой информации и риск ошибок при обновлении.                 |
|          | **Сохранение целостности данных** при их изменении. Правильно спроектированные взаимосвязи между таблицами гарантируют согласованное внесение изменений во всей базе данных. |
|          | **Улучшение производительности** — оптимизированные запросы к базе данных позволяют быстрее выполнять запросы и уменьшить время отклика системы.                             |
|          | **Упрощение структуры** — изменение структуры нормализованной базы затрагивает меньшее количество элементов, что снижает вероятность ошибок.                                 |

### Назовите и опишите три формы нормализации.

| Под темы           | Заметка                                                                                                                                                                                                                                                                                                                                                                                   |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Формы нормализации | **Первая нормальная форма (1НФ)**. Таблица находится в 1НФ, если она не содержит повторяющихся групп или повторяющихся столбцов для одного значения ключа. Каждое значение столбца должно быть атомарным, то есть его нельзя разложить дальше. Такая форма упрощает структуру таблицы и облегчает хранение и поиск данных.                                                                |
|                    | **Вторая нормальная форма (2НФ)**. Таблица находится в 2НФ, если она находится в 1НФ и все её неключевые столбцы полностью функционально зависят от первичного ключа. Это означает, что значение первичного ключа определяет значение каждого неключевого столбца. 2НФ гарантирует отсутствие частичной зависимости внутри структуры таблицы и дополнительно снижает избыточность данных. |
|                    | **Третья нормальная форма (3НФ)**. Таблица находится в 3НФ, если она находится в 2НФ и все её неключевые столбцы не зависят транзитивно от первичного ключа. Другими словами, ни один неключевой столбец не должен зависеть от других неключевых столбцов, определяемых первичным ключом. 3НФ устраняет транзитивные зависимости, повышая эффективность и согласованность данных.         |

### Какие проблемы могут возникнуть при неправильной нормализации базы данных?

| Под темы | Заметка                                                                                                                                                                                                                                                                                     |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Проблемы | Дублирование данных<br>**Хранение одних и тех же данных в нескольких местах** — признак неэффективной структуры. Это увеличивает объём хранилища и приводит к ошибкам при обновлении.                                                                                                       |
|          | Несогласованность<br>**Изменения в одной таблице не отражаются в других** — это ведёт к «висячим» записям и нарушает логику данных.                                                                                                                                                         |
|          | Аномалии обновления<br>**Изменения части данных в базе данных также требуют внесения изменений в другие строки или столбцы той же таблицы** — это может привести к несогласованности и ошибкам, если данные не обновляются правильно во всех затронутых местоположениях.                    |
|          | Снижение производительности<br>**В чрезмерно нормализованных базах данных запросы могут требовать множества операций соединения между таблицами**, что может увеличивать время ответа. Это может быть медленным и громоздким, особенно при работе со сложными или большими наборами данных. |

### Как нормализация помогает избежать дублирования данных?

| Под темы | Заметка                                                                                                                                         |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| Решения  | **Устранить повторяющиеся данные** — один и тот же фрагмент информации не повторяется в разных местах.                                          |
|          | **Уменьшить объём базы данных** — за счёт сокращения избыточной информации снижается объём хранимых данных.                                     |
|          | **Упростить обновления** — если нужно изменить или удалить данные в одной таблице, то остальные связанные с ней данные автоматически обновятся. |

### Объясните, что такое функциональная зависимость в контексте нормализации БД.

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                          |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение | **Функциональная зависимость в контексте нормализации базы данных (БД)** — это связь между атрибутами таблицы, при которой **значение одного атрибута определяет значение другого**. Это ключевое понятие нормализации, которое помогает организовать данные так, чтобы минимизировать избыточность и устранить аномалии при внесении изменений. |
|             | Если атрибут **В функционально зависит от атрибута А** (обозначается как **А → В**), то **каждое значение атрибута А связано только с одним значением атрибута В**. При этом атрибуты А и В могут состоять из одного или нескольких атрибутов.                                                                                                   |
| Особенности | Если две строки имеют одно и то же значение атрибута А, то они обязательно имеют одно и то же значение атрибута В.                                                                                                                                                                                                                               |
|             | При этом для одного значения атрибута В могут существовать несколько различных значений атрибута А.                                                                                                                                                                                                                                              |
|             | Атрибут А в этой зависимости ещё называется детерминантом.                                                                                                                                                                                                                                                                                       |

### Какие техники и подходы можно использовать для оптимизации процесса нормализации, чтобы минимизировать потери производительности и упростить управление данными?

| Под темы             | Заметка                                                                                                                                                                                                                                                           |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Проектирование схемы | **Тщательное планирование структуры данных** с учётом конкретных задач и нагрузок. Это включает определение оптимального количества таблиц, их взаимосвязей, а также выбор подходящих типов данных для каждого столбца.                                           |
|                      | **Использование ER-диаграмм** (Entity-Relationship диаграмм) для визуализации структуры базы данных. Они помогают наглядно представить структуру и выявить возможные проблемы на этапе проектирования.                                                            |
|                      | **Применение подхода «прогрессивной нормализации»**: начинать с минимально необходимой структуры, но проектировать её так, чтобы будущая нормализация была возможна без полной перестройки системы.                                                               |
| Индексация           | **Создание индексов для часто запрашиваемых столбцов** — это ускорит выполнение запросов.                                                                                                                                                                         |
|                      | **Использование составных индексов** — если в запросах часто используется несколько столбцов в предложении WHERE, составной индекс может уменьшить необходимость в нескольких одностолбцовых индексах.                                                            |
|                      | - **Применение покрывающих индексов** — они включают все столбцы, на которые ссылается запрос (SELECT, JOIN, WHERE), что позволяет базе данных получать данные непосредственно из индекса, не обращаясь к таблице.                                                |
|                      | **Избегание избыточного индексирования** — не создавать слишком много индексов, так как это может замедлить операции записи (например, INSERT, UPDATE, DELETE).                                                                                                   |
| Кэширование          | **Кэширование результатов запросов** — хранение в памяти результатов ресурсоёмких или часто выполняемых запросов. При повторном выполнении аналогичного запроса вместо извлечения данных из базы данных может быть возвращён кэшированный результат.              |
|                      | **Объектное кэширование** — представления данных (например, специфические для приложения объекты) хранятся в памяти, а не в записях базы данных. Это позволяет снизить необходимость многократного преобразования записей в специфические для приложения форматы. |
| Мониторинг           | **Анализ журнала выполнения запросов** — позволяет выявить долгие и часто выполняемые запросы, которые можно оптимизировать для улучшения общей производительности базы данных.                                                                                   |
|                      | **Отслеживание метрик производительности** — например, времени выполнения запроса до/после индексирования, коэффициентов попадания в индекс, влияния на производительность записи.                                                                                |
|                      | **Использование инструментов профилирования для баз данных** — они помогают определить медленные запросы и получить представление о тенденциях производительности с течением времени.                                                                             |

### Что такое денормализация и в каких случаях она может быть полезна?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                            |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Денормализация** — это процесс нормализации базы данных с последующим добавлением избыточных столбцов в таблицы базы данных.                                                                                                                                                                                                                     |
| Случаи      | **Повышение производительности запросов**. Когда часто выполняются сложные запросы, включающие несколько объединений, денормализация может упростить эти запросы за счёт предварительного расчёта и сохранения агрегированных или связанных данных в денормализованной структуре.                                                                  |
|             | **Оптимизация приложений, требующих интенсивного чтения**. Для приложений, которые сильно зависят от операций чтения, таких как платформы отчётности или аналитики, денормализация может ускорить извлечение данных.                                                                                                                               |
|             | **Минимизация сложности приложения**. Денормализация может упростить логику приложения, уменьшив сложность поиска и обработки данных.                                                                                                                                                                                                              |
|             | **Улучшение разделения и секционирования данных**. В средах распределённых баз данных денормализация может использоваться для секционирования и сегментирования данных на нескольких узлах.                                                                                                                                                        |
|             | **Кэширование и повышение производительности**. Денормализация может использоваться в сочетании с механизмами кэширования для дальнейшего повышения производительности приложений. Приложение может быстрее обслуживать запросы, не извлекая данные из базовой базы данных, сохраняя часто используемые данные в денормализованном формате в кэше. |

### Объясните концепцию BCNF (Нормальная форма Бойса-Кодда) и ее отличия от 3NF.

| Под темы    | Заметка                                                                                                                                                                       |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | Отношение находится в BCNF, если оно находится в 3NF и **для каждой функциональной зависимости X → A выполняется одно из условий**: X содержит ключ или A — часть ключа.      |
| Отличия     | BCNF устраняет все зависимости от неключевых атрибутов, в то время как 3NF устраняет только транзитивные зависимости.                                                         |
| Важно       | Требования BCNF предъявляются только к таблицам, у которых первичный ключ составной. Таблицы с простым первичным ключом, находящиеся в 3NF, автоматически находятся и в BCNF. |
