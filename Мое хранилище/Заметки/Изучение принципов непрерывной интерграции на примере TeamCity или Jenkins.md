### Что такое непрерывная интеграция?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Непрерывная интеграция (CI, англ. Continuous Integration)** — практика разработки программного обеспечения, при которой **разработчики часто объединяют изменения кода в общий репозиторий**. Каждая интеграция проверяется автоматизированными сборками, модульными и интеграционными тестами.<br>**Цель CI** — минимизировать время между написанием кода и его проверкой, чтобы выявлять ошибки как можно раньше. Это сокращает технический долг, улучшает взаимодействие между командами и ускоряет выпуск программного обеспечения.                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| Принцип     | - **Частота интеграции** — рекомендуется, чтобы разработчики отправляли изменения в общий репозиторий как минимум один раз в день, а в идеале — несколько раз в день.<br>- **Автоматизация процессов** — каждый такой коммит запускает конвейер (pipeline), который автоматически собирает приложение, запускает тесты, проверяет качество кода и сообщает о результатах. Если тесты падают, команда немедленно получает уведомление и может оперативно исправить проблему.<br>- **Единый репозиторий** — вся информация, необходимая для сборки проекта, должна храниться в одном месте — в системе управления версиями. Это касается не только исходного кода, но и файлов настройки, библиотек, скриптов сборки и тестов, структуры базы данных.<br>- **Частые коммиты в основную ветку** — вместо крупных изменений предпочтение отдаётся небольшим коммитам, которые сразу проходят проверку и вливаются в общий код. Это снижает вероятность конфликтов и упрощает откат, если что-то пойдёт не так. |

### Какие преимущества предоставляет использование систем непрерывной интеграции, таких как TeamCity или Jenkins?

| Под темы               | Заметка                                                                                                                                                                                                                                                                     |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ключевые  преимущества | **Автоматизация рутинных процессов**<br>    - сборка проекта выполняется  автоматически при каждом изменении кода;<br>    - автоматическое тестирование  (юнит‑тесты, интеграционные тесты);<br>    - развёртывание на тестовые/промышленные  серверы по заданным правилам. |
|                        | **Раннее обнаружение ошибок**<br>    - проблемы выявляются сразу после  коммита, а не на поздних этапах;<br>    - снижение стоимости исправления  дефектов за счёт их быстрого обнаружения;<br>    - минимизация «снежного кома»  ошибок перед релизом.                     |
|                        | **Постоянная обратная связь**<br>    - разработчики моментально получают  отчёты о результатах сборки и тестов;<br>    - наглядные дашборды со статусами  сборок, покрытия тестами, метриками качества;<br>    - уведомления о сбоях (email, Slack,  Telegram и др.).       |
|                        | **Стандартизация процессов**<br>    - единые сценарии сборки  и тестирования для всех разработчиков;<br>    - воспроизводимость сборок  (один и тот же процесс на всех этапах);<br>    - чёткие критерии «зелёной»  сборки (все тесты пройдены, покрытие ≥ X %).            |
|                        | **Улучшение коллаборации**<br>    - вся команда видит статус  проекта в реальном времени;<br>    - прозрачность вклада каждого  разработчика;<br>    - снижение конфликтов при  слиянии кода (частые малые интеграции).                                                     |
|                        | **Ускорение релизов**<br>    - готовые к развёртыванию артефакты  после каждой успешной сборки; <br>    - возможность быстрого отката к  предыдущей стабильной версии;<br>    - подготовка к переходу на  CI/CD (непрерывная доставка).                                     |
|                        | **Контроль качества кода**<br>    - интеграция со статическими  анализаторами (SonarQube, ESLint, Pylint);<br>    - измерение покрытия тестами;<br>    - проверка стандартов кодирования  и архитектурных правил.                                                           |
|                        | **Масштабируемость и гибкость**<br>    - поддержка множества языков и фреймворков;<br>    - распределённые сборки на  агентах (например, под Windows и Linux);<br>    - плагины и API для кастомизации процессов.                                                           |
|                        | **Документация процессов**<br>    - история всех сборок, изменений  и результатов тестов;<br>    - аудируемые логи для соответствия  стандартам (ISO, GDPR и др.);<br>    - отчётность для менеджмента  (время сборки, частота сбоев, тренды качества).                     |
|                        | **Экономия ресурсов**<br>    - снижение ручной работы QA и DevOps;<br>    - эффективное использование вычислительных  мощностей (облачные агенты, контейнеры);<br>    - предотвращение дорогостоящих инцидентов  в продакшене.                                              |

### Каковы основные компоненты системы непрерывной интеграции?

| Под темы                                   | Заметка                                                                                                                                                                                                                                                                                                                                                                                            |
| ------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Система контроля  версий (VCS)             | **Функция:** централизованное хранение исходного кода и управление изменениями. <br>**Примеры:** Git, Subversion (SVN), Mercurial, Perforce.  <br>**Роль в CI:**<br>- служит «точкой входа» для изменений кода;<br>- фиксирует историю правок;<br>- позволяет создавать ветки (branches) и сливать изменения (merge);<br>- запускает CI‑процесс при коммитах или пул‑реквестах.                    |
| CI‑сервер (CI‑агент/оркестратор)           | **Функция:** автоматизация сборки, тестирования и развёртывания.  <br>**Примеры:** Jenkins, TeamCity, GitLab CI/CD, GitHub Actions, CircleCI.  <br>**Роль в CI:**<br>- опрашивает репозиторий на наличие изменений;<br>- запускает пайплайн (pipeline) по триггеру (коммит, таймер, внешний вызов);<br>- распределяет задачи между агентами;<br>- собирает отчёты и отображает статус сборок.      |
| Среда сборки  (Build Environment)          | **Функция:** исполнение кода и создание артефактов (исполняемых файлов, пакетов, контейнеров).<br>**Варианты:**<br>- локальные агенты (машины/контейнеры на инфраструктуре команды);<br>- облачные агенты (например, GitHub Actions runners);<br>- контейнеры (Docker) для изоляции зависимостей.  <br>    **Роль в CI:** обеспечивает воспроизводимость сборок независимо от машины разработчика. |
| Система  автоматизации  сборки             | **Функция:** преобразование исходного кода в исполняемый продукт.  <br>**Примеры инструментов:** Maven (Java), Gradle (Java/Kotlin), npm/yarn (JavaScript), pip (Python), Make.  <br>**Роль в CI:**<br>- разрешает зависимости;<br>- компилирует код;<br>- упаковывает артефакты (JAR, WAR, Docker‑образ и т. п.).                                                                                 |
| Система  тестирования                      | **Функция:** проверка корректности кода на разных уровнях.  <br>**Типы тестов:**<br>- юнит‑тесты (JUnit, pytest, Mocha);<br>- интеграционные тесты (TestContainers, Postman);<br>- тесты производительности (JMeter, Gatling);<br>- UI‑тесты (Selenium, Cypress).  <br>    **Роль в CI:** автоматически запускает тесты после сборки и блокирует деплой при ошибках.                               |
| Репозиторий  артефактов                    | **Функция:** хранение собранных пакетов, образов и других выходных данных.  <br>**Примеры:** Nexus, Artifactory, Docker Hub, JFrog.  <br>**Роль в CI:**<br>- сохраняет версии артефактов для повторного использования;<br>- обеспечивает доставку в следующие среды (тест, продакшен).                                                                                                             |
| Система развёртывания  (Deployment System) | **Функция:** автоматическая доставка артефактов в целевые среды.  <br>**Примеры инструментов:** Ansible, Terraform, Kubernetes, AWS CodeDeploy.  <br>**Роль в CI:**<br>- разворачивает код в тестовых/продакшен‑средах;<br>- выполняет миграцию баз данных;<br>- перезапускает сервисы при необходимости.                                                                                          |
| Система  мониторинга и  отчётности         | **Функция:** визуализация статуса пайплайна и анализ метрик.  <br>**Компоненты:**<br>- дашборды CI‑сервера (статус сборок, время выполнения);<br>- отчёты о покрытии тестами (JaCoCo, Istanbul);<br>- логи сборок и тестов.  <br>    **Роль в CI:** даёт команде обратную связь о качестве кода и стабильности процесса.                                                                           |
| Система уведомлений                        | **Функция:** оповещение участников о событиях.  <br>**Каналы:** email, Slack, Microsoft Teams, Telegram.  <br>**Роль в CI:** мгновенно информирует разработчиков о сбоях, чтобы ускорить исправление.                                                                                                                                                                                              |
| Система управления  конфигурацией          | **Функция:** стандартизация окружений и параметров сборки.  <br>**Примеры:** YAML/JSON‑конфигурации (`.gitlab-ci.yml`, `Jenkinsfile`), Ansible playbooks.  <br>**Роль в CI:** описывает шаги пайплайна, переменные окружения, условия деплоя.                                                                                                                                                      |

### Как настроить автоматическое сборку и тестирование проекта на Jenkins?

| Под темы                              | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Установка и подготовка Jenkins        | **Установите Jenkins** на сервер или локальную машину.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
|                                       | **Установите необходимые плагины**. Например, Git Plugin для работы с репозиториями, Maven/Gradle Plugin для сборки, JUnit Plugin для публикации результатов тестов, Allure Report (если используется Allure для отчётов). Плагины можно установить через «Управление плагинами» в интерфейсе Jenkins.                                                                                                                                                                                                                                              |
|                                       | **Настройте глобальные инструменты** (JDK, Maven, Gradle и т. д.) в «Manage Jenkins» → «Global Tool Configuration». Укажите пути к установленным инструментам.                                                                                                                                                                                                                                                                                                                                                                                      |
|                                       | **Добавьте учётные данные** для доступа к репозиторию (логин, пароль, SSH-ключи) в «Manage Jenkins» → «Manage Credentials».                                                                                                                                                                                                                                                                                                                                                                                                                         |
| Создание пайплайна                    | 1. В интерфейсе Jenkins нажмите «New Item», введите имя проекта и выберите тип «Pipeline».<br>2. В конфигурации пайплайна в разделе «Pipeline» выберите «Pipeline script from SCM» (если скрипт хранится в репозитории) или введите скрипт напрямую.<br>3. Если используете SCM (например, Git), укажите:<br>    - URL репозитория;<br>    - учётные данные;<br>    - ветку для отслеживания (например, `main` или `develop`);<br>    - путь к файлу `Jenkinsfile` (если он находится не в корне репозитория).                                      |
| Интеграция с системой контроля версий | - В настройках проекта в разделе «Управление исходным кодом» выберите Git, укажите URL репозитория и учётные данные.<br>- Если репозиторий приватный, добавьте логин и пароль в разделе «Credentials».                                                                                                                                                                                                                                                                                                                                              |
| Настройка триггеров                   | Чтобы сборка запускалась автоматически при изменении кода, можно использовать:<br>- **Polling SCM**: Jenkins периодически проверяет репозиторий на наличие изменений. В настройках проекта в секции «Build Triggers» выберите «Poll SCM» и укажите расписание в формате cron (например, `H/15 * * * *` для проверки каждые 15 минут). <br>- **Webhooks**: система контроля версий (например, GitHub) сама уведомляет Jenkins о изменениях. В настройках проекта выберите «GitHub hook trigger for GITScm polling» и настройте вебхук в репозитории. |
| Настройка шагов сборки и тестирования | В `Jenkinsfile` (скрипт пайплайна) опишите этапы сборки и тестирования.                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| Публикация результатов тестов         | В разделе «Post-build Actions» добавьте действия для публикации результатов:<br>- **Publish JUnit test result report** — для публикации результатов в формате JUnit.<br>- **Publish Cobertura Coverage Report** — для отчётов о покрытии кода.<br>- **HTML Publisher** — для публикации HTML-отчётов. <br>Если используется Allure, установите плагин Allure Jenkins и в настройках послесборочных операций укажите путь к артефактам Allure (по умолчанию `allure-result`).                                                                        |

### В чем различие между TeamCity и Jenkins с точки зрения управления конфигурацией?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Jenkins  | **Конфигурация через скрипты и веб-интерфейс**. Jenkins позволяет настраивать пайплайны двумя способами:<br>- **Через веб-интерфейс**. Пользователь может вручную задавать параметры сборки, триггеры, шаги выполнения и другие настройки через графический интерфейс.<br>- **С помощью скриптов**. Для декларативного или скриптового описания пайплайнов используется язык Groovy. Также возможна конфигурация через XML-файлы. Это даёт гибкость, но требует навыков программирования и может быть сложным для новичков. <br>**Зависимость от плагинов**. Многие функции и интеграции в Jenkins реализуются через плагины. Это позволяет расширять функциональность, но усложняет управление конфигурацией: необходимо следить за совместимостью плагинов, их обновлениями и корректной установкой.<br>**Открытый исходный код и сообщество**. Jenkins — открытый проект, поддерживаемый сообществом. Это означает большую экосистему плагинов, но менее предсказуемую стабильность, так как качество и поддержка плагинов зависят от их разработчиков.                                                                                                                                                 |
| TeamCity | **Конфигурация как код на Kotlin DSL**. TeamCity использует Domain Specific Language (DSL) на базе Kotlin для описания конфигураций в виде кода. Это обеспечивает типобезопасность, удобство структурирования и поддержку со стороны IDE, что упрощает работу с конфигурациями и снижает вероятность ошибок. Также в TeamCity Pipelines доступны YAML-конфигурации (в версии Enterprise). <br>**Встроенные функции и интеграции**. Многие ключевые возможности (например, работа с Docker, Jira, Bugzilla, поддержка .NET) реализованы «из коробки», без необходимости устанавливать дополнительные плагины. Это упрощает начальную настройку и снижает зависимость от сторонних компонентов.<br>**Шаблоны конфигураций**. TeamCity предоставляет встроенные шаблоны, которые позволяют легко настраивать и переиспользовать сборки в разных проектах. Это ускоряет процесс создания новых проектов и обеспечивает единообразие конфигураций. <br>**Централизованное управление**. Как коммерческий продукт, TeamCity поддерживается командой JetBrains. Это обеспечивает более предсказуемую стабильность и безопасность, так как обновления и исправления ошибок контролируются разработчиками продукта. |

### Как можно интегрировать систему контроля версий с TeamCity или Jenkins для автоматического запуска сборок?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| TeamCity | TeamCity «из коробки» интегрируется с Git, Mercurial, Perforce, Subversion, Team Foundation (TFVC) и другими системами контроля версий. Для настройки необходимо создать VCS Root (источник контроля версий).<br>**Шаги для интеграции:**<br>1. В настройках проекта перейдите в раздел **VCS root**.<br>2. Нажмите **Create VCS root**.<br>3. Укажите тип VCS (например, Git) или выберите опцию _Guess from repository URL_, чтобы TeamCity автоматически определил тип по URL репозитория.<br>4. Введите URL репозитория (например, SSH или HTTPS).<br>5. Настройте аутентификацию: укажите логин и пароль или используйте SSH-ключ.<br>6. Укажите основную ветку, изменения в которой должен отслеживать TeamCity (параметр _Default branch_).<br>7. При необходимости настройте правила проверки кода (Checkout Rules), чтобы исключить некоторые папки или файлы из мониторинга. <br>**Триггеры для автоматического запуска сборок:**<br>- Изменения в определённой ветке или директории.<br>- Пул-реквесты (поддерживаются для GitHub, GitLab, Azure DevOps, Bitbucket Server). <br>- Отложенные файлы в Perforce (Perforce Shelve Trigger). <br>- Можно настроить запуск сборки при изменении определённого пользователя или при создании тега.<br>TeamCity также поддерживает публикацию статуса сборки в репозитории (например, в GitHub, GitLab), что позволяет видеть результаты прямо в интерфейсе VCS.                                                                                                 |
| Jenkins  | Jenkins интегрируется с Git, SVN, Mercurial и другими VCS через плагины. Для работы с Git требуется установить Git Plugin.<br>**Шаги для интеграции:**<br>1. Установите необходимый плагин (например, Git Plugin для Git, Subversion Plugin для SVN) через _Manage Jenkins → Manage Plugins_.<br>2. Настройте глобальные параметры Git в _Manage Jenkins → Configure System_, указав путь к исполняемому файлу Git. <br>3. Создайте новый джоб (задание) или пайплайн. В разделе **Source Code Management** выберите тип VCS (например, Git) и укажите URL репозитория. <br>4. Настройте учётные данные в _Manage Jenkins → Manage Credentials_, если репозиторий требует аутентификации (логин/пароль или SSH-ключ). <br>5. Укажите ветку, которую нужно отслеживать (например, `main` или `develop`). <br>**Способы автоматического запуска сборок:**<br>- **Polling SCM**. Jenkins периодически проверяет репозиторий на наличие изменений. В настройках джоба в разделе **Build Triggers** выберите **Poll SCM** и укажите расписание в формате cron (например, `H/15 * * * *` для проверки каждые 15 минут).<br>- **Webhooks**. Система контроля версий (например, GitHub, GitLab) сама уведомляет Jenkins о изменениях. Для GitHub нужно настроить вебхук в репозитории, указав URL Jenkins, и выбрать события _push_. В Jenkins активируйте опцию **GitHub hook trigger for GITScm polling** в настройках джоба. Аналогично для GitLab: настройте вебхук в репозитории и используйте плагин GitLab в Jenkins. |

Нажмите **Build**.<br>Jenkins запустит сборку с указанными параметрами, подставив их в шаги пайплайна или shell‑скрипты.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |

### Как использовать Docker в связке с TeamCity или Jenkins для создания изолированных сред для сборки и тестирования?

| Под темы          | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ             | Использование Docker в связке с TeamCity или Jenkins позволяет создавать изолированные среды для сборки и тестирования, обеспечивая воспроизводимость, масштабируемость и изоляцию зависимостей. Это особенно полезно для работы с разными версиями ПО, зависимостями или для интеграционного тестирования в контейнеризованных службах.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| TeamCity и Docker | TeamCity поддерживает несколько способов работы с Docker:<br>1. **Билд-раннер Docker**. Позволяет добавлять к сборке шаги выполнения команд Docker. Это полезно, если нужно динамически изменять аргументы Docker, например, запускать сборку в определённой версии операционной системы, которая определена на предыдущих этапах пайплайна. TeamCity автоматически очищает локальный кэш Docker-образов на билд-агентах, готовя их к следующим сборкам. <br>2. **Docker-обёртка**. Позволяет запускать шаги сборки в Docker-контейнере независимо от операционной системы хоста. Это гарантирует точные и воспроизводимые результаты. TeamCity отслеживает все события во время сборки и предоставляет отчёты в реальном времени, анализ тестов и другие возможности, например, трассировку стека из Docker-контейнера. <br>3. **Билд-раннер Docker Compose**. Используется для запуска Docker-контейнеров в начале сборки. Позволяет TeamCity закрыть их после завершения или остановки сборки. Основной сценарий использования — выполнение интеграционных тестов, когда компоненты запускаются в контейнерах, а их выполнение описывается с помощью файлов Docker Compose. |
| Jenkins и Docker  | В Jenkins для работы с Docker требуется установить плагины, например, Docker Pipeline.<br>**Основные способы использования Docker в Jenkins:**<br>**Использование Docker в качестве среды выполнения этапов пайплайна**. В Jenkinsfile можно указать образ Docker для выполнения конкретного этапа или всего пайплайна.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
|                   | **Использование Dockerfile для сборки образа**. В Jenkinsfile можно указать путь к Dockerfile, и Jenkins соберёт образ на его основе.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
|                   | **Запуск «sidecar»-контейнеров**. Это полезно, если для тестов требуется внешний сервис (например, база данных). С помощью Docker Pipeline плагина можно запустить один контейнер «в фоне», а работу выполнять в другом.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
|                   | **Использование Docker Compose**. Позволяет описать несколько сервисов в одном файле `docker-compose.yml` и запустить их вместе. Это удобно для интеграционного тестирования.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |

### Как настроить мониторинг и уведомления о состоянии сборки в системах непрерывной интеграции, чтобы оперативно реагировать на ошибки?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Jenkins  | В Jenkins для настройки мониторинга и уведомлений применяются плагины и встроенные возможности.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
|          | Email-уведомления<br>Для отправки email требуется установить плагин **Email Extension** и настроить SMTP-сервер. <br>**Шаги настройки:**<br>1. Перейдите в _Manage Jenkins → Manage Plugins_, найдите плагин Email Extension и установите его.<br>2. В _Manage Jenkins → Configure System_ настройте параметры SMTP: сервер, порт, аутентификацию (если требуется), SSL/TLS. <br>3. В конфигурации задания (job) в разделе **Post-build Actions** добавьте **E-mail Notification**, укажите адреса получателей и условия отправки (например, при неудаче сборки). <br>В Jenkinsfile можно определять уведомления напрямую, используя `emailext`. |
|          | Интеграция с Slack<br>Для уведомлений в Slack требуется плагин **Slack Notification**.<br>**Шаги настройки:**<br>1. Установите плагин через менеджер плагинов.<br>2. В Slack создайте интеграцию Jenkins CI и получите URL вебхука.<br>3. В Jenkins в _Manage Jenkins → Configure System_ укажите домен Slack и токен интеграции или URL вебхука.<br>4. В конфигурации задания в **Post-build Actions** добавьте **Slack Notifications** и настройте их.                                                                                                                                                                                         |
|          | Вебхуки<br>Для интеграции с внешними системами можно использовать вебхуки. В конфигурации задания добавьте действие **Webhook Notification**, укажите URL и события, которые должны триггерить уведомление (например, начало сборки, успех, неудача).                                                                                                                                                                                                                                                                                                                                                                                            |
|          | Скриптованные уведомления<br>В пайплайне можно использовать Groovy-скрипты для гибкой логики уведомлений.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| TeamCity | В TeamCity уведомления можно настраивать на уровне пользователя и на уровне конфигурации сборки.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
|          | Email-уведомления<br>**Шаги настройки:**<br>1. Администратор сервера должен настроить интеграцию с email-провайдером в _Administration → Email Notifier_ (указать SMTP-хост, порт, логин, пароль). <br>2. Пользователи могут настроить правила уведомлений в своём профиле (_Your Profile → Notification Rules_), указав проекты, ветки, события (например, сбой сборки).<br>3. Для уведомлений по конфигурации сборки добавьте функцию **Notifications** в настройки сборки. Укажите email-адреса, на которые должны отправляться уведомления при определённых событиях.                                                                        |
|          | Уведомления в Slack<br>Для Slack требуется создать отдельное приложение в Slack и подключить его к TeamCity. После аутентификации в Slack настройте правила уведомлений в профиле пользователя. Можно также настроить уведомления для конкретной конфигурации сборки, указав канал Slack.                                                                                                                                                                                                                                                                                                                                                        |
|          | Браузерные уведомления<br>TeamCity Browser Notifier — расширение для браузера, которое отображает уведомления о событиях сборки прямо в веб-браузере.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
|          | IDE-уведомления<br>TeamCity IDE Notifier — плагин для IDE (например, IntelliJ IDEA), который позволяет получать уведомления о событиях сборки в среде разработки.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
|          | CatLight<br>Приложение **CatLight** может мониторить сборки TeamCity и показывать десктоп-уведомления. Оно отображает статус сборок в трее и позволяет отслеживать несколько серверов TeamCity одновременно. Для настройки нужно:<br>1. Скачать приложение CatLight.<br>2. Подключиться к TeamCity, указав URL сервера.<br>3. Выбрать задания для мониторинга.                                                                                                                                                                                                                                                                                   |
