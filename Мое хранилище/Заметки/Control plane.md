| Под темы                                | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| --------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение                             | Существует два основных пути связи между уровнем управления (сервером API) и узлами. Первый — от сервера API к процессу kubelet, который работает на каждом узле кластера. Второй — от сервера API к любому узлу, модулю или сервису через прокси-функцию сервера API.                                                                                                                                                                                                                                                                                                                                                                     |
| от API server к kubelet                 | Соединения между сервером API и kubelet используются для: получения журналов для подов. Присоединения (обычно через kubectl) к запущенным подам. Обеспечения функции переадресации портов kubelet.                                                                                                                                                                                                                                                                                                                                                                                                                                         |
|                                         | Эти соединения завершаются на конечной точке HTTPS-сервера kubelet. По умолчанию сервер API не проверяет сертификат сервера kubelet, из-за чего соединение становится уязвимым для атак типа «злоумышленник в середине» и небезопасным для использования в ненадежных и/или общедоступных сетях.                                                                                                                                                                                                                                                                                                                                           |
|                                         | Чтобы проверить это соединение, используйте флаг --kubelet-certificate-authority, чтобы предоставить серверу API пакет корневых сертификатов для проверки обслуживающего сертификата kubelet.                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| от API server к nodes, pods, и services | По умолчанию соединения между сервером API и узлом, модулем или сервисом осуществляются по протоколу HTTP и не требуют аутентификации или шифрования. Их можно установить по защищенному HTTPS-соединению, добавив префикс https: к имени узла, модуля или сервиса в URL-адресе API, но при этом не будет проверяться сертификат, предоставленный конечной точкой HTTPS, и не будут использоваться учетные данные клиента. Таким образом, хотя соединение и будет зашифровано, оно не обеспечит никакой гарантии целостности данных. В настоящее время небезопасно устанавливать такие соединения через ненадежные или общедоступные сети. |
| SSH тунели                              | Kubernetes поддерживает SSH-туннели для защиты каналов связи между уровнем управления и узлами. В этой конфигурации сервер API инициирует SSH-туннель для каждого узла в кластере (подключаясь к SSH-серверу, прослушивающему порт 22), и весь трафик, предназначенный для kubelet, узла, пода или сервиса, проходит через туннель. Благодаря этому туннелю трафик не выходит за пределы сети, в которой работают узлы.                                                                                                                                                                                                                    |
| Konnectivity service                    | В качестве альтернативы SSH-туннелям сервис Konnectivity предоставляет прокси-сервер на уровне TCP для связи между уровнем управления и кластером. Сервис Konnectivity состоит из двух частей: сервера Konnectivity в сети уровня управления и агентов Konnectivity в сети узлов. Агенты Konnectivity устанавливают соединения с сервером Konnectivity и поддерживают сетевые соединения. После включения сервиса Konnectivity весь трафик между уровнем управления и узлами проходит через эти соединения.                                                                                                                                |
