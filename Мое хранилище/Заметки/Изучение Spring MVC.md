### Что такое Spring MVC?

| Под темы                                 | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ---------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                              | Spring MVC — это веб‑фреймворк в составе  Spring Framework, реализующий архитектурный  шаблон Model‑View‑Controller (MVC) для создания  веб‑приложений на Java.                                                                                                                                                                                                                                                                                                                                                                                                         |
| Суть шаблона MVC                         | Приложение разделяется на три взаимосвязанных компонента:<br><br>- Model (Модель) — хранит данные и бизнес‑логику (например, сущности, сервисы, репозитории).<br>- View (Представление) — отвечает за отображение данных пользователю (HTML, JSP, Thymeleaf и т. п.).<br>- Controller (Контроллер) — обрабатывает HTTP‑запросы, взаимодействует с Model и выбирает View для отображения.                                                                                                                                                                                |
| Центральный компонент: DispatcherServlet | Вся логика работы построена вокруг DispatcherServlet — «фронт‑контроллера», который:<br><br>1. Принимает все HTTP‑запросы.<br>2. С помощью HandlerMapping определяет, какой контроллер обработать запрос.<br>3. Передаёт запрос контроллеру.<br>4. С помощью ViewResolver находит нужное представление (View).<br>5. Возвращает сгенерированный ответ клиенту.                                                                                                                                                                                                          |
| Ключевые возможности                     | - Гибкость представлений: поддержка JSP, Thymeleaf, HTML, JSON/XML (включая REST API).<br>- Масштабируемость: удобное разделение кода на модули.<br>- Автоматизация: аннотации для настройки маршрутизации, валидации и т. д.<br>- Интеграция: работа с Spring Security (авторизация), Spring Data (доступ к БД) и др.<br>- Обработка ошибок: встроенные механизмы для перехвата и обработки исключений.<br>- Поддержка REST: нативная работа с RESTful API.<br>- Валидация форм: удобная проверка входных данных.<br>- Локализация: поддержка многоязычных приложений. |

### Какова основная роль аннотации @Controller в Spring MVC?

| Под темы         | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение      | Аннотация @Controller в Spring MVC выполняет  ключевую роль связующего звена между HTTP‑запросами  клиента и бизнес‑логикой приложения.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Основная роль    | @Controller маркирует класс как компонент‑контроллер в архитектуре MVC, который:<br><br>- принимает HTTP‑запросы (от DispatcherServlet);<br>- обрабатывает их (вызывает сервисы, работает с моделью);<br>- возвращает ответ (HTML‑страницу или данные).<br>    <br>По сути, контроллер выступает «официантом»: принимает заказ (запрос), передаёт на «кухню» (в сервисы), возвращает готовое «блюдо» (ответ).                                                                                                                                                                                                                                                                                                                                                                                                                              |
| Ключевые функции | 1. Маршрутизация запросов  <br>    Совместно с аннотациями @RequestMapping, @GetMapping,  @PostMapping определяет, какой метод обрабатывает конкретный URL и HTTP‑метод.<br>    <br>2. Обработка входных данных<br>    <br>    - принимает параметры запроса (@RequestParam);<br>    - извлекает переменные из URL (@PathVariable);<br>    - обрабатывает тело запроса (@RequestBody).<br>        <br>3. Взаимодействие с моделью  <br>    Вызывает сервисы и репозитории для работы с данными.<br>    <br>4. Выбор представления (View)  <br>    Возвращает имя шаблона (например, «hello» для hello.html), который будет отрендерен.<br>    <br>5. Формирование ответа  <br>    Может возвращать:<br>    <br>    - строку (имя представления);<br>    - объект модели (передаётся в View);<br>    - данные (с @ResponseBody — JSON/XML). |

### Какие шаги необходимо выполнить для создания простого веб-приложения на Spring MVC?

| Под темы                | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Настройка проекта       | **Создайте проект через Spring Initializr** (). Это онлайн-сервис, который автоматически сгенерирует базовый проект с нужными зависимостями.<br><br>- Выберите тип проекта (Maven или Gradle) и язык (Java).<br>- В разделе Dependencies добавьте **Spring Web** — это включит основные компоненты Spring MVC.<br>- При желании можно добавить дополнительные зависимости, например, **Thymeleaf** для шаблонизации или **Spring Boot DevTools** для удобства разработки. <br>- Нажмите Generate и скачайте архив с проектом.<br><br>Если вы работаете в IDE с интеграцией Spring Initializr, можно создать проект прямо в среде разработки. |
| Добавление зависимостей | Если вы настраиваете проект вручную, в файле `pom.xml` (для Maven) или `build.gradle` (для Gradle) добавьте зависимость `spring-boot-starter-web`. Эта зависимость включает встроенный веб-сервер (по умолчанию Tomcat), основные компоненты Spring MVC и Jackson для сериализации/десериализации JSON.                                                                                                                                                                                                                                                                                                                                      |
| Создание контроллера    | Создайте класс контроллера с аннотацией `@Controller` (или `@RestController` для REST API). Методы контроллера должны быть помечены аннотациями маршрутизации, например, `@GetMapping`, `@PostMapping` и т. д.                                                                                                                                                                                                                                                                                                                                                                                                                               |
| Создание представлений  | Создайте файлы шаблонов (например, JSP, Thymeleaf или HTML) и поместите их в соответствующую папку. По умолчанию Spring Boot ищет представления в папке `src/main/resources/templates/` (для Thymeleaf) или `src/main/webapp/WEB-INF/views/` (для JSP).                                                                                                                                                                                                                                                                                                                                                                                      |

### Как в Spring MVC реализуется обработка запросов на стороне сервера?

| Под темы                                                | Заметка                                                                                                                                                                                                                                                                                                                                                               |
| ------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Приём запроса  `DispatcherServlet`                      | Все HTTP‑запросы (независимо от URL)  сначала попадают в DispatcherServlet —  фронт‑контроллер приложения. Он:<br><br>- получает HttpServletRequest и HttpServletResponse;<br>- запускает цепочку обработки.                                                                                                                                                          |
| Маршрутизация: поиск обработчика через `HandlerMapping` | DispatcherServlet обращается к HandlerMapping, чтобы определить:<br><br>- какой контроллер и метод должны обработать запрос;<br>- соответствует ли URL и HTTP‑метод (GET, POST и т. д.) настроенным маршрутам.                                                                                                                                                        |
| Предварительная обработка через `Handl erInterceptor`   | Перед вызовом контроллера выполняются HandlerInterceptor (перехватчики):<br><br>- preHandle() — до вызова метода контроллера (например, проверка авторизации);<br>- если preHandle() вернул false, дальнейшая обработка прерывается.                                                                                                                                  |
| Вызов метода контроллера через  `HandlerAdapter`        | DispatcherServlet использует HandlerAdapter, чтобы вызвать нужный метод контроллера. HandlerAdapter:<br><br>- определяет, как вызвать метод (с учётом аннотаций, параметров, возвращаемого типа);<br>- передаёт в метод параметры запроса (через @RequestParam, @PathVariable, @RequestBody и т. п.);<br>- вызывает метод и получает результат.                       |
| Формирование  `ModelAndView`                            | Метод контроллера возвращает:<br><br>- Для MVC (@Controller) — объект ModelAndView, содержащий:<br>    - model — данные для представления (например, список товаров);<br>    - viewName — имя шаблона (например, "productList").<br>- Для REST (@RestController или @ResponseBody) — объект данных (например, List<Product>), который сразу сериализуется в JSON/XML. |
| Разрешение представления через `ViewResolver`           | Если возвращалось имя представления (не null), DispatcherServlet использует ViewResolver, чтобы:<br><br>- найти реальное представление (JSP, Thymeleaf‑шаблон и т. п.) по имени;<br>- создать объект View.                                                                                                                                                            |
| Рендеринг представления                                 | Объект View:<br><br>- берёт данные из model; <br>- применяет шаблон (например, заполняет HTML‑страницу);<br>- записывает результат в HttpServletResponse.<br>    <br>Для REST‑ответов (с @RestController) этот шаг пропускается: данные сериализуются напрямую в тело ответа.                                                                                         |
| Постобработка через `Handl erInterceptor`               | После выполнения контроллера вызывается postHandle() у HandlerInterceptor:<br><br>- можно модифицировать ModelAndView или ответ;<br>- обычно используется для логирования, добавления заголовков.                                                                                                                                                                     |
| Обработка исключений                                    | Если в процессе возникло исключение, DispatcherServlet передаёт его HandlerExceptionResolver:<br><br>- ExceptionHandlerExceptionResolver — вызывает методы с @ExceptionHandler;<br>- ResponseStatusExceptionResolver — преобразует исключения с @ResponseStatus в HTTP‑коды;<br>- DefaultHandlerExceptionResolver — обрабатывает стандартные исключения Spring MVC.   |
| Отправка ответа клиенту                                 | Готовый ответ (HTML, JSON, XML и т. п.) отправляется через HttpServletResponse обратно клиенту.                                                                                                                                                                                                                                                                       |

### Какие преимущества предоставляет использование ViewResolver в Spring MVC?

| Под темы              | Заметка                                                                                                                                                                                                                                                                                                                                                                                                   |
| --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение           | ViewResolver в Spring MVC — ключевой компонент,   обеспечивающий гибкое сопоставление  логических имён представлений с реальными  шаблонами (JSP, Thymeleaf, FreeMarker и др.). Его  использование даёт ряд существенных преимуществ.                                                                                                                                                                     |
| Основные преимущества | Абстракция от технологии представления<br>    <br>    - Контроллеры возвращают лишь логическое имя представления (например, "productList"), не зная, как оно будет отрендерено.<br>    - Можно менять движок шаблонов (с JSP на Thymeleaf) без правки контроллеров.                                                                                                                                       |
|                       | Централизованная конфигурация путей<br><br>- Задаётся единый префикс/суффикс для всех представлений (например, /WEB-INF/views/ и .html).                                                                                                                                                                                                                                                                  |
|                       | Кэширование представлений<br>    <br>    - Подклассы AbstractCachingViewResolver кэшируют экземпляры представлений, ускоряя повторные запросы.<br>    - Можно отключить кэш (setCache(false)) или очистить конкретный шаблон (removeFromCache(...)).                                                                                                                                                      |
|                       | Гибкость маршрутизации<br>    <br>    - Разные ViewResolver обрабатывают разные типы представлений:<br>        - InternalResourceViewResolver — для JSP/сервлетов.<br>        - FreeMarkerViewResolver — для шаблонов FreeMarker.<br>        - ContentNegotiatingViewResolver — выбор представления по Accept-заголовку (JSON/XML/PDF).<br>    - Возможность цепочки ViewResolver с приоритетами (order). |
|                       | Поддержка переадресаций и форвардов<br><br>- Специальные префиксы в имени представления:<br>    - redirect:/path — перенаправление клиента. <br>    - forward:/path — внутренний форвард через RequestDispatcher.                                                                                                                                                                                         |
|                       | Динамический выбор представления<br>    <br>    - ContentNegotiatingViewResolver выбирает шаблон на основе:<br>        - заголовка Accept (например, text/html vs application/json);<br>        - параметра запроса (format=pdf).<br>    - Позволяет возвращать разные форматы (HTML, JSON, PDF) для одного URL.                                                                                          |
|                       | Интеграция с Spring-бинами<br>    <br>    - BeanNameViewResolver трактует имя представления как имя бина в контексте Spring.<br>    - Даёт возможность использовать кастомные View-компоненты, определённые как Spring‑бины.                                                                                                                                                                              |
|                       | Упрощение тестирования<br>    <br>    - Логические имена представлений легко подменять в тестах (например, для проверки ошибок).<br>    - Изоляция логики контроллера от деталей рендеринга.                                                                                                                                                                                                              |
|                       | Расширяемость<br>    <br>    - Можно создавать собственные реализации ViewResolver для нестандартных сценариев.<br>    - Переопределение поведения через наследование (например, кастомная логика поиска шаблонов).                                                                                                                                                                                       |
|                       | Согласованность обработки ошибок<br>    <br>    - Если ViewResolver не находит представление, он возвращает null, что позволяет другим ViewResolver в цепочке попытаться обработать запрос.<br>    - InternalResourceViewResolver всегда ставят последним в цепочке, так как он не может заранее проверить существование JSP.                                                                             |

### Как можно обработать исключения в приложении Spring MVC?

| Под темы                                                              | Заметка                                                                                                                                                                                                                                                                                 |
| --------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `@ExceptionHandler`  (локальная  обработка в  контроллере)            | Позволяет обработать исключение  **внутри конкретного контроллера**. **Особенности:**<br><br>- Работает только для исключений в этом контроллере.<br>- Можно обрабатывать несколько   типов исключений разными методами.<br>- Подходит для специфичных ошибок контроллера.              |
| `@ControllerAdvice`  (глобальная обработка)                           | Централизованная обработка исключений **для всех контроллеров** приложения. **Преимущества:**<br>- Единая точка обработки ошибок приложения.<br>- Возможность возвращать структурированные ответы (например, JSON с timestamp).<br>- Поддержка всех контроллеров без дублирования кода. |
| `@ResponseStatus`  (установка HTTP‑статуса)                           | Можно комбинировать с `@ExceptionHandler` для явного указания HTTP‑кода ответа.                                                                                                                                                                                                         |
| `ResponseEntity ExceptionHandler`  (расширенная глобальная обработка) | Базовый класс для **централизованной обработки стандартных исключений Spring MVC** (например, `MethodArgumentNotValidException`).                                                                                                                                                       |
| Обработка ошибок валидации (`@Valid` +  `BindingResult`)              | Для валидации DTO/моделей используйте `@Valid` и проверяйте `BindingResult`.                                                                                                                                                                                                            |
| Пользовательский  `ErrorController`  (кастомные страницы ошибок)      | Для обработки ошибок на уровне контейнера (например, 404) реализуйте `ErrorController`.                                                                                                                                                                                                 |
| Конфигурация через  `SimpleMappingExcept ionResolver`                 | Для XML‑конфигурации можно использовать `SimpleMappingExceptionResolver`                                                                                                                                                                                                                |

### Как реализовать международную поддержку (i18n) в приложении Spring MVC?

| Под темы                                                     | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Создание файлов  свойств  с локализованными  сообщениями     | Создайте в `src/main/resources` файлы вида:<br><br>- `messages.properties` — базовый (обычно английский);<br>- `messages_ru.properties` — русский;<br>- `messages_fr.properties` — французский и т. д.                                                                                                                                                                                                                    |
| Настройка  `MessageSource`  (источник сообщений)             | В конфигурационном классе определите bean `ResourceBundleMessageSource`                                                                                                                                                                                                                                                                                                                                                   |
| Настройка  `LocaleResolver`  (определение текущей локали)    | Выберите стратегию определения локали                                                                                                                                                                                                                                                                                                                                                                                     |
| Добавление  `LocaleChange Interceptor`  (переключение языка) | Чтобы пользователь мог менять язык через параметр запроса (например, `?lang=fr`), добавьте перехватчик                                                                                                                                                                                                                                                                                                                    |
| Использование  сообщений в контроллерах                      | Внедрите `MessageSource` и используйте `getMessage()`                                                                                                                                                                                                                                                                                                                                                                     |
| Использование  сообщений в представлениях                    |                                                                                                                                                                                                                                                                                                                                                                                                                           |
| Обработка отсутствующих переводов                            | - Если ключ не найден в текущем `messages_xx.properties`, Spring ищет его в базовом `messages.properties`.<br>- Если и там нет — возвращается ключ (если `setUseCodeAsDefaultMessage(true)`) или бросается исключение.                                                                                                                                                                                                    |
| Тестирование                                                 | 1. Запустите приложение.<br>    <br>2. Откройте страницу и проверьте переводы.<br>    <br>3. Смените язык:<br>    <br>    - через куки (`lang=ru`);<br>    - через заголовок `Accept-Language`;  <br>    - через параметр `?lang=fr`.<br>        <br>4. Убедитесь, что:<br>    <br>    - сообщения подгружаются корректно;<br>    - кодировка не ломается (особенно для кириллицы);<br>    - переключение языка работает. |

### Какие стратегии можно применить для оптимизации производительности веб-приложения на Spring MVC?

| Под темы                                     | Заметка                                                                                                                                                                                                                                                                                                                                                                        |
| -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Использование Virtual Threads                | Virtual Threads (доступны в Java 21+) позволяют снизить затраты на создание потоков и улучшить масштабируемость приложений, выполняющих блокирующие операции (например, запросы к базе данных или HTTP-запросы). Они легковесны и не блокируют основные потоки обработки запросов, что особенно полезно для Spring MVC-приложений на базе Servlet-контейнеров (Tomcat, Jetty). |
| Class Data Sharing (CDS)                     | CDS — функция JVM, которая позволяет предварительно загружать классы в общий архив, ускоряя запуск приложения и снижая потребление памяти. В Spring Boot 3.3+ поддержка CDS интегрирована, что упрощает её использование.                                                                                                                                                      |
| Кэширование                                  | Кэширование часто используемых данных или результатов вычислений снижает нагрузку на базу данных и ускоряет обработку запросов. В Spring можно использовать аннотацию `@Cacheable` для кэширования методов сервисов или результатов запросов.                                                                                                                                  |
| Оптимизация доступа к базе данных            | - **Пул соединений.** Используйте пул соединений (например, HikariCP), чтобы избежать затрат на создание новых соединений при каждом запросе. <br>- **Кэширование запросов.** Spring Data JPA поддерживает кэширование результатов запросов.<br>- **Оптимизация запросов.** Анализируйте и оптимизируйте SQL-запросы, используйте индексы, избегайте N+1 проблемы.             |
| Асинхронная обработка запросов               | Spring MVC поддерживает асинхронную обработку запросов, что позволяет выполнять длительные операции в отдельном потоке, не блокируя основной поток обработки запроса. Это особенно полезно для операций, которые могут замедлять обработку (например, вызовы внешних API).                                                                                                     |
| Оптимизация маппинга запросов                | Используйте конкретные маппинги запросов вместо общих, чтобы сократить время на сопоставление запросов с контроллерами. Например, вместо `@RequestMapping("/")` лучше использовать `@GetMapping("/specific-path")`.                                                                                                                                                            |
| Управление памятью                           | - Избегайте создания ненужных объектов, используйте пул объектов там, где это возможно.<br>- Мониторьте использование памяти и оптимизируйте его, чтобы предотвратить утечки памяти. <br>- Рассмотрите использование Spring AOT (Ahead-of-Time) оптимизаций, которые могут снизить нагрузку на JVM во время выполнения.                                                        |
| Выбор веб-сервера                            | По умолчанию в Spring Boot используется Tomcat, но можно попробовать другие серверы (например, Jetty или Undertow), чтобы сравнить производительность в конкретных условиях.                                                                                                                                                                                                   |
| Мониторинг и профилирование                  | Регулярно мониторьте и профилируйте приложение, чтобы выявлять узкие места. Используйте инструменты вроде VisualVM, YourKit, New Relic или Micrometer.                                                                                                                                                                                                                         |
| Ограничение области сканирования компонентов | Если не ограничивать область сканирования компонентов Spring, это может замедлять запуск приложения. Ограничьте сканирование только необходимыми пакетами.                                                                                                                                                                                                                     |

### Как в Spring MVC реализовать аутентификацию и авторизацию с использованием Spring Security?

| Под темы                            | Заметка                                                                                                                                                                                                                                                                                                                                       |
| ----------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                         | Для реализации аутентификации и авторизации  в  Spring MVC с использованием Spring Security  необходимо настроить конфигурацию  безопасности,  определить механизмы  аутентификации и правила  доступа к ресурсам. Spring Security предоставляет  гибкие инструменты для управления  доступом и защиты приложения от распространённых  угроз. |
| Добавление зависимостей             | Если используется Spring Boot, достаточно добавить  зависимость spring-boot-starter-security в pom.xml или build.gradle. Для обычного Spring MVC потребуется отдельно подключить  spring-security-web и spring-security-config.                                                                                                               |
| Настройка конфигурации безопасности | Создайте класс конфигурации, аннотированный   @Configuration и  @EnableWebSecurity.  В этом классе определяется  SecurityFilterChain, где задаются правила  безопасности.                                                                                                                                                                     |
| Аутентификация                      | Spring Security поддерживает различные механизмы  аутентификации, включая форму входа, базовую аутентификацию,  OAuth2 и другие.  Рассмотрим пример с формой входа.                                                                                                                                                                           |
| Авторизация                         | Авторизация определяет, какие ресурсы доступны  пользователю в зависимости от его ролей. В конфигурации HttpSecurity с помощью authorizeHttpRequests задаются правила доступа к URL-адресам.                                                                                                                                                  |
