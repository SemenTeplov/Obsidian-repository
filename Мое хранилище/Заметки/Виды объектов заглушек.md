### Что такое заглушка (stub) в контексте модульного тестирования?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                  |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Заглушка (stub) в контексте модульного тестирования** — это **упрощённая версия реального компонента, которая возвращает заранее определённые ответы**. Она не содержит логики или сложного поведения.                                                                                                 |
|             | Заглушки используются, когда тесту нужны какие-то данные от зависимости, но сама зависимость не является объектом проверки. Прямое использование внешних систем (баз данных, API, файловой системы) в юнит-тестах нежелательно, потому что это может быть медленно, ненадёжно или сложно контролировать. |

### Какова основная цель использования заглушек в тестировании?

| Под темы  | Заметка                                                                                                                                                                                                                                                                            |
| --------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ     | Основная цель использования заглушек (тестовых двойников) в тестировании — упростить тестирование, заменив реальные компоненты системы упрощёнными имитациями.                                                                                                                     |
| Позволяет | **Изолировать тестируемый код от внешних зависимостей**. Например, от реальных баз данных, внешних сервисов или действий пользователей. Прямое использование этих зависимостей в тестах нежелательно, потому что они могут быть медленными, ненадёжными или сложно контролировать. |
|           | **Проверить работу системы без необходимости запускать всё «по-настоящему»**. Заглушки помогают проверить работу системы, имитируя поведение реального компонента, на тех входных данных, которые они могут получить в ходе тестирования.                                          |
|           | **Заставить тестируемую систему выполнить действия, которые она не выполнила бы в обычных условиях**. Это позволяет тесту переключить тестируемую систему на ветвь кода, не выполняемую в обычной ситуации.                                                                        |

### В чем разница между заглушками и макетами (mocks)?

| Под темы         | Заметка                                                                                                                                                                                                                                                                                                                   |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Заглушки (Stubs) | **Заглушка** — это простой объект, который предоставляет предопределённые («зашитые») ответы на вызовы методов во время теста. Обычно не проверяет вызовы, а только имитирует поведение.                                                                                                                                  |
|                  | **Использование**: заглушки применяются, когда тесту нужны какие-то данные от зависимости, но сама зависимость не является объектом проверки.                                                                                                                                                                             |
| Макет (Mocks)    | **Макет** — это более «умный» объект, который не только предоставляет ответы (как заглушки), но и позволяет проверять, как код взаимодействовал с зависимостью. Можно настроить ожидания (например, какой метод должен быть вызван, с какими аргументами, сколько раз) и затем проверить, были ли эти ожидания выполнены. |
|                  | **Использование**: макеты применяются, когда важно проверить, как код использует зависимость.                                                                                                                                                                                                                             |

### Какие есть типы заглушек и в чем их основные отличия?

| Под темы        | Заметка                                                                                                                                                                                                                       |
| --------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ           | **В Java для создания заглушек (фиктивных объектов) используются разные типы**: заглушки (Stub) и макеты (Mock). Также для работы с заглушками в Java существуют фреймворки, например Spock, и библиотеки, например WireMock. |
| Stub (заглушка) | **Простая заглушка**, которая возвращает заранее определённые ответы, но не содержит логики или сложного поведения.                                                                                                           |
| Mock (макет)    | **Объект, который имитирует поведение реального объекта**. Благодаря ему можно задать заранее предопределённый ответ на конкретные вызовы.                                                                                    |
| Spock           | **Фреймворк для создания заглушек и моков** для тестирования взаимодействия между компонентами.                                                                                                                               |
| WireMock        | Библиотека для создания заглушек и макетирования веб-сервисов.                                                                                                                                                                |

### Как заглушка может быть использована для имитации исключений в тестах?

| Под темы  | Заметка                                                                                                                                                                                                                          |
| --------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ     | **Заглушка (stub) может быть использована для имитации исключений в тестах** — это позволяет проверить, как система обрабатывает ошибочные ситуации (например, недоступность сервиса, ошибку базы данных или неправильный ввод). |
| Mockito   | **Mockito** позволяет имитировать исключения с помощью методов when и thenThrow. Например, можно указать, что метод должен выбрасывать исключение при определённых условиях.                                                     |
| PowerMock | **PowerMock** позволяет имитировать исключения, например, для статических методов.                                                                                                                                               |

### В каких случаях предпочтительнее использовать заглушки, а не реальные объекты в тестировании?

| Под темы | Заметка                                                                                                                                                                                                                                                                      |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Заглушки (Stubs) и моки (Mocks) в тестировании предпочтительнее использовать в случаях, когда нужно изолировать тестируемый код от внешних зависимостей**. Это позволяет упростить тестирование, проверить работу системы без необходимости запускать всё «по-настоящему». |
| Stubs    | **Используются, когда тесту нужны какие-то данные от зависимости**, но сама зависимость не является объектом проверки.                                                                                                                                                       |
| Mocks    | Не только предоставляют ответы (как заглушки), но и позволяют проверять, как код взаимодействовал с зависимостью.                                                                                                                                                            |

### Как реализовать параметризированную заглушку с использованием фреймворка для тестирования на Java?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                               |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Параметризированную заглушку (mock-объект) можно реализовать с помощью фреймворков для тестирования на Java**, например Mockito или JUnit 5. Эти инструменты позволяют имитировать поведение объекта, задавая ожидаемые ответы на вызовы методов, и запускать один и тот же тест с разными входными данными.                                                        |
| Mockito  | **Mockito** позволяет создать mock-объект любого класса, для которого сразу после создания характерно поведение по умолчанию (все методы возвращают заранее известные значения). Можно переопределить это поведение желаемым образом, в результате mock становится заглушкой с требуемыми свойствами.                                                                 |
| JUnit 5  | **JUnit 5** предлагает возможности для работы с параметризованными тестами, которые позволяют запускать один и тот же тестовый метод с разными входными значениями. **Основная аннотация** для параметризованных тестов — @ParameterizedTest. Она заменяет стандартную @Test и указывает, что метод может запускаться несколько раз с различными наборами параметров. |

### Можете ли вы привести пример использования заглушек для тестирования взаимодействия между слоями приложения?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                       |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ          | **Заглушки (stubs) можно использовать для тестирования взаимодействия между слоями приложения**. Например, при модульном тестировании сервиса, который зависит от репозитория (доступа к данным), вместо реального обращения к реальной базе данных используют заглушку для репозитория. Это позволяет тестировать сервис изолированно, проверяя взаимодействие между слоями. |
| Принцип работы | Заглушка — это объект, который подменяет зависимость тестируемого класса и возвращает заранее определённые данные. Обычно не проверяет вызовы, а только имитирует поведение. Заглушки не пытаются повторить логику партнёра по взаимодействию, а дают тестируемому коду контракт (чёткий набор ответов) и позволяют сосредоточиться на собственной бизнес-логике.             |

### Какие принципы следует учитывать при проектировании заглушек для обеспечения их эффективного использования?

| Под темы  | Заметка                                                                                                                                                                                                                                                                                                   |
| --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ     | **При проектировании заглушек (макетов) в Java для эффективного использования следует учитывать принципы изоляции, минимальности, простоты и читаемости, определённости поведения, не нарушения логики теста и настройки**. Также важно использовать специализированные библиотеки для создания заглушек. |
| Изоляция  | Заглушки должны **изолировать тестируемый код от внешних зависимостей** (БД, сети, API и т. п.). Это гарантирует, что тесты не зависят от внешнего состояния и стабильны.                                                                                                                                 |
| Поведение | **Минимальность** — заглушка должна реализовывать только необходимое поведение для текущего теста. Не следует добавлять лишнюю логику — это усложняет сопровождение и понимание.                                                                                                                          |
|           | **Определённость** — поведение заглушки должно быть предсказуемым и соответствовать сценарию теста.                                                                                                                                                                                                       |
|           | **Не нарушать логику теста** — заглушка не должна влиять на бизнес-логику, которую проверяют. Её задача — имитировать поведение внешнего компонента, а не быть частью логики теста.                                                                                                                       |
|           | **Повторное использование** — хорошая практика — выносить заглушки в общие методы или базовые классы, если они многократно используются.                                                                                                                                                                  |
|           | **Контроль взаимодействий** — использование mock-заглушек позволяет проверять, был ли вызван метод, сколько раз и с какими аргументами (например, с помощью метода verify() в Mockito).                                                                                                                   |
| Настройка | **Инициализировать заглушки** — например, с помощью аннотации @Mock. Забывание проинициализировать заглушки приводит к ошибкам выполнения тестов.                                                                                                                                                         |
|           | **Избегать избыточности** — создание объектов-заглушек для элементов, не влияющих на тест, усложняет тесты и снижает их читаемость.                                                                                                                                                                       |
|           | **Не использовать заглушки для всех зависимостей** — даже если они не участвуют в тесте, это усложняет код и увеличивает время выполнения тестов.                                                                                                                                                         |
