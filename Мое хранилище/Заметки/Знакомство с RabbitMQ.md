### Что такое RabbitMQ?

| Под темы    | Заметка                                                                                                                                                                                                  |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **RabbitMQ** — **программный брокер сообщений** с открытым исходным кодом, написанный на языке Erlang. Предназначен для обмена сообщениями между компонентами приложений.                                |
| Особенности | Поддерживает несколько протоколов, включая AMQP (Advanced Message Queuing Protocol), MQTT, STOMP.                                                                                                        |
|             | Гарантирует доставку сообщений получателю с помощью механизма подтверждения доставки (Acknowledgements). Если получатель не подтверждает получение сообщения, RabbitMQ автоматически повторяет отправку. |
|             | Поддерживает горизонтальное масштабирование для построения кластерных решений.                                                                                                                           |

### Для чего используется RabbitMQ?

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                                                                                                 |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ                  | **RabbitMQ используется для асинхронной обработки сообщений** между компонентами приложений.                                                                                                                                                                                                                                                                                            |
| Позволяет              | **Обмениваться данными и командами** без прямого взаимодействия друг с другом.                                                                                                                                                                                                                                                                                                          |
|                        | **Асинхронно обрабатывать большие объёмы данных** — это полезно для задач, требующих значительных вычислительных ресурсов или времени на обработку.                                                                                                                                                                                                                                     |
|                        | **Обеспечивать отказоустойчивость системы** — в случае сбоя одного из компонентов RabbitMQ сохраняет сообщения и передаёт их, как только проблемный компонент вновь сможет нормально работать.                                                                                                                                                                                          |
| Сценарии использования | **Интеграция приложений и микросервисов** — RabbitMQ служит посредником, позволяя им обмениваться данными и командами. Например, в крупной электронной коммерции платформе каждый сервис (например, корзина покупок, обработка заказов, учёт пользователей) может отправлять события через RabbitMQ, а другие сервисы подписываются на эти события и реагируют соответствующим образом. |
|                        | **Обработка больших объёмов данных** — RabbitMQ помогает организовать конвейеры обработки данных. Например, система может получать данные из множества источников (сенсоры, веб-запросы, логи), которые затем отправляются в RabbitMQ. Далее рабочие процессы могут забирать эти сообщения и обрабатывать их параллельно.                                                               |
|                        | **Асинхронные задачи** — в системах с высокой нагрузкой часто возникает необходимость выполнения задач асинхронно, чтобы избежать блокировки основного потока приложения. Например, при обработке пользовательских запросов в реальном времени можно отправить задачу на обработку в очередь RabbitMQ, а затем обработать её позже в фоновом режиме.                                    |
|                        | **Балансировка нагрузки** — RabbitMQ может использоваться для балансировки нагрузки между несколькими серверами или рабочими процессами. Например, если одна часть системы перегружена запросами, RabbitMQ может перенаправлять сообщения на менее загруженные узлы, тем самым распределяя нагрузку более равномерно.                                                                   |

### Какие основные принципы работы RabbitMQ вы знаете?

| Под темы         | Заметка                                                                                                                                                                                                                                                                                                                |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ            | **Основные принципы работы RabbitMQ** включают использование протокола AMQP, гибкую маршрутизацию сообщений, механизм доставки и масштабируемость системы.                                                                                                                                                             |
| Протокол         | RabbitMQ реализует **протокол AMQP (Advanced Message Queuing Protocol)** — открытый протокол прикладного уровня для передачи сообщений между компонентами системы. AMQP поддерживает различные механизмы маршрутизации и управления очередями, позволяя создавать гибкие архитектуры для взаимодействия микросервисов. |
| Маршрутизация    | В RabbitMQ есть понятие **обменников (exchange)** — точек входа для сообщений. Обменник по правилам распределяет входящие сообщения по очередям.                                                                                                                                                                       |
| Масштабируемость | RabbitMQ позволяет **масштабировать систему горизонтально**.                                                                                                                                                                                                                                                           |

### Какие типы обменов сообщениями поддерживает RabbitMQ?

| Под темы                | Заметка                                                                                                                                                                                                                                                          |
| ----------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ                   | **RabbitMQ поддерживает несколько типов обменов сообщениями** (exchanges). Они определяют правила маршрутизации сообщений от обмена к очередям.                                                                                                                  |
| Direct (прямой)         | **Доставляет сообщения в очереди**, чей binding_key точно совпадает с routing_key сообщения. Это простой механизм маршрутизации, часто используемый для связи «точка-точка» или когда сообщения должны доставляться в определённые, известные пункты назначения. |
| Особенности             | Если несколько очередей привязаны к прямому обмену с одинаковым binding_key, обмен будет распределять сообщения с соответствующим routing_key между всеми ними.                                                                                                  |
|                         | Сообщения с любым другим routing_key будут отброшены (если только не привязана очередь для перехвата всех сообщений).                                                                                                                                            |
| Fanout (веер)           | **Широковещательно рассылает все полученные сообщения всем привязанным к нему очередям**, независимо от routing_key сообщения. Предоставляемый производителем routing_key просто игнорируется.                                                                   |
| Особенности             | Сообщения доставляются во все привязанные очереди, что обеспечивает параллельную обработку.                                                                                                                                                                      |
|                         | Полезен для широковещательной рассылки (Broadcast Messaging), уведомлений, чат-приложений и обновлений в реальном времени.                                                                                                                                       |
| Topic (тематический)    | **Маршрутизирует сообщения на основе шаблонов ключей маршрутизации**. Вместо фиксированного ключа маршрутизации здесь используются символы подстановки.                                                                                                          |
| Особенности             | Маршрутизация выполняется в соответствии с шаблоном, указанным привязкой очереди.                                                                                                                                                                                |
|                         | Можно использовать два формата подстановки: # (hash) — соответствует нулю или более слов, * (star) — соответствует одному слову.                                                                                                                                 |
|                         | Сообщения, соответствующие критериям, отправляются в соответствующую очередь.                                                                                                                                                                                    |
| Headers (с заголовками) | **Позволяет определить критерии маршрутизации с использованием заголовков сообщений** вместо ключа маршрутизации. Заголовки — пары «ключ-значение», которые обмен использует для оценки и сопоставления с критериями привязки, указанными очередями.             |
| Особенности             | При определении привязки очереди к обмену можно указать заголовки, которые нужно сопоставлять, и whether все или any из указанных критериев должны быть выполнены.                                                                                               |
|                         | Есть возможность использовать различные техники фильтрации.                                                                                                                                                                                                      |

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                   |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Механизм обработки сообщений на стороне потребителя в RabbitMQ реализуется через подтверждение (Acknowledgement) и отклонение (Reject/Nack)**. RabbitMQ — брокер сообщений, который обеспечивает асинхронный обмен данными между компонентами распределённой системы, и действует как посредник, маршрутизируя сообщения от производителей (отправителей) к потребителям (получателям). |

### В чем разница между точкой-точка (point-to-point) и публикацией/подпиской (publish/subscribe) в контексте RabbitMQ?

| Под темы          | Заметка                                                                                                                                                                                                                                                                                    |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ             | Разница между паттернами «точка-точка» (point-to-point) и «публикация/подписка» (publish/subscribe) в контексте RabbitMQ заключается в том, что Point-to-Point предназначен для доставки сообщения одному получателю, а Publish/Subscribe — для рассылки сообщения нескольким подписчикам. |
| Point-to-Point    | **Суть**: один отправитель ставит сообщения в очередь брокера, откуда только один получатель их забирает. Каждое сообщение потребляется единожды одним получателем.                                                                                                                        |
| Publish/Subscribe | **Суть**: издатель публикует сообщение в определённую тему (топик), и каждый подписчик этой темы получает копию сообщения. Здесь продюсер не отправляет данные конкретному получателю, а скорее объявляет их в «эфир» на определённую тему.                                                |

### Как настроить масштабируемость и надежность в RabbitMQ с использованием кластеров?

| Под темы        | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                        |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Ответ           | **В RabbitMQ масштабируемость и надёжность можно настроить с использованием кластеров** — группы взаимосвязанных узлов, которые совместно используют конфигурационную информацию, очереди, обменники и привязки. Это позволяет распределить нагрузку между несколькими брокерами сообщений и обеспечить высокую доступность (HA).                                                                                              |
| Масштабирование | **Распределение нагрузки**. Узлы кластера разделяют общее состояние и ресурсы, что позволяет распределять рабочую нагрузку. Клиенты могут подключаться к любому узлу в кластере, а сообщения могут маршрутизироваться в очереди, расположенные на разных узлах.                                                                                                                                                                |
|                 | **Управление масштабированием**. При необходимости можно добавлять или убирать потребителей и очереди, чтобы адаптировать систему к изменяющимся нагрузкам.                                                                                                                                                                                                                                                                    |
|                 | **Использование типов узлов**. Например, в производственном кластере рекомендуется поддерживать большинство узлов дисковыми, чтобы обеспечить стабильную синхронизацию конфигурации и надёжное хранение сообщений.                                                                                                                                                                                                             |
| Надежность      | **Репликация данных сообщений и конфигурации** на нескольких узлах. Это гарантирует, что если узел становится недоступным, другой узел может взять на себя его обязанности.                                                                                                                                                                                                                                                    |
|                 | **Настройка зеркалирования очередей** с помощью политик. Политики — это правила, которые сопоставляют очереди по имени и применяют к ним набор аргументов. Например, можно указать, что очередь зеркалируется на всех узлах кластера, на указанное число узлов или на указанном списке узлов.                                                                                                                                  |
|                 | **Использование очередей-кворумов (Quorum Queues)**. Они используют алгоритм консенсуса Raft для обеспечения строгой согласованности и высокой доступности. Сообщения реплицируются только на узлы, обозначенные как участники очереди (реплики), до тех пор, пока кворум (большинство реплик) не подтвердит получение. Если выходит из строя меньшинство реплик, очередь остаётся доступной, пока большинство может общаться. |
|                 | **Настройка долговечности сообщений**.                                                                                                                                                                                                                                                                                                                                                                                         |
|                 | **Мониторинг и оповещение**. Нужно внедрить надёжный мониторинг для состояния узлов, метрик RabbitMQ и состояния кластера. Можно настроить оповещения о критических порогах (например, длина очереди превышает лимит, узел офлайн, высокая загрузка ЦП) для обеспечения быстрого реагирования на потенциальные проблемы.                                                                                                       |

### Какие стратегии обработки ошибок при работе с сообщениями вы можете применить в RabbitMQ?

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ                  | **В RabbitMQ можно применить стратегии обработки ошибок при работе с сообщениями на разных уровнях**: на уровне конфигурации, на уровне кода и с помощью мониторинга.                                                                                                                                                                                                                                                                          |
| На уровне конфигурации | **Использование Dead Letter Exchange (DLX)**. Повторно недоставленные сообщения перенаправляются в другую очередь для анализа. У очереди, куда приходят «умершие» сообщения, можно указать время их жизни (message-ttl), а свойства dead-letter-exchange и dead-letter-routing-key задать так, чтобы сообщения возвращались в исходную очередь. В результате сообщения автоматически отправляются на повторную обработку с заданной задержкой. |
|                        | **Переотправка через DLX с TTL (Time-To-Live)**. Сообщения с установленным TTL помещаются обратно в основную очередь после истечения срока.                                                                                                                                                                                                                                                                                                    |
|                        | **Указание alternate exchange**. Можно указать у исходного обменника, чтобы сообщения, которые не удалось маршрутизировать, направлялись в него, а не в /dev/null.                                                                                                                                                                                                                                                                             |
|                        | **Защита от потери сообщений**. Например, можно попросить брокер удалять сообщения только после получения подтверждения от потребителя. В AMQP режим подтверждений включается свойством no-ack=false в basic.consume.                                                                                                                                                                                                                          |
| На уровне кода         | **Реализация retry-логики** для повторной отправки сообщений при временных ошибках. Это позволяет потребителям повторять обработку неудачных сообщений перед отправкой их в DLX.                                                                                                                                                                                                                                                               |
|                        | **Использование методов basic_nack и basic_reject** для отклонения сообщений.                                                                                                                                                                                                                                                                                                                                                                  |
|                        | **Настройка глобального обработчика ошибок** — это позволяет обрабатывать исключения в любом потребителе. Например, можно определить пользовательский ErrorHandler, который будет повторно отправлять в очередь только определённые исключения.                                                                                                                                                                                                |
| С помощью мониторинга  | **Отслеживание ключевых метрик**.                                                                                                                                                                                                                                                                                                                                                                                                              |
|                        | **Использование плагина управления RabbitMQ (Management Plugin)** — веб-плагин предоставляет представление о состоянии очереди в реальном времени.                                                                                                                                                                                                                                                                                             |
|                        | **Настройка пороговых значений оповещения** — например, на основе абсолютной глубины очереди (messages_ready > X) и устойчиво высокой скорости публикации.                                                                                                                                                                                                                                                                                     |
|                        | **Автоматизация масштабирования** — если это возможно, метрики мониторинга (например, messages_ready) можно привязать к механизму масштабирования потребителей, чтобы автоматически увеличивать количество потребителей при начале формирования отставания.                                                                                                                                                                                    |

### Какие практики лучше всего подходят для обеспечения высокой доступности сообщений в RabbitMQ?

| Под темы                      | Заметка                                                                                                                                                                                                                                                                                                        |
| ----------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ                         | **Для обеспечения высокой доступности сообщений в RabbitMQ подходят практики, связанные с кластеризацией, зеркалированием очередей, мониторингом и автоматическим восстановлением**. Эти подходы помогают системе оставаться функциональной даже при возникновении сбоев, минимизируя простои и потерю данных. |
| Кластеризация                 | **Кластеризация** позволяет распределить нагрузку и управление состоянием между несколькими узлами, что повышает доступность и масштабируемость.                                                                                                                                                               |
| Зеркалирование очередей       | **Зеркалирование очередей** (queue mirroring) — функция, которая позволяет копировать данные очереди на несколько узлов кластера.                                                                                                                                                                              |
|                               | **Настройка зеркалирования** осуществляется с помощью политик, а не параметров для каждой очереди. Это позволяет гибко централизованно управлять зеркалированием.                                                                                                                                              |
| Мониторинг                    | **Непрерывный мониторинг** состояния узлов, очередей и потоков данных помогает быстро обнаруживать проблемы и предотвратить их эскалацию.                                                                                                                                                                      |
| Автоматическое восстановление | **Системы могут быть настроены на автоматическое восстановление** после сбоев, например, перезапуск узлов или пересоздание очередей.                                                                                                                                                                           |
