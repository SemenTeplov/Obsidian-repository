### Что такое монолитная архитектура?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Монолитная архитектура** — это подход к разработке программного обеспечения, при котором все составляющие программы объединены в единый комплекс — монолит. Все функциональные элементы, включая логику бизнес-процессов, механизмы взаимодействия с базами данных, пользовательский интерфейс и API, интегрированы в рамках одного приложения.<br>Процессы разработки, внедрения и масштабирования осуществляются для всей системы как единого целого, без разделения на отдельные компоненты.<br>**Примеры монолитных приложений**: сайты-визитки, простые онлайн-магазины или небольшие внутренние корпоративные системы.                                                                            |
| Принцип     | Монолитная архитектура строится вокруг принципа единства — всё приложение существует как одна логическая единица. Некоторые принципы проектирования: <br>- **Модульность** — даже в рамках единой кодовой базы компоненты должны быть логически разделены.<br>- **Управление зависимостями** — несмотря на общую кодовую базу, зависимости между модулями должны быть минимизированы и направлены в одну сторону — от высокоуровневых модулей к низкоуровневым.<br>- **Централизованное управление версиями и релизами** — команда должна координировать изменения и придерживаться общего графика релизов, что может замедлять процесс доставки новых функций, но упрощает контроль целостности системы. |

### В чем основное преимущество монолитной архитектуры перед микросервисной?

| Под темы                   | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| -------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                | **Основное преимущество монолитной архитектуры перед микросервисной** заключается в **простоте разработки и развёртывания**. Это связано с тем, что в монолитной архитектуре все компоненты приложения (код, база данных, пользовательский интерфейс) объединены в единую неделимую единицу — монолит. В микросервисной архитектуре приложение разбивается на множество независимых компонентов — микросервисов, каждый из которых отвечает только за одну конкретную функцию и разрабатывается, тестируется и развёртывается изолированно от остальных частей системы.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| Монолитная архитектура     | **Преимущества**:<br>- **Простота разработки** — все компоненты находятся в одном репозитории, что упрощает работу команды.<br>- **Упрощённое развёртывание** — приложение разворачивается на одном сервере или виртуальной машине, что делает процесс выпуска, установки и активации ПО быстрым.<br>- **Простая коммуникация** — внутри монолита коммуникация между компонентами может происходить напрямую, без использования удалённого вызова процедур (RPC) или межпроцессного взаимодействия (IPC).<br>- **Вариативность масштабирования** — приложение на монолите можно наращивать и усложнять как горизонтальным путём (через добавление дополнительных ресурсов), так и вертикальным (улучшая производительность сервера и самого приложения).<br>- **Простота обновлений** — апгрейд программы на монолитной архитектуре может происходить легче, чем на микросервисной, ведь в первом случае потребуется обновить только одну кодовую базу, во втором — базу каждого микросервиса.<br>**Недостатки**:<br>- **Сложность масштабирования** — трудно масштабировать отдельные компоненты, что может приводить к избыточному использованию ресурсов.<br>- **Ограниченная гибкость** — изменения в одной части системы могут требовать модификаций в других частях, что снижает гибкость разработки.<br>- **Длительное развёртывание** — внедрение новых версий может занимать больше времени, так как нужно развернуть всё приложение целиком.                                                                                                                                                                                                                                                                                                           |
| Микросервисная архитектура | **Преимущества**:<br>- **Быстрая разработка** — каждая команда может работать над своим сервисом независимо от остальных, что позволяет выпускать обновления поэтапно, без остановки всего продукта, и быстрее тестировать гипотезы.<br>- **Гибкость технологий** — разные микросервисы можно писать на разных языках программирования и использовать подходящие базы данных или фреймворки.<br>- **Устойчивость к сбоям** — ошибка в одном сервисе не «роняет» всю систему: остальные продолжают работать. Это особенно ценно для сервисов с высокой нагрузкой — банков, маркетплейсов, онлайн-игр.<br>- **Масштабирование по частям** — вместо того чтобы увеличивать мощности всего приложения, можно масштабировать только нужные компоненты, например сервис оплаты в периоды пиковых продаж.<br>- **Прозрачность** — логирование, мониторинг и метрики позволяют точно видеть, где возникают проблемы и как ведёт себя каждый компонент, что облегчает поддержку и повышает надёжность системы.<br>**Недостатки**:<br>- **Сложность проектирования и управления** — микросервисная архитектура требует продуманного проектирования, грамотного разбиения системы и чётких интерфейсов.<br>- **Рост инфраструктурных расходов** — для каждой команды нужны отдельные среды, CI/CD-конвейеры, контейнеры, системы оркестрации и мониторинга, что увеличивает стоимость поддержки.<br>- **Сложность коммуникации между сервисами** — синхронизация данных и управление зависимостями становятся нетривиальной задачей.<br>- **Задержки и сетевые риски** — в отличие от монолита, где всё выполняется в рамках одного процесса, микросервисы обмениваются данными по сети, и любая нестабильность каналов или ошибка в API могут повлиять на скорость работы. |

### Почему монолитная архитектура может стать проблемой при масштабировании проекта?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Причины  | Некоторые проблемы монолитной архитектуры при масштабировании:<br>- **Проблемы с горизонтальным масштабированием** — добавлением дополнительных серверов для распределения нагрузки. Монолитные системы зачастую плохо поддаются этому виду масштабирования, поскольку разработаны для работы на одном сервере. Разделение такой системы на несколько серверов требует значительных усилий и может привести к сложностям с синхронизацией данных и операциями.<br>- **Проблемы с вертикальным масштабированием** — усилением уже существующего сервера за счёт увеличения его ресурсов (процессорная мощность, память и т. д.). Хотя это возможно для монолитных систем, оно имеет свои пределы: существует ограничение на то, насколько мощным может быть один сервер, а также это может быть дорогостоящим решением.<br>- **Нерациональное использование ресурсов** — если дополнительные ресурсы нужны только для определённых компонентов, это может вылиться в их нерациональное использование. Например, если есть сложный продукт с множеством опций, то вся система может начать работать медленнее, когда для работы одной из таких функций потребуются дополнительные ресурсы.<br>Эти ограничения масштабируемости особенно заметны в условиях быстрого роста или изменений в бизнесе. Компании, сталкивающиеся с резким увеличением числа пользователей или объёма данных, могут обнаружить, что их монолитная система не способна эффективно справиться с увеличенной нагрузкой, что может привести к снижению производительности и даже к простоям. |
| Решения  | Для преодоления проблем с масштабируемостью в монолитной архитектуре можно рассмотреть **переход на микросервисную архитектуру**. Она подразумевает разделение монолитного приложения на независимые сервисы, которые могут масштабироваться отдельно друг от друга. Это гибкий вариант, ведь можно усилить только те части, на которые ложится наибольшая нагрузка.<br>Также можно рассмотреть **стратегии перехода** от монолитной к микросервисной архитектуре, например:<br>- **«Стратегия большого взрыва»** — вся команда разработчиков одновременно прекращает поддержку монолитной архитектуры и начинает перестраивать её в микросервисную.<br>- **«Стратегия „Душитель“»** — создание микросервисной архитектуры параллельно с существованием монолита: разработчики заменяют старые части монолита новыми микросервисами и добавляют новые функции уже в виде микросервисов.<br>- **«Переход по функциям»** — идентификация ключевых бизнес-функций и постепенное выделение этих функций в микросервисы.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |

### Какие признаки указывают на то, что пора разделять монолит на микросервисы?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Проблемы | - **Трудности масштабирования**. В монолите все модули приложения связаны в единое целое и обычно разворачиваются как один сервис. Когда нагрузка возрастает, приходится масштабировать всё приложение сразу, даже если только один модуль подвергается особой нагрузке. Это приводит к избыточному потреблению ресурсов, усложняет процесс релизов и повышает риск отказов.<br>- **Усложнение релизного цикла**. При монолитной архитектуре любой незначительный апдейт требует пересборки и деплоя всего приложения. Это увеличивает время вывода новых фич на рынок и повышает вероятность ошибок: если что-то пошло не так, откатить систему к предыдущей версии может быть сложно, ведь весь функционал тесно связан.<br>- **Сложность кода и зависимости**. По мере роста системы кодовая база «разбухает»: возникает всё больше зависимостей между модулями, логика становится запутанной, увеличивается время на тестирование и отладку.<br>- **Сложности в обеспечении отказоустойчивости**. Для многих приложений критически важно поддержание высокой доступности и устойчивости к сбоям. В монолитной архитектуре отказ одного компонента системы может привести к выходу из строя всего приложения, что недопустимо для критически важных систем. |
| Критерии | Решение о переходе на микросервисы не должно быть импульсивным. Необходимо тщательно спланировать, какие именно части монолита будут преобразованы и в какой последовательности. Некоторые факторы, которые стоит учесть:<br>- **Размер проекта**. Микросервисы наиболее эффективны в масштабных приложениях, где сложность управления и разработки может быть снижена разделением функциональности на отдельные сервисы. Если проект маленький или средний по размеру, миграция на микросервисы может привести к ненужной сложности и издержкам.<br>- **Недостаток ресурсов**. Микросервисы требуют значительных затрат на сетап и поддержку. Если у команды недостаточно ресурсов (квалифицированных разработчиков, времени, денег), переход на микросервисы может оказаться рискованным.<br>- **Устойчивость текущей архитектуры**. Если текущий монолитный проект функционирует хорошо, стабилен и соответствует бизнес-требованиям, переход на микросервисы может быть ненужным и даже рискованным шагом.                                                                                                                                                                                                                                                 |

### Какие риски несет разделение монолитного приложения на микросервисы?

| Под темы                                       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ---------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Увеличение  сложности  системы                 | Управление множеством независимых сервисов сложнее, чем одним монолитом. Каждый микросервис имеет собственные зависимости, конфигурации и требования к развёртыванию. Это требует глубоких знаний о системе, тщательного определения границ ответственности сервисов и их взаимодействия.                                                                                                                                                                                                                                                                                                                   |
| Сложность координации между командами          | Если над разными микросервисами работают отдельные команды, это может привести к проблемам в коммуникации, управлении проектом и согласовании изменений. Особенно это актуально в крупных организациях. Требуется организация кросс-функциональных команд, которые будут отвечать за полный жизненный цикл микросервиса.                                                                                                                                                                                                                                                                                    |
| Изменение процессов разработки и развёртывания | Переход на микросервисы требует радикальных изменений в процессах разработки, тестирования и развёртывания. Традиционные подходы часто оказываются неэффективными. Необходимо внедрение CI/CD для автоматизации процессов, инструментов для автоматического тестирования и мониторинга микросервисов.                                                                                                                                                                                                                                                                                                       |
| Проблемы с коммуникацией и интеграцией         | Микросервисы взаимодействуют друг с другом по сети, что может приводить к задержкам, проблемам с безопасностью и надёжностью. Неправильная организация коммуникации между сервисами может привести к потере их автономии и нивелированию преимуществ микросервисной архитектуры. Если сервисы слишком детализированы, между ними возрастает трафик, что может снизить производительность системы.                                                                                                                                                                                                           |
| Управление данными и согласованность           | При переходе на микросервисы каждый сервис обычно имеет независимое хранилище данных. Это создаёт сложности с разделением данных монолитного приложения на фрагменты, подходящими для отдельных микросервисов. Требуется углублённый анализ и понимание границ домена. Обеспечение согласованности данных между различными микросервисами, особенно при работе с распределёнными транзакциями, становится сложной задачей. Разработчикам приходится использовать такие стратегии, как архитектура, управляемая событиями, или шаблон Saga.                                                                  |
| Увеличение числа точек отказа                  | В микросервисной архитектуре больше компонентов, которые могут выйти из строя. Если один микросервис перестаёт отвечать на запросы, другие продолжают работать, но такие ситуации требуют специальной обработки. В распределённой системе неизбежно возникают ситуации, когда сервис оказывается под большой нагрузкой и не может своевременно ответить. Требуется внедрение механизмов отказоустойчивости, например, шаблона «прерыватель цепи» (circuit breaker), который быстро обрабатывает сбои и может обеспечить откат с возвратом данных по умолчанию вместо ожидания ответа от зависимого сервиса. |
| Высокие первоначальные затраты                 | Переход на микросервисы требует значительных начальных инвестиций. Для каждого микросервиса может потребоваться отдельная команда разработчиков. Необходимо наладить процессы автоматического тестирования и развёртывания (CI/CD), использовать инструменты оркестрации (например, Kubernetes), нанять инженеров DevOps.                                                                                                                                                                                                                                                                                   |
| Сложность тестирования                         | Хотя тестировать отдельные микросервисы проще, чем монолит, по мере масштабирования системы общая сложность тестирования возрастает. Требуется проверять не только каждый сервис в отдельности, но и их взаимодействие. Интеграционное тестирование микросервисов может быть сложнее, чем в монолитной архитектуре.                                                                                                                                                                                                                                                                                         |
| Риски при декомпозиции монолита                | Неправильная декомпозиция монолита на микросервисы может привести к неэффективной миграции. Например, чрезмерно дробные сервисы усложняют понимание, управление и отладку системы. Также функционально узкий сервис может быть очень ресурсозатратным.                                                                                                                                                                                                                                                                                                                                                      |
| Необходимость в квалифицированных специалистах | Для успешной работы с микросервисами требуются специалисты, владеющие навыками работы с распределёнными системами, контейнеризацией, оркестрацией, мониторингом и другими технологиями. Это может стать проблемой, если в команде нет таких специалистов. <br>Чтобы минимизировать риски, важно тщательно планировать переход на микросервисы, проводить обучение команды, использовать подходящие инструменты автоматизации и мониторинга, а также постепенно внедрять изменения, например, с помощью паттернов вроде Strangler Fig или Parallel Run.                                                      |

### Какие преимущества микросервисной архитектуры можно использовать для обоснования ее внедрения вместо монолита?

| Под темы         | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Масштабируемость | - **Возможность точечного масштабирования**. В отличие от монолитных приложений, где требуется масштабировать всю систему целиком, микросервисы позволяют увеличивать мощности только там, где это действительно необходимо. Например, во время распродаж можно масштабировать только сервис обработки заказов, не затрагивая остальные компоненты системы.<br>- **Поддержка горизонтального масштабирования**. Микросервисы позволяют увеличивать производительность за счёт запуска дополнительных копий только нужных сервисов. Например, если нагрузка растёт на модуль обработки платежей, администратор может увеличить количество инстансов только этого сервиса, не затрагивая остальные части приложения.<br>- **Независимое развёртывание**. Обновление одного микросервиса не требует остановки всей системы.                                                                                                                                                  |
| Гибкость         | - **Возможность развивать микросервисы независимо друг от друга**. Например, новый функционал приложения можно добавлять без изменения сервиса, отвечающего за управление библиотекой треков.<br>- **Технологическая гибкость**. Каждый микросервис может быть реализован с использованием наиболее подходящего для его задач технологического стека. Это даёт возможность постепенно вводить новые технологии и экспериментировать с ними без риска для всей системы.<br>- **Ускорение процессов разработки**. Микросервисная архитектура позволяет разным командам работать над различными компонентами системы независимо друг от друга, что ускоряет процесс разработки и внедрения новых функций. Каждая команда может придерживаться своего темпа разработки и графика релизов, не влияя на работу других команд.                                                                                                                                                   |
| Устойчивость     | - **Изолированность компонентов**. Сбой в работе одного микросервиса не приводит к отказу всей системы. Если, например, временно недоступен сервис рекомендаций, основной функционал интернет-магазина продолжит работать. Это достигается за счёт грамотного проектирования системы с учётом возможных сбоев и применения паттернов отказоустойчивости, таких как Circuit Breaker.<br>- **Прозрачность**. Логирование, мониторинг и метрики позволяют точно видеть, где возникают проблемы и как ведёт себя каждый компонент. Это облегчает поддержку и повышает надёжность системы.<br>Важно отметить, что переход от монолита к микросервисам должен быть постепенным и обоснованным. Преждевременное введение микросервисной архитектуры может создать больше проблем, чем решит. Часто оптимальный подход — **гибридный**: начать с хорошо структурированного модульного монолита, но спроектировать его так, чтобы в будущем можно было выделить отдельные сервисы. |
| Тестируемость    | - **Возможность вносить изменения и тестировать их изолированно**. Это упрощает управление кодом и уменьшает риски ошибок.<br>- **Упрощённое внедрение обновлений** без остановки всей системы. Поскольку каждый микросервис разрабатывается и развёртывается независимо, обновления могут происходить частично, без остановки всего приложения. Это снижает риски и позволяет внедрять новые функции или исправлять ошибки более оперативно.<br>Однако есть и недостатки микросервисной архитектуры, например, сложности тестирования, которые требуют иного подхода по сравнению с монолитными приложениями. Помимо модульных тестов для каждого сервиса, необходимо обеспечить качественное интеграционное тестирование, которое проверяет корректность взаимодействия между сервисами.                                                                                                                                                                                |

### Какие стратегии разделения монолитного приложения на микросервисы вы знаете?

| Под темы            | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Основные  стратегии | **Strangler Fig (паттерн «душителя»)**<br>- **Суть**: постепенное замещение  функционала монолита новыми микросервисами.<br>- **Как работает**:<br>    - вокруг монолита разворачиваются новые сервисы;<br>    - трафик для отдельных функций перенаправляется  на микросервисы через API‑шлюз;<br>    - монолит постепенно «сжимается»,  пока не остаётся только вспомогательный код.<br>- **Плюсы**: минимальный риск, нет необходимости  в единовременном переходе.<br>- **Минусы**: длительное сосуществование двух  архитектур, временный рост сложности.<br>- **Когда применять**: при необходимости плавного  перехода  без остановки работы системы. |
|                     | **Decomposition by Business Capability (по бизнес‑возможностям)**<br>- **Суть**: разбиение монолита в соответствии  с бизнес‑доменами (платежи, каталог, пользователи и т. д.).<br>- **Как работает**: каждый микросервис реализует одну  бизнес‑функцию и владеет своими данными.<br>- **Плюсы**: чёткое соответствие бизнес‑процессам,  простота поддержки.<br>- **Минусы**: требует глубокого понимания предметной области.<br>- **Пример**: сервис заказов, сервис платежей, сервис уведомлений.                                                                                                                                                         |
|                     | **Decomposition by Subdomain (по поддоменам DDD)**<br>- **Суть**: использование подхода Domain‑Driven  Design (DDD) для выделения микросервисов на основе поддоменов.<br>- **Как работает**:<br>    - определяются основные и вспомогательные поддомены;<br>    - для каждого поддомена создаётся отдельный сервис.<br>- **Плюсы**: минимизация зависимостей, высокая  автономность сервисов.<br>- **Минусы**: сложность моделирования доменной области.                                                                                                                                                                                                     |
|                     | **Decomposition by Data (по данным)**<br>- **Суть**: разделение монолита на основе источников данных.<br>- **Как работает**: каждый сервис получает собственную  БД или доступ к выделенному фрагменту данных.<br>- **Плюсы**: изоляция данных, упрощение  масштабирования хранилищ.<br>- **Минусы**: риск дублирования данных,  сложности с согласованностью.<br>- **Пример**: сервис пользователей с собственной БД,  сервис продуктов с отдельной БД.                                                                                                                                                                                                     |
|                     | **Decomposition by UI/User Journey (по пользовательским сценариям)**<br>- **Суть**: выделение микросервисов для отдельных  пользовательских путей (например, оформление заказа, поиск товаров).<br>- **Как работает**: функционал, связанный с конкретным  сценарием, изолируется в отдельный сервис.<br>- **Плюсы**: удобство тестирования пользовательских сценариев.<br>- **Минусы**: возможна избыточность кода.                                                                                                                                                                                                                                         |
|                     | **Decomposition by Deployment Frequency (по частоте развёртывания)**<br>- **Суть**: группировка компонентов, которые обновляются с одинаковой частотой.<br>- **Как работает**: часто изменяемые модули  выделяются в отдельные сервисы, стабильные — остаются в монолите или объединяются.<br>- **Плюсы**: снижение риска сбоев при частых релизах.<br>- **Минусы**: не всегда соответствует бизнес‑логике.                                                                                                                                                                                                                                                  |
|                     | **Decomposition by Technical Boundary (по техническим границам)**<br>- **Суть**: разделение на основе технологических  различий (например, сервисы для машинного обучения, обработки файлов).<br>- **Как работает**: компоненты с особыми требованиями  (к ресурсам, стеку технологий) выносятся в отдельные сервисы.<br>- **Плюсы**: возможность использовать специализированные технологии.<br>- **Минусы**: рост числа сервисов без явной бизнес‑ценности.                                                                                                                                                                                                |
|                     | **Parallel Run (параллельный запуск)**<br>- **Суть**: временное дублирование функционала — новый  микросервис работает параллельно с монолитом.<br>- **Как работает**:<br>    - запросы обрабатываются и монолитом, и микросервисом;<br>    - результаты сравниваются для проверки корректности;<br>    - после валидации трафик переключается на микросервис.<br>- **Плюсы**: низкий риск ошибок при миграции.<br>- **Минусы**: высокие затраты на поддержку двух систем.                                                                                                                                                                                   |
|                     | **Branch By Abstraction (ветвление через абстракцию)**<br>- **Суть**: внедрение абстракций в монолит для постепенного выделения сервисов.<br>- **Как работает**:<br>    - в монолите создаются интерфейсы (абстракции) для взаимодействия с компонентами;<br>    - новые реализации подключаются через эти интерфейсы;<br>    - старые компоненты заменяются микросервисами без изменения API.<br>- **Плюсы**: плавный переход, минимум изменений в клиентском коде.<br>- **Минусы**: временная сложность архитектуры.                                                                                                                                       |
|                     | **Big Bang (полный переход за один этап)**<br>- **Суть**: единовременное разделение монолита на микросервисы.<br>- **Как работает**: система останавливается, код рефакторится, сервисы разворачиваются.<br>- **Плюсы**: быстрое достижение целевой архитектуры.<br>- **Минусы**: высокий риск сбоев, необходимость длительной подготовки.<br>- **Когда применять**: только для небольших монолитов с низкой критичностью.                                                                                                                                                                                                                                   |

### Какие технические и организационные аспекты необходимо учитывать при переходе от монолита к микросервисам?

| Под темы                | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Технические  аспекты    | **Декомпозиция монолита**. Необходимо правильно разделить монолит на автономные микросервисы с чётко определёнными функциональными обязанностями. Непродуманное разделение может привести к тому, что новые микросервисы будут либо слишком большими (и мало чем отличаться от монолита), либо слишком мелкими (с бесконечным числом пересечений). Для декомпозиции часто используют подходы, основанные на бизнес-возможностях (Business Capability), поддоменах DDD (Domain-Driven Design) или других критериях. |
|                         | **Взаимодействие между сервисами**. Требуется организовать коммуникацию между микросервисами. Для этого используют API Gateway (единая точка входа для внешних запросов), брокеры сообщений (Kafka, RabbitMQ) для событийно-ориентированной архитектуры, балансировщики нагрузки и другие инструменты. Важно избежать тесной связанности сервисов, чтобы сохранить их автономность.                                                                                                                                |
|                         | **Управление данными**. В микросервисной архитектуре каждый сервис может управлять своими данными и иметь отдельную базу данных. Это требует решения проблем согласованности данных между сервисами, особенно в условиях распределённой архитектуры. Могут потребоваться механизмы для синхронизации данных и реализации надёжных протоколов управления распределёнными транзакциями.                                                                                                                              |
|                         | **Оркестрация и развёртывание**. Микросервисы требуют управления множеством независимых компонентов. Для этого используют оркестраторы контейнеров (Kubernetes, Docker Swarm), CI/CD-пайплайны для автоматизации сборки, тестирования и развёртывания. Важно обеспечить независимость развёртывания сервисов.                                                                                                                                                                                                      |
|                         | **Мониторинг и логирование**. Распределённая система усложняет наблюдение за её состоянием. Рекомендуется использовать централизованные системы логирования (ELK Stack, Loki, Splunk), распределённый трейсинг (Jaeger, Zipkin) для отслеживания пути запроса через все сервисы, уникальные корреляционные ID для каждого запроса.                                                                                                                                                                                 |
|                         | **Безопасность**. Микросервисы обмениваются данными по сети, поэтому важно уделять внимание шифрованию трафика (TLS), аутентификации (OAuth2, JWT) и авторизации. Могут потребоваться специализированные решения для управления секретами (HashiCorp Vault, Consul).                                                                                                                                                                                                                                               |
|                         | **Обработка сбоев и отказоустойчивость**. В распределённой системе неизбежны ситуации, когда сервис может оказаться под большой нагрузкой или стать недоступным. Необходимо реализовать механизмы изоляции сбоев (например, circuit breaker), чтобы предотвратить каскадные отказы.                                                                                                                                                                                                                                |
| Организационные аспекты | **Стратегия и планирование**. Прежде чем начать процесс, необходимо тщательно спланировать, какие именно части монолита будут преобразованы и в какой последовательности. План должен учитывать технические, организационные и культурные аспекты изменений. Важно определить цели перехода, объём проекта, основные этапы, роли и обязанности в команде, технологический стек, требования к инфраструктуре, бюджет и распределение ресурсов.                                                                      |
|                         | **Обучение команды**. Переход требует повышения квалификации специалистов. Разработчикам необходимо освоить концепции, связанные с микросервисной архитектурой (например, шина данных, Outbox-паттерн), новые технологии и инструменты (контейнеры, оркестраторы, системы мониторинга). Команде нужно адаптироваться к изменению процессов разработки: разделению задач на более мелкие компоненты, использованию DevOps-практик.                                                                                  |
|                         | **Изменение структуры команд**. При микросервисном подходе каждая команда несёт полную ответственность за свой сервис (end-to-end), включая работу в проде. Это требует перехода к модели, где команды организованы вокруг сервисов, а не вокруг этапов разработки. Не все могут быть готовы к такой автономии, поэтому может потребоваться координация и согласование архитектурных решений.                                                                                                                      |
|                         | **Управление изменениями**. Необходимо разработать стратегию управления изменениями, включая планирование, тестирование и внедрение новых функций и возможностей. Важно обеспечить коммуникацию и сотрудничество между членами команды и другими участниками проекта.                                                                                                                                                                                                                                              |
|                         | **Культурная трансформация**. Переход на микросервисы требует изменения организационной культуры. Нужно сформировать подход «you build it, you run it» (ты разрабатываешь — ты и поддерживаешь), где команды полностью отвечают за жизненный цикл своих сервисов.                                                                                                                                                                                                                                                  |
|                         | **Согласование с бизнесом**. Решение о переходе не должно быть исключительной ответственностью ИТ-отдела. В процессе должны участвовать бизнес-подразделения и все заинтересованные стороны, так как миграция — сложный, дорогостоящий и долгосрочный процесс.                                                                                                                                                                                                                                                     |

### Какие паттерны проектирования микросервисов помогут минимизировать проблемы с зависимостями и коммуникацией между сервисами?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Паттерны | **Database per Service (база данных для каждого сервиса)**. Каждый микросервис имеет собственную базу данных и управляет только своими данными. Это обеспечивает изоляцию данных, снижает зависимости между сервисами и позволяет каждому сервису масштабироваться независимо. Доступ к данным одного сервиса другими сервисами возможен только через публичные API, что соблюдает принцип инкапсуляции. Однако этот паттерн усложняет реализацию глобальных транзакций и запросов, затрагивающих несколько сервисов.                  |
|          | **Service Discovery (обнаружение сервисов)**. Механизм, который позволяет микросервисам динамически находить друг друга. Сервисы регистрируются в центральном реестре (например, Netflix Eureka, Consul, Zookeeper), а при необходимости взаимодействия с другим сервисом запрашивают у реестра его текущий адрес. Это устраняет жёсткую привязку к конкретным адресам и упрощает управление в динамически изменяющейся среде.                                                                                                         |
|          | **API Gateway (API-шлюз)**. Единая точка входа для всех клиентских запросов. Шлюз принимает запросы, маршрутизирует их к соответствующим микросервисам, может выполнять аутентификацию, авторизацию, балансировку нагрузки и другие задачи. Это упрощает взаимодействие клиентов с системой, скрывает сложность внутренней архитектуры и централизует управление сквозной функциональностью. Однако шлюз может стать единой точкой отказа, поэтому требует тщательного мониторинга.                                                    |
|          | **Circuit Breaker (предохранитель)**. Паттерн, который предотвращает каскадные сбои в системе. Он мониторит взаимодействие между микросервисами и временно блокирует запросы к неисправному сервису, давая ему время на восстановление. Работает в трёх состояниях: closed (нормальный режим), open (блокировка запросов при превышении порога ошибок) и half-open (проверка возможности восстановления сервиса после тайм-аута). Это повышает устойчивость системы и помогает избежать полного отказа из-за проблем с одним сервисом. |
|          | **Bulkhead (отсек)**. Изолирует компоненты системы друг от друга, чтобы сбой в одной части не повлиял на другие. Ресурсы (пулы потоков, соединений и т. д.) разделяются на группы, каждая из которых выделяется для взаимодействия с конкретным удалённым сервисом. Если один сервис испытывает проблемы, это повлияет только на его «отсек», а остальные сервисы смогут продолжать работу.                                                                                                                                            |
|          | **Async Messaging (асинхронный обмен сообщениями)**. Вместо синхронного взаимодействия микросервисы используют очереди сообщений (например, Kafka, RabbitMQ) для асинхронной коммуникации. Отправитель публикует сообщение в очередь, а получатель обрабатывает его, когда сможет. Это разрывает временную связь между сервисами, снижает зависимость от мгновенного ответа и повышает масштабируемость системы.                                                                                                                       |
|          | **Saga (сага)**. Паттерн для управления длительными транзакциями, которые включают несколько этапов, выполняемых разными сервисами. Сага разбивает сложную бизнес-операцию на серию независимых этапов, каждый из которых обрабатывается отдельным микросервисом. Есть два подхода: хореографический (сервисы общаются через события) и оркестровый (есть центральный координатор — оркестратор). Это позволяет поддерживать согласованность данных в распределённой системе.                                                          |
|          | **Smart Endpoints, Dumb Pipes (умные конечные точки, глупые каналы)**. Бизнес-логика размещается в самих микросервисах (умные конечные точки), а инфраструктура коммуникаций (каналы) остаётся простой и занимается только маршрутизацией сообщений. Это упрощает систему и делает её более гибкой.                                                                                                                                                                                                                                    |
