### Что такое многоуровневая архитектура приложений?

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение    | **Многоуровневая (многослойная) архитектура приложений** — это модель организации программных систем, в которой функциональность разделяется на несколько логических уровней. Каждый уровень выполняет определённую роль и взаимодействует только с соседними уровнями, что обеспечивает чёткое разделение ответственности и упрощает поддержку системы.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| Структура      | Традиционная многоуровневая архитектура включает следующие уровни:<br>- **Уровень представления (Presentation Layer)** — интерфейс пользователя, отображение данных и обработка пользовательских действий.<br>- **Уровень приложения (Application Layer)** — координация действий, управление сеансами и рабочими потоками.<br>- **Уровень бизнес-логики (Business Layer)** — реализация бизнес-правил, обработка данных и бизнес-процессов.<br>- **Уровень доступа к данным (Data Access Layer)** — взаимодействие с базами данных и другими источниками информации.<br>- **Уровень данных (Data Layer)** — физическое хранение данных в базах данных, файловых системах.<br>Современные интерпретации многоуровневой архитектуры часто дополняются специализированными уровнями, такими как уровень API, уровень интеграции, уровень кэширования или уровень безопасности, в зависимости от требований конкретной системы.<br>**Ключевой принцип** — взаимодействие только между смежными уровнями. Например, уровень представления не должен напрямую обращаться к уровню данных, минуя промежуточные уровни. Такое ограничение обеспечивает высокую степень изоляции и возможность замены реализации отдельных уровней без влияния на остальную систему. |
| Принцип работы | Запрос может последовательно проходить все слои ПО сверху вниз (от слоя представления до слоя хранилища данных). После выполнения запроса формируется ответ. Ответ может проходить все слои ПО снизу вверх (от слоя хранилища данных до слоя представления), чтобы передать пользователю результат выполнения функции. Вместе с тем запрос и ответ могут пропускать некоторые слои ПО. Обязательность прохождения через слой определяется для каждого слоя по отдельности.<br>**Преимущества многоуровневой архитектуры**:<br>- чёткое разделение ответственности между компонентами системы;<br>- возможность независимой разработки и тестирования каждого уровня;<br>- гибкость в замене или модификации отдельных уровней без изменения всей системы;<br>- повышенная безопасность благодаря изоляции критических компонентов;<br>- возможность горизонтального масштабирования отдельных уровней под нагрузкой;<br>- упрощение поддержки и эволюции системы в долгосрочной перспективе.                                                                                                                                                                                                                                                                 |

### В чем заключается основная идея паттерна MVC (Model-View-Controller)?

| Под темы                | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение             | **Основная идея паттерна MVC (Model-View-Controller) заключается в том, что у каждого раздела кода есть своя цель**. Приложение разделено на три логических компонента: модель, представление и контроллер, и каждый из них имеет чётко определённую функцию.<br>Это позволяет:<br>- **Писать независимые друг от друга блоки кода** — их можно менять отдельно, не затрагивая остальные. Например, переписывать способ отображения данных, не трогая способ их обработки.<br>- **Структурировать код** и упростить тестирование и поддержку: с этим паттерном модернизировать можно отдельные блоки, а не всё приложение целиком.<br>- **Разделить обязанности**: пока один программист пишет обработчик данных, другой занят их отображением. Это экономит время: каждый работает над своей задачей, не нужно вникать в чужой код, а изменения в одном блоке не влияют на другой. |
| Модель (Model)          | **Отвечает за бизнес-логику приложения и управление данными**. Некоторые задачи модели:<br>- хранит данные и связанную с ними логику;<br>- закрепляет структуру приложения;<br>- реагирует на команды из контроллера и выдаёт информацию и/или изменяет своё состояние.<br>Модель не должна содержать логику взаимодействия с пользователем и не должна определять механизм обработки запроса.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| Представление (View)    | **Отвечает за визуализацию данных** и взаимодействие с пользователем. Некоторые функции представления:<br>- отображает данные на уровне пользовательского интерфейса, например, в виде таблицы или списка;<br>- определяет внешний вид приложения и способы взаимодействия с ним;<br>- перехватывает действия пользователя и передаёт их контроллеру.<br>Представление может обращаться к модели за данными и событиями, но не может её менять.                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| Контроллер (Controller) | **Служит посредником между моделью и представлением**. Некоторые задачи контроллера:<br>- принимает входные данные от пользователя из интерфейса;<br>- обрабатывает их с помощью бизнес-логики модели;<br>- а затем передаёт данные в представление.<br>Контроллер определяет, как приложение будет реагировать на действия пользователя, и может отвечать за фильтрацию данных и авторизацию.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |

### Какие преимущества дает использование микросервисной архитектуры перед монолитной?

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                                                                               |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Основные  преимущества | **Гибкое масштабирование**<br>    - Можно масштабировать **отдельные сервисы**  (а не всю систему целиком), направляя ресурсы туда, где нагрузка максимальна.<br>    - Пример: увеличить число экземпляров  сервиса заказов при пиковом спросе, не трогая сервис профилей пользователей.                                                                              |
|                        | **Независимое развёртывание и обновление**<br>    - Изменения в одном сервисе не требуют перезапуска всего приложения.<br>    - Команды могут выпускать обновления  **параллельно** и чаще, снижая время выхода на рынок (time‑to‑market).<br>    - Меньше рисков: сбой при обновлении  одного сервиса не ломает всю систему.                                         |
|                        | **Повышенная отказоустойчивость**<br>    - Изоляция сервисов: сбой одного микросервиса  **не приводит к падению всей системы**.<br>    - Возможность реализовать механизмы отказоустойчивости  (Circuit Breaker, Timeout, Retry) для отдельных компонентов.                                                                                                           |
|                        | **Технологическая гибкость**<br>    - Каждый сервис может использовать **свой  технологический стек** (язык, фреймворк, БД), оптимальный для конкретной задачи.<br>    - Проще внедрять новые технологии: можно  переписать один сервис, не переделывая всю систему.<br>    - Пример: сервис аналитики на Python + PostgreSQL, сервис уведомлений на Node.js + Redis. |
|                        | **Организационная масштабируемость**<br>    - Разные команды могут работать над разными  сервисами **независимо**, без жёсткой координации.<br>    - Упрощается onboarding новых разработчиков:  достаточно разбираться в одном сервисе.<br>    - Подходит для крупных распределённых команд.                                                                         |
|                        | **Упрощённое тестирование и отладка**<br>    - Тестирование отдельных сервисов **проще и быстрее**, чем всего монолита.<br>    - Локализация ошибок: проблема в одном сервисе  не маскируется другими компонентами.                                                                                                                                                   |
|                        | **Поэтапное внедрение и эволюция системы**<br>    - Можно начинать с монолита и **постепенно  выделять микросервисы** по мере роста сложности.<br>    - Легче экспериментировать с новыми  функциями: достаточно добавить новый сервис.                                                                                                                               |
|                        | **Лучшая поддержка высоконагруженных сценариев**<br>    - Горизонтальное масштабирование отдельных  сервисов позволяет справляться с пиковыми нагрузками.<br>    - Возможность использовать специализированные  хранилища (кэш, поисковики) для конкретных задач.                                                                                                     |
|                        | **Чёткие границы ответственности**<br>    - Каждый сервис отвечает за **конкретную  бизнес‑функцию** (например, «платежи», «поиск», «профили»).<br>    - Уменьшается «спагетти‑код»: логика не размывается между компонентами.                                                                                                                                        |
|                        | **Упрощение интеграции с внешними системами**<br>    - Внешние API и сервисы подключаются через  отдельные микросервисы, не затрагивая ядро системы.<br>    - Пример: интеграция с платёжными системами  или CRM через специализированный адаптер‑сервис.                                                                                                             |

### Какие основные принципы SOLID применимы к архитектуре приложений?

| Под темы                              | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| SRP (Single Responsibility Principle) | **Суть**: каждый класс должен иметь только одну причину для изменения. Класс не должен брать на себя функции, которые не относятся к его основной задаче.<br>**Пример**: класс, который генерирует отчёт и сразу же его печатает, нарушает SRP, так как у него две ответственности. Правильное решение — разделить эту логику на два разных класса.<br>**Важно**: если класс выполняет несколько задач, изменения в одной из них могут случайно сломать другую. Например, если изменить логику сохранения данных в базу, это может затронуть обработку данных пользователя.                                                                                                                                                                                                                                          |
| OCP (Open-Closed Principle)           | **Суть**: программные компоненты должны быть открыты для расширения, но закрыты для изменения. Добавление новой функциональности не должно требовать изменения существующих классов или функций. Вместо этого новые возможности добавляются через расширение (наследование, внедрение интерфейсов, композицию и другие механизмы).<br>**Пример**: в системе обработки платежей: вместо изменения существующего кода при добавлении нового способа оплаты (например, криптовалюты) можно создать новый класс, реализующий уже существующий интерфейс платежа.<br>**Важно**: OCP снижает риски и стоимость изменений: когда новые требования или бизнес-изменения приводят к минимальным модификациям существующего кода, это снижает вероятность ошибок, упрощает тестирование, поддержку и развитие системы в целом. |
| LSP (Liskov Substitution Principle)   | **Суть**: объекты в программе должны быть заменяемыми экземплярами их базовых типов, не нарушая корректность программы. Это означает, что код, который работает с базовым типом, должен работать и с любым его подтипом, не вызывая ошибок или неожиданного поведения.<br>**Пример**: если PremiumCustomer наследует Customer, то все функции, работающие с Customer, должны корректно работать и с PremiumCustomer.<br>**Важно**: если подкласс не соблюдает контракт суперкласса, он нарушает принцип подстановки Лисков. Например, возврат объекта, несовместимого с объектом, возвращаемым методом суперкласса, или создание нового исключения, которое не было создано методом суперкласса.                                                                                                                     |
| DIP (Dependency Inversion Principle)  | **Суть**: модули верхнего уровня (бизнес-логика, основные компоненты системы) не должны зависеть от модулей нижнего уровня (конкретных реализаций, деталей). Вместо этого они должны зависеть от абстракций (интерфейсов, абстрактных классов).<br>**Пример**: OrderService должен зависеть от интерфейса IPaymentGateway, а не от конкретных классов StripePaymentGateway или PayPalPaymentGateway. Это позволяет легко заменять компоненты системы.<br>**Важно**: DIP помогает уменьшить жёсткие связи между компонентами, что делает код более гибким и удобным для модификации. DIP часто реализуется через внедрение зависимостей (Dependency Injection, DI) — паттерн, который позволяет передавать зависимости через конструктор, сеттеры или параметры методов.                                              |

### Как реализовать обмен данными между микросервисами?

| Под темы                                              | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ----------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Синхронный обмен                                      | При синхронном взаимодействии сервис отправляет запрос и ждёт ответа от другого сервиса. Это похоже на телефонный звонок: пока собеседник не ответит, дальнейшее выполнение задачи невозможно. <br>**Наиболее распространённый способ — использование REST API**. REST (Representational State Transfer) — архитектурный стиль, который использует стандартные HTTP-методы (GET, POST, PUT, DELETE) для работы с ресурсами. Каждый ресурс идентифицируется уникальным URI (Uniform Resource Identifier). Данные обычно передаются в формате JSON или XML.<br>**Преимущества REST API:**<br>- простота и универсальность;<br>- широкая поддержка языков программирования и инструментов;<br>- подходит для публичных API, мобильных приложений, веб-сервисов с преимущественно CRUD-операциями. <br>**Недостатки:**<br>- если вызываемый сервис недоступен, нельзя отложить передачу или получение информации на более позднее время;<br>- требуется прямое обращение к искомому сервису (peer-to-peer);<br>- текстовый формат данных (JSON/XML) может генерировать избыточный трафик.                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| Асинхронный обмен с использованием брокеров сообщений | При асинхронном взаимодействии сервис отправляет сообщение, не ожидая немедленного ответа. Это похоже на отправку электронной почты: отправитель продолжает работу, а получатель обрабатывает сообщение, когда сможет. <br>Для управления асинхронным обменом часто используют **брокеры сообщений** (message brokers) — посредники, которые хранят и доставляют сообщения, обеспечивая надёжность и гибкость.<br>**Популярные брокеры:**<br>- **Apache Kafka.** Высокопроизводительный брокер для обработки больших объёмов данных в реальном времени. Данные организованы в темы (topics), которые делятся на разделы (partitions). Поддерживает персистентность — сообщения сохраняются на диск.<br>- **RabbitMQ.** Брокер с гибкой маршрутизацией сообщений. Использует протокол AMQP (Advanced Message Queuing Protocol). Сообщения отправляются в очереди (queues), а обмены (exchanges) выполняют роль маршрутизаторов, направляя сообщения в нужные очереди в соответствии с правилами.<br>- **NATS.** Лёгкий и быстрый брокер для асинхронных коммуникаций в распределённых системах. <br>**Паттерны асинхронного взаимодействия:**<br>- **Point-to-Point (один к одному).** Каждое сообщение доставляется ровно одному потребителю. Подходит для сценариев, где требуется строгое соответствие отправителя и получателя (например, финансовые транзакции). <br>- **Publish/Subscribe (публикация-подписка).** Одно сообщение может быть обработано несколькими подписчиками. Используется для распространения событий, уведомлений и т. д.. |
| gRPC                                                  | **gRPC** — фреймворк для удалённого вызова процедур (RPC), разработанный Google. Он использует протокол HTTP/2 и формат сериализации данных Protocol Buffers (Protobuf), что обеспечивает высокую производительность и эффективность.<br>**Особенности gRPC:**<br>- поддерживает четыре типа взаимодействий: унарный (один запрос — один ответ), потоковая передача данных с сервера, потоковая передача данных от клиента, двунаправленная потоковая передача;<br>- автоматическая генерация кода на основе файлов .proto, где описываются сервисы и сообщения;<br>- поддержка многоязычных систем — клиенты и серверы могут быть написаны на разных языках;<br>- возможность использования TLS для безопасной связи. <br>gRPC подходит для сценариев, где важна низкая задержка, высокая производительность и строгая типизация данных.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |

### Опишите процесс проектирования системы с использованием CQRS и Event Sourcing

| Под темы       | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение    | **Процесс проектирования системы с использованием паттернов CQRS (Command Query Responsibility Segregation) и Event Sourcing** включает разделение операций чтения и записи, а также сохранение состояния системы в виде последовательности событий. Эти паттерны часто используются вместе, так как дополняют друг друга.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| CQRS           | **CQRS** разделяет операции изменения данных (команды) и чтения данных (запросы) на отдельные модели. Это позволяет:<br>- **Оптимизировать каждую часть системы** для выполнения своих задач. Например, если нагрузка на чтение превышает нагрузку на запись, можно задействовать дополнительные реплики базы данных для обработки запросов.<br>- **Упростить масштабирование** системы, поскольку запросы и команды могут обрабатываться разными механизмами и даже распределяться по разным серверам.<br>- **Упростить тестирование** — когда логика чтения и изменения данных разделена, модули проще тестировать независимо друг от друга.<br>**Реализация CQRS** может включать:<br>- **Определение команд и запросов** — команды — действия, которые изменяют состояние (например, создают заказ, обновляют информацию о клиенте), запросы — извлекают данные.<br>- **Создание обработчиков команд** — они проверяют команду, применяют бизнес-правила и обновляют модель записи.<br>- **Использование разных моделей данных** для разных целей: для чтения можно применить упрощённые структуры ради скорости, а для записи — строгую и безопасную модель, которая гарантирует корректные изменения.                                                                                                                                                                |
| Event Sourcing | **Event Sourcing** сохраняет состояние системы в виде последовательного набора событий, изменяющих состояние. Это позволяет:<br>- **Отслеживать историю изменений** — текущее состояние системы можно определить по истории событий.<br>- **Восстанавливать состояние** — при необходимости система «восстанавливает» или обрабатывает все прошлые события, чтобы построить состояние с нуля.<br>- **Обеспечить интеграцию с внешними сервисами** — каждое событие может обрабатывать несколько задач, что обеспечивает простую интеграцию с системами, которые только слушают новые события.<br>**Реализация Event Sourcing** может включать:<br>- **Хранение событий** — они последовательно записываются в специализированное хранилище событий (event store).<br>- **Транслирование событий** — события транслируются с использованием подхода публикации/подписки (pub/sub), чтобы информировать микросервисы-потребители об изменении данных.<br>- **Учёт структуры событий** — расширяемая схема событий помогает отличать новые события от старых, а периодические моментальные снимки базы данных дают возможность выделить серьёзные изменения структуры событий.<br>**Ограничения Event Sourcing**: этот паттерн подходит для проектирования микросервисов, но не для их интеграции между собой, поскольку ориентирован на применение в ограниченном контексте. |
| Интеграция     | **CQRS и Event Sourcing часто используются вместе**. Например, в архитектуре микросервисов:<br>- **Command side** — принимает команды, меняет состояние через генерацию событий.<br>- **Query side** — подписывается на события и строит удобные проекции для чтения.<br>**Вместе эти подходы позволяют**:<br>- легко масштабировать чтение и запись;<br>- хранить полную историю изменений;<br>- откатывать или пересобирать состояние;<br>- гибко интегрировать новые сервисы через события.<br>**Однако внедрение CQRS и Event Sourcing может быть сложным** — это требует от разработчиков пересмотра традиционных методик проектирования и управления состояниями приложений, создания отдельных обработчиков для команд и событий. Некоторые проблемы:<br>- управление версиями событий;<br>- масштабирование хранилища событий;<br>- обеспечение консистентности данных между моделями команд и запросов.                                                                                                                                                                                                                                                                                                                                                                                                                                                           |

### Какие вызовы и проблемы связаны с переходом от монолитной архитектуры к микросервисной?

| Проблемы и вызовы         | **Сложность разбиения монолита**<br>    - трудно корректно выделить границы  сервисов (особенно при запутанной кодовой базе);<br>    - риск получить «распределённый  монолит» — систему с сильной связанностью микросервисов;<br>    - необходимость анализа бизнес‑доменов и  применения DDD (Domain‑Driven Design) для грамотного разделения.                                  |
|                           | **Управление данными и согласованностью**<br>    - монолитный БД нужно разбить на  отдельные хранилища для сервисов;<br>    - переход от строгой консистентности (ACID) к eventual consistency;<br>    - сложности с транзакциями, охватывающими  несколько сервисов (требуется Saga, CQRS и др.);<br>    - синхронизация данных между сервисами  и миграция исторических данных. |

|                           | **Операционные сложности**<br>    - резкое увеличение числа развёртываемых компонентов;<br>    - потребность в оркестрации (Kubernetes, Docker Swarm);<br>    - автоматизация CI/CD для множества сервисов;<br>    - настройка мониторинга, логирования  и трейсинга (Prometheus, ELK, Jaeger);<br>    - управление конфигурациями и секретами.                                   |

|                           | **Инфраструктурные требования**<br>    - необходимость контейнеризации (Docker);<br>    - внедрение сервис‑меша (Istio, Linkerd);<br>    - организация API Gateway и балансировки нагрузки;<br>    - обеспечение безопасности межсервисных  вызовов (mTLS, OAuth2).                                                                                                               |
|                           | **Организационные и культурные изменения**<br>    - переход к кросс‑функциональным  командам (DevOps, product teams);<br>    - принцип «you build it, you run it»  (команда отвечает за весь жизненный цикл сервиса);<br>    - обучение разработчиков новым практикам  (микросервисы, контейнеризация, Observability);<br>    - изменение процессов разработки и релизов.         |
|                           | **Производительность и задержки**<br>    - накладные расходы на сериализацию/десериализацию данных;<br>    - сетевые задержки между сервисами;<br>    - необходимость кэширования и оптимизации маршрутов вызовов.                                                                                                                                                                |

|                           | **Затраты и сроки**<br>    - высокие начальные инвестиции в  инфраструктуру и инструменты;<br>    - длительный период миграции (поэтапный переход);<br>    - риск простоя бизнеса при переключении на новую архитектуру.                                                                                                                                                          |

### В чем заключается подход Идемпотентности API и как он влияет на архитектуру системы?

| Под темы               | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ---------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение            | **Подход идемпотентности в API** заключается в обеспечении согласованных и предсказуемых результатов, на которые не влияют несколько однотипных запросов. Это важно для проектирования систем, особенно в параллельных и распределённых системах, где повторные действия могут привести к нежелательным последствиям.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| Принцип                | **Идемпотентность** означает, что многократное выполнение одного и того же запроса приводит к одинаковому результату. Первый запрос может изменить состояние системы, но последующие идентичные запросы не должны вызывать дополнительных изменений.<br>**Пример**: если клиент отправляет запрос PUT для обновления ресурса определёнными данными и из-за проблем с сетью запрос дублируется, идемпотентный API гарантирует, что ресурс каждый раз обновляется одними и теми же данными, оставляя систему в согласованном состоянии даже после нескольких последующих запросов.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| Реализация             | **Для обеспечения идемпотентности в API используются, например**:<br>- **Уникальные идентификаторы** — они генерируются на стороне клиента и отправляются в заголовках запросов. Когда сервер получает запрос с уникальным токеном идемпотентности, он проверяет, был ли уже обработан этот токен. Если да, клиенту отправляется ранее кэшированный ответ, а если нет, выполняется новая операция.<br>- **Управление состоянием** — система отслеживает выполненные операции и их результаты, чтобы распознавать и пропускать уже обработанные запросы.<br>- **Ограничения базы данных** — например, уникальные ключи, которые предотвращают дублирование записей.<br>**Важно**: идемпотентность определяется по изменениям, которые операция вносит в систему, а не по ответу. Например, если API предназначен для регистрации нового пользователя, идемпотентность означает, что при многократном вызове с одинаковыми данными (например, тем же адресом электронной почты и паролем) аккаунт создаётся только один раз, и последующие вызовы не имеют эффекта. |
| Влияние на архитектуру | **Принцип идемпотентности влияет на архитектуру системы, например**:<br>- **Повышение надёжности** — повторяющиеся запросы дают один и тот же результат, что позволяет избежать непредвиденного поведения.<br>- **Упрощение обработки ошибок** — идемпотентность позволяет безопасно повторять попытки, не вызывая дублирования действий.<br>- **Обеспечение целостности данных** — идемпотентные операции предотвращают повреждение данных в результате повторных или неудачных запросов. Это важно для транзакционных систем, таких как обработка платежей и управление заказами.<br>- **Поддержка масштабируемости** — идемпотентные API позволяют безопасно выполнять повторные попытки и распределять нагрузку по нескольким серверам.<br>**Однако есть и недостатки**: необходимость дополнительного проектирования и реализации механизмов для контроля идемпотентности может увеличить сложность системы. Также в некоторых случаях строгое применение идемпотентности может быть ограничено из-за специфики бизнес-логики или технических ограничений.   |
