### Что означает аббревиатура ACID в контексте транзакций в базах данных?

| Под темы    | Заметка                                                                                                                                                                                                                     |
| ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | Аббревиатура ACID в контексте транзакций в базах данных означает **набор свойств, обеспечивающих надёжное выполнение транзакций**.                                                                                          |
| Свойства    | **Атомарность (Atomicity)**. Исключает сценарии, при которых операции внутри транзакции остаются частично выполненными. Все изменения либо фиксируются целиком, либо не фиксируются вовсе.                                  |
|             | **Согласованность (Consistency)**. Гарантирует, что транзакция переводит базу данных из одного корректного состояния в другое. Все ограничения и правила бизнес-логики должны соблюдаться.                                  |
|             | **Изолированность (Isolation)**. Обеспечивает отсутствие взаимного влияния параллельных транзакций друг на друга. Результат выполнения параллельных транзакций будет такой же, как если бы они выполнялись последовательно. |
|             | **Долговечность (Durability)**. После успешного завершения транзакции, изменения, произведённые ею, надёжно сохраняются и не могут быть утеряны даже в случае системных сбоев.                                              |

### Что такое транзакция в контексте баз данных?

| Под темы    | Заметка                                                                                                                                                                                     |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Транзакция в контексте баз данных** — это **последовательность операций, которые выполняются как единое целое**. Такой набор включает один или несколько запросов, связанных между собой. |
|             | **Цель транзакций** — обеспечить согласованность и целостность данных, особенно в условиях многопользовательского доступа, сбоев или ошибок.                                                |

### Какие основные уровни изолированности существуют?

| Под темы         | Заметка                                                                                                                                                 |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Read Uncommitted | **Самый низкий уровень изоляции**. Позволяет транзакции видеть изменения данных, произведённые другими незавершёнными транзакциями.                     |
| Особенности      | Может приводить к проблеме «грязного чтения» — чтению данных, добавленных или изменённых транзакцией, которая впоследствии не подтвердится (откатится). |
|                  | Используется редко, обычно в сценариях отчётности, где актуальность важнее точности.                                                                    |
| Read Committed   | Гарантирует, что транзакция видит только подтверждённые изменения других транзакций.                                                                    |
| Особенности      | Предотвращает «грязное чтение», но возможны «неповторяемое чтение» и «фантомные чтения».                                                                |
|                  | Уровень по умолчанию в большинстве СУБД, включая PostgreSQL.                                                                                            |
| Repeatable Read  | Гарантирует, что если транзакция дважды прочитает одни и те же данные, она получит одинаковые результаты.                                               |
| Особенности      | Предотвращает «грязное чтение» и «неповторяемое чтение», но возможны «фантомные чтения».                                                                |
|                  | Реализуется через механизмы многоверсионности (MVCC) в большинстве современных СУБД.                                                                    |
| Serializable     | **Наивысший уровень изоляции**. Гарантирует, что результат параллельного выполнения транзакций эквивалентен некоторому последовательному их выполнению. |
| Особенности      | Предотвращает все известные аномалии параллельного доступа.                                                                                             |
|                  | Обычно реализуется с помощью блокировок или более сложных механизмов.                                                                                   |
|                  | Может приводить к снижению производительности из-за увеличения времени ожидания между транзакциями.                                                     |

### Каковы основные различия между уровнями изолированности транзакций READ UNCOMMITTED и READ COMMITTED?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                   |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | **Основные различия между уровнями изолированности транзакций READ UNCOMMITTED и READ COMMITTED** заключаются в том, что **READ UNCOMMITTED** позволяет транзакциям видеть изменения данных, произведённые другими незавершёнными транзакциями, а **READ COMMITTED** гарантирует, что транзакция видит только подтверждённые изменения других транзакций. |

### Как SERIALIZABLE уровень изолированности влияет на выполнение транзакций?

| Под темы    | Заметка                                                                                                                                                                                                                                                        |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ       | **Уровень изолированности SERIALIZABLE (сериализуемость) влияет на выполнение транзакций, обеспечивая строгую изоляцию**. Этот уровень гарантирует, что результат параллельного выполнения транзакций эквивалентен некоторому последовательному их выполнению. |
| Особенности | Транзакции полностью изолированы друг от друга.                                                                                                                                                                                                                |
|             | Каждая транзакция выполняется так, будто параллельных транзакций нет.                                                                                                                                                                                          |
|             | Предотвращает все известные аномалии параллельного доступа.                                                                                                                                                                                                    |

### Какие стратегии и подходы используются для оптимизации производительности при работе с транзакциями?

| Под темы              | Заметка                                                                                                                                                                                                                                                                                        |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Настройка базы данных | **Уменьшение длительности транзакций**. Чем быстрее завершается транзакция, тем меньше вероятность, что она столкнётся с блокировками.                                                                                                                                                         |
|                       | **Использование правильного уровня изоляции**. Например, для большинства случаев подходит уровень READ_COMMITTED. Высокие уровни изоляции (например, SERIALIZABLE) уменьшают риск аномалий в данных за счёт снижения параллельности и производительности.                                      |
|                       | **Пакетная обработка данных** (Batch Processing) — если необходимо выполнить множество операций записи или обновления данных, это снижает количество обращений к базе данных.                                                                                                                  |
| Код                   | **Правильный выбор границы транзакции**. Не делать транзакцию глобальной для всей цепочки вызовов, обернуть транзакцией только те методы, которые действительно изменяют состояние данных.                                                                                                     |
|                       | **Придерживаться принципа единой ответственности транзакции** — один метод — одна логическая транзакция. Не смешивать несколько разнородных операций в одной транзакции «про запас» — лучше разделить их на несколько методов/транзакций и при необходимости управлять ими явно.               |
| Потоки                | **Настройка пула потоков** в зависимости от конкретных нагрузок. Например, для I/O-интенсивных задач увеличить количество потоков в пуле, чтобы не блокировать выполнение, для CPU-интенсивных операций — уменьшить количество потоков до числа, соответствующего физическим ядрам процессора. |
|                       | **Использование асинхронных вызовов** с использованием CompletableFuture — это позволяет снизить нагрузку на потоки и эффективно обрабатывать независимые операции параллельно.                                                                                                                |
| Кэширование           | **Кэширование часто запрашиваемых данных** — это сокращает время отклика приложения и снижает нагрузку на ресурсы.                                                                                                                                                                             |

### Как можно реализовать механизм оптимистической блокировки в контексте управления транзакциями?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                      |
| -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| SQL      | **Часто используется версионирование данных**. В таблицу добавляется колонка, которая хранит текущую версию. При выполнении обновления в запросе в секции where передаётся версия данных, которая была взята на изменение. Если update вернул 0 изменённых строк, значит данные были уже изменены, и транзакцию необходимо запускать заново. |
| ORM      | **В некоторых технологиях оптимистическая блокировка реализуется с помощью специального поля-версии в сущности**. Например, в JPA оптимистичная блокировка — встроенный механизм при наличии поля @Version.                                                                                                                                  |

### Какие стратегии и подходы используются для оптимизации производительности при работе с транзакциями в распределенных системах?

| Под темы                    | Заметка                                                                                                                                                                                                                                                                                                                                                   |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Алгоритмы планирования      | **Оптимистическое управление транзакциями** — минимизация блокировок и увеличение параллелизма.                                                                                                                                                                                                                                                           |
|                             | **Оптимичная блокировка** — операции выполняются параллельно с проверкой на конфликт только в момент фиксации, что повышает пропускную способность системы.                                                                                                                                                                                               |
|                             | **Асинхронная обработка** — выполнение операций асинхронно, что позволяет системе быстро реагировать на запросы пользователя, при этом выполняя более длительные задачи в фоновом режиме.                                                                                                                                                                 |
| Кэширование                 | **Локальный (частный) кэш** — данные сохраняются на устройстве или сервере, где работает экземпляр приложения, позволяя каждому экземпляру иметь собственный набор кэшированных данных. Это снижает задержки, поскольку доступ осуществляется непосредственно, но ограничивает использование общей информации между экземплярами.                         |
|                             | **Общий (глобальный) кэш** — располагается на выделенном сервере или кластере, доступен для всех экземпляров приложения, гарантируя консистентность данных. Этот подход масштабируем, так как позволяет добавлять узлы для повышения производительности, но доступ к данным занимает больше времени, чем в локальном кэше, из-за удалённого расположения. |
| Балансировка нагрузки       | **Использование балансировщиков нагрузки** — они равномерно распределяют входящий трафик между несколькими серверами, чтобы избежать перегрузки одного сервера.                                                                                                                                                                                           |
| Управление согласованностью | **Протокол двухфазной фиксации (2PC)** — гарантирует, что все участники транзакции согласуют свои действия перед её завершением. Однако 2PC имеет ограничения, такие как блокировка ресурсов и уязвимость к отказам координатора.                                                                                                                         |
|                             | **Протоколы, основанные на согласовании**, — например, Paxos или Raft, которые обеспечивают более высокую доступность и устойчивость к сбоям.                                                                                                                                                                                                             |
|                             | **Механизмы для автоматического восстановления после сбоев** — это упрощает процесс управления распределёнными транзакциями.                                                                                                                                                                                                                              |

### Как можно обеспечить атомарность транзакций, выполняющихся параллельно, без ухудшения производительности?

| Под темы | Заметка                                                                                                                                                                                                                                                                |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| SQL      | **Разделяемые блокировки** — позволяют нескольким транзакциям одновременно читать данные и не мешать друг другу, что обеспечивает высокую степень параллелизма.                                                                                                        |
|          | **Исключительные блокировки** — запрещают доступ к информации другим транзакциям до завершения операции, которая изменяет данные, но могут снизить производительность из-за необходимости ждать освобождения блокировки.                                               |
|          | **Оптимизировать уровни изоляции транзакций**. Выбрать наиболее оптимальный уровень в соответствии с требованиями приложения. Более высокие уровни изоляции обеспечивают более строгие гарантии согласованности, но могут отрицательно повлиять на производительность. |
|          | **Обеспечить правильную обработку ошибок**. Проверять наличие потенциальных ошибок и сбоев во время выполнения транзакции и использовать чёткие механизмы обработки ошибок.                                                                                            |
