### Какими способами можно определить нужный класс при внедрении интерфейса?

| Под темы                              | Заметка                                                                                                        |
| ------------------------------------- | -------------------------------------------------------------------------------------------------------------- |
| [[Заметки/@Qualifier.md\|@Qualifier]] | Позволяет явно указать имя бина, который нужно  внедрить.                                                      |
|                                       | **Когда применять:** когда в контексте есть несколько реализаций одного интерфейса и нужно выбрать конкретную. |
| Использование имени поля              | Spring может резолвить бин по имени поля, если оно совпадает с именем бина.                                    |
|                                       | **Ограничение:** работает только если имя поля точно соответствует имени бина.                                 |
| Использование конструкторов           | При внедрении через конструктор Spring автоматически подбирает подходящую реализацию.                          |
|                                       | **Преимущества:**<br>- явная зависимость;<br>- неизменяемость объекта (final);<br>- простота тестирования.     |
| Внедрение коллекций                   | Если нужно использовать все реализации интерфейса, можно внедрить их в коллекцию.                              |
|                                       | **Когда полезно:** для плагинных систем, обработчиков событий.                                                 |
| [[Заметки/@Primary.md\|@Primary]]     | Помечает бин как приоритетный при неоднозначности.                                                             |
|                                       | **Важно:** если есть `@Qualifier`, он переопределит `@Primary`.                                                |
| Конфигурация через Java‑код           | В классе конфигурации можно явно указать, какой бин возвращать.                                                |
|                                       | **Плюсы:** полный контроль над созданием бина.                                                                 |
| `@Profile`                            | Выбирает реализацию в зависимости от активного профиля Spring.                                                 |
| `ObjectProvider`                      | Позволяет безопасно получать бины, даже если их нет в контексте.                                               |

### Какие реализации IoC вы можете перечислить?

| Под темы                                         | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Контейнер Spring IoC                             | **Описание:** центральный механизм Spring  Framework.  Управляет жизненным циклом бинов, внедряет  зависимости, обрабатывает события. <br>**Ключевые компоненты:**<br>- `BeanFactory` — базовый интерфейс контейнера; <br>- `ApplicationContext` — расширенная версия с дополнительными  возможностями (AOP, события, ресурсы);<br>- `WebApplicationContext` — для веб‑приложений.  <br>    **Особенности:**<br>- поддержка DI через аннотации (`@Autowired`,  `@Inject`);<br>- XML‑ и Java‑конфигурация;<br>- обширная экосистема модулей. |
| Google Guice                                     | **Описание:** легковесный IoC‑контейнер от Google.  <br>**Особенности:**<br>- минималистичный API;<br>- акцент на Java‑конфигурацию (без XML);<br>- гибкая система привязок (bindings);<br>- поддержка AOP через `MethodInterceptor`.                                                                                                                                                                                                                                                                                                       |
| CDI (Contexts and Dependency Injection, JSR‑299) | **Описание:** стандартный механизм IoC для Java EE/Jakarta EE.  <br>**Ключевые аннотации:**<br>- `@Inject` — внедрение зависимостей;<br>- `@Named` — именованные бины;<br>- `@Scope` — управление жизненным циклом.  <br>    **Интеграция:** работает в серверах приложений (WildFly, Payara).                                                                                                                                                                                                                                              |
| Apache Deltaspike                                | **Описание:** расширение CDI с дополнительными модулями (data, security, config).  <br>**Особенности:**<br>- совместимость с CDI;<br>- расширенные возможности конфигурации;<br>- поддержка мультиарендности.                                                                                                                                                                                                                                                                                                                               |
| PicoContainer                                    | **Описание:** минималистичный IoC‑контейнер.  <br>**Особенности:**<br>- простой API (менее 10 классов);<br>- только конструкторное внедрение;<br>- высокая производительность.  <br>    **Подходит:** для встраиваемых систем, тестов.                                                                                                                                                                                                                                                                                                      |
| Avalon (Apache)                                  | **Описание:** устаревший контейнер (активно использовался в 2000‑х).  <br>**Особенности:**<br>- компонентная модель;<br>- управление жизненным циклом через интерфейсы (`Initializable`, `Disposable`).  <br>    **Статус:** не рекомендуется для новых проектов.                                                                                                                                                                                                                                                                           |
| JBoss Weld                                       | **Описание:** эталонная реализация CDI (JSR‑299).  <br>**Особенности:**<br>- полная поддержка спецификации CDI;<br>- интеграция с Java EE;<br>- модульная архитектура.                                                                                                                                                                                                                                                                                                                                                                      |
| Dagger (для Android/Java)                        | **Описание:** статический IoC‑контейнер от Google (оптимизирован для Android).  <br>**Особенности:**<br>- генерация кода на этапе компиляции (нет рефлексии);<br>- низкая нагрузка на runtime;<br>- поддержка scoped‑зависимостей.  <br>    **Ограничения:** требует настройки через аннотации и модули.                                                                                                                                                                                                                                    |
| Micronaut                                        | **Описание:** современный IoC‑контейнер для микросервисов и serverless.  <br>**Особенности:**<br>- предварительная компиляция (AOT);<br>- нулевой overhead рефлексии;<br>- встроенная поддержка реактивного программирования;<br>- совместимость с Spring‑аннотациями (`@Inject`, `@Singleton`).                                                                                                                                                                                                                                            |
| Quarkus (Unified CDI)                            | **Описание:** контейнер для GraalVM и Kubernetes.  <br>**Особенности:**<br>- оптимизация для native‑образов;<br>- быстрая инициализация;<br>- интеграция с Kubernetes Operators.                                                                                                                                                                                                                                                                                                                                                            |

### Какие существуют области видимости (scopes) бинов в Spring и в чем их особенности?

| Под темы        | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `singleton`     | - **Суть:** один экземпляр бина на весь   `ApplicationContext`   (контейнер Spring).<br>- **Когда создаётся:**  при старте контекста (или при первом запросе, если `lazy‑init`).<br>- **Особенности:**<br>    - все запросы на этот бин возвращают **один и тот же объект**;<br>    - потокобезопасность нужно обеспечивать вручную (если  бин  имеет состояние);<br>    - оптимально для сервисов, репозиториев,  конфигурационных  объектов без состояния ( `stateless`).<br>- **Пример применения:** сервисы бизнес‑логики, DAO, утилиты.       |
| `prototype`     | - **Суть:** новый экземпляр бина создаётся **при каждом  запросе** (включая внедрение в другие бины).<br>- **Когда создаётся:** каждый раз при вызове `applicationContext.getBean()` или при внедрении в другой  бин.<br>- **Особенности:**<br>    - экземпляры **не управляются контейнером** после  создания (Spring не вызывает методы уничтожения);<br>    - подходит для объектов с состоянием (`stateful`), где  нужна изоляция между использованиями.<br>- **Пример применения:** команды, DTO с валидацией,  объекты с временными данными. |
| `request`       | - **Суть:** один экземпляр бина на **один HTTP‑запрос**.<br>- **Когда создаётся:** в начале HTTP‑запроса; уничтожается после его завершения.<br>- **Особенности:**<br>    - доступен только в `WebApplicationContext`;<br>    - удобен для хранения данных, специфичных для запроса (например, параметры формы).<br>- **Пример применения:** обработчики запросов, объекты контекста запроса.                                                                                                                                                      |
| `session`       | - **Суть:** один экземпляр бина на **HTTP‑сессию** пользователя.<br>- **Когда создаётся:** при старте сессии; уничтожается при её завершении или истечении.<br>- **Особенности:**<br>    - данные сохраняются в течение всей сессии;<br>    - полезно для хранения пользовательских настроек, корзины покупок.<br>- **Пример применения:** сервисы пользовательских данных, менеджеры сессий.                                                                                                                                                      |
| `application`   | - **Суть:** один экземпляр бина на **ServletContext** (всё веб‑приложение).<br>- **Когда создаётся:** при старте приложения; живёт до его остановки.<br>- **Особенности:**<br>    - похож на `singleton`, но ограничен веб‑контекстом;<br>    - редко используется (обычно хватает `singleton`).<br>- **Пример применения:** глобальные кэши, настройки приложения.                                                                                                                                                                                |
| `websocket`     | - **Суть:** один экземпляр бина на **соединение  WebSocket**.<br>- **Когда создаётся:** при установлении соединения;  уничтожается при его закрытии.<br>- **Особенности:**<br>    - изолирует данные для каждого клиента WebSocket;<br>    - полезен для чат‑приложений, real‑time сервисов.<br>- **Пример применения:** менеджеры соединений,  обработчики сообщений.                                                                                                                                                                             |
| `globalSession` | - **Суть:** один экземпляр на глобальную сессию в  Portlet‑среде.<br>- **Статус:** не используется в современных приложениях  (Portlet устарели).                                                                                                                                                                                                                                                                                                                                                                                                  |

### Перечислите этапы поднятия контекста в Spring.

| Под темы                                                      | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Загрузка и парсинг  конфигурации                              | - Spring считывает метаданные о бинах из  источников:<br>    - XML‑файлы (`applicationContext.xml`);<br>    - Java‑классы с `@Configuration`,  `@ComponentScan`;<br>    - аннотации на классах (`@Component`, `@Service` и  др.).<br>- На основе метаданных формируются  `BeanDefinition` — объекты, описывающие:<br>    - класс бина;<br>    - область видимости (`scope`);<br>    - зависимости;<br>    - методы инициализации/уничтожения и т. п.                                                    |
| Создание экземпляров бинов (только для singleton)             | - Для бинов с областью видимости `singleton`   Spring создаёт экземпляры **на этапе поднятия контекста**.<br>- Для `prototype`, `request`, `session` и др.   экземпляры создаются позже (по требованию).<br>- Используется рефлексия или фабричные методы   (если задано в `BeanDefinition`).                                                                                                                                                                                                           |
| Внедрение зависимостей (DI)                                   | - Spring анализирует зависимости бина (поля, конструкторы, сеттеры с `@Autowired`, `@Inject`).<br>    - Для каждой зависимости:<br>    - находит подходящий бин в контексте (по типу,  имени, квалификатору);<br>    - внедряет его в целевой объект.<br>- Обрабатывает циклические зависимости  (например, через сеттеры).                                                                                                                                                                             |
| Вызов методов Aware‑интерфейсов                               | - Если бин реализует специальные интерфейсы, Spring передаёт ему служебную информацию:<br>    - `BeanNameAware` →  `setBeanName(String name)`;<br>    - `BeanFactoryAware` → `setBeanFactory(BeanFactory beanFactory)`;<br>    - `ApplicationContextAware` →  `setApplicationContext(ApplicationContext context)`.<br>- Это позволяет бину получить доступ к контейнеру и его сервисам.                                                                                                                 |
| Предварительная настройка через BeanPostProcessor             | - До вызова методов инициализации Spring прогоняет бин через все зарегистрированные `BeanPostProcessor`.<br>- Вызываются два метода:<br>    1. `postProcessBeforeInitialization(Object bean, String beanName)` — до инициализации;<br>    2. `postProcessAfterInitialization(Object bean, String beanName)` — после инициализации.<br>- Примеры использования:<br>    - внедрение значений через аннотации (`@Value`);<br>    - проксирование (AOP, транзакционность);<br>    - валидация конфигурации. |
| Инициализация бина                                            | - Вызываются методы инициализации в следующем порядке:<br>    1. Методы, помеченные `@PostConstruct`;<br>    2. Метод `afterPropertiesSet()` (если бин реализует `InitializingBean`);<br>    3. Пользовательский метод инициализации (указанный в `init-method` или `@Bean(initMethod = "...")`).<br>- На этом этапе бин обычно:<br>    - заполняет внутренние структуры;<br>    - открывает соединения (БД, кэш);<br>    - загружает конфигурационные данные.                                          |
| Регистрация в контексте                                       | - Готовый бин помещается в контейнер (`BeanFactory`).<br>- Теперь он доступен для внедрения в другие бины и получения через `applicationContext.getBean(...)`.                                                                                                                                                                                                                                                                                                                                          |
| Обработка событий контекста                                   | - После полной инициализации бинов Spring публикует события:<br>    - `ContextRefreshedEvent` — контекст готов к работе;<br>    - другие кастомные события (если настроены слушатели).<br>- Слушатели (`@EventListener`) могут выполнить дополнительные действия.                                                                                                                                                                                                                                       |
| Завершение поднятия контекста                                 | - Контекст считается «поднятым» (ready).<br>- Приложение начинает обрабатывать запросы (в веб‑приложениях — после старта сервера).                                                                                                                                                                                                                                                                                                                                                                      |
| Управление жизненным циклом (на протяжении работы приложения) | - Для `prototype`‑бинов: новые экземпляры создаются по запросу.<br>- Для веб‑scopes (`request`, `session`):<br>    - экземпляры создаются при старте HTTP‑запроса/сессии;<br>    - уничтожаются при завершении запроса/сессии.                                                                                                                                                                                                                                                                          |

### Как можно создать бин в Spring без использования аннотаций?

| Под темы                                                                      | Заметка                                                                                                                            |
| ----------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| XML‑конфигурация                                                              | Определяете бины в XML‑файле контекста  (например,  `applicationContext.xml`).                                                     |
| Java‑конфигурация без `@Bean`                                                 | Можно программно  регистрировать бины в `BeanFactory` до инициализации контекста.                                                  |
| Программная регистрация через `ApplicationContext`                            | В некоторых случаях (например,  в тестовом  коде или динамических сценариях) можно добавлять  бины прямо в   `ApplicationContext`. |
| Использование  `BeanDefinitionRegistry` и  `BeanDefinition`                   | Можно создавать и   регистрировать  `BeanDefinition` вручную.                                                                      |
| Динамическая фабрика через `ServiceLocator FactoryBean` (для prototype‑бинов) | Позволяет создавать бины с  параметрами во время выполнения без аннотаций.                                                         |
| Через  `AutowireCapable BeanFactory` (для создания с внедрением зависимостей) | Если нужно создать бин и  автоматически  внедрить его зависимости (как если бы  он был управляемым Spring).                        |

### Как в Spring реализуется Dependency Injection и какие существуют его типы?

| Под темы | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ    | Spring реализует DI через **IoC‑контейнер** (`ApplicationContext` /  `BeanFactory`), который:<br>- создаёт объекты (бины);<br>- управляет их жизненным циклом;<br>- автоматически внедряет зависимости согласно конфигурации.<br><br>**Ключевые механизмы:**<br>- сканирование компонентов (`@ComponentScan`);<br>- регистрация бинов (через аннотации, XML, Java‑конфигурацию);<br>- резолвинг зависимостей (поиск подходящих бинов по типу/имени/квалификатору);<br>- внедрение через конструкторы, поля или сеттеры;<br>- управление областями видимости (`scope`) бинов. |
| Типы     | Constructor Injection (внедрение через конструктор)<br>**Суть:** зависимости передаются через параметры конструктора. **Преимущества:**<br>- неизменяемость объекта (можно использовать `final`);<br>- явная зависимость (без «магических» полей);<br>- простота тестирования (легко подставить моки);<br>- гарантирует наличие всех зависимостей при создании объекта.<br>**Рекомендации:** предпочтительный способ для обязательных зависимостей.                                                                                                                          |
|          | Setter Injection (внедрение через сеттер)<br>**Суть:** зависимости устанавливаются через публичные сеттер‑методы. **Преимущества:**<br>- подходит для необязательных зависимостей;<br>- позволяет менять зависимости после создания объекта;<br>- совместим с JavaBeans‑стандартом.<br>**Недостатки:**<br>- объект может быть частично сконфигурирован;<br>- сложнее тестировать (нужно вызывать сеттеры).                                                                                                                                                                   |
|          | Field Injection (внедрение через поля)<br>**Суть:** зависимости внедряются прямо в поля класса через аннотации. **Преимущества:**<br>- минимум кода;<br>- простота понимания.<br>**Недостатки:**<br>- нарушает инкапсуляцию (поля часто делают `private`, но Spring обходит это через рефлексию);<br>- затрудняет тестирование (нельзя передать зависимости через конструктор);<br>- делает класс зависимым от Spring (т. к. аннотация — часть Spring);<br>- проблемы с наследованием и проксированием.                                                                      |
|          | Method Injection (внедрение через метод)<br>**Суть:** зависимости внедряются в произвольный метод (не только сеттер). **Преимущества:**<br>- можно выполнить дополнительную логику при внедрении;<br>- подходит для группировки зависимостей.                                                                                                                                                                                                                                                                                                                                |
|          | Внедрение коллекций<br>Spring может внедрять **все бины определённого типа** в коллекцию                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
|          | Использование `@Qualifier`<br>Для выбора конкретной реализации при наличии нескольких бинов одного типа                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
|          | Использование `@Primary`<br>Помечает бин как приоритетный при неоднозначности                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
|          | Внедрение через `ObjectProvider`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |

**Что такое Spring Expression Language (SpEL) и как она используется в Spring Core?**

| Под темы                            | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ----------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение                         | **SpEL** — это мощный, гибкий язык выражений,  встроенный в Spring Framework. Он позволяет:<br>- динамически вычислять значения во время  выполнения;<br>- обращаться к свойствам объектов, методам, полям; <br>- выполнять операции (арифметические, логические,  строковые);<br>- работать с коллекциями и массивами;<br>- использовать условные конструкции и тернарные  операторы.<br>SpEL напоминает EL (Expression Language) из JSP, но  значительно богаче по возможностям. |

### Что такое циклические зависимости и как можно решить проблему с циклическими зависимостями?

| Под темы        | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение     | **Циклическая зависимость** — ситуация, когда два или  более бина (компонентов) Spring взаимно зависят друг от  друга, образуя замкнутый цикл.                                                                                                                                                                                                                                                                                                           |
| Проблемы        | 1. **Нарушение жизненного цикла бинов** — контейнер не может завершить инициализацию.<br>2. **Снижение поддерживаемости** — сложно понять и изменить зависимости.<br>3. **Потенциальные утечки памяти** — прокси‑объекты могут задерживать освобождение ресурсов.<br>4. **Проблемы с тестированием** — трудно изолировать компоненты.                                                                                                                    |
| Способы решения | Использование `@Lazy` (рекомендуемый быстрый способ)<br>Откладывает создание бина до момента фактического использования. **Плюсы:**<br>- минимум изменений кода;<br>- работает с constructor injection.<br>**Минусы:**<br>- создаёт прокси‑объект (небольшая нагрузка);<br>- не устраняет архитектурную проблему.                                                                                                                                        |
|                 | Переход на setter injection<br>Внедрение зависимости через сеттер вместо конструктора. **Плюсы:**<br>- Spring создаёт бины по очереди, затем устанавливает зависимости;<br>- нет прокси‑объектов.<br>**Минусы:**<br>- объект становится изменяемым (нарушает принцип неизменности);<br>- сложнее тестировать (нужно вызывать сеттеры).                                                                                                                   |
|                 | Пересмотр архитектуры (лучшее долгосрочное решение)<br>**Варианты:**<br>- **Выделить общий сервис** — создать третий бин, который будет использоваться обоими компонентами.<br>- **Разделить логику** — разбить один из бинов на более мелкие компоненты.<br>- **Использовать интерфейс-посредник** — ввести абстракцию между зависимостями.<br>- **Пересмотреть границы ответственности** — возможно, бины слишком тесно связаны и их стоит объединить. |
|                 | Использование `ApplicationContext` (крайний случай)<br>Получить зависимость через контекст в момент использования. **Минусы:**<br>- нарушает принцип DI (явная зависимость от Spring);<br>- усложняет тестирование;<br>- снижает читаемость кода.                                                                                                                                                                                                        |
|                 | Настройка Spring (не рекомендуется)<br>В Spring Boot 2.6+ можно включить разрешение циклических зависимостей. **Риски:**<br>- скрывает архитектурные проблемы;<br>- может привести к непредсказуемому поведению;<br>- не работает с constructor injection в некоторых случаях.                                                                                                                                                                           |
