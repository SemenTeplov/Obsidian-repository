### Что такое unit-тестирование?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                                                                                                        |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Определение | **Модульное (unit) тестирование** (юнит-тестирование) — это метод тестирования программного обеспечения, при котором **проверяются отдельные модули или компоненты программы** на предмет их корректности. Модуль в этом контексте — небольшая, изолированная часть программы, такая как функция, метод, класс или объект, которая выполняет определённую задачу в приложении. |

### Что такое заглушка (stub) в контексте тестирования ПО?

| Под темы    | Заметка                                                                                                                                                                                                                                                                                           |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Определение | **Заглушка (stub) в контексте тестирования ПО** — это **простой объект, который предоставляет предопределённые («зашитые») ответы на вызовы методов во время теста**. Заглушки используются, когда тесту нужны какие-то данные от зависимости, но сама зависимость не является объектом проверки. |
|             | Цель заглушек — **изолировать тестируемый код от внешних зависимостей** (баз данных, API, файловой системы и др.).                                                                                                                                                                                |

### Какие преимущества дают фреймворки для модульного тестирования?

| Под темы      | Заметка                                                                                                                                                                                                                                                                                                 |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Автоматизация | Фреймворки автоматически запускают проверки, анализируют результаты и формируют отчёты.                                                                                                                                                                                                                 |
| Преимущества  | **Быстрое прогонирование сотен тестов**.                                                                                                                                                                                                                                                                |
|               | **Мгновенная проверка корректности внесённых изменений** — автоматизированные тесты позволяют мгновенно проверить, не нарушают ли изменения работу ранее функционировавших компонентов.                                                                                                                 |
|               | **Выявление регрессий** — тесты помогают обнаружить ситуации, когда новые изменения непреднамеренно нарушают работу ранее функционировавших компонентов.                                                                                                                                                |
|               | **Стандартизация** — фреймворки гарантируют максимальное покрытие тестов, поскольку набор кодов с самого начала выполняется стандартным образом.                                                                                                                                                        |
| Повтор        | Фреймворки позволяют создавать модульные тесты, которые можно легко повторно использовать в различных сценариях. Это экономит время на создание новых тестов и обеспечивает согласованность и надёжность всего набора тестов.                                                                           |
| Преимущества  | **Параметризация** — позволяет определить один тестовый случай, который будет выполняться с различными наборами параметров, что уменьшает дублирование кода тестов.                                                                                                                                     |
|               | **Фикстуры** — используются для создания и предварительной настройки тестового окружения, могут быть определены как функции, которые будут выполняться перед каждым тестовым случаем или перед группой тестовых случаев.                                                                                |
| Отчетность    | Фреймворки предоставляют возможности для анализа результатов тестирования, включая вывод информации о покрытии кода тестами, создание отчётов.                                                                                                                                                          |
| Преимущества  | **Allure Framework** — открытый инструмент для создания детальных отчётов о выполнении тестов. Позволяет визуализировать тестовые данные в виде графиков и диаграмм, детализировать каждый тестовый шаг с возможностью прикрепления скриншотов и логов, классифицировать ошибки по типам и серьёзности. |
|               | **pytest** — фреймворк, который предоставляет возможности для анализа результатов тестирования, включая вывод информации о покрытии кода тестами, создание отчётов.                                                                                                                                     |
| Интеграция    | Автоматизированное тестирование становится эффективным, когда оно интегрировано в процесс непрерывной интеграции и доставки (CI/CD).                                                                                                                                                                    |
| Преимущества  | **Автоматически запускать тесты** при каждом изменении кода.                                                                                                                                                                                                                                            |
|               | **Обеспечивать раннее обнаружение проблем** — запуск автоматизированных тестов сразу после внесения изменений в код позволяет выявлять ошибки на ранних этапах.                                                                                                                                         |
|               | **Гарантировать, что каждое изменение кода проходит полный набор тестов** до его внедрения в рабочую среду.                                                                                                                                                                                             |

### В чем разница между заглушкой (stub) и макетом (mock)?

| Под темы        | Заметка                                                                                                                                                                                                                                                              |
| --------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Заглушка (Stub) | **Предоставляет предопределённые ответы** на вызовы методов во время теста. Не проверяет взаимодействие, а просто возвращает предопределённые значения при вызове метода с аргументами.                                                                              |
| Макет (Mock)    | **Не только предоставляет ответы**, но и **позволяет проверять, как код взаимодействовал с зависимостью**. Можно настроить ожидания (например, какой метод должен быть вызван, с какими аргументами, сколько раз) и затем проверить, были ли эти ожидания выполнены. |

### Как можно использовать макеты (mocks) для тестирования взаимодействия между компонентами?

| Под темы       | Заметка                                                                                                                                                 |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Принцип работы | **Эмулировать вызовы методов** и проверять, какие методы были вызваны и с какими параметрами.                                                           |
|                | **Настроить ожидания** (например, какой метод должен быть вызван, с какими аргументами, сколько раз) и затем проверить, были ли эти ожидания выполнены. |
|                | **Проверить, как код обрабатывает ошибки** от внешних зависимостей — устанавливать mock-объекты так, чтобы они возвращали ошибочные результаты.         |

### Какие существуют библиотеки для создания заглушек и макетов в Java?

| Под темы              | Заметка                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Для создания заглушек | **AnyStub**. Библиотека для создания заглушек соединений в Java. Оборачивает вызовы функций, пытается найти совпадающие вызовы, которые были уже записаны. Если существует совпадающий вызов, AnyStub восстанавливает записанный результат, ассоциированный с этим вызовом, и возвращает его. Если нет совпадающего вызова и обращение во внешнюю систему разрешено, AnyStub выполняет этот вызов, записывает результат и возвращает его. |
|                       | **MockFtpServer**. Библиотека, которая предоставляет две разные реализации FTP-сервера («заглушка» и «обманка»), которые можно использовать для тестирования различных сценариев.                                                                                                                                                                                                                                                         |
| Для создания макетов  | **Mockito**. Фреймворк для создания моков (заглушек) в Java. Создаёт прокси-объекты, которые имитируют поведение реальных классов. Когда вызывается метод на моке, Mockito перехватывает вызов и возвращает заранее настроенный результат или выполняет заданное действие.                                                                                                                                                                |
|                       | **PowerMock**. Расширение библиотек Mockito и EasyMock, предоставляет дополнительные возможности для имитации статических методов, конструкторов, а также финальных классов и методов.                                                                                                                                                                                                                                                    |
|                       | **JMockit**. Библиотека, которая использует ASM для динамического переопределения байткода класса, позволяет имитировать все методы, включая статические, приватные, конструкторы и статические инициализаторы.                                                                                                                                                                                                                           |

### Какие принципы следует учитывать при проектировании тестов, использующих заглушки и макеты?

| Под темы | Заметка                                                                                                                                                                                                                                                           |
| -------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Заглушки | **Изоляция**. Заглушки должны изолировать тестируемый код от внешних зависимостей (БД, сети, API и т. п.). Это гарантирует, что тесты не зависят от внешнего состояния и стабильны.                                                                               |
|          | **Минимальность**. Заглушка должна реализовывать только необходимое поведение для текущего теста. Не следует добавлять лишнюю логику — это усложняет сопровождение и понимание.                                                                                   |
|          | **Простота и читаемость**. Код заглушек должен быть легко читаемым и понятным. Лучше использовать фреймворки, чем писать заглушки вручную.                                                                                                                        |
|          | **Определённость**. Поведение заглушки должно быть предсказуемым и соответствовать сценарию теста.                                                                                                                                                                |
|          | **Не нарушать логику теста**. Заглушка не должна влиять на бизнес-логику, которую проверяют. Её задача — имитировать поведение внешнего компонента, а не быть частью логики теста.                                                                                |
|          | **Повторное использование**. Хорошая практика — выносить заглушки в общие методы или базовые классы, если они многократно используются.                                                                                                                           |
| Моки     | **Мокать только то, что необходимо**. Злоупотребление моками делает тесты хрупкими и сложными для поддержки. Типичные кандидаты для мокирования: внешние API и веб-сервисы, базы данных и хранилища, файловые системы, сервисы отправки email или SMS.            |
|          | **Соблюдать принцип единой ответственности**. Каждый тест должен проверять только одно конкретное поведение. Если для тестирования одного метода требуется настроить множество моков со сложной логикой, это может указывать на проблемы в архитектуре кода.      |
|          | **Тестировать контракты, а не реализацию**. Моки должны имитировать контрактное поведение зависимостей, а не их конкретную реализацию. Это обеспечивает устойчивость тестов при изменении внутренней структуры зависимостей.                                      |
|          | **Обеспечивать читаемость тестов**. Настройка моков должна быть понятной и лаконичной. Использовать фабрики и вспомогательные методы для создания типичных моковых объектов, чтобы сохранить основной код теста ясным и сфокусированным на проверяемом поведении. |
|          | - **Стремиться к идемпотентности тестов**. Тесты с моками должны давать одинаковые результаты независимо от порядка их выполнения и окружения. После каждого теста все моки должны быть сброшены в исходное состояние.                                            |

### Какие подходы и методологии могут быть использованы для тестирования ПО в микросервисной архитектуре?

| Под темы                    | Заметка                                                                                                                                                                                                                                                                                             |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Юнит-тестирование           | **Проверяет функциональность отдельных компонентов каждого сервиса в изоляции**. Для микросервисов важно уделить особое внимание тестированию бизнес-логики, доменных объектов и механизмов обработки ошибок.                                                                                       |
| Интеграционное тестирование | **Проверяет взаимодействие между сервисами и их зависимостями**, включая БД и API. Цель — обнаружить проблемы, связанные с интеграцией компонентов, например несовместимость интерфейсов или проблемы с передачей данных.                                                                           |
| Контрактное тестирование    | **Проверяет, что интерфейс одного компонента (обычно микросервиса) соответствует ожиданиям других компонентов**, которые от него зависят. Это особенно важно в микросервисной архитектуре, где сервисы активно взаимодействуют друг с другом по сетевым протоколам (REST, gRPC, messaging и т. д.). |
| Нагрузочное тестирование    | **Проверяет работу системы под высокой нагрузкой**, выявляя узкие места. Включает измерение времени отклика, пропускной способности и анализ производительности микросервисов в условиях пикового трафика.                                                                                          |

### Как реализовать параметризованные тесты в JUnit для проверки поведения с различными входными данными?

| Под темы  | Заметка                                                                                                                                                                                                                                                                                 |
| --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Ответ     | **В JUnit 5 для реализации параметризованных тестов, проверяющих поведение с различными входными данными, используется аннотация @ParameterizedTest**. Она позволяет запустить один и тот же тест несколько раз с разными аргументами.                                                  |
|           | **Важно**: функциональность параметризованных тестов не входит в базовую библиотеку JUnit 5 — для её доступности нужно включить библиотеку **junit-jupiter-params** в проект.                                                                                                           |
| Синтаксис | В параметризованных тестах не нужно использовать аннотацию @Test, вместо этого используется только @ParameterizedTest. **Нужно объявить по крайней мере один источник аргументов**, который предоставляет аргументы для каждого вызова, которые будут использоваться в тестовом методе. |
